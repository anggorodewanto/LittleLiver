## Summary

Phases 22-24 introduce three capabilities: (1) account deletion anonymization integration tests, (2) R2-compatible object storage with photo upload endpoint, and (3) photo linking/unlinking across 5 metric handler files (stools, abdomen, bruising, skin observations, general notes). The changes add a `storage.ObjectStore` interface, a `MemoryStore` test double, `photo_uploads` table support, HEIC-to-JPEG conversion, thumbnail generation, and plumb photo resolution (converting stored R2 keys into signed URLs) into every metric handler's create/update/delete/list/get paths.

The architecture is sound -- the `ObjectStore` interface is clean, the `handlePhotoLinking` helper correctly centralizes link/unlink logic, and the test coverage is thorough. However, there are several code quality issues worth addressing.

### [CRITICAL] Router does not pass objStore to metric handlers -- photos will never resolve in production
- **File:** `backend/internal/handler/router.go`
- **Line(s):** L85-L114
- **Perspective:** QA
- **Finding:** All metric handlers (stools, abdomen, bruising, skin, general_notes) accept `objStores ...storage.ObjectStore` as a variadic parameter, but the `registerMetricCRUD` calls on lines 85-114 only pass `cfg.db` -- never `cfg.objStore`. In production, `objStore` will be `nil` inside every metric handler, causing `resolvePhotos` (helpers.go L113) to short-circuit and return `[]photoResponse{}` for every response.
- **Reasoning:** The variadic signature was designed for backward compatibility in tests, but the router was never updated to thread `cfg.objStore` through `registerMetricCRUD`. Tests pass because they construct handlers directly with `handler.CreateStoolHandler(db, objStore)`, bypassing the router entirely.
- **Consequence:** In production, all API responses will show `"photos": []` regardless of how many photos are linked. The upload and linking logic works correctly, but users will never see their photos in any metric response.
- **Suggestion:** Either pass `cfg.objStore` through `registerMetricCRUD` (which would require changing its signature), or pass it to each handler constructor directly. For example:
  ```go
  registerMetricCRUD(mux, "/api/babies/{id}/stools", rateMw, authMw, csrfMw,
      CreateStoolHandler(cfg.db, cfg.objStore), ListStoolsHandler(cfg.db, cfg.objStore),
      GetStoolHandler(cfg.db, cfg.objStore), UpdateStoolHandler(cfg.db, cfg.objStore),
      DeleteStoolHandler(cfg.db, cfg.objStore))
  ```

### [HIGH] Delete handlers use `strings.Split` on comma-separated format, but data is stored as JSON array
- **File:** `backend/internal/handler/stools.go`
- **Line(s):** L291-L292
- **Perspective:** QA
- **Finding:** All 5 delete handlers parse `PhotoKeys` using `strings.Split(*existing.PhotoKeys, ",")`, but photo keys are stored as a JSON array (e.g., `["photos/abc.jpg","photos/def.jpg"]`). `strings.Split` on this string would produce malformed keys like `["photos/abc.jpg"` and `"photos/def.jpg"]` with JSON brackets and quotes embedded.
- **Reasoning:** The create and update paths correctly use `parsePhotoKeysJSON` (which does `json.Unmarshal`) via `handlePhotoLinking`, but the delete path was written with an inconsistent assumption about the storage format.
- **Consequence:** Unlinking photos on delete will always fail silently (the malformed keys will not match any `r2_key` in `photo_uploads`), leaving orphaned `linked_at` timestamps and preventing those photos from ever being re-linked to another entry.
- **Suggestion:** Replace `strings.Split` with `parsePhotoKeysJSON` in all delete handlers:
  ```go
  // In all 5 delete handlers, replace:
  oldKeys := strings.Split(*existing.PhotoKeys, ",")
  // With:
  oldKeys := parsePhotoKeysJSON(existing.PhotoKeys)
  ```
  This would also allow removing the `"strings"` import from those files. The same bug exists in:
  - `backend/internal/handler/abdomen.go` L210-L211
  - `backend/internal/handler/bruising.go` L432-L433
  - `backend/internal/handler/skin_observations.go` L738-L739
  - `backend/internal/handler/general_notes.go` L677-L678

### [HIGH] Copy-paste boilerplate: identical objStore extraction pattern repeated 25+ times
- **File:** `backend/internal/handler/stools.go`, `abdomen.go`, `bruising.go`, `skin_observations.go`, `general_notes.go`
- **Line(s):** N/A - structural issue
- **Perspective:** Tech Lead
- **Finding:** The following 4-line block is copy-pasted into every handler constructor (Create, List, Get, Update -- 4 per file, 5 files = 20 occurrences):
  ```go
  var objStore storage.ObjectStore
  if len(objStores) > 0 {
      objStore = objStores[0]
  }
  ```
  Additionally, the `toXxxResponseWithPhotos` functions are near-identical across all 5 files, differing only in the model type and response type names. The delete handler photo-unlinking block is also copy-pasted 5 times.
- **Reasoning:** This volume of repetition increases the surface area for bugs (as demonstrated by the `strings.Split` inconsistency above) and makes future changes error-prone -- any fix must be applied to all 5 files manually.
- **Consequence:** Maintenance burden scales linearly with the number of metric types. When a new metric with photos is added, all this boilerplate must be manually duplicated again, increasing the risk of subtle divergence.
- **Suggestion:** Extract the variadic objStore resolution into a helper:
  ```go
  func firstObjStore(stores []storage.ObjectStore) storage.ObjectStore {
      if len(stores) > 0 {
          return stores[0]
      }
      return nil
  }
  ```
  For the delete-path photo unlinking, extract a helper like:
  ```go
  func unlinkExistingPhotos(db *sql.DB, photoKeys *string) {
      keys := parsePhotoKeysJSON(photoKeys)
      if len(keys) > 0 {
          if err := store.UnlinkPhotos(db, keys); err != nil {
              log.Printf("unlink photos on delete: %v", err)
          }
      }
  }
  ```

### [MEDIUM] Parameter sprawl in store functions -- `CreateStoolWithPhotos` has 10 parameters
- **File:** `backend/internal/store/stools.go`
- **Line(s):** L51
- **Perspective:** Tech Lead
- **Finding:** `CreateStoolWithPhotos(db, babyID, loggedBy, timestamp, colorRating, colorLabel, consistency, volumeEstimate, photoKeys, notes)` takes 10 parameters. `CreateBruisingWithPhotos` and `UpdateSkinObservationWithPhotos` have 11. These long positional parameter lists make call sites fragile -- swapping two `*string` arguments (e.g., `color` and `notes`) compiles without error but produces incorrect behavior.
- **Reasoning:** Each time a new field is added (like `photoKeys`), another parameter is appended, making the functions increasingly unwieldy. Go lacks named parameters, so the only guardrail is careful manual ordering.
- **Consequence:** Future changes risk silently swapping arguments of the same type, producing data corruption bugs that pass compilation and potentially even tests if the test data does not distinguish the fields.
- **Suggestion:** Consider introducing a params struct pattern for store functions with more than ~6 parameters:
  ```go
  type CreateStoolParams struct {
      BabyID, LoggedBy, Timestamp string
      ColorRating    int
      ColorLabel     *string
      Consistency    *string
      VolumeEstimate *string
      PhotoKeys      *string
      Notes          *string
  }
  ```
  This is a larger refactor and may be deferred, but is worth tracking as technical debt.

### [MEDIUM] Variadic `objStores ...storage.ObjectStore` is a leaky workaround for dependency injection
- **File:** `backend/internal/handler/stools.go` (and 4 others)
- **Line(s):** L103 (CreateStoolHandler signature)
- **Perspective:** Tech Lead
- **Finding:** The variadic parameter `objStores ...storage.ObjectStore` is used solely to avoid breaking existing callers that pass only `db`. This effectively makes the object store an optional, undiscoverable dependency. The signature does not communicate that an ObjectStore is needed for correct behavior -- callers see `db` as required and may never realize photos will silently degrade to empty arrays.
- **Reasoning:** The variadic pattern is typically used for truly optional behavior (like functional options). Here, the ObjectStore is semantically required for correct photo rendering, but the API hides that requirement.
- **Consequence:** As demonstrated by the router bug above, this pattern directly caused a production defect. Future maintainers will also struggle to understand which handlers need an ObjectStore and which do not.
- **Suggestion:** Make the ObjectStore a required parameter. Callers that do not have one can pass `nil` explicitly, which makes the decision visible:
  ```go
  func CreateStoolHandler(db *sql.DB, objStore storage.ObjectStore) http.HandlerFunc {
  ```
  Alternatively, bundle `db` and `objStore` into a shared `HandlerDeps` struct threaded through all handlers.

### [MEDIUM] N+1 query pattern in resolvePhotos and GetPhotoUploadsByR2Keys
- **File:** `backend/internal/store/photo_uploads.go`
- **Line(s):** L107-L119
- **Perspective:** Performance Engineer
- **Finding:** `GetPhotoUploadsByR2Keys` queries the database one row at a time in a loop. This function is called by `resolvePhotos` for every metric entry in list responses. For a list page of 20 entries, each with 4 photos, this produces 80 individual queries per page load.
- **Reasoning:** SQLite can handle this volume in a personal-use app, but the pattern sets a poor precedent and will be costly if the backing store ever moves to a network database.
- **Consequence:** With 20 entries x 4 photos, a single list API call executes ~80 DB queries plus ~80 signed URL generations. While acceptable for SQLite, this could cause noticeable latency with a networked database.
- **Suggestion:** Batch the query using `WHERE r2_key IN (?, ?, ...)`:
  ```go
  func GetPhotoUploadsByR2Keys(db *sql.DB, keys []string) ([]model.PhotoUpload, error) {
      if len(keys) == 0 { return nil, nil }
      placeholders := strings.Repeat("?,", len(keys))
      placeholders = placeholders[:len(placeholders)-1]
      query := fmt.Sprintf("SELECT ... FROM photo_uploads WHERE r2_key IN (%s)", placeholders)
      args := make([]any, len(keys))
      for i, k := range keys { args[i] = k }
      rows, err := db.Query(query, args...)
      // ...
  }
  ```

### [LOW] `targetWidth` is a magic number in thumbnail generation
- **File:** `backend/internal/handler/uploads.go`
- **Line(s):** L189
- **Perspective:** Tech Lead
- **Finding:** The thumbnail target width of 300 pixels is hard-coded as a local variable (`targetWidth := 300`) inside `generateThumbnail`. The comment on L177 documents it, but it is not a named constant and cannot be configured or reused elsewhere.
- **Reasoning:** This is a domain-significant value (thumbnail size) that ideally lives alongside `maxUploadSize` and `MaxPhotosPerMetric` as a named constant.
- **Consequence:** Minor -- if the thumbnail size needs changing, the developer must search for the magic number rather than updating a named constant.
- **Suggestion:**
  ```go
  const thumbnailWidth = 300
  ```

### [LOW] Nearest-neighbor resize produces low-quality thumbnails
- **File:** `backend/internal/handler/uploads.go`
- **Line(s):** L222-L234
- **Perspective:** Product Owner
- **Finding:** The `resizeNearestNeighbor` function uses a simple nearest-neighbor algorithm for thumbnailing. This produces noticeably pixelated results, especially for images with text, gradients, or fine details (which medical/stool photos often have).
- **Reasoning:** The stdlib-only constraint is respected, and nearest-neighbor is the simplest approach. However, `image/draw` in the standard library provides bilinear interpolation via `draw.CatmullRom` or `draw.ApproxBiLinear` (available since Go 1.1 via `golang.org/x/image/draw` or manually), which would produce significantly better results.
- **Consequence:** Thumbnails may appear jagged or unclear, reducing their usefulness for quick visual scanning.
- **Suggestion:** Consider using `golang.org/x/image/draw` (quasi-stdlib) for higher quality scaling, or accept this as a known trade-off for the minimal-dependency constraint.
