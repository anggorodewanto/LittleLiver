## Summary

Phase 37 implements an integration test for the complete photo lifecycle: upload, link to stool entry, verify signed URLs, unlink, and cleanup. It also tests 5MB size limit rejection, invalid MIME type rejection, and the 4-photo-per-entry limit. All four tests pass, run in parallel, and complete in ~40ms total. The implementation matches the phase description and spec requirements.

### [LOW] Duplicated helper code between integration test files
- **File:** `backend/internal/integration/photo_flow_test.go`
- **Line(s):** L56-L167
- **Perspective:** Tech Lead
- **Finding:** The `photoTestClient` type and its `doJSON` method duplicate the logic already present in the `testClient` type used by other integration tests (e.g., `multi_parent_test.go`). The only addition is the `uploadPhoto` method for multipart uploads.
- **Reasoning:** The existing `testClient` in `multi_parent_test.go` already has `doJSON` with identical logic. Having two parallel client types increases maintenance burden -- any fix to request handling must be applied in both places.
- **Consequence:** Future changes to auth cookies, CSRF handling, or request patterns require updating two separate client implementations, risking divergence and subtle test bugs.
- **Suggestion:** Embed or extend the existing `testClient` type rather than creating a parallel `photoTestClient`. Add the `uploadPhoto` method to `testClient` or create a composition:
  ```go
  type photoTestClient struct {
      *testClient
  }
  func (pc *photoTestClient) uploadPhoto(...) { ... }
  ```

### [LOW] Missing error message assertion in invalid MIME test
- **File:** `backend/internal/integration/photo_flow_test.go`
- **Line(s):** L400-L404
- **Perspective:** QA
- **Finding:** `TestPhotoFlow_InvalidMIMERejection` only checks the HTTP status code (400) but does not assert anything about the error message content. The 5MB test checks for `"5MB"` in the response, but the MIME test does not check for mention of accepted types.
- **Reasoning:** Without an error message assertion, the test would pass even if the 400 was returned for a completely different reason (e.g., a parsing error), reducing the test's diagnostic value.
- **Consequence:** A regression that changes the rejection reason from MIME type to something else would go undetected by this test.
- **Suggestion:** Add an assertion on the error message:
  ```go
  raw, _ := resp["_raw"].(string)
  if !strings.Contains(raw, "unsupported file type") {
      t.Errorf("expected error mentioning unsupported file type, got: %v", resp)
  }
  ```

### [LOW] Hardcoded reference date could become stale
- **File:** `backend/internal/integration/photo_flow_test.go`
- **Line(s):** L329
- **Perspective:** QA
- **Finding:** The cleanup test uses a hardcoded reference time `time.Date(2026, 3, 18, 12, 0, 0, 0, time.UTC)` to compute the cutoff for orphan cleanup. While this works correctly because `CleanupPhotos` accepts a `now` parameter, using a hardcoded date is unnecessary.
- **Reasoning:** A relative time (e.g., `time.Now().UTC()`) or any arbitrary fixed time would serve the same purpose since the test controls both `uploaded_at` and the reference time. The hardcoded 2026 date adds no value and may confuse future readers about whether the specific date matters.
- **Consequence:** Minor readability issue only; the test logic is correct regardless of the date chosen.
- **Suggestion:** Use `time.Now().UTC()` or a clearly arbitrary date with a comment explaining it is arbitrary:
  ```go
  refTime := time.Now().UTC()
  oldUploadedAt := refTime.Add(-25 * time.Hour).Format(time.DateTime)
  ```
