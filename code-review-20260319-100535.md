## Summary

Phases 46-48 add seven Chart.js chart wrapper components (StoolColorChart, WeightChart, TemperatureChart, AbdomenGirthChart, FeedingChart, DiaperChart, LabTrendsChart), a TrendsView orchestrator with date range selection, and four medication management components (MedicationList, MedicationForm, DoseLogForm, MedLogList). The changes also wire a "Medication Given" quick-log button through the TodayDashboard. Test coverage is thorough with a shared chart mock utility. The code is generally well-structured, but there are several copy-paste and duplication issues across the chart and medication components.

### [MEDIUM] Copy-paste boilerplate across all 7 chart components
- **File:** `frontend/src/lib/components/AbdomenGirthChart.svelte` (and 6 siblings)
- **Line(s):** All chart components
- **Perspective:** Tech Lead
- **Finding:** Every chart component repeats an identical skeleton: `import { onMount, onDestroy }`, declare `canvas: HTMLCanvasElement`, declare `chart: Chart | null = null`, `onMount` that creates the chart, `onDestroy` that calls `chart?.destroy()`, and the template `{#if isEmpty}<p>No data</p>{:else}<canvas bind:this={canvas}></canvas>{/if}`. This identical lifecycle pattern is duplicated 7 times with only the Chart.js config varying.
- **Reasoning:** Any future change to chart lifecycle management (e.g., reactive data updates, resize handling, accessibility attributes on canvas) must be applied to all 7 files independently. This is the textbook case for extracting a shared wrapper.
- **Consequence:** A bug fix or enhancement to chart lifecycle (such as adding `$effect` to react to prop changes instead of only rendering once on mount) would require editing 7 files in lockstep, with high risk of inconsistency.
- **Suggestion:** Extract a generic `ChartWrapper.svelte` component or a `useChart` action/helper that accepts a `(canvas) => ChartConfiguration` factory function. Each chart component becomes a thin config generator. Example:
  ```svelte
  <!-- ChartWrapper.svelte -->
  <script lang="ts">
    import { onMount, onDestroy } from 'svelte';
    import { Chart } from 'chart.js';
    interface Props { config: () => ChartConfiguration; isEmpty?: boolean; }
    let { config, isEmpty = false }: Props = $props();
    let canvas: HTMLCanvasElement;
    let chart: Chart | null = null;
    onMount(() => { if (!isEmpty) chart = new Chart(canvas, config()); });
    onDestroy(() => { chart?.destroy(); });
  </script>
  {#if isEmpty}<p>No data</p>{:else}<canvas bind:this={canvas}></canvas>{/if}
  ```

### [MEDIUM] Duplicate Percentiles/PercentilePoint interfaces in TrendsView and WeightChart
- **File:** `frontend/src/lib/components/TrendsView.svelte` and `frontend/src/lib/components/WeightChart.svelte`
- **Line(s):** TrendsView L20-L31, WeightChart L11-L22
- **Perspective:** Tech Lead
- **Finding:** The `PercentilePoint` and `Percentiles` interfaces are defined identically in both files. This is a textbook case of duplicated type definitions.
- **Reasoning:** If the WHO percentile data shape changes (e.g., adding p1 or p99), both files must be updated in sync.
- **Consequence:** Divergent type definitions across files, leading to type mismatches that may not surface until runtime.
- **Suggestion:** Extract to a shared types file, e.g., `$lib/types/percentiles.ts`, and import in both components.

### [MEDIUM] Duplicate Medication interface in DoseLogForm and MedicationList
- **File:** `frontend/src/lib/components/DoseLogForm.svelte` and `frontend/src/lib/components/MedicationList.svelte`
- **Line(s):** DoseLogForm L13-L19, MedicationList L5-L17
- **Perspective:** Tech Lead
- **Finding:** Both files define a `Medication` interface locally. DoseLogForm's version is a subset (5 fields) of MedicationList's (10 fields), but they represent the same entity.
- **Reasoning:** Having two `Medication` types that are supposed to represent the same API entity creates a maintenance burden and risks divergence.
- **Consequence:** If the API adds a field that DoseLogForm needs, the developer may not realize MedicationList already has it, or vice versa.
- **Suggestion:** Define a single canonical `Medication` interface in `$lib/types/medication.ts` and use `Pick<Medication, 'id' | 'name' | 'dose' | 'frequency' | 'active'>` where a subset is needed.

### [MEDIUM] Frequency string literals used without a shared type or constant
- **File:** `frontend/src/lib/components/MedicationForm.svelte` and `frontend/src/lib/components/MedicationList.svelte`
- **Line(s):** MedicationForm L55-L63, MedicationList L35-L48
- **Perspective:** Tech Lead
- **Finding:** Frequency values (`once_daily`, `twice_daily`, `three_times_daily`, `as_needed`, `custom`) are raw strings scattered across MedicationForm (in `getTimeSlotCount`, in the template `<option>` values, and in conditional logic) and MedicationList (in `formatFrequency`). The `formatFrequency` switch in MedicationList and the `getTimeSlotCount` switch in MedicationForm both enumerate the same set of frequency values.
- **Reasoning:** This is stringly-typed code. A typo in any one location would silently fall through to the `default` case, and the two switch statements must be kept in sync manually.
- **Consequence:** Adding a new frequency option (e.g., `four_times_daily`) requires finding and updating every switch/conditional across multiple files with no compiler assistance.
- **Suggestion:** Define a shared `MedicationFrequency` union type and a `FREQUENCY_OPTIONS` constant array with `{ value, label, timeSlots }` in a shared module. Both components can derive their behavior from this single source of truth.

### [MEDIUM] Charts do not update when props change (onMount-only rendering)
- **File:** `frontend/src/lib/components/AbdomenGirthChart.svelte` (and all chart siblings)
- **Line(s):** All chart components, `onMount` blocks
- **Perspective:** QA
- **Finding:** All 7 chart components create the Chart.js instance in `onMount` only. If `data` props change after initial mount (e.g., when the user switches date ranges in TrendsView), the charts will not re-render with new data. The TrendsView component works around this because Svelte's `{#if loading}` block destroys and re-creates the chart components on each fetch cycle, but this is an implicit and fragile coupling.
- **Reasoning:** The correctness of chart updates depends entirely on TrendsView's loading state toggling, which causes full component destruction/recreation. If someone removes the loading guard or optimizes to avoid the flash, charts will show stale data.
- **Consequence:** If the loading state pattern changes or charts are reused in a context without loading toggling, users will see stale chart data with no indication it is outdated.
- **Suggestion:** Either (a) add a `$effect` that calls `chart.update()` or destroys/recreates the chart when `data` changes, or (b) document prominently that these components must be keyed or conditionally rendered to trigger remounting. Option (a) is more robust. With the `ChartWrapper` suggested in finding 1, this logic would only need to exist once.

### [LOW] Inconsistent empty-data handling across chart components
- **File:** `frontend/src/lib/components/StoolColorChart.svelte` and `frontend/src/lib/components/TemperatureChart.svelte`
- **Line(s):** StoolColorChart L80 (template), TemperatureChart L79 (template)
- **Perspective:** QA
- **Finding:** AbdomenGirthChart, FeedingChart, DiaperChart, and LabTrendsChart show "No data" and skip chart creation when data is empty. StoolColorChart and TemperatureChart always render a canvas and create a Chart.js instance even with empty data arrays. WeightChart also always renders.
- **Reasoning:** Inconsistent UX -- some charts show a helpful message, others show an empty chart frame. The tests even verify this divergent behavior, locking it in.
- **Consequence:** Users see blank chart canvases for some metrics and "No data" text for others, which is confusing. Empty Chart.js instances consume memory unnecessarily.
- **Suggestion:** Standardize: all chart components should show "No data" when their data array is empty. This becomes trivial with a shared chart wrapper component.

### [LOW] Hard-coded medical constants without documentation or sourcing
- **File:** `frontend/src/lib/components/LabTrendsChart.svelte`
- **Line(s):** L23-L32
- **Perspective:** Product Owner
- **Finding:** The `NORMAL_RANGES` object contains hard-coded clinical reference ranges for lab values (total_bilirubin: 0-2.0, ALT: 0-40, etc.) and the `FEVER_THRESHOLD` in TemperatureChart is hard-coded at 38.0C. These are medical constants with no source citation or documentation about whether they are pediatric-specific, age-adjusted, or from a particular clinical guideline.
- **Reasoning:** For a health-tracking app used for a post-Kasai baby, incorrect reference ranges could provide false reassurance or unnecessary alarm to caregivers.
- **Consequence:** If ranges are incorrect or need age-adjustment, there is no way to know the original source to verify correctness.
- **Suggestion:** Add a comment citing the clinical source (e.g., "Pediatric reference ranges per [source], applicable to infants 0-12 months"). Ideally, move these to a shared `$lib/medical-constants.ts` file with documentation.

### [LOW] MedLogList constructs query string by concatenation
- **File:** `frontend/src/lib/components/MedLogList.svelte`
- **Line(s):** L47-L48
- **Perspective:** SecOps
- **Finding:** The API URL is built via template literal string concatenation: `` `/babies/${babyId}/med-logs?medication_id=${medicationId}` ``. While `babyId` and `medicationId` are props (not user input), this pattern does not URL-encode the values.
- **Reasoning:** If IDs ever contain characters that need URL encoding (unlikely with UUIDs but possible with other ID schemes), the request would be malformed.
- **Consequence:** Malformed API request if an ID contains special characters.
- **Suggestion:** Use `encodeURIComponent()` for query parameter values, or use a `URLSearchParams` builder: `const params = new URLSearchParams({ medication_id: medicationId }); apiClient.get(\`/babies/${babyId}/med-logs?\${params}\`)`.
