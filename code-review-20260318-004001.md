### [MEDIUM] Inconsistent test setup pattern between WHO percentile handler tests and all other handler tests
- **File:** `backend/internal/handler/who_percentiles_test.go`
- **Line(s):** L15-L31
- **Perspective:** Tech Lead
- **Finding:** The `setupWHOMux` function creates a full application mux via `handler.NewMux()`, while every other handler test file (feedings, stools, push, dashboard, etc.) manually wraps individual handlers with `middleware.Auth` and `middleware.CSRF`, then registers them on a per-test `http.NewServeMux()`. The WHO test is the only file using the full-mux approach.
- **Reasoning:** Two different test setup patterns for the same layer creates confusion for future contributors. If the full-mux approach is preferred, all handler tests should migrate; if the manual-wrap approach is preferred, the WHO test should align. Mixed patterns make it unclear which is "correct."
- **Consequence:** New contributors will cargo-cult whichever pattern they find first, leading to further inconsistency. The full-mux approach also tests more than the handler itself (all middleware, all routes), making failures harder to isolate.
- **Suggestion:** Align the WHO percentile test with the established pattern used by all other handler tests. Replace `setupWHOMux` with the manual middleware wrapping approach:
  ```go
  func TestWHOPercentiles_MaleCurvesReturnExpectedValues(t *testing.T) {
      t.Parallel()
      db := testutil.SetupTestDB(t)
      defer db.Close()
      user := testutil.CreateTestUser(t, db)

      req := testutil.AuthenticatedRequest(t, db, user.ID, testCookieName, testSecret, http.MethodGet,
          "/api/who/percentiles?sex=male&from_days=0&to_days=10")

      authMw := middleware.Auth(db, testCookieName)
      h := authMw(http.HandlerFunc(handler.WHOPercentilesHandler()))
      rr := httptest.NewRecorder()
      h.ServeHTTP(rr, req)
      // ...
  }
  ```

### [LOW] Push test boilerplate: repeated middleware wrapping could use a local helper
- **File:** `backend/internal/handler/push_test.go`
- **Line(s):** L35-L38, L73-L75, L140-L142, L173-L174, L206-L208, L250-L252, L275-L277, L301-L303
- **Perspective:** Tech Lead
- **Finding:** Every test function in `push_test.go` repeats the same 3-line middleware wrapping sequence: `authMw := middleware.Auth(...)`, `csrfMw := middleware.CSRF(...)`, `h := authMw(csrfMw(...))`. This pattern appears 8 times in the file.
- **Reasoning:** While this matches the established pattern across the handler test suite (so it is consistent), the repetition within a single file is high. Other test files with similar repetition exist (feedings, stools), so this is a codebase-wide pattern rather than a new-code-specific issue. However, the push tests could benefit from a local helper, especially since all push handler tests use the same handler chain structure.
- **Consequence:** Minor maintenance burden; adding a new middleware to the chain requires updating every test function individually.
- **Suggestion:** Consider extracting a file-local helper in `push_test.go`:
  ```go
  func pushHandler(t *testing.T, db *sql.DB, h http.HandlerFunc) http.Handler {
      t.Helper()
      authMw := middleware.Auth(db, testCookieName)
      csrfMw := middleware.CSRF(db, testCookieName, testSecret)
      return authMw(csrfMw(h))
  }
  ```

### [LOW] notify.Subscription duplicates fields from model.PushSubscription
- **File:** `backend/internal/notify/notify.go`
- **Line(s):** L19-L24
- **Perspective:** Tech Lead
- **Finding:** The `notify.Subscription` struct has three fields (`Endpoint`, `P256dh`, `Auth`) that are a subset of `model.PushSubscription`. When sending notifications, code must convert between the two types.
- **Reasoning:** The `notify` package is designed as a standalone push-sending abstraction, so having its own wire type is defensible for decoupling. However, in a personal-use app with minimal dependencies philosophy, an adapter function or direct use of the model type could reduce the conversion overhead.
- **Consequence:** Minor friction: any code that fetches subscriptions from the store and sends notifications must map `model.PushSubscription` to `notify.Subscription`. This is a trivial conversion but adds boilerplate.
- **Suggestion:** This is acceptable as-is given the clean separation of concerns. If it becomes tedious, add a conversion helper:
  ```go
  func SubscriptionFromModel(m model.PushSubscription) Subscription {
      return Subscription{Endpoint: m.Endpoint, P256dh: m.P256dh, Auth: m.Auth}
  }
  ```
