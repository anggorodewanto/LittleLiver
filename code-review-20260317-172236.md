## Summary

This review covers Phases 13-15 of LittleLiver: the `UnlinkSelfHandler`, account deletion (`DeleteAccountHandler` + `store.DeleteAccount`), and feeding CRUD (`feedings.go` handler + store). The code is generally well-structured with consistent error handling and early-return style. The findings below focus on the five requested quality dimensions: redundant state, parameter sprawl, copy-paste with variation, leaky abstractions, and stringly-typed code.

---

### [MEDIUM] Redundant `extractBabyID` Call After `requireBabyAccess`
- **File:** `backend/internal/handler/babies.go` (L235-L240), `backend/internal/handler/feedings.go` (L95-L100, L136-L141, L192-L197, L230-L235, L279-L284)
- **Line(s):** L235-L240 (babies.go); L95-L100, L136-L141, L192-L197, L230-L235, L279-L284 (feedings.go)
- **Perspective:** Tech Lead
- **Finding:** `requireBabyAccess` already calls `extractBabyID` internally and loads the baby (which contains `baby.ID`). Yet every feeding handler and `UnlinkSelfHandler` discard the returned `*model.Baby` with `_` and then immediately call `extractBabyID(r)` again to get the same baby ID. This is redundant state extraction -- the baby ID is parsed from the path twice per request.
- **Reasoning:** The pattern `_, ok = requireBabyAccess(...)` followed by `babyID := extractBabyID(r)` repeats 6 times across two files. It obscures intent and means `requireBabyAccess` does wasted work (loading the full baby record from the DB) when only the ID is needed.
- **Consequence:** If `requireBabyAccess` evolves (e.g., path parameter name changes), the separate `extractBabyID` call could silently diverge. The unnecessary DB read on every feeding request also wastes a query.
- **Suggestion:** Return `baby.ID` from `requireBabyAccess` even when callers only need the ID. Alternatively, refactor into two helpers: `requireBabyID` (validates path param + parent check, returns ID) and `requireBabyAccess` (returns full baby). Example for feeding handlers:
  ```go
  baby, ok := requireBabyAccess(w, r, db, user.ID)
  if !ok {
      return
  }
  babyID := baby.ID
  ```

### [MEDIUM] Copy-Paste Pattern: Feeding Handler Boilerplate
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L88-L126, L129-L182, L185-L220, L223-L269, L272-L304
- **Perspective:** Tech Lead
- **Finding:** All five feeding handlers repeat an identical preamble: `requireUser` -> `requireBabyAccess` (discarding baby) -> `extractBabyID` -> (for Get/Update/Delete) extract `entryID` with identical empty-check. The `entryID` extraction block (PathValue + empty check + 400 response) is copy-pasted verbatim in `GetFeedingHandler`, `UpdateFeedingHandler`, and `DeleteFeedingHandler`.
- **Reasoning:** This is copy-paste with slight variation. The entry ID extraction could be a small helper like `requireEntryID(w, r) (string, bool)` analogous to the existing `requireBabyAccess` pattern.
- **Consequence:** When a fourth metric type is added (e.g., stools, medications), this pattern will be duplicated again, increasing maintenance burden and inconsistency risk.
- **Suggestion:** Extract a helper:
  ```go
  func requireEntryID(w http.ResponseWriter, r *http.Request) (string, bool) {
      entryID := r.PathValue("entryId")
      if entryID == "" {
          http.Error(w, "missing entry ID", http.StatusBadRequest)
          return "", false
      }
      return entryID, true
  }
  ```

### [MEDIUM] Stringly-Typed Sentinel: `'deleted_user'` Repeated as Raw String
- **File:** `backend/internal/store/delete_account.go`
- **Line(s):** L74, L82, L92
- **Perspective:** Tech Lead
- **Finding:** The sentinel value `'deleted_user'` appears as a raw string literal 3 times in `delete_account.go`. It also appears in the migration SQL and tests. There is no constant defined for it.
- **Reasoning:** This is stringly-typed code. If the sentinel value ever changes (or needs to be referenced from other packages for validation), every occurrence must be found and updated manually.
- **Consequence:** A typo in one location (e.g., `'deleted-user'` vs `'deleted_user'`) would silently break anonymization, leaving real user IDs or creating orphaned references.
- **Suggestion:** Define a constant in the `model` package:
  ```go
  const DeletedUserSentinel = "deleted_user"
  ```
  Then reference `model.DeletedUserSentinel` in `delete_account.go` and tests.

### [MEDIUM] Parameter Sprawl: `store.CreateFeeding` and `store.UpdateFeeding`
- **File:** `backend/internal/store/feedings.go`
- **Line(s):** L70, L183
- **Perspective:** Tech Lead
- **Finding:** `CreateFeeding` takes 8 parameters; `UpdateFeeding` takes 10 parameters. The same pattern exists in `store.CreateBaby` (8 params) and `store.UpdateBaby` (8 params). As more metric types are added, this parameter-list approach will not scale.
- **Reasoning:** Long parameter lists are error-prone (easy to swap two `*string` or two `*float64` arguments at call sites) and make the API harder to extend. The handler already has a validated struct (`feedingRequest`) that mirrors these parameters.
- **Consequence:** Adding a new field (e.g., `supplement_type`) requires changing the function signature, every call site, and introduces risk of argument ordering bugs that the compiler cannot catch.
- **Suggestion:** Pass a struct instead of individual parameters:
  ```go
  type CreateFeedingParams struct {
      BabyID      string
      LoggedBy    string
      Timestamp   string
      FeedType    string
      VolumeMl    *float64
      CalDensity  *float64
      DurationMin *int
      Notes       *string
  }
  func CreateFeeding(db *sql.DB, p CreateFeedingParams) (*model.Feeding, error) { ... }
  ```

### [MEDIUM] Redundant Response Type: `feedingListResponse` Duplicates `MetricPage`
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L82-L85
- **Perspective:** Tech Lead
- **Finding:** `feedingListResponse` is defined as `struct { Data []feedingResponse; NextCursor *string }` -- structurally identical to `model.MetricPage[feedingResponse]`. The generic `MetricPage[T]` already exists in `model/types.go` (L67-L70) but is not used in the handler layer.
- **Reasoning:** This is redundant state/type. The generic `MetricPage` was clearly designed for this exact purpose, yet the handler defines its own parallel struct.
- **Consequence:** When pagination metadata evolves (e.g., adding `total_count` or `has_more`), two types must be updated instead of one.
- **Suggestion:** Use `model.MetricPage[feedingResponse]` directly:
  ```go
  page := model.MetricPage[feedingResponse]{
      Data: make([]feedingResponse, 0, len(page.Data)),
  }
  ```

### [LOW] Date Format Strings Repeated in Store Layer
- **File:** `backend/internal/store/feedings.go`
- **Line(s):** L110, L114, L120, L125
- **Perspective:** Tech Lead
- **Finding:** The format strings `"2006-01-02"` and `"2006-01-02T15:04:05Z"` are used as raw literals in `store/feedings.go`. The handler layer already defines constants `dateFormat` and `dateTimeFormat` for these same values, but they are unexported and unavailable to the store package.
- **Reasoning:** The same magic strings exist in two packages with no shared constant. This is stringly-typed duplication across package boundaries.
- **Consequence:** If the date format changes in one place but not the other, date parsing silently breaks or produces incorrect time boundaries.
- **Suggestion:** Move the format constants to the `model` package (or a shared `internal/timeutil` package) and reference them from both handler and store:
  ```go
  // model/formats.go
  const (
      DateFormat     = "2006-01-02"
      DateTimeFormat = "2006-01-02T15:04:05Z"
  )
  ```

### [LOW] SQL Injection Surface via `AnonymizeTables` Package Variable
- **File:** `backend/internal/handler/delete_account.go` (L14), `backend/internal/store/delete_account.go` (L72-L88)
- **Line(s):** L14 (handler), L72-L88 (store)
- **Perspective:** SecOps
- **Finding:** `AnonymizeTables` is an exported `[]string` package variable whose values are interpolated directly into SQL via `fmt.Sprintf("UPDATE %s SET ...")`. While currently only set internally, the exported variable means any package can append arbitrary table names, which flow unsanitized into SQL.
- **Reasoning:** Table names cannot be parameterized in SQL, but the exported mutable slice creates a wider attack surface than necessary. If any future code path allows user input to influence this slice, it becomes a SQL injection vector.
- **Consequence:** In the worst case, a compromised or buggy caller could inject `"users SET name = 'hacked' WHERE 1=1; --"` as a table name.
- **Suggestion:** Either (1) validate table names against an allow-list of known tables before interpolation, or (2) make the slice unexported and provide an `init()`-time registration function that validates names:
  ```go
  func RegisterAnonymizeTable(name string) {
      if !validTableName.MatchString(name) {
          panic("invalid table name: " + name)
      }
      anonymizeTables = append(anonymizeTables, name)
  }
  ```

### [LOW] End-of-Day Boundary Off by One Second
- **File:** `backend/internal/store/feedings.go`
- **Line(s):** L124-L125
- **Perspective:** QA
- **Finding:** The "to" date boundary is calculated as `t.Add(24*time.Hour - time.Second)`, producing `23:59:59`. Any feeding logged at exactly `23:59:59.500` (sub-second precision) would be included, but one logged at `23:59:59.001` in a system with sub-second timestamp storage would also be included. More importantly, using `<= 23:59:59` means a feeding at exactly `00:00:00` of the next day is excluded, but so is anything in the last 999ms of the day.
- **Reasoning:** The standard pattern for date range queries is to use an exclusive upper bound (`timestamp < next_day_start`) rather than an inclusive `<= end_of_day`, which avoids sub-second boundary issues entirely.
- **Consequence:** Feedings logged in the last sub-second of a day could be missed in filtered queries, though this is unlikely to matter in practice with SQLite's typical datetime precision.
- **Suggestion:** Use exclusive upper bound:
  ```go
  utcTo := t.Add(24 * time.Hour).UTC().Format("2006-01-02T15:04:05Z")
  conditions = append(conditions, "timestamp < ?")
  ```
