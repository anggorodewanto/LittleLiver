## Summary

This review covers phases 10-12 of the LittleLiver project: rate limiting middleware, baby CRUD endpoints (store + handlers), and the invite code generation/join flow. The changes are well-structured with good test coverage. The issues below focus on the five requested categories: redundant state, parameter sprawl, copy-paste with slight variation, leaky abstractions, and stringly-typed code.

---

### [MEDIUM] Copy-paste: Baby row scanning duplicated between GetBabyByID and GetBabiesByUserID
- **File:** `backend/internal/store/babies.go`
- **Line(s):** L48-L91 and L148-L183
- **Perspective:** Tech Lead
- **Finding:** The row-scanning logic (declaring temp variables, calling `Scan`, parsing date strings, handling nullable fields) is copied almost identically between `GetBabyByID` and the `rows.Next()` loop in `GetBabiesByUserID`. The two blocks differ only in that one uses `db.QueryRow` and the other iterates `rows`.
- **Reasoning:** When a column is added to the `babies` table (which is inevitable as the app grows), both scan sites must be updated in lockstep. This is the exact kind of near-duplicate that leads to subtle divergence bugs.
- **Consequence:** A future column addition updated in one function but missed in the other would cause silent data loss or a runtime scan error that only surfaces in the list-babies path (or vice versa).
- **Suggestion:** Extract a `scanBaby` helper that accepts a `scanner` interface (satisfied by both `*sql.Row` and `*sql.Rows`):
```go
type scanner interface {
    Scan(dest ...any) error
}

func scanBaby(s scanner) (*model.Baby, error) {
    var b model.Baby
    var dobStr string
    var diagStr, kasaiStr, notesStr sql.NullString
    var createdStr string
    if err := s.Scan(&b.ID, &b.Name, &b.Sex, &dobStr, &diagStr, &kasaiStr,
        &b.DefaultCalPerFeed, &notesStr, &createdStr); err != nil {
        return nil, err
    }
    // ... parsing logic once ...
    return &b, nil
}
```

```go
        return "sex must be 'male' or 'female'"
    }
    // ... date parsing ...
    return ""
}
```

### [MEDIUM] Stringly-typed: Sex validation uses raw string literals instead of model.ValidSex
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L86-L89 and L246-L249
- **Perspective:** Tech Lead
- **Finding:** The handler validates sex with `req.Sex != "male" && req.Sex != "female"`, but the `model` package already provides `model.ValidSex(v string) bool` which uses a centralized `validSet("male", "female")`. The handler ignores this existing abstraction and uses raw string comparisons.
- **Reasoning:** The `model.ValidSex` function exists precisely to centralize enum values. Bypassing it means that if the valid sex values ever change, the handler will be out of sync with the model layer.
- **Consequence:** If additional sex values are added to the model (or values are renamed), the handler will incorrectly reject valid input while the store layer accepts it.
- **Suggestion:** Replace the inline check with the model validator:
```go
if !model.ValidSex(req.Sex) {
    http.Error(w, "sex must be 'male' or 'female'", http.StatusBadRequest)
    return
}
```

### [MEDIUM] Parameter sprawl: CreateBaby and UpdateBaby take 8-9 positional parameters
- **File:** `backend/internal/store/babies.go`
- **Line(s):** L12 and L109
- **Perspective:** Tech Lead
- **Finding:** `CreateBaby` accepts `(db, creatorID, name, sex, dob string, diagnosisDate, kasaiDate *string, defaultCalPerFeed *float64, notes *string)` -- 9 parameters. `UpdateBaby` has 9 as well. At the call sites in the handlers (L108-L109, L268-L269), this produces long, hard-to-read lines where positional parameter ordering is error-prone.
- **Reasoning:** With this many parameters of similar types (multiple `string` and `*string` values), it is easy to accidentally swap `diagnosisDate` and `kasaiDate` at a call site and the compiler will not catch it.
- **Consequence:** A transposition bug (e.g., passing kasai_date as diagnosis_date) would silently store incorrect medical dates, which is particularly dangerous for a health-tracking application.
- **Suggestion:** Introduce a params struct:
```go
type BabyParams struct {
    Name              string
    Sex               string
    DateOfBirth       string
    DiagnosisDate     *string
    KasaiDate         *string
    DefaultCalPerFeed *float64
    Notes             *string
}

func CreateBaby(db *sql.DB, creatorID string, p BabyParams) (*model.Baby, error) { ... }
```

### [MEDIUM] Copy-paste: Authorization check (get baby + check parent) repeated across 3 handlers
- **File:** `backend/internal/handler/babies.go` and `backend/internal/handler/invites.go`
- **Line(s):** babies.go L211-L233 (Update), invites.go L48-L70 (CreateInvite), and a variation at babies.go L167-L187 (Get)
- **Perspective:** Tech Lead
- **Finding:** The pattern of "extract baby ID, check baby exists via `store.GetBabyByID`, handle `ErrNoRows` as 404, then call `store.IsParentOfBaby` and handle 403" is repeated three times with only log message differences. `GetBabyHandler` is slightly different (it uses the fetched baby object) but `UpdateBabyHandler` and `CreateInviteHandler` are structurally identical -- they discard the fetched baby entirely.
- **Reasoning:** This authorization check is the most security-sensitive code in the handlers. Having it repeated in three places increases the risk that a future endpoint gets a subtly incorrect version.
- **Consequence:** A future handler copying this pattern might miss a check, leading to unauthorized access to baby data.
- **Suggestion:** Extract a helper like `requireParentOfBaby(db, w, r) (babyID string, ok bool)` that performs the full baby-exists + parent-check flow, writes the error response, and returns whether to proceed.

### [HIGH] Leaky abstraction / Redundant state: CreateInvite fabricates CreatedAt instead of reading from DB
- **File:** `backend/internal/store/invites.go`
- **Line(s):** L79-L85
- **Perspective:** QA
- **Finding:** After committing the transaction, `CreateInvite` returns a hand-built `model.Invite` with `CreatedAt: time.Now().UTC()`. This is a second call to `time.Now()` that will differ from the actual `created_at` default set by the database. The `ExpiresAt` was also computed with a separate `time.Now()` call at L51, so there are potentially three distinct timestamps in play (DB's `created_at`, the Go-side `ExpiresAt`, and the Go-side `CreatedAt`).
- **Reasoning:** The returned object does not match what the database actually stored. While this may not matter functionally today, it establishes a pattern where the in-memory object and the database row can diverge. Compare with `CreateBaby`, which correctly re-reads from the database after commit via `GetBabyByID`.
- **Consequence:** If any consumer relies on the returned `CreatedAt` for display or logic, they will see a value that does not match the database. Clock skew between the two `time.Now()` calls (L51 vs L84) could also cause `CreatedAt` to appear after `ExpiresAt` in edge cases.
- **Suggestion:** Either re-read the invite from the database after commit (consistent with the `CreateBaby` pattern), or at minimum capture `time.Now().UTC()` once at the top of the function and reuse it:
```go
now := time.Now().UTC()
expiresAt := now.Add(InviteExpiry)
// ... later ...
return &model.Invite{
    // ...
    ExpiresAt: expiresAt,
    CreatedAt: now,
}, nil
```

### [MEDIUM] Stringly-typed: Date format layout "2006-01-02" repeated as bare string 9+ times
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L45, L51, L55, L91, L96, L102, L251, L256, L262
- **Perspective:** Tech Lead
- **Finding:** The Go date layout string `"2006-01-02"` appears as a raw string literal at least 9 times across the handler file (in `toBabyResponse` for formatting and in both Create/Update handlers for parsing). Similarly, `"2006-01-02T15:04:05Z"` appears in response formatting.
- **Reasoning:** These are semantic constants (the project's date format). Using raw strings means a typo in one instance (e.g., `"2006-01-2"`) would cause a subtle parsing bug that the compiler cannot catch.
- **Consequence:** Inconsistent date formatting if one instance is accidentally modified; harder to change the date format project-wide.
- **Suggestion:** Define package-level constants:
```go
const (
    dateFormat     = "2006-01-02"
    dateTimeFormat = "2006-01-02T15:04:05Z"
)
```

### [MEDIUM] Leaky abstraction: RateLimiter sessions map grows unboundedly
- **File:** `backend/internal/middleware/ratelimit.go`
- **Line(s):** L34-L58
- **Perspective:** Performance Engineer
- **Finding:** The `sessions` map in `RateLimiter` adds a new entry for every unique session cookie value it sees but never removes expired entries. The only "cleanup" is resetting the count when a known session's window expires, but entries for sessions that never return are retained forever.
- **Reasoning:** In a long-running server, expired or invalidated sessions will accumulate in memory. While this is a personal-use app, a malicious actor or session rotation could inflate this map.
- **Consequence:** Slow memory leak proportional to the number of distinct session IDs seen over the server's lifetime. In a worst case (e.g., bot scanning with fabricated cookies), memory could grow without bound.
- **Suggestion:** Add a periodic cleanup goroutine or a lazy eviction in the `allow` method that purges entries older than the window. A simple approach:
```go
// In allow(), after acquiring the lock, periodically sweep:
if len(rl.sessions) > someThreshold {
    for k, b := range rl.sessions {
        if now.Sub(b.windowStart) >= rl.window {
            delete(rl.sessions, k)
        }
    }
}
```

### [LOW] Stringly-typed: Magic number 67 for default calories per feed
- **File:** `backend/internal/store/babies.go`
- **Line(s):** L22
- **Perspective:** Tech Lead
- **Finding:** The SQL query uses `COALESCE(?, 67)` with the default calories-per-feed value hardcoded as the integer `67`. This same default appears in test assertions (`resp.DefaultCalPerFeed != 67`) across multiple test files. The value has medical significance (67 kcal/100mL is standard for infant formula) but is not defined as a named constant anywhere.
- **Reasoning:** If the default ever needs to change (e.g., different formula recommendation), every occurrence must be found and updated manually.
- **Consequence:** Inconsistency between store layer and test expectations if updated in one place but not others.
- **Suggestion:** Define a constant in the model or store package:
```go
const DefaultCalPerFeed = 67.0
```
Then use it in the SQL via a Go-side default: `COALESCE(?, ?)` with the constant as the second parameter.
