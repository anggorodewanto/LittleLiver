## Summary

This review covers phases 40-42 of LittleLiver, which add a report generation integration test (backend), frontend stores for user and baby management, Svelte UI components (BabySelector, CreateBabyForm, JoinBabyForm, FirstLogin, login page), and comprehensive frontend unit tests. The review focuses specifically on **code reuse** opportunities -- identifying duplicate functions, repeated data structures, and misplaced helpers.

### [MEDIUM] Duplicate JPEG creation helper between report and photo tests
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L236-L249
- **Perspective:** Tech Lead
- **Finding:** `createReportTestJPEG` in `report_generation_test.go` duplicates the existing `createTestJPEGData` in `photo_flow_test.go` (L88-L101). Both create a small in-memory JPEG for testing. The only differences are the hardcoded dimensions (20x20 vs parameterized) and pixel color.
- **Reasoning:** Having two nearly identical JPEG-creation helpers in the same package means any future change to JPEG test infrastructure (e.g., switching to PNG, adding metadata) must be applied twice. The parameterized version in `photo_flow_test.go` is strictly more flexible.
- **Consequence:** Maintenance burden grows as more test files copy-paste this pattern. A bug in JPEG generation logic (e.g., if the image library API changes) would need to be fixed in multiple places.
- **Suggestion:** Delete `createReportTestJPEG` and reuse `createTestJPEGData` from `photo_flow_test.go` instead:
  ```go
  // In report_generation_test.go, replace:
  //   thumbPNG := createReportTestJPEG(t)
  // With:
  thumbPNG := createTestJPEGData(t, 20, 20)
  ```

### [MEDIUM] doRawBytes helper defined in wrong file
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L252-L271
- **Perspective:** Tech Lead
- **Finding:** The `doRawBytes` method on `testClient` is defined in `report_generation_test.go` rather than in `helpers_test.go`, where all other `testClient` methods (`doJSON`, `doJSONWithHeaders`, `doJSONList`, `doJSONArray`, `doRaw`) are co-located.
- **Reasoning:** The helpers file establishes a clear convention: all `testClient` methods live in `helpers_test.go`. Scattering methods across test files makes them harder to discover for future test authors, who may create yet another duplicate when they need raw-bytes functionality.
- **Consequence:** Future integration tests needing raw byte responses may not find `doRawBytes` and will re-implement it, leading to further duplication. Test infrastructure becomes fragmented across files.
- **Suggestion:** Move `doRawBytes` from `report_generation_test.go` to `helpers_test.go`, alongside the other `testClient` methods. This also avoids the `session_id` vs `auth.CookieName` inconsistency -- `doRawBytes` hardcodes `"session_id"` as the cookie name (L258), while all helpers in `helpers_test.go` use the constant `auth.CookieName` (e.g., L94, L139).

### [LOW] Hardcoded cookie name in doRawBytes diverges from constant
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L258
- **Perspective:** QA
- **Finding:** `doRawBytes` uses the hardcoded string `"session_id"` for the cookie name, while every other helper method in `helpers_test.go` uses the constant `auth.CookieName`. If the constant's value ever changes, `doRawBytes` would silently break, causing unauthenticated requests.
- **Reasoning:** The constant `auth.CookieName` exists precisely to avoid this kind of silent breakage from hardcoded strings.
- **Consequence:** If `auth.CookieName` is changed, the report generation test would send requests with the wrong cookie name, producing confusing 401 errors unrelated to the actual report logic.
- **Suggestion:** Replace the hardcoded string with the constant:
  ```go
  req.AddCookie(&http.Cookie{Name: auth.CookieName, Value: tc.sessionID})
  ```
  This also requires adding `auth` to the import list in `report_generation_test.go`.

### [LOW] Duplicate mockBabies fixture across frontend test files
- **File:** `frontend/src/tests/baby-selector.test.ts`
- **Line(s):** L7-L24
- **Perspective:** Tech Lead
- **Finding:** The `mockBabies` array in `baby-selector.test.ts` (L7-24) is nearly identical to the one in `baby-store.test.ts` (L16-33). Both define the same two babies (Alice with `baby-1` and Bob with `baby-2`) with the same field values and structure.
- **Reasoning:** Duplicated test fixtures across files mean any change to the `Baby` interface requires updating the fixture in multiple places.
- **Consequence:** If the `Baby` type gains a required field, both fixtures must be updated independently. One may be missed, causing confusing test failures.
- **Suggestion:** Extract the shared fixture into a test helpers file:
  ```typescript
  // frontend/src/tests/fixtures.ts
  import type { Baby } from '$lib/stores/baby';
  
  export const mockBabies: Baby[] = [
    { id: 'baby-1', name: 'Alice', date_of_birth: '2025-06-01', sex: 'female', diagnosis_date: '2025-06-15', kasai_date: '2025-06-20' },
    { id: 'baby-2', name: 'Bob', date_of_birth: '2025-09-01', sex: 'male', diagnosis_date: null, kasai_date: null },
  ];
  ```
