## Summary

Phase 50 implements a Report generation UI component (`ReportPage.svelte`) with a date range picker, preview summary, loading state, error handling, and PDF download trigger. The test file (`report-page.test.ts`) covers 11 test cases. Both files are untracked and not yet committed. The implementation is generally clean and minimal, but has several issues that need attention before approval.

### [HIGH] No Route Page for ReportPage Component
- **File:** `frontend/src/routes/` (missing `report/+page.svelte` or similar)
- **Line(s):** N/A - structural issue
- **Perspective:** Product Owner
- **Finding:** `ReportPage.svelte` is a standalone component with no corresponding SvelteKit route page. There is no `frontend/src/routes/report/+page.svelte` (or similar) that renders `<ReportPage>`. Additionally, the layout (`+layout.svelte`) contains no navigation elements that would link to a report page.
- **Reasoning:** Without a route, users cannot navigate to the report page. The component exists in isolation with no way to access it in the running application.
- **Consequence:** The report feature is unreachable by users. Phase 50 proof of progress requires "Report page renders" -- but it only renders in tests, not in the actual application.
- **Suggestion:** Create a route file (e.g., `frontend/src/routes/report/+page.svelte`) that imports and renders `ReportPage` with the appropriate `babyId` and `babyName` props. This likely requires integration with the baby selector/context pattern used elsewhere.

### [HIGH] Uses Raw fetch Instead of apiClient
- **File:** `frontend/src/lib/components/ReportPage.svelte`
- **Line(s):** L36-L39
- **Perspective:** Tech Lead
- **Finding:** The component uses raw `fetch()` directly instead of the project's `apiClient` from `$lib/api`. Every other component in the codebase (`TodayDashboard`, `TrendsView`, `MedicationList`, `MedLogList`, `DoseLogForm`) uses `apiClient` for API calls.
- **Reasoning:** The `apiClient` handles CSRF tokens, timezone headers, 401 redirects, and consistent error handling. Bypassing it means the report endpoint will not send the `X-Timezone` header and will not handle 401 responses (redirect to login). While the report is a GET request (no CSRF needed), the timezone header and auth redirect are still important.
- **Consequence:** If the user's session expires, clicking "Generate Report" will silently fail instead of redirecting to login. The timezone header expected by the backend will be missing. This is an inconsistency that creates maintenance burden.
- **Suggestion:** The `apiClient` currently returns parsed JSON (`response.json()`), which does not support blob responses needed for PDF download. Two options: (1) Add a `getBlob` method to `apiClient` that returns the raw response or blob, then use it in ReportPage, or (2) Add a `getRaw` method that returns the `Response` object. Example:
```typescript
// In api.ts
async getRaw(path: string): Promise<Response> {
    return requestRaw(path); // similar to request() but returns Response directly
}

// In ReportPage.svelte
const response = await apiClient.getRaw(`/babies/${babyId}/report?from=${fromDate}&to=${toDate}`);
```

### [MEDIUM] Test Mocks apiClient But Component Uses Raw fetch
- **File:** `frontend/src/tests/report-page.test.ts`
- **Line(s):** L5-L9
- **Perspective:** QA
- **Finding:** The test file mocks `$lib/api` and its `apiClient.get` method (lines 5-9), but the component never uses `apiClient` -- it uses raw `fetch()`. The mock is entirely unused dead code. The tests work because they separately stub `fetch` via `vi.stubGlobal('fetch', ...)` in the relevant test cases.
- **Reasoning:** This indicates either the component was supposed to use `apiClient` (and was changed without updating the mock) or the mock was copy-pasted from another test. Either way, it is misleading and suggests incomplete implementation.
- **Consequence:** Future developers may be confused about which API mechanism this component uses. If the component is later refactored to use `apiClient`, the existing mock setup may mask bugs.
- **Suggestion:** Remove the unused `vi.mock('$lib/api', ...)` block (lines 5-9). If the component is refactored to use `apiClient`, add the appropriate mock at that time.

### [MEDIUM] No Cleanup of Stubbed Global fetch
- **File:** `frontend/src/tests/report-page.test.ts`
- **Line(s):** L88, L107, L132, L160
- **Perspective:** QA
- **Finding:** Several tests call `vi.stubGlobal('fetch', ...)` to mock `fetch`, but there is no `afterEach` that restores the original `fetch`. While `vi.resetAllMocks()` in `beforeEach` resets mock state, it does not restore `vi.stubGlobal` stubs.
- **Reasoning:** Test isolation can be compromised if tests run in a different order or if additional tests are added that depend on the real `fetch`. The `URL` global stub in `beforeEach` has the same issue.
- **Consequence:** Tests that run after these may unexpectedly use the mocked `fetch`, leading to false passes or confusing failures.
- **Suggestion:** Add `vi.restoreAllMocks()` or use `vi.unstubAllGlobals()` in an `afterEach` block:
```typescript
afterEach(() => {
    vi.unstubAllGlobals();
});
```

### [LOW] Error Message Is Generic
- **File:** `frontend/src/lib/components/ReportPage.svelte`
- **Line(s):** L53
- **Perspective:** Product Owner
- **Finding:** The error message is always "Failed to generate report" regardless of the actual error (network failure, 403, 500, etc.). The response status code is available but not used.
- **Reasoning:** For a personal-use app this is acceptable, but distinguishing between "server error" and "network error" would help debugging.
- **Consequence:** Users cannot tell whether the issue is a network problem (fixable by retrying) or a server-side error.
- **Suggestion:** Consider including the HTTP status code in the error message when the response exists: `error = \`Failed to generate report (status ${response.status})\``.

### [LOW] Date Comparison Uses String Comparison
- **File:** `frontend/src/lib/components/ReportPage.svelte`
- **Line(s):** L14-L16
- **Perspective:** QA
- **Finding:** The date validation `fromDate <= toDate` uses string comparison. This works correctly for ISO date format (`YYYY-MM-DD`) since the format is lexicographically ordered, so this is not a bug. However, it relies on the implicit guarantee that `<input type="date">` always produces `YYYY-MM-DD` strings.
- **Reasoning:** This is a valid approach for ISO date strings. No fix needed, but worth noting that this correctness depends on the input format.
- **Consequence:** No practical consequence as HTML date inputs consistently use ISO format.
- **Suggestion:** No change needed. This is informational only.
