## Summary

This review covers phases 19-21 of the LittleLiver project, which add four new metric resource types: skin observations, bruising observations, lab results, and general notes. Each resource follows the established CRUD pattern (Create/List/Get/Update/Delete) at both the handler and store layers. The code is functionally correct and consistent with earlier phases, but the accumulated duplication across 10 metric resources has reached a level that warrants attention. The new code introduces approximately 1,400 lines across 8 files (4 handler + 4 store), the vast majority of which is structurally identical to existing metric files.

### [MEDIUM] Parameter Sprawl in Store Create/Update Functions
- **File:** `backend/internal/store/bruising.go`, `skin_observations.go`, `lab_results.go`, `general_notes.go`
- **Line(s):** bruising.go L46, L151; skin_observations.go L46, L151; lab_results.go L44, L149; general_notes.go L43, L148
- **Perspective:** Tech Lead
- **Finding:** Store Create and Update functions accept 7-10 individual parameters. For example, `CreateBruising` takes `(db, babyID, loggedBy, timestamp, location, sizeEstimate string, sizeCm *float64, color, notes *string)` (9 parameters), and `UpdateBruising` takes 10 parameters. This pattern is repeated across all 10 metric store files. The existing `model.BruisingObservation` struct already contains all these fields.
- **Reasoning:** Go convention discourages functions with more than 5-6 parameters. The model structs already exist and contain all these fields. Passing individual parameters forces callers to maintain exact parameter ordering, which is error-prone and makes signature changes ripple across handler and test code. This is a pre-existing pattern from earlier phases, but each new metric resource compounds the problem.
- **Consequence:** As more fields are added to any metric (e.g., adding a `severity` field to bruising), every caller must be updated and parameter position bugs become likely. A misplaced `*string` argument would compile without error since multiple parameters share the same type.
- **Suggestion:** Accept a struct parameter for domain-specific fields. For example:
  ```go
  type CreateBruisingParams struct {
      BabyID       string
      LoggedBy     string
      Timestamp    string
      Location     string
      SizeEstimate string
      SizeCm       *float64
      Color        *string
      Notes        *string
  }

  func CreateBruising(db *sql.DB, p CreateBruisingParams) (*model.BruisingObservation, error) {
  ```
  Alternatively, reuse the model struct directly (setting ID/timestamps server-side).

### [MEDIUM] Copy-Paste ListWithTZ Implementation Across All Store Files
- **File:** `backend/internal/store/bruising.go`, `skin_observations.go`, `lab_results.go`, `general_notes.go` (and 6 earlier files)
- **Line(s):** bruising.go L76-L148, skin_observations.go L76-L148, lab_results.go L74-L146, general_notes.go L73-L145
- **Perspective:** Tech Lead
- **Finding:** The `ListXxxWithTZ` function body is identical across all 10 store files, differing only in (1) the table name string, (2) the column constant, and (3) the scan function. Each copy is approximately 70 lines. Total: ~700 lines of near-identical pagination/filtering logic across the store layer.
- **Reasoning:** The project already uses generics (`model.MetricPage[T]`) and a shared `deleteByID` helper, demonstrating willingness to extract common patterns. The list function is the single largest source of duplication and is purely mechanical -- the only varying parts are the table name, column set, and scan function.
- **Consequence:** A bug in pagination logic (e.g., off-by-one in cursor handling) or a change to the date filtering behavior would need to be applied identically in 10 files. The risk of drift between implementations increases with each new metric.
- **Suggestion:** Extract a generic list helper:
  ```go
  func listMetricWithTZ[T any](
      db *sql.DB, table, columns string,
      scanFn func(scanner) (*T, error),
      babyID string, from, to, cursor *string,
      limit int, loc *time.Location,
  ) (*model.MetricPage[T], error) {
      // shared implementation
  }
  ```
  Each store file then becomes a one-liner delegating to this helper.

### [MEDIUM] Copy-Paste Handler CRUD Structure Across All Handler Files
- **File:** `backend/internal/handler/bruising.go`, `skin_observations.go`, `lab_results.go`, `general_notes.go`
- **Line(s):** N/A - structural issue
- **Perspective:** Tech Lead
- **Finding:** Each handler file follows an identical 5-handler pattern (Create, List, Get, Update, Delete) with the same authentication/authorization preamble (~12 lines per handler), JSON decode, validation, store call, and response mapping. Across 10 metric handler files, this amounts to ~2,272 lines total, of which roughly 60-70% (approximately 1,400 lines) is boilerplate that differs only by type names and store function references. The `registerMetricCRUD` helper in `router.go` already acknowledges this pattern but does not reduce the handler code itself.
- **Reasoning:** The handlers have no resource-specific logic beyond (a) the request/response struct definitions, (b) the validate() method, and (c) the store function to call. The auth preamble, JSON decode, error handling, and response writing are identical.
- **Consequence:** Adding a new metric resource requires copying an entire handler file and performing a find-and-replace. Any improvement to error handling or response formatting must be replicated across 10 files.
- **Suggestion:** Consider a generic CRUD handler factory that takes a configuration struct:
  ```go
  type MetricCRUD[Req any, Model any, Resp any] struct {
      Validate   func(*Req) (string, bool)
      ToResp     func(*Model) Resp
      Create     func(db *sql.DB, babyID, userID string, req *Req) (*Model, error)
      // ... List, Get, Update, Delete
  }
  ```
  This is a larger refactor and the current approach is perfectly functional. The existing pattern is idiomatic Go and easy to understand. This is a judgment call -- but at 10 resources, the duplication cost is becoming concrete.

### [LOW] Stringly-Typed Category Validation Message in General Notes
- **File:** `backend/internal/handler/general_notes.go`
- **Line(s):** L30
- **Perspective:** QA
- **Finding:** The validation error message hard-codes the valid category values as a string literal: `"invalid category: must be one of behavior, sleep, vomiting, irritability, skin, other"`. Meanwhile, the actual valid values are defined in `model/enums.go` via `validSet("behavior", "sleep", "vomiting", "irritability", "skin", "other")`. If the enum values are updated in `enums.go`, this error message will become stale.
- **Reasoning:** This is the only validation in the new code that embeds the valid values directly in the error message. Other validations (e.g., `bruising.go` L36: `"invalid size_estimate"`) do not enumerate values, which is more resilient to change.
- **Consequence:** After adding a new category to `noteCategoryValues`, users would receive an error message listing the old set, causing confusion.
- **Suggestion:** Either remove the enumeration from the message (matching the style of other validators), or generate the message from the source of truth:
  ```go
  // Option A: match existing style
  return "invalid category", false

  // Option B: derive from enum (requires exporting values or adding a helper)
  return fmt.Sprintf("invalid category: must be one of %s", model.NoteCategoryList()), false
  ```

### [LOW] Redundant Response Structs Mirror Model Structs
- **File:** `backend/internal/handler/bruising.go`, `skin_observations.go`, `lab_results.go`, `general_notes.go`
- **Line(s):** bruising.go L44-L58, skin_observations.go L35-L49, lab_results.go L38-L51, general_notes.go L36-L47
- **Perspective:** Tech Lead
- **Finding:** Each handler file defines a response struct (e.g., `bruisingResponse`) that is field-for-field identical to the corresponding model struct, except that `time.Time` fields are replaced with `string`. The `toXxxResponse` mapping functions then manually copy every field. Across 4 new files, this adds roughly 100 lines of pure field-mapping boilerplate (and ~250 lines total across all 10 metric handlers).
- **Reasoning:** The response structs exist solely to format timestamps as strings. While this is a valid concern (controlling JSON serialization), the mapping is mechanical and error-prone if a field is added to the model but forgotten in the response struct.
- **Consequence:** Adding a field to a model struct without updating the corresponding response struct and mapping function would silently drop the field from API responses.
- **Suggestion:** Consider implementing a custom `MarshalJSON` on the model types, or using a shared `TimeString` type with a custom JSON marshaler, to eliminate the response struct layer entirely. Alternatively, accept this as acceptable boilerplate for explicit API contract control, but add a test that verifies response struct field counts match model struct field counts.
