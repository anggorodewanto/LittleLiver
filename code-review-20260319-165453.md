## Summary

Phase 48 implements the medication management UI for LittleLiver: medication list view with active/inactive distinction, create/edit medication forms with pre-populated suggestions and schedule time pickers, deactivation toggle, med-log history view per medication, dose logging form with medication pre-fill via URL param, and a "Med Given" quick-log button on the Today dashboard. All 327 tests pass. The implementation is well-structured overall but has several spec compliance issues and one functional concern.

### [MEDIUM] Missing "Other" in medication name suggestions
- **File:** `frontend/src/lib/components/MedicationForm.svelte`
- **Line(s):** L25-L33
- **Perspective:** Product Owner
- **Finding:** The `MEDICATION_SUGGESTIONS` array omits `Other` from the list. The spec (section 3.9) explicitly lists the pre-populated suggestions as: `UDCA (ursodiol)`, `Sulfamethoxazole-Trimethoprim (Bactrim)`, `Vitamin A`, `Vitamin D`, `Vitamin E (TPGS)`, `Vitamin K`, `Iron`, `Other`.
- **Reasoning:** While users can type any name into the free-text field, the "Other" suggestion provides a discoverable option for parents who may not know whether their medication fits the pre-defined list, improving UX clarity.
- **Consequence:** Parents may not realize they can enter custom medication names, or may feel uncertain about doing so without an explicit "Other" option.
- **Suggestion:** Add `'Other'` to the `MEDICATION_SUGGESTIONS` array:
  ```typescript
  const MEDICATION_SUGGESTIONS = [
      'UDCA (ursodiol)',
      'Sulfamethoxazole-Trimethoprim (Bactrim)',
      'Vitamin A',
      'Vitamin D',
      'Vitamin E (TPGS)',
      'Vitamin K',
      'Iron',
      'Other'
  ];
  ```

### [MEDIUM] Extra "Medication" quick-log button not in spec
- **File:** `frontend/src/lib/components/QuickLogButtons.svelte`
- **Line(s):** L12-L18
- **Perspective:** Product Owner
- **Finding:** The component renders 6 buttons: Feed, Wet Diaper, Stool, Temp, Medication, and Med Given. The spec (section 7.1) defines exactly 5 quick-log buttons: "Feed, Diaper (wet), Diaper (stool), Temp, Medication Given." There is no separate "Medication" button in the spec -- only "Medication Given."
- **Reasoning:** The extra "Medication" button creates ambiguity for users. "Medication" could mean managing medication definitions or logging a dose. The spec is clear that the quick-log area is for rapid data entry, so only dose logging ("Medication Given") belongs here.
- **Consequence:** User confusion about the difference between "Medication" and "Med Given" buttons. The `MetricType` union type also includes `'medication'` which has no clear handler behavior distinct from `'med_given'`.
- **Suggestion:** Remove the "Medication" button, keeping only "Med Given" (or rename it to "Medication Given" to match spec wording exactly). Update the `MetricType` to remove `'medication'`:
  ```svelte
  <button type="button" onclick={() => onselect('feeding')}>Feed</button>
  <button type="button" onclick={() => onselect('urine')}>Wet Diaper</button>
  <button type="button" onclick={() => onselect('stool')}>Stool</button>
  <button type="button" onclick={() => onselect('temperature')}>Temp</button>
  <button type="button" onclick={() => onselect('med_given')}>Med Given</button>
  ```

### [MEDIUM] Svelte compiler warnings about non-reactive state references
- **File:** `frontend/src/lib/components/MedicationForm.svelte`
- **Line(s):** L37-L40
- **Perspective:** Tech Lead
- **Finding:** The Svelte compiler emits warnings: "This reference only captures the initial value of `initialData`. Did you mean to reference it inside a derived instead?" for lines 37-40 where `$state` variables are initialized from `initialData` props. Similarly, `DoseLogForm.svelte` line 37 has the same warning for `medicationId`.
- **Reasoning:** While the current behavior is functionally correct for create/edit forms (initial data is set once), these compiler warnings indicate the code does not properly react to prop changes. If a parent navigates between editing different medications without unmounting the component, the form would show stale data.
- **Consequence:** If the component is reused without remounting (e.g., navigating from editing medication A to editing medication B), the form fields would retain medication A's values instead of updating to medication B.
- **Suggestion:** Use `$derived` for initial values or restructure to react to prop changes:
  ```typescript
  let name = $derived(initialData?.name ?? '');
  // Or use an $effect to reset when initialData changes
  $effect(() => {
      name = initialData?.name ?? '';
      dose = initialData?.dose ?? '';
      frequency = initialData?.frequency ?? '';
      scheduleTimes = initialData?.schedule_times ?? [];
  });
  ```

### [LOW] MedicationList toggleDeactivate only deactivates, no reactivation path
- **File:** `frontend/src/lib/components/MedicationList.svelte`
- **Line(s):** L65-L72
- **Perspective:** Product Owner
- **Finding:** The `toggleDeactivate` function always sends `{ active: false }`. The Deactivate button only appears for active medications (line 99-101). There is no way to reactivate a medication from this view. Per the spec, `PUT /api/babies/:id/medications/:id` supports updating `active` to either value.
- **Reasoning:** Phase 48 says "Deactivation toggle" which implies bidirectional toggling. While a strict reading could mean one-way deactivation, providing no reactivation path means a parent who accidentally deactivates a medication must use the Edit form to restore it (if that path even exists). The function is also named `toggleDeactivate` suggesting toggle semantics.
- **Consequence:** If a parent accidentally deactivates a medication, there is no obvious UI path to reactivate it without editing the medication. This could cause missed medication reminders.
- **Suggestion:** Make the toggle bidirectional:
  ```typescript
  async function toggleActive(med: Medication): Promise<void> {
      try {
          await apiClient.put(`/babies/${babyId}/medications/${med.id}`, { active: !med.active });
          await fetchMedications();
      } catch {
          error = 'Failed to update medication';
      }
  }
  ```
  And show a "Reactivate" button for inactive medications.

### [LOW] DoseLogForm silently fails on medication list fetch error
- **File:** `frontend/src/lib/components/DoseLogForm.svelte`
- **Line(s):** L43-L49
- **Perspective:** QA
- **Finding:** The `fetchMedications` function has a bare `catch` block with a comment "silent fail for medication loading." If the API call fails, the medication dropdown will be empty with no indication to the user about why.
- **Reasoning:** The dose log form is a critical path -- it is opened from notification clicks (`/log/med?medication_id=X`). If medication loading fails silently, the user sees an empty dropdown and cannot log a dose, with no actionable feedback.
- **Consequence:** A parent clicking a medication notification while having network issues would see an empty medication dropdown with no error message, potentially missing a dose log.
- **Suggestion:** Show an error state when medications fail to load:
  ```typescript
  let medicationLoadError = $state('');
  async function fetchMedications(): Promise<void> {
      try {
          const data = await apiClient.get<MedicationsResponse>(`/babies/${babyId}/medications`);
          medications = data.medications.filter((m) => m.active);
      } catch {
          medicationLoadError = 'Failed to load medications. Please try again.';
      }
  }
  ```

### [LOW] MedicationForm notes field not populated from initialData in edit mode
- **File:** `frontend/src/lib/components/MedicationForm.svelte`
- **Line(s):** L41
- **Perspective:** QA
- **Finding:** The `notes` state is always initialized to an empty string (`let notes = $state('');`) and is not populated from `initialData`. The `MedicationInitialData` interface does not include a `notes` field. When editing an existing medication, any previously saved notes would be lost.
- **Reasoning:** If medications have notes stored on the backend, editing a medication would clear the notes field since it is not pre-filled.
- **Consequence:** Editing a medication could silently clear any existing notes, losing information parents previously recorded.
- **Suggestion:** Add `notes` to `MedicationInitialData` and initialize from it:
  ```typescript
  export interface MedicationInitialData {
      name: string;
      dose: string;
      frequency: string;
      schedule_times: string[];
      active: boolean;
      notes?: string;
  }
  // ...
  let notes = $state(initialData?.notes ?? '');
  ```
