## Summary

This review covers phases 10-12 of the LittleLiver project, which introduce baby CRUD endpoints, invite code generation/redemption, and per-session rate limiting middleware. The review focuses exclusively on efficiency issues: redundant DB queries, unbounded memory growth, unnecessary work on the hot path, and TOCTOU patterns.

### [MEDIUM] Redundant DB round-trip: GetBabyByID followed by IsParentOfBaby in UpdateBabyHandler and CreateInviteHandler
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L212-L233
- **Perspective:** Performance Engineer
- **Finding:** `UpdateBabyHandler` calls `store.GetBabyByID` (line 212) solely to check existence, discards the result, then calls `store.IsParentOfBaby` (line 224) as a second DB round-trip. The same pattern appears in `CreateInviteHandler` at `backend/internal/handler/invites.go` lines 49-67.
- **Reasoning:** The existence check is redundant because `IsParentOfBaby` already implies the baby exists if `count > 0`, and `UpdateBaby` itself returns `sql.ErrNoRows` when the baby does not exist. You can collapse both checks into a single query: `SELECT 1 FROM baby_parents WHERE baby_id = ? AND user_id = ?` with an additional check that the baby exists via a LEFT JOIN or by simply relying on the downstream operation's own not-found error.
- **Consequence:** Every update and invite-create request executes two sequential queries (SELECT all baby columns + SELECT COUNT from baby_parents) when one would suffice. Under load this doubles the per-request DB round-trip time for these endpoints.
- **Suggestion:** For `UpdateBabyHandler`, remove the `GetBabyByID` existence check entirely. `IsParentOfBaby` returning false covers both "baby doesn't exist" and "not linked". If you need to distinguish 404 from 403, write a single query like `SELECT EXISTS(SELECT 1 FROM babies WHERE id = ?) AS baby_exists, EXISTS(SELECT 1 FROM baby_parents WHERE baby_id = ? AND user_id = ?) AS is_parent` to get both answers in one round-trip. For `CreateInviteHandler`, apply the same approach.

### [MEDIUM] Extra DB round-trip after CreateBaby and UpdateBaby via re-SELECT
- **File:** `backend/internal/store/babies.go`
- **Line(s):** L43, L130
- **Perspective:** Performance Engineer
- **Finding:** `CreateBaby` inserts a baby within a transaction, commits it, then calls `GetBabyByID(db, id)` on line 43 to return the created record. `UpdateBaby` does the same on line 130. Both functions already know the values they just wrote -- the only unknown is `created_at` (which is a server-side DEFAULT).
- **Reasoning:** This is an extra SELECT after every INSERT and UPDATE. For CreateBaby, you can read `created_at` with a final `SELECT created_at FROM babies WHERE id = ?` inside the same transaction (much cheaper than re-reading all columns), or use `RETURNING` if the SQLite version supports it (3.35+). For UpdateBaby, the caller already supplied every field except `created_at`, so the full re-read fetches data the caller already has.
- **Consequence:** Every create and update operation performs an unnecessary full-row SELECT, doubling the query count for write operations.
- **Suggestion:** For `CreateBaby`, query only `created_at` within the transaction or construct the model directly from the input parameters plus a `time.Now()` approximation. For `UpdateBaby`, do the same -- the only field the caller does not control is `created_at`, which does not change on update and could be fetched once or passed through.

### [HIGH] Unbounded memory growth in RateLimiter session map -- no eviction of expired entries
- **File:** `backend/internal/middleware/ratelimit.go`
- **Line(s):** L16-L29 (struct definition) and L34-L58 (allow method)
- **Perspective:** Performance Engineer
- **Finding:** The `sessions` map in `RateLimiter` grows without bound. Entries are created for every unique session cookie value but are never deleted. When a window expires, the bucket is reset in-place (line 47-49), but only if that session makes another request. Sessions that become inactive after their window are never evicted.
- **Reasoning:** Over time, as sessions expire and users stop visiting, the map accumulates stale entries indefinitely. Each entry is small (~40 bytes for the pointer + bucket), but in a long-running process with thousands of distinct sessions over days/weeks, this constitutes a slow memory leak.
- **Consequence:** In a long-running production instance, the map will grow monotonically. For a personal-use app this may take a long time to matter, but it is a design flaw that violates the expectation of bounded memory. If session IDs are ever spoofed or rotated frequently, growth accelerates.
- **Suggestion:** Add a periodic cleanup goroutine that runs on a timer (e.g., every 2x the window duration) and removes entries whose `windowStart` is older than the window. Example:
  ```go
  func (rl *RateLimiter) startCleanup() {
      go func() {
          ticker := time.NewTicker(rl.window * 2)
          defer ticker.Stop()
          for range ticker.C {
              rl.mu.Lock()
              now := time.Now()
              for k, b := range rl.sessions {
                  if now.Sub(b.windowStart) >= rl.window {
                      delete(rl.sessions, k)
                  }
              }
              rl.mu.Unlock()
          }
      }()
  }
  ```

### [MEDIUM] GetBabyHandler: two sequential DB queries that could be one
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L167-L187
- **Perspective:** Performance Engineer
- **Finding:** `GetBabyHandler` calls `store.GetBabyByID` (line 167) to fetch the full baby row, then calls `store.IsParentOfBaby` (line 178) to verify authorization. These are two independent sequential queries.
- **Reasoning:** Since the handler needs both the baby data and the authorization check, these could be combined into a single query with a JOIN: `SELECT b.* FROM babies b JOIN baby_parents bp ON b.id = bp.baby_id WHERE b.id = ? AND bp.user_id = ?`. This returns the baby only if the user is linked, or no rows if either the baby does not exist or the user is not linked. You lose the ability to distinguish 404 from 403, but you could add `SELECT EXISTS(SELECT 1 FROM babies WHERE id = ?)` as a fallback only when the first query returns no rows.
- **Consequence:** Every GET /api/babies/:id request makes two sequential DB round-trips when one would suffice. This is the most frequently called read path for baby data.
- **Suggestion:** Add a `GetBabyByIDForUser(db, babyID, userID)` store function that JOINs babies with baby_parents and returns the baby if linked, or sql.ErrNoRows otherwise.

### [LOW] IsParentOfBaby uses COUNT(*) instead of EXISTS pattern
- **File:** `backend/internal/store/babies.go`
- **Line(s):** L94-L104
- **Perspective:** Performance Engineer
- **Finding:** `IsParentOfBaby` uses `SELECT COUNT(*) FROM baby_parents WHERE user_id = ? AND baby_id = ?` to check existence. While functionally correct, `SELECT 1 FROM baby_parents WHERE user_id = ? AND baby_id = ? LIMIT 1` is marginally more efficient because the query engine can stop after finding the first match rather than counting all matches.
- **Reasoning:** The baby_parents table has a composite primary key, so COUNT(*) will scan at most one row via index anyway. This is a minor optimization that matters more as the table grows or if the schema changes.
- **Consequence:** Negligible impact in practice for this table structure, but using the EXISTS/LIMIT 1 pattern communicates intent more clearly and is the idiomatic approach for existence checks.
- **Suggestion:** Change to `SELECT 1 FROM baby_parents WHERE user_id = ? AND baby_id = ? LIMIT 1` and check for `sql.ErrNoRows` instead of scanning a count.

### [LOW] TOCTOU in UpdateBabyHandler: existence check before update
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L212-L221
- **Perspective:** QA
- **Finding:** `UpdateBabyHandler` first checks that the baby exists via `GetBabyByID` (line 212), then later calls `store.UpdateBaby` (line 268). Between these two calls, the baby could theoretically be deleted by another request. The `UpdateBaby` store function already handles the "not found" case by checking `RowsAffected() == 0` and returning `sql.ErrNoRows`.
- **Reasoning:** The initial existence check is not only redundant (as noted above) but introduces a time-of-check-to-time-of-use gap. If the baby is deleted after the check but before the update, the handler would get a confusing 500 error ("failed to update baby") instead of the expected 404.
- **Consequence:** In a concurrent environment, a race between delete and update operations could produce misleading error responses. Low severity because the app is personal-use and concurrent deletes are unlikely.
- **Suggestion:** Remove the `GetBabyByID` check; let `store.UpdateBaby` return `sql.ErrNoRows` and handle it in the handler as a 404.

### [LOW] CreateInvite returns a fabricated CreatedAt instead of the DB value
- **File:** `backend/internal/store/invites.go`
- **Line(s):** L79-L85
- **Perspective:** QA
- **Finding:** After committing the transaction, `CreateInvite` constructs the returned `model.Invite` with `CreatedAt: time.Now().UTC()` (line 84). This is a second call to `time.Now()`, distinct from the one that might be used by the DB's DEFAULT for `created_at`. There is no guarantee these timestamps match.
- **Reasoning:** While this is not strictly an efficiency issue, it represents unnecessary imprecision. The function already avoids re-reading from the DB (which is good for efficiency), but it fabricates a timestamp that may differ from the actual stored value by several milliseconds.
- **Consequence:** Minor inconsistency between the returned invite's `CreatedAt` and the actual DB row. Not a performance issue, but a data correctness concern worth noting.
- **Suggestion:** If the `created_at` column uses a DB-side DEFAULT, either read it back with a `SELECT created_at FROM invites WHERE code = ?` inside the transaction, or set it explicitly in the INSERT so you control the value returned.
