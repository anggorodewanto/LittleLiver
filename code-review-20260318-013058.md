## Summary

This review covers phases 34-36 of LittleLiver, focusing on **efficiency only**: the medication notification scheduler (`notify/scheduler.go`), periodic cleanup tasks (`cron/cleanup.go`), the cron runner (`cron/runner.go`), and their associated tests. The changes introduce a minute-by-minute scheduler that checks active medications and sends push notifications, plus hourly cleanup of expired invites, sessions, and orphaned photos.

### [HIGH] N+1 Query Pattern in Scheduler Tick
- **File:** `backend/internal/notify/scheduler.go`
- **Line(s):** L71-L73, L155, L187-L205, L229-L247
- **Perspective:** Performance Engineer
- **Finding:** For each active medication that is due, `Tick()` issues up to 3 separate database queries in sequence: (1) `isDoseSuppressed` queries `med_logs` per medication per schedule time (L155), (2) `sendNotifications` queries `baby_parents` per medication (L187-L205), and (3) `queryPushSubscriptions` queries `push_subscriptions` per parent (L229-L247). With M medications due across B babies and P parents, this is O(M * S + M + M*P) queries per tick.
- **Reasoning:** This is a classic N+1 pattern. While the personal-use scale is small (likely a handful of medications), this architecture issues many small queries every minute. The medication loop at L71-L73 processes each med sequentially, and each `processMedication` call can fan out to multiple queries.
- **Consequence:** At current scale this is tolerable, but if the app ever supports multiple families or the number of scheduled medications grows, each tick could issue dozens of queries against SQLite, holding the WAL lock repeatedly and adding latency. More critically, each query opens and closes a statement on every tick, generating unnecessary GC pressure.
- **Suggestion:** Batch the suppression check into a single query by collecting all (medication_id, baby_id, window_start, window_end) tuples first, then issuing one query with `IN (...)` or a CTE. Similarly, pre-fetch all parent-to-subscription mappings for relevant baby_ids in a single JOIN query before the medication loop:
  ```sql
  SELECT bp.baby_id, ps.user_id, ps.endpoint, ps.p256dh, ps.auth
  FROM baby_parents bp
  JOIN push_subscriptions ps ON ps.user_id = bp.user_id
  WHERE bp.baby_id IN (?, ?, ...)
  ```
  This collapses the per-medication fan-out into two queries total: one for suppression, one for subscriptions.

### [MEDIUM] Row-by-Row R2 Deletion in CleanupPhotos
- **File:** `backend/internal/cron/cleanup.go`
- **Line(s):** L72-L90
- **Perspective:** Performance Engineer
- **Finding:** `CleanupPhotos` iterates over orphaned photos one at a time, issuing an `objStore.Delete` call per R2 key (up to 2 per photo: original + thumbnail) and then a `db.Exec("DELETE ... WHERE id = ?")` per row. Each R2 delete is a network round-trip to Cloudflare.
- **Reasoning:** R2 (and S3-compatible stores) support batch delete operations (up to 1000 keys per call in S3). The current implementation makes 2N+N network calls for N orphaned photos -- two R2 deletes plus one DB delete per photo. The DB deletes could also be batched into a single `DELETE ... WHERE id IN (...)`.
- **Consequence:** If orphaned photos accumulate (e.g., user uploads many photos without linking them), this cleanup could take a very long time due to sequential network round-trips. With 100 orphaned photos, that is 300 sequential HTTP calls to R2 plus 100 sequential DB statements.
- **Suggestion:** Collect all R2 keys to delete, then issue a batch delete if the `ObjectStore` interface supports it (or add a `DeleteBatch(ctx, keys []string)` method). For the DB side, batch the IDs:
  ```go
  // After successful R2 deletes, batch-delete DB rows
  ids := make([]string, len(successfullyDeleted))
  // ...
  db.Exec("DELETE FROM photo_uploads WHERE id IN (" + placeholders + ")", args...)
  ```
  Even without changing the `ObjectStore` interface, the DB deletes can be batched immediately.

### [MEDIUM] Sequential Execution of Independent Cleanup Tasks in RunAll
- **File:** `backend/internal/cron/cleanup.go`
- **Line(s):** L96-L124
- **Perspective:** Performance Engineer
- **Finding:** `RunAll` executes `CleanupInvites`, `CleanupSessions`, and `CleanupPhotos` sequentially. The first two are fast single-statement DELETEs, but `CleanupPhotos` involves network I/O to R2. All three tasks are logically independent.
- **Reasoning:** `CleanupInvites` and `CleanupSessions` are pure DB operations and fast, so running them in parallel with each other gains little. However, `CleanupPhotos` can be slow due to R2 network calls. Running it concurrently with the DB-only tasks would reduce total cleanup wall time. Additionally, the current early-return-on-error pattern means if `CleanupInvites` fails, sessions and photos are never cleaned up.
- **Consequence:** A transient failure in invite cleanup prevents session and photo cleanup from running at all for that cycle. In the worst case, expired sessions and orphaned R2 objects accumulate until the next successful invites cleanup.
- **Suggestion:** Run all three tasks concurrently with `errgroup`, or at minimum change to a collect-all-errors approach so one failure does not block the others:
  ```go
  var errs []error
  invites, err := CleanupInvites(db, now)
  if err != nil {
      errs = append(errs, err)
  }
  // ... continue with sessions and photos regardless
  return errors.Join(errs...)
  ```

### [LOW] Repeated time.LoadLocation Calls Per Medication Per Tick
- **File:** `backend/internal/notify/scheduler.go`
- **Line(s):** L109
- **Perspective:** Performance Engineer
- **Finding:** `time.LoadLocation` is called for every medication on every tick (once per minute). While Go caches timezone data internally after the first load, this still involves a map lookup and string comparison per call. Multiple medications sharing the same timezone (e.g., all in "America/New_York") will redundantly look up the same location.
- **Reasoning:** This is a minor inefficiency. `time.LoadLocation` is not expensive after the first call due to internal caching, but a local cache within `Tick` or `Scheduler` would be cleaner and more explicit.
- **Consequence:** Negligible performance impact at current scale. This is more of a code hygiene issue.
- **Suggestion:** Add a simple timezone cache within the `Tick` method:
  ```go
  tzCache := map[string]*time.Location{}
  for _, med := range meds {
      tz := "UTC"
      if med.Timezone != nil && *med.Timezone != "" {
          tz = *med.Timezone
      }
      loc, ok := tzCache[tz]
      if !ok {
          loc, err = time.LoadLocation(tz)
          // ...
          tzCache[tz] = loc
      }
      // use loc
  }
  ```

### [LOW] CleanupPhotos Loads All Orphan Rows Into Memory Before Processing
- **File:** `backend/internal/cron/cleanup.go`
- **Line(s):** L58-L65
- **Perspective:** Performance Engineer
- **Finding:** All orphaned photo rows are collected into a slice (`var photos []orphanPhoto`) before any deletion begins. This means the entire result set is held in memory.
- **Reasoning:** For a personal-use baby tracker app, the number of orphaned photos is likely small. However, if a bug or misuse leads to many unlinked uploads, this could consume significant memory. Processing rows in a streaming fashion (delete as you scan) would bound memory usage.
- **Consequence:** Low risk at current scale. If thousands of orphaned photos accumulate, memory usage spikes during cleanup.
- **Suggestion:** Process rows as they are scanned rather than collecting them first, or add a `LIMIT` clause to cap the batch size per cleanup cycle.
