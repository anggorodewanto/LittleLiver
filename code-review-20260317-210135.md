## Summary

Phases 25-27 introduce Medication CRUD, Med-Log dose logging/editing/deletion, and a Dashboard aggregation endpoint. The changes span new store functions, handler files, migrations, and model types. This review focuses exclusively on efficiency issues in the production code paths.

### [HIGH] Six sequential SQL queries in GetDashboardSummary where several could be combined
- **File:** `backend/internal/store/dashboard.go`
- **Line(s):** L39-L123
- **Perspective:** Performance Engineer
- **Finding:** `GetDashboardSummary` executes 6 sequential `QueryRow` calls against the database. The first three queries (feeds count+calories, urine count, stools count) all filter on the same `baby_id` and date range against different tables. The stools count (query 3) and most-recent stool color (query 4) both hit the `stools` table with the same WHERE clause.
- **Reasoning:** Each `QueryRow` is a full database round-trip. On SQLite this is cheap (in-process), but if the project ever migrates to a networked database (as the spec allows), this becomes 6 serial round-trips per dashboard load -- the single most latency-sensitive endpoint in the app. Even on SQLite, queries 3 and 4 scan the same `stools` rows twice.
- **Consequence:** Dashboard latency scales linearly with the number of queries. With a networked DB, a single dashboard request would take 6x the single-query latency. The duplicate stools table scan wastes CPU on every request today.
- **Suggestion:** Combine the stools count and most-recent color into one query:
  ```sql
  SELECT COUNT(*),
         (SELECT color_label FROM stools
          WHERE baby_id = ? AND timestamp >= ? AND timestamp < ?
          ORDER BY timestamp DESC LIMIT 1)
  FROM stools
  WHERE baby_id = ? AND timestamp >= ? AND timestamp < ?
  ```
  Longer-term, consider combining all date-range aggregations into a single query using subqueries or a CTE, and run the two date-independent queries (last temp, last weight) concurrently with an `errgroup`.

### [HIGH] Three sequential store calls in DashboardHandler that are fully independent
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L55-L74
- **Perspective:** Performance Engineer
- **Finding:** The handler calls `GetDashboardSummary`, `GetStoolColorTrend`, and `GetUpcomingMeds` sequentially. These three operations are completely independent -- none depends on the result of another.
- **Reasoning:** This is the dashboard endpoint, likely the most frequently loaded page in the app. Running independent database work sequentially on a per-request hot path unnecessarily increases P50 latency by the sum of all three call durations rather than the max.
- **Consequence:** Dashboard response time is the sum of all three queries' durations rather than the maximum. With 6 internal SQL queries in `GetDashboardSummary` alone, plus 2 more for the other two calls, the endpoint performs 8 serial database operations when the minimum is 6 (with the merge above) or even fewer with concurrency.
- **Suggestion:** Use `sync.WaitGroup` or `golang.org/x/sync/errgroup` to run all three store calls concurrently:
  ```go
  var (
      summary *store.DashboardSummary
      trend   []store.StoolColorEntry
      meds    []store.UpcomingMed
  )
  g, _ := errgroup.WithContext(r.Context())
  g.Go(func() error {
      var err error
      summary, err = store.GetDashboardSummary(db, baby.ID, from, to)
      return err
  })
  g.Go(func() error {
      var err error
      trend, err = store.GetStoolColorTrend(db, baby.ID)
      return err
  })
  g.Go(func() error {
      var err error
      meds, err = store.GetUpcomingMeds(db, baby.ID)
      return err
  })
  if err := g.Wait(); err != nil {
      // handle error
  }
  ```

### [MEDIUM] UpdateMedication performs 3 SQL round-trips (SELECT + UPDATE + SELECT)
- **File:** `backend/internal/store/medications.go`
- **Line(s):** L104-L135
- **Perspective:** Performance Engineer
- **Finding:** `UpdateMedication` calls `GetMedicationByID` to read the current `active` state, then performs the UPDATE, then calls `GetMedicationByID` again to return the updated row. This is 3 SQL operations for a single update.
- **Reasoning:** The initial SELECT exists solely to preserve the current `active` value when the caller passes `active=nil`. The handler always provides all fields in the PUT body, so this read is avoidable if the handler resolved the `active` value itself (it already has the request body). The post-update SELECT is a pattern used across all store update functions, so it is at least consistent -- but the pre-update SELECT is unique to this function.
- **Consequence:** Every medication update takes 3 round-trips instead of 2. For a personal-use app this is minor, but it is unnecessary work on every PUT request.
- **Suggestion:** Have the handler resolve the `active` value before calling the store (e.g., default to `true` if not provided in the request), and pass a concrete `bool` instead of `*bool`. This eliminates the pre-update SELECT entirely, dropping the operation from 3 queries to 2.

### [MEDIUM] ListMedications returns all medications with no pagination or limit
- **File:** `backend/internal/store/medications.go`
- **Line(s):** L75-L101
- **Perspective:** Performance Engineer
- **Finding:** `ListMedications` has no LIMIT clause and no pagination. It returns every medication (active and inactive) for a baby, loading all rows into memory.
- **Reasoning:** While a single baby is unlikely to have hundreds of medications, this is a pattern that differs from other list endpoints in the codebase (e.g., `ListMedLogs` has cursor-based pagination). An unbounded query establishes a precedent that could be copied into higher-cardinality tables. The `ListMedicationsHandler` also calls `toMedicationResponse` per item, which does `json.Unmarshal` on the schedule field -- this work grows linearly with no bound.
- **Consequence:** If a baby accumulates many medications over time (never deleted, only deactivated), memory usage and response size grow without bound. The `json.Unmarshal` per item makes this slightly worse than a simple SELECT.
- **Suggestion:** Either add a reasonable LIMIT (e.g., 100) or document that this endpoint is intentionally unbounded because baby medication counts are naturally small. At minimum, pre-allocate the slice with a capacity hint: `meds := make([]model.Medication, 0, 16)`.

### [MEDIUM] time.LoadLocation called per medication on every dashboard request
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L122-L131
- **Perspective:** Performance Engineer
- **Finding:** `computeNextDoseAt` calls `time.LoadLocation(*tz)` for each medication in the upcoming meds list. `time.LoadLocation` reads from the filesystem tzdata on each call (unless the location has been previously loaded and cached by the runtime).
- **Reasoning:** If multiple medications share the same timezone (the common case -- a baby is in one timezone), the same tzdata file is read repeatedly. While Go's stdlib does cache loaded locations internally, the cache lookup and string comparison still happen on every call. More importantly, in minimal container images that bundle tzdata via `time/tzdata`, this involves map lookups per call.
- **Consequence:** Minor overhead per dashboard request that scales with the number of active medications. Not critical for a personal-use app, but easily avoidable.
- **Suggestion:** Cache the `*time.Location` by timezone string in a local map within the handler loop:
  ```go
  locCache := make(map[string]*time.Location)
  for _, m := range meds {
      // ... existing code ...
      if m.Timezone != nil {
          if _, ok := locCache[*m.Timezone]; !ok {
              loc, err := time.LoadLocation(*m.Timezone)
              if err == nil {
                  locCache[*m.Timezone] = loc
              }
          }
      }
      resp.NextDoseAt = computeNextDoseAtWithLoc(resp.ScheduleTimes, locCache[*m.Timezone])
  }
  ```

### [LOW] Stool color trend query has no upper bound on result set
- **File:** `backend/internal/store/dashboard.go`
- **Line(s):** L128-L163
- **Perspective:** Performance Engineer
- **Finding:** `GetStoolColorTrend` queries all stool entries from the last 7 days with no LIMIT. The comment says "one entry per stool (multiple per day possible)" so this result set is unbounded within the 7-day window.
- **Reasoning:** For a baby health app, the typical number of stools per day is small (1-10). However, if data entry errors or bulk imports create many entries, the result set could be unexpectedly large. Every other list query in the codebase either has a LIMIT or pagination.
- **Consequence:** In pathological cases, a large number of stool entries in a 7-day window would cause the dashboard response to balloon in size and memory usage.
- **Suggestion:** Add a reasonable LIMIT (e.g., `LIMIT 100`) to the query to prevent pathological cases while still covering normal usage.
