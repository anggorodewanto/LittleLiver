## Summary

This review covers Phases 46-48 of the LittleLiver frontend: 7 Chart.js chart components (StoolColorChart, WeightChart, TemperatureChart, AbdomenGirthChart, FeedingChart, DiaperChart, LabTrendsChart), a DateRangeSelector, a TrendsView orchestrator, 4 medication management components (MedicationList, MedicationForm, MedLogList, DoseLogForm), supporting modules (chart-setup.ts, chart-mock.ts), and comprehensive tests. The review focuses exclusively on code reuse opportunities and duplication.

### [HIGH] Chart components share identical boilerplate structure that should be extracted

- **File:** `frontend/src/lib/components/AbdomenGirthChart.svelte`, `TemperatureChart.svelte`, `WeightChart.svelte`, `FeedingChart.svelte`, `DiaperChart.svelte`, `LabTrendsChart.svelte`
- **Line(s):** N/A - structural issue across 6 of 7 chart components
- **Perspective:** Tech Lead
- **Finding:** All 7 chart components repeat the same structural pattern: (1) `let canvas: HTMLCanvasElement`, (2) `let chart: Chart | null = null`, (3) `onMount` that early-returns on empty data then calls `new Chart(canvas, config)`, (4) `onDestroy(() => chart?.destroy())`, (5) a template with `{#if isEmpty}<p>No data</p>{:else}<canvas bind:this={canvas}></canvas>{/if}`. This is approximately 15-20 lines of identical scaffolding per component. Four of the charts (StoolColorChart, TemperatureChart, AbdomenGirthChart, LabTrendsChart) additionally duplicate the same x-axis date-formatting tick callback: `(value) => new Date(value as number).toLocaleDateString()`.
- **Reasoning:** Seven copies of the same lifecycle and DOM pattern is a maintenance burden. If Chart.js changes its API, or the empty-state message needs updating, or the cleanup pattern needs adjusting, all 7 files must be modified in lockstep.
- **Consequence:** A future developer updating chart cleanup logic (e.g., adding animation teardown) must find and update all 7 components. Missing one produces a memory leak in that specific chart only, which would be hard to diagnose.
- **Suggestion:** Extract a reusable Svelte action or wrapper component. For example, a `createChart` action:
  ```typescript
  // src/lib/chart-action.ts
  import { Chart, type ChartConfiguration } from 'chart.js';

  export function chartAction(canvas: HTMLCanvasElement, config: ChartConfiguration) {
    const chart = new Chart(canvas, config);
    return {
      update(newConfig: ChartConfiguration) { /* update logic */ },
      destroy() { chart.destroy(); }
    };
  }
  ```
  Alternatively, a `<ChartWrapper data={data} config={configFn} />` component that handles the canvas binding, isEmpty check, onMount/onDestroy, and the "No data" fallback in one place.

### [MEDIUM] Duplicated x-axis date tick callback across 4 chart components

- **File:** `frontend/src/lib/components/AbdomenGirthChart.svelte` (L51), `TemperatureChart.svelte` (L64), `StoolColorChart.svelte` (L53), `LabTrendsChart.svelte` (L103)

- **Perspective:** Tech Lead
- **Finding:** The identical callback `(value) => new Date(value as number).toLocaleDateString()` appears in 4 chart components for x-axis tick formatting.
- **Reasoning:** This is a formatting concern that could be centralized. If the date format needs to change (e.g., to show short month names or ISO format), 4 files need updating.
- **Consequence:** Inconsistent date formatting if one component is updated but others are missed.
- **Suggestion:** Extract to a shared constant in `chart-setup.ts` or a new `chart-utils.ts`:
  ```typescript
  export const dateTickCallback = (value: string | number) =>
    new Date(value as number).toLocaleDateString();
  ```

### [MEDIUM] Medication interface defined three times across medication components

- **File:** `frontend/src/lib/components/MedicationList.svelte` (L5-L17), `DoseLogForm.svelte` (L13-L20)
- **Line(s):** MedicationList L5-L17, DoseLogForm L13-L20
- **Perspective:** Tech Lead
- **Finding:** The `Medication` interface is defined in both `MedicationList.svelte` (with 10 fields) and `DoseLogForm.svelte` (with 5 fields as a subset). The `MedicationsResponse` wrapper interface is also duplicated in both. These are the same API entity.
- **Reasoning:** When the API's medication schema changes (e.g., adding a `notes` field), both interfaces need updating. The DoseLogForm version is already a divergent subset, meaning it could break if code starts relying on fields present in one definition but not the other.
- **Consequence:** Schema drift between the two definitions. A developer adds a field to one and forgets the other, causing type confusion or runtime access of undefined properties.
- **Suggestion:** Create a shared types file `src/lib/types/medication.ts`:
  ```typescript
  export interface Medication {
    id: string;
    baby_id: string;
    name: string;
    dose: string;
    frequency: string;
    schedule: string | null;
    timezone: string | null;
    active: boolean;
    created_at: string;
    updated_at: string;
  }
  export interface MedicationsResponse { medications: Medication[]; }
  ```

### [MEDIUM] `formatFrequency` in MedicationList should be a shared utility

- **File:** `frontend/src/lib/components/MedicationList.svelte`
- **Line(s):** L35-L49
- **Perspective:** Tech Lead
- **Finding:** The `formatFrequency()` function maps frequency codes like `once_daily` to display strings like "Once daily". The same frequency codes are used in `MedicationForm.svelte` (L37-L47 in `getTimeSlotCount`) and will likely be needed in any future component displaying medication details.
- **Reasoning:** Frequency display formatting is a cross-cutting concern for the medication domain. `MedicationForm` already has a parallel switch statement over the same frequency values for time slot counting.
- **Consequence:** If a new frequency type is added (e.g., `four_times_daily`), the format function and time-slot-count function both need updating independently.
- **Suggestion:** Extract to `src/lib/medication-utils.ts`:
  ```typescript
  export const FREQUENCY_LABELS: Record<string, string> = {
    once_daily: 'Once daily',
    twice_daily: 'Twice daily',
    three_times_daily: 'Three times daily',
    as_needed: 'As needed',
    custom: 'Custom'
  };
  export function formatFrequency(freq: string): string {
    return FREQUENCY_LABELS[freq] ?? freq;
  }
  ```

### [MEDIUM] `formatDate` in TrendsView duplicates logic that could live in datetime.ts

- **File:** `frontend/src/lib/components/TrendsView.svelte`
- **Line(s):** L58-L60
- **Perspective:** Tech Lead
- **Finding:** `TrendsView` defines `function formatDate(date: Date): string { return date.toISOString().split('T')[0]; }`. The project already has `src/lib/datetime.ts` which is the designated home for date utilities. This ISO date extraction pattern (`toISOString().split('T')[0]`) is a common need and belongs in the shared module.
- **Reasoning:** The `datetime.ts` module already exists and is imported by form components. Adding `formatDate` there keeps date logic centralized.
- **Consequence:** Minor: if the date format for API calls needs to change, developers may not find this inline definition.
- **Suggestion:** Add to `src/lib/datetime.ts`:
  ```typescript
  export function formatDateISO(date: Date): string {
    return date.toISOString().split('T')[0];
  }
  ```

### [MEDIUM] Pervasive fetch-on-mount pattern with `$effect` + `untrack` could use a helper

- **File:** `frontend/src/lib/components/MedicationList.svelte` (L52-L55), `MedLogList.svelte` (L51-L54), `DoseLogForm.svelte` (L63-L65), `TrendsView.svelte` (L115-L117), `TodayDashboard.svelte` (L176-L179)
- **Line(s):** N/A - structural issue across 5 components
- **Perspective:** Tech Lead
- **Finding:** Five components use the same reactive fetch pattern:
  ```typescript
  $effect(() => {
    void someReactiveDep;
    untrack(() => { void fetchSomething(); });
  });
  ```
  Combined with the `loading`/`error`/`data` state triple, this is repeated verbatim across all data-fetching components.
- **Reasoning:** This is an emerging pattern in the codebase. While each instance is small, the `untrack` + `void` incantation is non-obvious to new developers and error-prone if the reactive dependency tracking is done incorrectly.
- **Consequence:** Inconsistent error handling or loading state management across components if one is modified without updating others.
- **Suggestion:** This is a lower-priority extraction since Svelte 5 does not have a built-in `useQuery` equivalent, but consider a thin helper:
  ```typescript
  // src/lib/reactive-fetch.ts
  export function reactiveFetch<T>(deps: () => void, fetcher: () => Promise<T>) {
    // encapsulates the $effect + untrack pattern
  }
  ```
  Alternatively, document this pattern in a CONTRIBUTING.md so it is applied consistently.
