## Summary

Phase 44 implements six metric entry forms (Stool, Abdomen, Skin, Bruising, Lab, Notes) plus a shared PhotoUpload component. The implementation covers all forms specified in the phase description with appropriate tests. All 179 tests pass. The forms handle validation, payload construction, photo upload integration, stool color swatches (7 tappable CSS swatches with labels), lab quick-pick buttons, skin lighting hint, and notes multi-photo support. However, there is one critical field naming mismatch between the frontend and backend API that will cause data loss.

### [CRITICAL] StoolForm sends `color_ref` but backend API expects `color_label`
- **File:** `frontend/src/lib/components/StoolForm.svelte`
- **Line(s):** L7, L33, L44, L67
- **Perspective:** QA
- **Finding:** The StoolForm payload uses the field name `color_ref` for the stool color label, but the backend API handler (`backend/internal/handler/stools.go` L18) expects `color_label` in the JSON body. The backend model type (`backend/internal/model/types.go` L86) also uses `ColorLabel` with JSON tag `color_label`. The database column is `color_label`.
- **Reasoning:** This is a complete mismatch between frontend and backend. When the StoolForm submits `{ color_ref: "green" }`, the backend will deserialize it and find no `color_label` field, resulting in a null/empty `color_label` being stored. The stool color label will be silently lost on every submission.
- **Consequence:** Every stool entry logged through this form will have a null `color_label` in the database, despite the parent selecting a color swatch. The dashboard stool color trend chart and clinical reports rely on `color_label` for display, meaning parents and clinicians will see incomplete stool data. This is especially dangerous because stool color is the single most critical metric for post-Kasai monitoring (acholic stool detection).
- **Suggestion:** Rename `color_ref` to `color_label` throughout `StoolForm.svelte` and `stool-form.test.ts`:
```typescript
// StoolForm.svelte - payload interface
export interface StoolPayload {
    timestamp: string;
    color_rating: number;
    color_label: string;  // was: color_ref
    // ...
}

// StoolForm.svelte - COLOR_SWATCHES
{ rating: 1, label: 'White', color: '#F5F5DC', colorLabel: 'white' },
// ...

// StoolForm.svelte - state and payload
let colorLabel = $state('');
// In handleSubmit:
color_label: colorLabel
```

### [MEDIUM] NotesForm does not enforce the 4-photo limit specified in the spec
- **File:** `frontend/src/lib/components/NotesForm.svelte`
- **Line(s):** L83
- **Perspective:** Product Owner
- **Finding:** The spec (section 3.11) states "Photos | image[] | Up to 4 per entry" for general notes. The NotesForm enables `multiple={true}` on the PhotoUpload component but does not enforce or communicate a 4-photo maximum. The PhotoUpload component itself has no limit mechanism.
- **Reasoning:** Without enforcement, a parent could attach more than 4 photos to a note entry. If the backend enforces this limit, submissions will fail with an unclear error. If the backend does not enforce it, storage costs increase and the photo appendix in clinical reports becomes unwieldy.
- **Consequence:** Either silent backend rejection of entries with >4 photos (bad UX) or unbounded photo attachment (spec violation). Neither outcome is desirable.
- **Suggestion:** Add a photo count check in the NotesForm or PhotoUpload component, and display a message when the limit is reached. Also add a test verifying the limit behavior.

### [MEDIUM] PhotoUpload only passes the first file even when `multiple` is enabled
- **File:** `frontend/src/lib/components/PhotoUpload.svelte`
- **Line(s):** L17
- **Perspective:** QA
- **Finding:** When `multiple={true}`, the file input allows selecting multiple files, but the `handleChange` function only passes `input.files[0]` to the `onupload` callback. All subsequent files are silently discarded.
- **Reasoning:** The NotesForm relies on `multiple={true}` to support multi-photo uploads (spec requires up to 4 photos per note). With the current implementation, only the first selected file will ever be uploaded, making multi-photo selection non-functional.
- **Consequence:** Parents attempting to attach multiple photos to a general note will only get the first photo uploaded. The remaining photos are silently dropped with no error message, leading to incomplete clinical records.
- **Suggestion:** Either iterate over all files in `handleChange` and call `onupload` for each, or change the callback signature to accept a `FileList`/`File[]`:
```typescript
function handleChange(event: Event) {
    const input = event.target as HTMLInputElement;
    if (!input.files || input.files.length === 0) {
        return;
    }
    for (let i = 0; i < input.files.length; i++) {
        onupload(input.files[i]);
    }
}
```

### [LOW] Duplicated `defaultTimestamp()` utility function across all forms
- **File:** `frontend/src/lib/components/StoolForm.svelte`
- **Line(s):** L25-L29 (and identical copies in AbdomenForm, SkinForm, BruisingForm, LabForm, NotesForm)
- **Perspective:** Tech Lead
- **Finding:** The `defaultTimestamp()` function is identically copy-pasted across all six form components. Each implementation computes a local ISO timestamp string in the same way.
- **Reasoning:** This violates DRY principles and creates a maintenance burden. If the timestamp format or logic needs to change (e.g., timezone handling adjustment), it must be updated in six places.
- **Consequence:** Inconsistency risk if one form is updated but others are not. Increases maintenance effort for any timestamp-related change.
- **Suggestion:** Extract to a shared utility:
```typescript
// frontend/src/lib/utils/timestamp.ts
export function defaultTimestamp(): string {
    const now = new Date();
    const offset = now.getTimezoneOffset();
    const local = new Date(now.getTime() - offset * 60000);
    return local.toISOString().slice(0, 16);
}
```

### [LOW] NotesForm does not display attached photo count feedback
- **File:** `frontend/src/lib/components/NotesForm.svelte`
- **Line(s):** L83
- **Perspective:** Product Owner
- **Finding:** The NotesForm accepts `photoKeys` as a prop (array of strings) and includes them in the payload, but there is no visual feedback showing how many photos have been attached. Other forms show "Photo attached" via the `photoKey` prop on PhotoUpload, but NotesForm passes neither `photoKey` nor `photoKeys` to PhotoUpload.
- **Reasoning:** Parents have no way to confirm that their photos were successfully uploaded before submitting the note. This creates uncertainty about whether the upload worked.
- **Consequence:** Poor user experience -- parents cannot see attachment status and may re-upload or abandon the form unnecessarily.
- **Suggestion:** Add visual feedback showing the count of attached photos, and pass appropriate state to the PhotoUpload component or add a separate indicator.
