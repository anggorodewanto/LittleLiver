## Summary

This review examines production code across 3 phases for **code reuse issues only**. The changes introduce Auth/CSRF middleware (`middleware.go`), API handlers (`api.go`), router wiring (`router.go`), and supporting store/model/test code. Test code duplication is out of scope per instructions (consolidated in Phase 8).

---

### [MEDIUM] Duplicated "read cookie + lookup session" pattern across 3 production locations

- **File:** `backend/internal/handler/api.go` (L18-28), `backend/internal/middleware/middleware.go` (L43-53, L109-120)
- **Line(s):** L18-L28 in `api.go`; L43-L53 and L109-L120 in `middleware.go`
- **Perspective:** Tech Lead
- **Finding:** The pattern of reading a session cookie from the request and calling `store.GetSessionByID(db, cookie.Value)` is repeated in three production locations: (1) the Auth middleware, (2) the CSRF middleware fallback path, and (3) the `CSRFTokenHandler`. All three perform the identical two-step operation of extracting the cookie by name, then querying the session store.
- **Reasoning:** The CSRF middleware already documents its fallback as "if not found in context, fall back to reading the session cookie and querying the DB." The `CSRFTokenHandler` performs the exact same fallback logic but cannot benefit from it because it is not behind the Auth middleware. If a small helper like `SessionFromCookie(db, cookieName, r) (*model.Session, error)` existed in the `middleware` or `store` package, all three sites could use it, reducing the surface area for inconsistent error handling.
- **Consequence:** When session validation logic changes (e.g., adding IP-binding, rate limiting, or audit logging to session lookups), the change must be applied in three separate places. Missing one creates a security inconsistency.
- **Suggestion:** Extract a helper function:
  ```go
  // In package store or middleware
  func SessionFromRequest(db *sql.DB, cookieName string, r *http.Request) (*model.Session, error) {
      cookie, err := r.Cookie(cookieName)
      if err != nil {
          return nil, err
      }
      return store.GetSessionByID(db, cookie.Value)
  }
  ```
  Then use it in Auth middleware, CSRF middleware fallback, and `CSRFTokenHandler`.

### [LOW] Repeated JSON response boilerplate could use a shared helper

- **File:** `backend/internal/handler/api.go` (L32-35, L96-99), `backend/internal/handler/health.go` (L11-14)
- **Line(s):** L32-L35 and L96-L99 in `api.go`; L11-L14 in `health.go`
- **Perspective:** Tech Lead
- **Finding:** Three handler functions in the `handler` package repeat the same three-line pattern: set `Content-Type: application/json`, encode to JSON, log on error. As more API endpoints are added (the spec implies many more), this boilerplate will multiply.
- **Reasoning:** This is a minor concern today with only 3 occurrences, but the project roadmap includes multiple additional API endpoints (babies CRUD, feeding logs, stool logs, lab results, weight entries). Each will likely repeat this pattern.
- **Consequence:** Inconsistent error handling across handlers (e.g., one handler might forget the Content-Type header or silently swallow encode errors). Minor risk at current scale, increases with growth.
- **Suggestion:** Consider a small helper in the `handler` package:
  ```go
  func writeJSON(w http.ResponseWriter, status int, v interface{}) {
      w.Header().Set("Content-Type", "application/json")
      w.WriteHeader(status)
      if err := json.NewEncoder(w).Encode(v); err != nil {
          log.Printf("writeJSON: %v", err)
      }
  }
  ```
