## Summary

This review covers phases 19-21 of the LittleLiver project, which add four new CRUD resource types: bruising observations, skin observations, lab results, and general notes. Each follows the same "metric pattern" established in earlier phases: handler with validate/response mapping, store with INSERT-then-SELECT and paginated list, and a multi-parent integration test. The review focuses exclusively on efficiency concerns.

### [MEDIUM] INSERT-then-SELECT pattern doubles write-path round trips to SQLite

- **File:** `backend/internal/store/bruising.go`, `skin_observations.go`, `lab_results.go`, `general_notes.go`
- **Line(s):** bruising.go L46-58, skin_observations.go L46-58, lab_results.go L44-56, general_notes.go L43-55
- **Perspective:** Performance Engineer
- **Finding:** Every `Create*` function performs an INSERT followed by a full SELECT (via `Get*ByID`) to return the created object. The same pattern applies to all `Update*` functions. Since the application generates the ID client-side (ULID), knows all input values, and SQLite sets `created_at`/`updated_at` to `CURRENT_TIMESTAMP`, the only server-generated values are the two timestamps.
- **Reasoning:** This doubles the number of database round trips on every create and update operation. For SQLite specifically, since `CURRENT_TIMESTAMP` is deterministic within a transaction, the created object could be constructed in-memory by reading `time.Now().UTC()` rather than re-querying. However, this is an established pattern across the entire codebase (feedings, stools, urine, weights, temperatures, abdomen all do the same), so changing it only in these new files would create inconsistency.
- **Consequence:** Each create/update operation takes roughly 2x the time it needs to. For a personal-use app with SQLite (sub-millisecond queries), this is unlikely to cause user-visible latency, but it is wasted work on every write.
- **Suggestion:** This is a codebase-wide pattern and not specific to phases 19-21. If addressed, it should be refactored uniformly across all store files. One approach: use SQLite's `RETURNING` clause (supported since SQLite 3.35.0) to get `created_at`/`updated_at` in a single round trip, or construct the model in-memory after INSERT. Low priority given the app's scale.

### [MEDIUM] List queries ORDER BY id DESC but indexes are on (baby_id, timestamp)

- **File:** `backend/internal/store/bruising.go`, `skin_observations.go`, `lab_results.go`, `general_notes.go`
- **Line(s):** bruising.go L108-112, skin_observations.go L108-112, lab_results.go L106-110, general_notes.go L105-109
- **Perspective:** Performance Engineer
- **Finding:** All list queries use `ORDER BY id DESC` with cursor pagination based on ULID comparison (`id < ?`), but the only indexes are `(baby_id, timestamp)`. The `id` column is a TEXT PRIMARY KEY which creates an implicit index on `id` alone, but a query filtering on `baby_id` and ordering by `id` cannot use either index efficiently -- it must filter rows by `baby_id` (using the timestamp index or table scan) and then sort by `id`.
- **Reasoning:** Since ULIDs are time-ordered, `ORDER BY id DESC` is effectively equivalent to `ORDER BY timestamp DESC, id DESC` for records created in sequence. However, the database engine does not know this. Without a `(baby_id, id)` composite index, SQLite must perform a filesort on the filtered results. This is again a codebase-wide pattern (all 10+ metric tables have the same mismatch).
- **Consequence:** For small datasets (personal-use app, likely hundreds to low thousands of records per baby), the impact is negligible. If data grows, list queries will degrade as SQLite cannot avoid a sort step. The `timestamp` filter conditions can use the `(baby_id, timestamp)` index for range filtering, but the final sort still requires a temporary sort.
- **Suggestion:** Add a composite index `(baby_id, id DESC)` to each table if query performance ever becomes a concern. Alternatively, switch pagination to use `timestamp` as the cursor (matching the existing index) instead of `id`. This is a cross-cutting concern, not specific to these four files.

### [LOW] List query builds dynamic SQL via fmt.Sprintf with string concatenation

- **File:** `backend/internal/store/bruising.go`, `skin_observations.go`, `lab_results.go`, `general_notes.go`
- **Line(s):** bruising.go L108-112, skin_observations.go L108-112, lab_results.go L106-110, general_notes.go L105-109
- **Perspective:** Performance Engineer
- **Finding:** Each list call builds the SQL query string dynamically using `fmt.Sprintf` and `strings.Join`. While the values are parameterized (safe from injection), the query string itself changes shape depending on which filters are applied (`from`, `to`, `cursor`). This means SQLite's prepared statement cache cannot reuse query plans across calls with different filter combinations.
- **Reasoning:** SQLite compiles each unique SQL string into a bytecode program. With 3 optional filters, there are up to 8 distinct query strings per table. For a personal-use app this is not impactful, but it prevents statement caching.
- **Consequence:** Minor overhead from repeated query compilation. Negligible at this scale.
- **Suggestion:** No action needed at current scale. If optimization is desired later, pre-build the 8 query variants at initialization time and select the appropriate one at runtime.

### [LOW] Scan functions allocate on heap via pointer return for list iteration

- **File:** `backend/internal/store/bruising.go`, `skin_observations.go`, `lab_results.go`, `general_notes.go`
- **Line(s):** bruising.go L16-43 (scanBruising), bruising.go L122-128 (list loop)
- **Perspective:** Performance Engineer
- **Finding:** Each `scan*` function returns a `*model.XxxObservation` (pointer), but the list loop immediately dereferences it with `*b` to append a value copy into the slice. This means each row scanned during a list operation allocates a model struct on the heap (via the pointer return), only to be immediately copied and the pointer discarded.
- **Reasoning:** The scan function returns a pointer because it is also used for single-row queries (`GetByID`) where returning a pointer is idiomatic Go. The list loop then copies the value. This results in one unnecessary heap allocation per row during list operations.
- **Consequence:** For typical page sizes (20-50 items), this creates 20-50 small, short-lived heap allocations per list request. The GC overhead is trivial at this scale.
- **Suggestion:** No action needed. This is a reasonable trade-off for code reuse between single-row and multi-row scan paths. If micro-optimization is ever desired, a separate `scanBruisingValue` function returning by value could be added for list paths.

### [LOW] requireBabyAccess performs two sequential DB queries on every request

- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L100-130 (called from all new handlers)
- **Perspective:** Performance Engineer
- **Finding:** Every handler in the new files calls `requireBabyAccess`, which performs `GetBabyByID` (one query) then `IsParentOfBaby` (a second query). The baby object returned from `GetBabyByID` is not used in most handlers (e.g., Create handlers only use `baby.ID`, which is the same value they already have from the URL path). The two queries could be combined into a single query that checks both existence and parent linkage.
- **Reasoning:** This is a pre-existing pattern used by all metric handlers across the codebase. Each CRUD request makes 2 auth-check queries + 1 business query (or 2 for create/update with the re-select). A single joined query could verify baby existence and parent access simultaneously.
- **Consequence:** Every API request performs one more DB round trip than necessary for authorization. At SQLite speeds on a personal-use app, this adds microseconds.
- **Suggestion:** Combine into a single query: `SELECT b.* FROM babies b JOIN baby_parents bp ON b.id = bp.baby_id WHERE b.id = ? AND bp.user_id = ?`. This is a cross-cutting improvement, not specific to phases 19-21.
