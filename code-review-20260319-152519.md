## Summary

This review focuses on **code reuse opportunities** across the 10 new form components (FeedingForm, UrineForm, TemperatureForm, WeightForm, StoolForm, AbdomenForm, SkinForm, BruisingForm, LabForm, NotesForm), the PhotoUpload component, the QuickLogButtons component, and the TodayDashboard component introduced in phases 43-45. The dominant finding is extensive copy-paste duplication of the same utility function, prop interface fragments, template blocks, and payload-building patterns across all form files.

---

### [HIGH] `defaultTimestamp()` duplicated verbatim in 10 form components
- **File:** `frontend/src/lib/components/FeedingForm.svelte`, `UrineForm.svelte`, `TemperatureForm.svelte`, `WeightForm.svelte`, `StoolForm.svelte`, `AbdomenForm.svelte`, `SkinForm.svelte`, `BruisingForm.svelte`, `LabForm.svelte`, `NotesForm.svelte`
- **Line(s):** FeedingForm L19-23, UrineForm L16-20, TemperatureForm L17-21, WeightForm L17-21, StoolForm L25-29, AbdomenForm L24-28, SkinForm L25-29, BruisingForm L25-29, LabForm L19-23, NotesForm L22-26
- **Perspective:** Tech Lead
- **Finding:** The exact same 5-line `defaultTimestamp()` function is copy-pasted into all 10 form components. The function computes a local-timezone ISO datetime string for `datetime-local` inputs.
- **Reasoning:** This is the single most duplicated piece of logic in the diff. Any bug fix or change (e.g., rounding to the nearest minute, timezone handling adjustment) must be applied in 10 separate files. This violates DRY and increases maintenance burden significantly.
- **Consequence:** A bug in timestamp formatting (or a timezone edge case fix) would need to be patched in 10 files independently, with high risk of inconsistency and missed fixes.
- **Suggestion:** Extract to a shared utility in `frontend/src/lib/utils.ts` (or `frontend/src/lib/datetime.ts`):
  ```typescript
  // frontend/src/lib/datetime.ts
  export function defaultTimestamp(): string {
    const now = new Date();
    const offset = now.getTimezoneOffset();
    const local = new Date(now.getTime() - offset * 60000);
    return local.toISOString().slice(0, 16);
  }
  ```
  Then import it in each form: `import { defaultTimestamp } from '$lib/datetime';`

### [MEDIUM] Duplicated error/validation template block across all forms
- **File:** All 10 form components plus `CreateBabyForm.svelte` and `JoinBabyForm.svelte`
- **Line(s):** e.g., FeedingForm L103-110, UrineForm L66-69, WeightForm L80-87, AbdomenForm L102-109, BruisingForm L115-122, etc.
- **Perspective:** Tech Lead
- **Finding:** Every form repeats the same validation error and API error display template:
  ```svelte
  {#if validationError}
    <p role="alert">{validationError}</p>
  {/if}
  {#if error}
    <p role="alert">{error}</p>
  {/if}
  ```
  This identical 6-line block appears in 12 components.
- **Reasoning:** If the styling, accessibility attributes, or error display logic changes (e.g., adding an error icon, adding ARIA live regions, changing from `<p>` to a styled `<div>`), 12 files must be updated.
- **Consequence:** UI inconsistency if some forms are updated and others are missed. Increased effort for any cross-cutting error display change.
- **Suggestion:** Extract a `FormErrors.svelte` component:
  ```svelte
  <!-- FormErrors.svelte -->
  <script lang="ts">
    interface Props { validationError?: string; error?: string; }
    let { validationError = '', error = '' }: Props = $props();
  </script>
  {#if validationError}<p role="alert">{validationError}</p>{/if}
  {#if error}<p role="alert">{error}</p>{/if}
  ```

### [MEDIUM] Duplicated submit button pattern across all forms
- **File:** All 10 form components
- **Line(s):** e.g., FeedingForm L111-113, UrineForm L70-72, AbdomenForm L110-112
- **Perspective:** Tech Lead
- **Finding:** Every form ends with the same button pattern:
  ```svelte
  <button type="submit" disabled={submitting}>
    {submitting ? 'Logging...' : 'Log <MetricName>'}
  </button>
  ```
  The only variation is the metric name string.
- **Reasoning:** This is a minor but consistent duplication. The `disabled` behavior, button type, and loading text pattern are identical.
- **Consequence:** If the submit UX changes (e.g., adding a spinner, changing disabled styling logic), 10 files need updating.
- **Suggestion:** Consider a `SubmitButton.svelte` component that accepts `submitting` and `label` props, or combine it with the `FormErrors` component into a `FormFooter.svelte` component.

### [MEDIUM] Duplicated `Props` interface with `submitting?` and `error?` in every form
- **File:** All 10 form components
- **Line(s):** e.g., FeedingForm L13-14, UrineForm L10-11, WeightForm L11-12, AbdomenForm L16-17
- **Perspective:** Tech Lead
- **Finding:** Every form Props interface includes `submitting?: boolean; error?: string;` as common fields. Photo-capable forms additionally share `onphotoupload`, `uploading?`, and `photoKey?`.
- **Reasoning:** These common props represent the shared "form submission lifecycle" contract. There is no shared type for this.
- **Consequence:** If a new common prop is needed (e.g., `disabled?: boolean` for the whole form), it must be added to each interface separately.
- **Suggestion:** Define shared base prop types in a common file:
  ```typescript
  // frontend/src/lib/form-types.ts
  export interface BaseFormProps {
    submitting?: boolean;
    error?: string;
  }
  export interface PhotoFormProps extends BaseFormProps {
    onphotoupload: (file: File) => void;
    uploading?: boolean;
    photoKey?: string;
  }
  ```
  Then each form's Props interface extends the appropriate base.

### [MEDIUM] Duplicated `photo_keys` payload building logic in 4 photo-capable forms
- **File:** `AbdomenForm.svelte`, `BruisingForm.svelte`, `SkinForm.svelte`, `StoolForm.svelte`
- **Line(s):** AbdomenForm L57, BruisingForm L67, SkinForm L57, StoolForm L77
- **Perspective:** Tech Lead
- **Finding:** Four forms repeat the identical pattern `if (photoKey) { payload.photo_keys = [photoKey]; }` to wrap a single photo key into an array for the payload.
- **Reasoning:** This logic couples form components to the API contract in an identical way across files.
- **Consequence:** If the photo key handling changes (e.g., supporting multiple keys, or changing the field name), 4 files need identical changes.
- **Suggestion:** Extract a helper: `function buildPhotoKeys(photoKey?: string): string[] | undefined { return photoKey ? [photoKey] : undefined; }` in a shared utility file, or handle this in a shared payload builder.

### [MEDIUM] Duplicated `notes.trim()` optional field pattern across 9 forms
- **File:** `FeedingForm.svelte`, `UrineForm.svelte`, `TemperatureForm.svelte`, `WeightForm.svelte`, `StoolForm.svelte`, `AbdomenForm.svelte`, `SkinForm.svelte`, `BruisingForm.svelte`, `LabForm.svelte`
- **Line(s):** e.g., FeedingForm L45-47, UrineForm L35-37, AbdomenForm L60-62
- **Perspective:** Tech Lead
- **Finding:** Nine forms repeat the same conditional optional field pattern: `if (notes.trim()) { payload.notes = notes.trim(); }`. Several forms also apply this same pattern to other string fields (e.g., `color.trim()`, `rashes.trim()`).
- **Reasoning:** This "set-if-truthy-trimmed" pattern is a common payload builder concern.
- **Consequence:** Minor inconsistency risk. If trimming behavior changes (e.g., null instead of omitting), many files need updating.
- **Suggestion:** Extract a utility function:
  ```typescript
  export function setIfPresent<T extends Record<string, unknown>>(
    obj: T, key: string, value: string
  ): void {
    const trimmed = value.trim();
    if (trimmed) { (obj as Record<string, unknown>)[key] = trimmed; }
  }
  ```

### [MEDIUM] Stool color mapping logic duplicated between StoolForm and TodayDashboard
- **File:** `frontend/src/lib/components/StoolForm.svelte`, `frontend/src/lib/components/TodayDashboard.svelte`
- **Line(s):** StoolForm L32-40 (`COLOR_SWATCHES`), TodayDashboard L118-123 (`stoolDotColor`)
- **Perspective:** Tech Lead
- **Finding:** Stool color rating interpretation exists in two places: `StoolForm` defines `COLOR_SWATCHES` with rating-to-color mappings, and `TodayDashboard` defines `stoolDotColor()` with rating-to-status-color mappings. Both components encode domain knowledge about the 1-7 color rating scale independently.
- **Reasoning:** The stool color rating scale is a core domain concept for this app (post-Kasai monitoring). Having it defined in multiple places means a change to the scale (e.g., adding a rating 8 or adjusting boundaries) requires updates in multiple components.
- **Consequence:** Rating scale drift between form entry and dashboard display. A new color added to the form would not appear in the dashboard trend without a separate change.
- **Suggestion:** Create a shared `stool-colors.ts` module in `$lib/`:
  ```typescript
  export const STOOL_COLOR_SWATCHES = [ ... ] as const;
  export function stoolStatusColor(rating: number): string { ... }
  ```

### [LOW] Duplicated test setup boilerplate across all form test files
- **File:** All 12 test files in `frontend/src/tests/`
- **Line(s):** e.g., `abdomen-form.test.ts` L6-10, `bruising-form.test.ts` L6-10, `feeding-form.test.ts` L5-9
- **Perspective:** Tech Lead
- **Finding:** Every test file repeats the same `beforeEach` pattern:
  ```typescript
  let onsubmit: ReturnType<typeof vi.fn>;
  beforeEach(() => { onsubmit = vi.fn(); });
  ```
  Photo-capable form tests additionally duplicate `let onphotoupload: ReturnType<typeof vi.fn>;` with the same setup.
- **Reasoning:** While test files tend to tolerate more duplication than production code, a shared test factory or helper could reduce noise.
- **Consequence:** Minor maintenance burden when changing test setup patterns.
- **Suggestion:** Consider a test helper file `frontend/src/tests/helpers.ts` that exports a factory function:
  ```typescript
  export function createFormMocks() {
    return { onsubmit: vi.fn(), onphotoupload: vi.fn() };
  }
  ```

### [LOW] Timestamp field template duplicated in every form
- **File:** All 10 form components
- **Line(s):** e.g., FeedingForm L67-69, UrineForm L45-47, AbdomenForm L68-70
- **Perspective:** Tech Lead
- **Finding:** Every form includes the identical timestamp field markup:
  ```svelte
  <div>
    <label for="XXX-timestamp">Timestamp</label>
    <input id="XXX-timestamp" type="datetime-local" bind:value={timestamp} />
  </div>
  ```
  Only the `id` prefix varies.
- **Reasoning:** This is the most universal field across all metric forms.
- **Consequence:** Minor inconsistency risk if timestamp field behavior changes (e.g., adding a "now" button or default value display).
- **Suggestion:** Could be extracted into a `TimestampField.svelte` component, though the benefit is marginal given the simplicity.
