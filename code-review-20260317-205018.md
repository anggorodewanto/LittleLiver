## Summary

This review covers phases 28-30 of LittleLiver, which add dashboard chart data series (7 query functions), an alert system (acholic stool, fever, jaundice, missed medication), and an integration test. The review is scoped exclusively to **efficiency** concerns, evaluated in the context of a personal-use SQLite app with 1-2 users and 1 baby.

**Overall verdict: The efficiency is acceptable for this use case.** SQLite operates in-process with no network round-trip per query, so ~12 sequential queries against small tables will complete in single-digit milliseconds total. The findings below are ranked by whether they are worth addressing.

---

### 1. [MEDIUM] N+1 Query Pattern in Missed Medication Alert Checker

- **File:** `backend/internal/store/alerts.go`
- **Line(s):** L284
- **Perspective:** Performance Engineer
- **Finding:** `IsDoseCovered` is called once per medication per schedule time per day (yesterday + today). With 5 medications x 2 daily doses x 2 days = up to 20 individual `SELECT COUNT(*)` queries, each scanning `med_logs`.
- **Reasoning:** This is a textbook N+1 pattern. Each call is a separate SQLite query with `datetime()` function calls on every row.
- **Consequence:** For this personal-use app with tiny data volumes, the total added latency is likely under 5ms. This is a non-issue at current scale. If medication count or schedule density grew significantly (unlikely for 1 baby), it could become noticeable.
- **Suggestion:** Could be replaced with a single query that fetches all med_log entries in the 24-hour window, then checks coverage in Go memory. However, **not worth fixing now** -- the current approach is clearer, and `IsDoseCovered` is explicitly designed for reuse by the Phase 34 scheduler.

**Worth fixing?** No. Acceptable for personal use.

---

### 2. [LOW] 12 Sequential DB Queries in Dashboard Handler

- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L95-L172
- **Perspective:** Performance Engineer
- **Finding:** The dashboard handler makes ~12 sequential database calls: `GetDashboardSummary` (which itself runs ~6 sub-queries), `GetStoolColorTrend`, `GetUpcomingMeds`, 7 chart data queries, and `GetActiveAlerts` (3-4+ sub-queries). Total is approximately 16-20+ round-trips.
- **Reasoning:** With a networked database (Postgres, MySQL), this would be a significant concern. With SQLite (in-process, no network overhead), each query on small tables takes microseconds. The sequential nature also simplifies error handling and keeps the code readable.
- **Consequence:** Total dashboard latency is likely 5-20ms even with all queries. No user-perceptible delay.
- **Suggestion:** If this ever becomes a bottleneck, the independent chart queries (lines 117-163) could be run concurrently using goroutines, since SQLite in WAL mode supports concurrent readers. Alternatively, some chart queries hitting the same tables could be combined (e.g., `GetStoolColorSeries` and `GetDiaperDaily` both touch `stools`). However, combining queries would sacrifice the clean separation of concerns.

**Worth fixing?** No. The code clarity benefit outweighs the negligible performance cost.

---

### 3. [LOW] GetDashboardSummary Runs 6 Separate Queries Against Overlapping Tables

- **File:** `backend/internal/store/dashboard.go`
- **Line(s):** L41-L130 (approximate, based on grep output)
- **Perspective:** Performance Engineer
- **Finding:** `GetDashboardSummary` runs individual queries for feedings count/calories, urine count, stools count, most recent stool color, last temperature, and last weight. The feedings and stools queries scan tables that are also queried by `GetFeedingDaily` and `GetDiaperDaily`/`GetStoolColorSeries` in the same dashboard request.
- **Reasoning:** The summary queries use `COUNT(*)` and `SUM()` while the chart queries return row-level data. They serve different purposes but scan overlapping data. With SQLite's page cache, the second scan of each table will be served from memory.
- **Consequence:** Negligible. SQLite's page cache means redundant table scans are nearly free after the first access.
- **Suggestion:** The summary counts could theoretically be derived from the chart data arrays in Go (e.g., sum `FeedingDailyEntry.FeedCount`), eliminating 4-5 queries. This would introduce coupling between chart data and summary logic, so it is a tradeoff.

**Worth fixing?** No. SQLite page caching makes this effectively free.

---

### 4. [LOW] Unbounded Result Sets in Chart Data Queries

- **File:** `backend/internal/store/chart_data.go`
- **Line(s):** L85-L359 (all 7 query functions)
- **Perspective:** Performance Engineer
- **Finding:** None of the chart data queries have a `LIMIT` clause. The result set size is bounded only by the `from`/`to` date range provided by the caller. The default range is "today" (1 day), but the API accepts arbitrary ranges.
- **Reasoning:** For a single baby tracked over months, even a year-wide query would return hundreds of rows at most (a few entries per day). This is well within acceptable memory bounds.
- **Consequence:** If someone passed `from=2020-01-01&to=2030-01-01`, the query would return all data, but for 1 baby this would still be trivially small. No real risk.
- **Suggestion:** Could add a maximum date range validation (e.g., max 365 days) in the handler if desired, but this is defensive coding rather than a real need.

**Worth fixing?** No. Data volume for 1 baby is inherently bounded.

---

### 5. [LOW] Repeated `parseDateRange` Calls Across 7 Chart Data Functions

- **File:** `backend/internal/store/chart_data.go`
- **Line(s):** L86, L132, L176, L213, L253, L290, L327
- **Perspective:** Performance Engineer
- **Finding:** `parseDateRange(from, to)` is called 7 times with the same `from` and `to` values, each time parsing the same date strings and computing the same time boundaries.
- **Reasoning:** `time.Parse` is extremely fast (nanoseconds). The duplication is computational noise.
- **Consequence:** Adds perhaps 1 microsecond total. Zero practical impact.
- **Suggestion:** Could parse once in the handler and pass pre-computed `fromTime`/`toTime` strings to the store functions. This would be a minor API cleanliness improvement but is not worth the refactoring effort for performance reasons.

**Worth fixing?** No. Purely cosmetic from a performance standpoint.

---

### 6. [LOW] `time.LoadLocation` Called Per Medication in Missed Medication Checker

- **File:** `backend/internal/store/alerts.go`
- **Line(s):** L248
- **Perspective:** Performance Engineer
- **Finding:** `time.LoadLocation` is called for each medication. If multiple medications share the same timezone (very likely for a single baby), this is redundant.
- **Reasoning:** Go caches timezone data after the first load, so subsequent calls for the same timezone string are fast. With 1-5 medications, this is irrelevant.
- **Consequence:** None. Go's internal caching handles this.
- **Suggestion:** No change needed.

**Worth fixing?** No.

---

## Final Assessment

**All findings are acceptable for the personal-use context.** The key insight is that SQLite operates in-process -- there is zero network latency per query. What would be a serious N+1 or sequential-query problem with PostgreSQL or MySQL over a network (adding 1-5ms per round-trip) is effectively free with SQLite where each simple query completes in microseconds.

The code is clean, well-separated by concern, and easy to maintain. Optimizing these queries would add complexity (goroutines, combined queries, shared state) that would hurt readability and maintainability with no user-visible benefit.

**The only finding worth bookmarking for the future** is finding #1 (N+1 in missed medication alerts). If the app ever migrated to a networked database or supported multiple babies, that would be the first thing to batch. The comment at line 40-41 noting reuse by the Phase 34 scheduler suggests this function's API is intentionally kept simple for that purpose, which is a reasonable design choice.
