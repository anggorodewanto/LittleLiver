## Summary

This review examines phases 25-27 (Medication CRUD, Med-log endpoints, and Dashboard essentials) for **code reuse opportunities**. The changes add three new handler files (`medications.go`, `med_logs.go`, `dashboard.go`), two new store files (`medications.go`, `med_logs.go`, `dashboard.go`), two migrations, and corresponding tests. The focus is on identifying duplicated logic that could leverage existing helpers and utilities.

### [MEDIUM] Inline entry ID extraction duplicates `requireEntryID` helper
- **File:** `backend/internal/handler/med_logs.go`
- **Line(s):** L212-L215, L241-L244, L281-L284
- **Perspective:** Tech Lead
- **Finding:** The `GetMedLogHandler`, `UpdateMedLogHandler`, and `DeleteMedLogHandler` functions all inline the pattern `entryID := r.PathValue("entryId"); if entryID == "" { http.Error(...) }` instead of calling the existing `requireEntryID(w, r)` helper defined in `backend/internal/handler/helpers.go` at line 29.
- **Reasoning:** Every other handler in the codebase (feedings, stools, urine, weights, temperatures, abdomen, skin observations, bruising, lab results, general notes) uses `requireEntryID`. This is an established convention with over 30 call sites.
- **Consequence:** Three instances of duplicated boilerplate that will diverge if the error message or status code needs to change, and inconsistency makes the codebase harder to navigate.
- **Suggestion:** Replace each inline block with the existing helper:
  ```go
  entryID, ok := requireEntryID(w, r)
  if !ok {
      return
  }
  ```

### [MEDIUM] Inline medication ID extraction should use a similar helper pattern
- **File:** `backend/internal/handler/medications.go`
- **Line(s):** L180-L183, L209-L212
- **Perspective:** Tech Lead
- **Finding:** `GetMedicationHandler` and `UpdateMedicationHandler` inline `medID := r.PathValue("medId"); if medID == "" { http.Error(w, "missing medication ID", ...) }`. While the path value key is `"medId"` rather than `"entryId"`, the pattern is identical to `requireEntryID` and could be generalized or a `requirePathValue(w, r, key, label)` helper introduced.
- **Reasoning:** The project already has `requireEntryID` as a precedent. A parameterized version would eliminate this repeated boilerplate pattern and make adding new path-value-based resources trivial.
- **Consequence:** Minor maintenance burden; two more inline instances of the same guard logic.
- **Suggestion:** Extract a generalized helper:
  ```go
  func requirePathValue(w http.ResponseWriter, r *http.Request, key, label string) (string, bool) {
      val := r.PathValue(key)
      if val == "" {
          http.Error(w, "missing "+label, http.StatusBadRequest)
          return "", false
      }
      return val, true
  }
  ```
  Then update `requireEntryID` to delegate to it, and use it for `medId` as well.

### [MEDIUM] Duplicated schedule JSON unmarshaling logic between `dashboard.go` and `medications.go`
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L87-L97
- **Perspective:** Tech Lead
- **Finding:** The dashboard handler manually unmarshals `m.Schedule` from JSON into `[]string` with a nil/empty guard and fallback to empty slice. The exact same logic exists in `toMedicationResponse` at `backend/internal/handler/medications.go` lines 57-63. Both perform: check nil, check empty, `json.Unmarshal`, log error, default to empty slice.
- **Reasoning:** This is a textbook case of duplicated transformation logic operating on the same data shape (`*string` containing a JSON array of time strings). Changes to the schedule format would need updating in two places.
- **Consequence:** If the schedule storage format changes (e.g., to a struct with time + label), both locations must be updated independently, risking inconsistency.
- **Suggestion:** Extract a shared helper in the handler package:
  ```go
  func parseScheduleTimes(schedule *string) []string {
      if schedule == nil || *schedule == "" {
          return []string{}
      }
      var times []string
      if err := json.Unmarshal([]byte(*schedule), &times); err != nil {
          log.Printf("unmarshal schedule: %v", err)
          return []string{}
      }
      return times
  }
  ```
  Then use it in both `toMedicationResponse` and `DashboardHandler`.

### [MEDIUM] `ListMedLogs` re-implements pagination logic already in `listMetricWithTZ`
- **File:** `backend/internal/store/med_logs.go`
- **Line(s):** L104-L177
- **Perspective:** Tech Lead
- **Finding:** The `ListMedLogs` function manually builds WHERE conditions with `conditions []string` / `args []any`, appends `baby_id`, `from`/`to` date filters, cursor, and limit+1, then iterates rows, builds a `MetricPage`, and handles the nil-to-empty-slice conversion. This is nearly identical to the generic `listMetricWithTZ` helper in `backend/internal/store/metric_helpers.go` (lines 64-141), which already does all of this.
- **Reasoning:** `listMetricWithTZ` was specifically created to avoid this duplication. All other metric types (feedings, stools, urine, weights, temperatures, abdomen, skin, bruising, lab results, general notes) use it. Med-logs differ only in having an extra `medication_id` filter and filtering by `created_at` instead of `timestamp`, but these could be accommodated with minor generalization (e.g., an additional conditions parameter, or the existing flexibility of the `table` and `columns` parameters).
- **Consequence:** ~70 lines of duplicated pagination boilerplate that must be maintained separately from the generic helper. Any bug fix to pagination logic (e.g., off-by-one in cursor) would need to be applied in two places.
- **Suggestion:** Either (a) extend `listMetricWithTZ` to accept optional extra conditions/args, or (b) create a slightly more generic variant. The medication_id filter and created_at column difference are the only divergences. For example:
  ```go
  // Add an extraConditions parameter to listMetricWithTZ
  func listMetricWithTZ[T any](
      db *sql.DB, table, columns, babyID string,
      from, to, cursor *string, limit int, loc *time.Location,
      scan func(scanner) (*T, error),
      getID func(*T) string,
      extraConditions []string, extraArgs []any,
  ) (*model.MetricPage[T], error) {
  ```

### [MEDIUM] Duplicated X-Timezone header extraction in medications handler
- **File:** `backend/internal/handler/medications.go`
- **Line(s):** L122-L125, L228-L231
- **Perspective:** Tech Lead
- **Finding:** Both `CreateMedicationHandler` and `UpdateMedicationHandler` contain identical inline blocks to extract the timezone from the `X-Timezone` header: `var tz *string; if tzHeader := r.Header.Get("X-Timezone"); tzHeader != "" { tz = &tzHeader }`. The existing `parseListParams` in `helpers.go` (line 69) already reads the same header to create a `*time.Location`, but for medications the raw string is needed for storage.
- **Reasoning:** This pattern appears twice in the same file and is a good candidate for a small helper, especially since the project values "no duplication."
- **Consequence:** Minor; two copies of the same 4-line block. If the header name changes, both need updating.
- **Suggestion:** Add a helper to `helpers.go`:
  ```go
  func optionalTimezone(r *http.Request) *string {
      if tz := r.Header.Get("X-Timezone"); tz != "" {
          return &tz
      }
      return nil
  }
  ```

### [LOW] `err == sql.ErrNoRows` instead of `errors.Is` in `CreateMedLogHandler`
- **File:** `backend/internal/handler/med_logs.go`
- **Line(s):** L148
- **Perspective:** QA
- **Finding:** The handler uses direct equality `if err == sql.ErrNoRows` instead of `errors.Is(err, sql.ErrNoRows)`, which is the pattern used everywhere else in the handler package (e.g., `helpers.go:84`, `babies.go:109`). Moreover, the existing `handleStoreError` helper already does this check correctly.
- **Reasoning:** If `GetMedicationBabyID` ever wraps the error with `fmt.Errorf("...: %w", err)`, the direct equality check will fail to match, causing a 500 instead of the intended 400.
- **Consequence:** Potential silent regression if error wrapping is added to the store layer, leading to incorrect HTTP status codes.
- **Suggestion:** Use `errors.Is`:
  ```go
  if errors.Is(err, sql.ErrNoRows) {
      http.Error(w, "medication not found", http.StatusBadRequest)
      return
  }
  ```

### [LOW] Dashboard `from/to` date range construction duplicates `GetDashboardSummary` pattern
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L45-L53
- **Perspective:** Tech Lead
- **Finding:** The dashboard handler defaults `from` and `to` to today's date using `time.Now().UTC().Format("2006-01-02")`. This is a minor instance, but the `model.DateFormat` constant (`"2006-01-02"`) already exists and should be used instead of the inline format string for consistency.
- **Reasoning:** All other date formatting in the codebase references `model.DateFormat`. Using the raw string creates a maintenance risk if the date format constant were ever changed.
- **Consequence:** Trivial inconsistency; using the constant is more self-documenting.
- **Suggestion:** Replace `"2006-01-02"` with `model.DateFormat`:
  ```go
  today := time.Now().UTC().Format(model.DateFormat)
  ```
