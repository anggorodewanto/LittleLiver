## Summary

Phases 43-45 add frontend metric entry forms (FeedingForm, UrineForm, TemperatureForm, WeightForm, StoolForm, LabForm, AbdomenForm, BruisingForm, SkinForm, NotesForm), a PhotoUpload component, QuickLogButtons, and a TodayDashboard view that consumes the dashboard API. The review focuses exclusively on efficiency concerns: unnecessary work, hot-path bloat, memory issues, no-op updates, and missed concurrency.

### [MEDIUM] Unguarded $effect triggers fetchDashboard on every reactive dependency change, not just babyId
- **File:** `frontend/src/lib/components/TodayDashboard.svelte`
- **Line(s):** L171-L174
- **Perspective:** Performance Engineer
- **Finding:** The `$effect` block calls `fetchDashboard()` which reads and writes several reactive state variables (`loading`, `error`, `dashboard`, `dismissedAlertIds`). In Svelte 5, `$effect` automatically tracks all reactive reads inside its callback. Because `fetchDashboard` is defined in component scope and touches these reactive variables, the effect may re-subscribe to more dependencies than intended. If the Svelte compiler inlines the function body, mutations to `loading`, `error`, or `dashboard` could retrigger the effect, causing an infinite fetch loop or at minimum a double-fetch on mount.
- **Reasoning:** The intent is to re-fetch only when `babyId` changes. The current pattern relies on the compiler treating `void fetchDashboard()` as opaque, but this is fragile. An explicit dependency on `babyId` with an untrack guard is the idiomatic Svelte 5 pattern.
- **Consequence:** If the effect tracks more than `babyId`, every state mutation inside `fetchDashboard` will re-trigger the fetch, creating an infinite loop of API calls that hammers the server and freezes the UI.
- **Suggestion:** Explicitly read `babyId` in the effect and wrap the fetch call in `untrack`:
```ts
import { untrack } from 'svelte';

$effect(() => {
    // Explicitly depend on babyId only
    const id = babyId;
    untrack(() => {
        void fetchDashboard();
    });
});
```

### [MEDIUM] localStorage read + JSON.parse + write on every dashboard fetch even when dismissed set is empty
- **File:** `frontend/src/lib/components/TodayDashboard.svelte`
- **Line(s):** L149-L169 (inside fetchDashboard)
- **Perspective:** Performance Engineer
- **Finding:** Every call to `fetchDashboard` unconditionally calls `loadDismissedAlerts()` (reads localStorage, JSON.parses), then `cleanupDismissedAlerts()` (allocates two new Sets), then iterates and `add`s to `dismissedAlertIds` one-by-one, then calls `saveDismissedAlerts()` (JSON.stringifies, writes localStorage). This happens even when there are zero dismissed alerts and zero active alerts, performing 4 allocations and 2 localStorage round-trips for nothing.
- **Reasoning:** localStorage I/O is synchronous and blocks the main thread. On a dashboard that could be refreshed frequently (pull-to-refresh pattern), this is unnecessary work.
- **Consequence:** Each unnecessary localStorage write takes ~1-5ms on mobile devices. Combined with the Set allocations, this adds latency to the critical rendering path after the API response arrives.
- **Suggestion:** Short-circuit when there is nothing to clean up:
```ts
const dismissed = loadDismissedAlerts();
if (dismissed.size === 0) {
    // Nothing to reconcile
    return;
}
const cleaned = cleanupDismissedAlerts(data.active_alerts, dismissed);
// Only write back if the set actually changed
if (cleaned.size !== dismissed.size) {
    saveDismissedAlerts(cleaned);
}
dismissedAlertIds = cleaned;
```

### [MEDIUM] Medication countdown text is static after render -- never updates
- **File:** `frontend/src/lib/components/TodayDashboard.svelte`
- **Line(s):** L95-L116, L270-L275
- **Perspective:** Product Owner
- **Finding:** `formatCountdown` computes the time difference using `Date.now()` at the moment of render. Since there is no interval or reactive timer, the countdown text ("in 2 h 15 min", "overdue by 5 min") becomes stale immediately and never updates. A user staring at the dashboard for 10 minutes will see the same countdown value.
- **Reasoning:** For a medication tracking app where overdue alerts are clinically significant, showing stale countdown values is a functional deficiency that also wastes the computation -- it gives the user a false sense of accuracy.
- **Consequence:** A parent may believe they still have time before a dose is due when in fact it is already overdue, potentially missing a critical medication window.
- **Suggestion:** Add a reactive timer that ticks every 60 seconds to force countdown recalculation:
```ts
let now = $state(Date.now());
$effect(() => {
    const interval = setInterval(() => { now = Date.now(); }, 60_000);
    return () => clearInterval(interval);
});
```
Then use `now` inside `formatCountdown` instead of `Date.now()` so the template re-renders.

### [LOW] defaultTimestamp() is duplicated verbatim across 9 form components
- **File:** `frontend/src/lib/components/FeedingForm.svelte` (and 8 others)
- **Line(s):** L19-L23 in FeedingForm (identical in AbdomenForm, BruisingForm, LabForm, NotesForm, SkinForm, StoolForm, TemperatureForm, UrineForm, WeightForm)
- **Perspective:** Tech Lead
- **Finding:** The exact same `defaultTimestamp()` function is copy-pasted into every form component. This is not strictly an efficiency issue at runtime but creates maintenance overhead and increases bundle size (the function is not tree-shaken since it appears in 9 separate modules).
- **Reasoning:** A shared utility function would reduce bundle size by ~0.5-1KB after minification and eliminate the risk of forms drifting out of sync if the timestamp format ever needs to change.
- **Consequence:** If the timestamp format needs adjustment (e.g., to include seconds, or to handle timezone edge cases), all 9 files must be updated independently, risking inconsistency.
- **Suggestion:** Extract to a shared utility:
```ts
// frontend/src/lib/utils/timestamp.ts
export function defaultTimestamp(): string {
    const now = new Date();
    const offset = now.getTimezoneOffset();
    const local = new Date(now.getTime() - offset * 60000);
    return local.toISOString().slice(0, 16);
}
```
Import in each form.

### [LOW] PhotoUpload input has hardcoded id="photo-upload" causing duplicate IDs when multiple instances on same page
- **File:** `frontend/src/lib/components/PhotoUpload.svelte`
- **Line(s):** L29-L37
- **Perspective:** QA
- **Finding:** The `<input id="photo-upload">` and `<label for="photo-upload">` use a hardcoded ID. When multiple forms with PhotoUpload are rendered on the same page (e.g., NotesForm with multi-photo), or if two photo-capable forms appear simultaneously, duplicate IDs break label-input association and accessibility.
- **Reasoning:** While currently forms are likely shown one at a time, this is a latent bug that will surface when the app evolves to show multiple forms or when testing renders multiple instances.
- **Consequence:** Clicking the "Photo" label could target the wrong file input, causing uploads to go to the wrong form. Screen readers will also report invalid markup.
- **Suggestion:** Accept an optional `id` prop or generate a unique ID per instance:
```ts
let inputId = $props().id ?? `photo-upload-${crypto.randomUUID().slice(0, 8)}`;
```

### [LOW] handleQuickLog uses full page navigation instead of client-side routing
- **File:** `frontend/src/lib/components/TodayDashboard.svelte`
- **Line(s):** L144-L147
- **Perspective:** Performance Engineer
- **Finding:** `handleQuickLog` sets `window.location.href = /log/${type}`, which triggers a full page reload. In a SvelteKit app, this discards the entire component tree, re-downloads JavaScript, and re-initializes application state. This is the most expensive navigation pattern available.
- **Reasoning:** SvelteKit provides `goto()` from `$app/navigation` for client-side navigation that preserves app state and is near-instant.
- **Consequence:** Every quick-log button tap incurs 500ms-2s of navigation overhead on mobile instead of the ~50ms a client-side route transition would take. Dashboard state is lost, requiring a fresh API call on return.
- **Suggestion:** Replace with SvelteKit's `goto`:
```ts
import { goto } from '$app/navigation';

function handleQuickLog(type: string): void {
    goto(`/log/${type}`);
}
```
