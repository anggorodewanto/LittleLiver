## Summary

This diff adds CSRF protection middleware, Auth middleware, API handlers (`/api/csrf-token`, `/api/me`), a `testutil` package for shared test helpers, new store functions (`GetUserByID`, `UpdateUserTimezone`, `GetBabiesByUserID`), and integration tests covering the full OAuth flow. It also consolidates duplicated test setup code across packages.

### 1. [CRITICAL] SessionSecret silently dropped by NewHandlers

- **File:** `backend/internal/auth/auth.go`
- **Line(s):** L60-L68
- **Perspective:** SecOps
- **Finding:** `NewHandlers` constructs a new `Config` struct but omits `SessionSecret`. When `WithAuthConfig` passes a `Config` with `SessionSecret` set, `router.go` L30 reads `cfg.authConfig.SessionSecret` (which works because it reads from the original config pointer), but the `Handlers.Config` copy has `SessionSecret` as empty string. This is fragile and will silently break if anyone accesses `h.Config.SessionSecret` later.
- **Reasoning:** The `SessionSecret` field was added to `auth.Config` specifically so it can be threaded through to the router, but `NewHandlers` does not preserve it. This is a data-loss bug in the constructor that will bite the next developer who expects `Handlers.Config` to be a faithful copy.
- **Consequence:** Any future code that reads `h.Config.SessionSecret` from a `Handlers` instance will get an empty string, causing CSRF token derivation with an empty key -- effectively disabling CSRF protection.
- **Suggestion:** Add `SessionSecret` to the copied config in `NewHandlers`:
  ```go
  return &Handlers{
      DB: db,
      Config: Config{
          ClientID:      cfg.ClientID,
          ClientSecret:  cfg.ClientSecret,
          RedirectURL:   cfg.RedirectURL,
          TokenURL:      tokenURL,
          UserInfoURL:   userInfoURL,
          SessionSecret: cfg.SessionSecret,
      },
      // ...
  }
  ```

### 2. [MEDIUM] Copy-paste: session cookie lookup duplicated in CSRFTokenHandler and Auth middleware

- **File:** `backend/internal/handler/api.go`
- **Line(s):** L17-L29
- **Perspective:** Tech Lead
- **Finding:** `CSRFTokenHandler` repeats the exact cookie-read-then-`GetSessionByID` pattern from the `Auth` middleware (middleware.go L43-L53). The CSRF token endpoint could instead be placed behind the `Auth` middleware and read the session token from context via `middleware.SessionTokenFromContext`.
- **Reasoning:** The comment on the router (L60: "needs session but not auth middleware context") is misleading -- the Auth middleware does exactly what this handler needs (validate session, put token in context). Adding Auth middleware to this route would eliminate the duplicated session lookup and ensure consistent session validation (e.g., sliding window extension, timezone update).
- **Consequence:** If session validation logic changes in Auth middleware (e.g., adding IP binding), `CSRFTokenHandler` will silently diverge, creating an inconsistent security posture.
- **Suggestion:** Route `/api/csrf-token` through the `authMw` middleware and simplify the handler:
  ```go
  func CSRFTokenHandler(secret string) http.HandlerFunc {
      return func(w http.ResponseWriter, r *http.Request) {
          sessionToken := middleware.SessionTokenFromContext(r.Context())
          csrfToken := middleware.CSRFToken(sessionToken, secret)
          w.Header().Set("Content-Type", "application/json")
          json.NewEncoder(w).Encode(map[string]string{"csrf_token": csrfToken})
      }
  }
  ```

### 3. [MEDIUM] Stringly-typed: `"session_id"` hardcoded in test files instead of using `auth.CookieName`

- **File:** `backend/internal/handler/api_test.go`, `backend/internal/middleware/middleware_test.go`, `backend/internal/handler/router_test.go`
- **Line(s):** api_test.go L15, middleware_test.go L14, router_test.go L232/L254/L306
- **Perspective:** Tech Lead
- **Finding:** Multiple test files define `const testCookieName = "session_id"` or hardcode `"session_id"` directly, duplicating the value from `auth.CookieName`. The integration test file correctly uses `auth.CookieName`.
- **Reasoning:** If `auth.CookieName` changes, these tests will silently pass while testing the wrong cookie name, giving a false sense of security.
- **Consequence:** Tests would continue passing with a stale cookie name, masking a real misconfiguration bug.
- **Suggestion:** Replace all `const testCookieName = "session_id"` and hardcoded `"session_id"` in test files with `auth.CookieName`:
  ```go
  const testCookieName = auth.CookieName
  ```

### 4. [MEDIUM] Redundant/misplaced state: `SessionSecret` on `auth.Config` is a router concern, not an auth concern

- **File:** `backend/internal/auth/auth.go`
- **Line(s):** L34-L35
- **Perspective:** Tech Lead
- **Finding:** `SessionSecret` was added to `auth.Config` solely as a pass-through vehicle for the router (the comment itself says "passed through for router setup"). The `auth` package never uses this field -- it is only read in `router.go`.
- **Reasoning:** This couples the `auth` config to an unrelated concern (CSRF middleware configuration). The `SessionSecret` logically belongs on the router's `options` struct.
- **Consequence:** As more middleware configuration is needed, `auth.Config` accumulates unrelated fields, making the auth package's API confusing and its tests harder to maintain.
- **Suggestion:** Move `SessionSecret` to the router's `options` struct and add a `WithSessionSecret(string) Option` function:
  ```go
  type options struct {
      db            *sql.DB
      authConfig    *auth.Config
      sessionSecret string
  }

  func WithSessionSecret(s string) Option {
      return func(o *options) { o.sessionSecret = s }
  }
  ```

### 5. [LOW] Dead code: `csrfMw` assigned then blanked with `_ = csrfMw`

- **File:** `backend/internal/handler/router.go`
- **Line(s):** L57-L58
- **Perspective:** Tech Lead
- **Finding:** The CSRF middleware is instantiated and immediately discarded: `csrfMw := middleware.CSRF(...)` followed by `_ = csrfMw`. The comment says "used by future state-changing API routes."
- **Reasoning:** Dead code in production files adds noise and signals incomplete work. The middleware allocation at startup is trivial, but the pattern is misleading to readers.
- **Consequence:** Minor: reader confusion and lint warnings. The middleware could be added when it is actually needed.
- **Suggestion:** Remove the `csrfMw` lines entirely. Add it when the first POST/PUT/DELETE API route is introduced.

### 6. [LOW] Copy-paste with slight variation: test user/session setup repeated across test files

- **File:** `backend/internal/middleware/middleware_test.go`, `backend/internal/handler/api_test.go`
- **Line(s):** Multiple (e.g., middleware_test.go L70-L77, api_test.go L25-L31)
- **Perspective:** Tech Lead
- **Finding:** Nearly every test function contains the same 5-line block: `db.Exec("INSERT INTO users ...")` then `store.CreateSession(db, "u1")`. The `testutil` package already has `CreateTestUser` which does exactly this, yet some test files still use raw SQL inserts.
- **Reasoning:** The `testutil.CreateTestUser` helper exists precisely to avoid this repetition. Raw SQL inserts in tests are fragile if the schema changes (e.g., adding a NOT NULL column).
- **Consequence:** A schema change to the `users` table will require updating dozens of test functions instead of one helper.
- **Suggestion:** Replace raw SQL inserts with `testutil.CreateTestUser(t, db)` across all test files, and use `testutil.AuthenticatedRequest` where session creation is also needed.
