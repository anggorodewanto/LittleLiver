## Summary

This review covers phases 40-42 of the LittleLiver project: a PDF report generation integration test (Go), frontend API client enhancements (CSRF, timezone, 204 handling), authentication/user stores, baby management stores, and UI components for login, baby creation, baby joining, and baby selection. The code is generally clean and well-structured. The issues found are relatively minor.

### [MEDIUM] Type Inconsistency for `sex` Field Between FirstLogin and CreateBabyForm
- **File:** `frontend/src/lib/components/FirstLogin.svelte`
- **Line(s):** L6-L12
- **Perspective:** Tech Lead
- **Finding:** The `oncreate` callback prop in `FirstLogin.svelte` declares `sex` as `string` (line 9), while `CreateBabyForm.svelte` declares it as `'male' | 'female'` (line 6) and the `CreateBabyInput` interface in `baby.ts` also uses `'male' | 'female'`. The `FirstLogin` component wraps `CreateBabyForm` and passes through `oncreate`, but widens the type from a union literal to `string`.
- **Reasoning:** This defeats the purpose of the typed union. A caller consuming the `FirstLogin` component's `oncreate` prop would receive a `string` for `sex` instead of the narrower `'male' | 'female'`, requiring a cast or losing type safety at the call site.
- **Consequence:** Type errors may go undetected at the boundary between `FirstLogin` and its parent. A future refactor could introduce invalid sex values without a compile-time error.
- **Suggestion:** Align the type in `FirstLogin.svelte` to match:
  ```typescript
  oncreate: (data: {
      name: string;
      date_of_birth: string;
      sex: 'male' | 'female';
      diagnosis_date?: string;
      kasai_date?: string;
  }) => void;
  ```
  Or better, import and reuse `CreateBabyInput` from `$lib/stores/baby` to avoid duplicating the type entirely.

### [MEDIUM] Duplicated Type Definition for Baby Creation Payload
- **File:** `frontend/src/lib/components/FirstLogin.svelte` and `frontend/src/lib/components/CreateBabyForm.svelte`
- **Line(s):** FirstLogin L6-L12, CreateBabyForm L3-L9
- **Perspective:** Tech Lead
- **Finding:** Both `FirstLogin.svelte` and `CreateBabyForm.svelte` inline the same object shape `{ name, date_of_birth, sex, diagnosis_date?, kasai_date? }` in their callback prop types. This shape is already defined as `CreateBabyInput` in `$lib/stores/baby.ts`.
- **Reasoning:** Three copies of the same type spread across files is a maintenance liability. If a field is added or renamed, all three must be updated in sync.
- **Consequence:** A field change (e.g., adding `birth_weight`) could be applied to the store type but missed in one or both component props, causing silent type mismatches.
- **Suggestion:** Import and reuse the canonical type:
  ```typescript
  import type { CreateBabyInput } from '$lib/stores/baby';
  // then use: oncreate: (data: CreateBabyInput) => void;
  ```

### [MEDIUM] Redundant `clearCsrfToken` and `_resetCsrfToken` Functions
- **File:** `frontend/src/lib/api.ts`
- **Line(s):** L22-L29
- **Perspective:** Tech Lead
- **Finding:** `clearCsrfToken()` (line 22-24) and `_resetCsrfToken()` (line 27-29) have identical implementations -- both set `csrfToken = null`. Two functions exist for the same operation.
- **Reasoning:** This is copy-paste duplication. The private `clearCsrfToken` and the exported `_resetCsrfToken` do exactly the same thing. The private function could simply call the exported one, or they could be consolidated.
- **Consequence:** Minor maintenance burden. A future change to token-clearing logic (e.g., also clearing a refresh token) might be applied to one but not the other.
- **Suggestion:** Remove `clearCsrfToken` and use `_resetCsrfToken` internally, or have `clearCsrfToken` call `_resetCsrfToken`:
  ```typescript
  function clearCsrfToken(): void {
      _resetCsrfToken();
  }
  ```

### [LOW] CSRF Token Race Condition on Concurrent State-Changing Requests
- **File:** `frontend/src/lib/api.ts`
- **Line(s):** L9-L20
- **Perspective:** QA
- **Finding:** If two state-changing requests are initiated concurrently before the CSRF token is cached, both will call `fetchCsrfToken()`, which will issue two parallel `fetch('/api/csrf-token')` calls. This is because the `if (csrfToken)` check is not aware of in-flight fetches.
- **Reasoning:** While this is unlikely to cause a bug (both requests will eventually get a valid token), it results in a redundant network request on first load.
- **Consequence:** Minor: one extra unnecessary CSRF token fetch request. In theory, if the server invalidates tokens on re-issue, the first-returned token could become stale, though this is unlikely.
- **Suggestion:** Use a promise-based deduplication pattern:
  ```typescript
  let csrfPromise: Promise<string> | null = null;
  async function fetchCsrfToken(): Promise<string> {
      if (csrfToken) return csrfToken;
      if (!csrfPromise) {
          csrfPromise = fetch('/api/csrf-token', { credentials: 'include' })
              .then(r => r.json())
              .then(data => { csrfToken = data.csrf_token; csrfPromise = null; return csrfToken!; });
      }
      return csrfPromise;
  }
  ```

### [LOW] Content-Type Header Set Unconditionally When Body Is Present
- **File:** `frontend/src/lib/api.ts`
- **Line(s):** L40-L42
- **Perspective:** Tech Lead
- **Finding:** The `request` function sets `Content-Type: application/json` whenever `options?.body` is truthy. This would be incorrect if a future caller passes a `FormData` or `Blob` body (e.g., for file uploads).
- **Reasoning:** Currently all callers use `JSON.stringify`, so this is safe today. However, the generic `request` function signature accepts any `RequestInit`, making it a leaky abstraction that assumes JSON bodies.
- **Consequence:** Adding file upload support through this function would silently set the wrong Content-Type, causing server-side parsing failures that are non-obvious to debug.
- **Suggestion:** Either check if body is a string before setting the header, or document that this function is JSON-only:
  ```typescript
  if (options?.body && typeof options.body === 'string') {
      headers['Content-Type'] = 'application/json';
  }
  ```
