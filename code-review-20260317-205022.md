## Summary

Reviewed phases 28-30 of LittleLiver for **code reuse** issues only. These phases add chart data query functions (`chart_data.go`), alert computation logic (`alerts.go`), and corresponding handler/test changes. The review focuses on duplicated logic, redundant type definitions, and missed opportunities to use existing utilities.

### [MEDIUM] Duplicated date-range parsing logic
- **File:** `backend/internal/store/chart_data.go`
- **Line(s):** L70-L82
- **Perspective:** Tech Lead
- **Finding:** `parseDateRange()` duplicates the exact same date parsing and boundary computation found in `GetDashboardSummary` (`dashboard.go` L42-L52) and `ListMedLogs` (`med_logs.go` L106-L121). All three locations parse `from`/`to` with `model.DateFormat`, then compute `fromDate.Format(model.DateTimeFormat)` and `toDate.Add(24*time.Hour).Format(model.DateTimeFormat)`.
- **Reasoning:** The `parseDateRange` function was correctly extracted as a helper, but it only serves `chart_data.go`. The identical logic in `dashboard.go` and `med_logs.go` was not refactored to use it. This means three separate copies of the same date boundary computation exist in the store package.
- **Consequence:** A future change to date range semantics (e.g., timezone-awareness, different boundary logic) requires updating three locations instead of one, creating a maintenance risk and a potential source of inconsistent behavior.
- **Suggestion:** Move `parseDateRange` to `metric_helpers.go` (which already holds shared helpers like `nullStr` and `nullFloat`) and refactor `GetDashboardSummary` and `ListMedLogs` to call it:
  ```go
  // In metric_helpers.go
  func ParseDateRange(from, to string) (string, string, error) { ... }
  
  // In dashboard.go
  fromTime, toTime, err := ParseDateRange(from, to)
  ```

### [MEDIUM] `alertResponse` type duplicates `store.Alert` field-for-field
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L54-L60
- **Perspective:** Tech Lead
- **Finding:** The `alertResponse` struct has identical fields to `store.Alert` (EntryID, AlertType, Method, Value, Timestamp) with the same JSON tags. The handler then manually copies each field from `store.Alert` to `alertResponse` in a loop (L174-L183).
- **Reasoning:** When the handler response type is structurally identical to the store type, the extra type adds boilerplate without providing any decoupling benefit. Other chart data types (`store.FeedingDailyEntry`, `store.TemperatureSeriesEntry`, etc.) are already used directly in `chartDataSeriesResponse` without a handler-layer wrapper.
- **Consequence:** Every time a field is added to `store.Alert`, the developer must also update `alertResponse` and the mapping loop, or the field silently gets dropped from the API response.
- **Suggestion:** Use `store.Alert` directly in `dashboardResponseJSON`, consistent with how `chartDataSeriesResponse` already embeds store types:
  ```go
  type dashboardResponseJSON struct {
      // ...
      ActiveAlerts []store.Alert `json:"active_alerts"`
  }
  ```
  Then remove the manual mapping loop and assign `storeAlerts` directly.

### [MEDIUM] `dashboardResp` test type duplicated across two test files
- **File:** `backend/internal/handler/dashboard_test.go` and `backend/internal/integration/dashboard_aggregation_test.go`
- **Line(s):** `dashboard_test.go` L27-L97, `dashboard_aggregation_test.go` L19-L93
- **Perspective:** Tech Lead
- **Finding:** The `dashboardResp` struct is defined independently in both test files with nearly identical structure. The handler test version uses a pointer for `ChartDataSeries` while the integration test uses a value type, but otherwise they are the same ~75-line type definition.
- **Reasoning:** Duplicating large test response types across packages creates a maintenance burden: when the dashboard response shape changes, both definitions must be updated in lockstep.
- **Consequence:** A schema change to the dashboard response could cause one test file to silently pass with stale expectations while the other catches the issue, leading to false confidence in test coverage.
- **Suggestion:** Define a shared `DashboardTestResponse` type in `backend/internal/testutil/` and import it from both test files. Alternatively, since the handler already defines the response types (`dashboardResponseJSON`, `chartDataSeriesResponse`), the handler tests could import and reuse those directly.

### [LOW] `loginAndGetAuthClient` and `createBabyForIntegration` are not shared across integration tests
- **File:** `backend/internal/integration/dashboard_aggregation_test.go`
- **Line(s):** L122-L178
- **Perspective:** Tech Lead
- **Finding:** `loginAndGetAuthClient` (OAuth login flow + user ID extraction) and `createBabyForIntegration` (baby insertion + parent linking) are defined only in this file. While no other integration test currently uses identical helpers, `auth_flow_test.go` performs the same OAuth login flow inline, and `createBabyForIntegration` duplicates what `testutil.CreateTestBaby` does (insert baby + insert baby_parents), just without the `GetBabiesByUserID` verification step.
- **Reasoning:** `createBabyForIntegration` manually inserts into `babies` and `baby_parents` tables, bypassing the existing `testutil.CreateTestBaby` helper which does the same thing plus validates the result. As more integration tests are added, these helpers will likely be copied again.
- **Consequence:** Minor: if the babies table schema changes (e.g., a new required column), `createBabyForIntegration` will break separately from `testutil.CreateTestBaby`, requiring fixes in multiple places.
- **Suggestion:** For `createBabyForIntegration`, consider using `testutil.CreateTestBaby` directly (it returns `*model.Baby` with the ID). For `loginAndGetAuthClient`, extract it to a shared integration test helper file (e.g., `integration/helpers_test.go`) so future integration tests can reuse it.

### [LOW] Repetitive nil-to-empty-slice pattern across all chart data query functions
- **File:** `backend/internal/store/chart_data.go`
- **Line(s):** L124-L127, L168-L170, L204-L207, L244-L246, L282-L285, L319-L322
- **Perspective:** Tech Lead
- **Finding:** Every chart data function ends with the same nil-to-empty-slice guard pattern:
  ```go
  if entries == nil {
      entries = make([]T, 0)
  }
  ```
  This pattern is repeated six times in the file.
- **Reasoning:** While this is a common Go idiom for ensuring JSON serialization produces `[]` instead of `null`, repeating it in every function is boilerplate. The `nullStr`/`nullFloat` helpers in `metric_helpers.go` show the project already uses the pattern of extracting repetitive nil-handling into shared helpers.
- **Consequence:** Minimal direct risk, but contributes to visual noise and makes it easier to accidentally omit the guard in a future query function, causing an inconsistent `null` vs `[]` response.
- **Suggestion:** Consider a generic helper (Go 1.18+) or handle this at the JSON serialization layer. For example:
  ```go
  func emptySliceIfNil[T any](s []T) []T {
      if s == nil {
          return make([]T, 0)
      }
      return s
  }
  ```
