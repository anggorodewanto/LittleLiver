## Summary

Phase 23 implements the R2 storage client interface, in-memory mock, photo upload endpoint (`POST /api/babies/:id/upload`), `photo_uploads` table migration, and supporting store/model code. The implementation covers file size validation (5MB), MIME type validation (JPEG/PNG/HEIC), HEIC-to-JPEG conversion via injectable converter, thumbnail generation (~300px wide) using Go stdlib, storage of both original and thumbnail in the object store, DB row creation, and R2 key response. All 13 upload handler tests and 5 storage tests pass. The migration matches the spec DDL.

### [MEDIUM] Context parameter ignored in convertHEICToJPEGWithCmd
- **File:** `backend/internal/handler/uploads.go`
- **Line(s):** L248-L249
- **Perspective:** Tech Lead
- **Finding:** The function accepts a `context.Context` parameter but creates a fresh `context.Background()` for the `exec.CommandContext` call, ignoring the passed context entirely.
- **Reasoning:** This defeats the purpose of using `CommandContext` -- if the HTTP request is cancelled, the ImageMagick process will not be terminated.
- **Consequence:** A cancelled or timed-out upload request will leave an orphaned ImageMagick process running until completion, wasting resources.
- **Suggestion:** Use the passed context parameter instead of `context.Background()`:
  ```go
  func convertHEICToJPEGWithCmd(ctx context.Context, data []byte) ([]byte, error) {
      cmd := exec.CommandContext(ctx, "convert", "-", "jpg:-")
  ```

### [MEDIUM] No UNIQUE constraint on r2_key column in photo_uploads migration
- **File:** `backend/migrations/007_photo_uploads.sql`
- **Line(s):** L5
- **Perspective:** QA
- **Finding:** The `r2_key` column has no UNIQUE constraint. While ULIDs in key generation make collisions extremely unlikely, the spec's DDL also lacks this constraint, so this is spec-consistent. However, `GetPhotoUploadByR2Key` assumes at most one row per key.
- **Reasoning:** If a duplicate key were somehow inserted, `GetPhotoUploadByR2Key` would silently return one arbitrary row. Adding a UNIQUE constraint would make the guarantee explicit and catch bugs early.
- **Consequence:** In the astronomically unlikely ULID collision case, data integrity issues could occur when linking photos to metrics.
- **Suggestion:** Consider adding `UNIQUE` to `r2_key`:
  ```sql
  r2_key TEXT NOT NULL UNIQUE,
  ```
  Since the spec DDL does not include UNIQUE, this is optional but recommended.

### [LOW] DefaultHEICConverter creates new context instead of passing through request context
- **File:** `backend/internal/handler/uploads.go`
- **Line(s):** L37-L39
- **Perspective:** Tech Lead
- **Finding:** `DefaultHEICConverter` calls `convertHEICToJPEGWithCmd(context.Background(), data)`, discarding any request context. The `HEICConverter` function type does not accept a context parameter.
- **Reasoning:** The converter function type `func(data []byte) ([]byte, error)` has no context parameter, so the request context cannot propagate to the ImageMagick process in production use.
- **Consequence:** Request cancellation will not stop in-progress HEIC conversions. For a personal-use app this is low risk, but it is an architectural gap.
- **Suggestion:** Consider adding context to the `HEICConverter` type signature:
  ```go
  type HEICConverter func(ctx context.Context, data []byte) ([]byte, error)
  ```
  Then pass `r.Context()` in the handler when calling the converter.

### [LOW] Log output during test run for HEIC conversion failure
- **File:** `backend/internal/handler/uploads.go`
- **Line(s):** L115
- **Perspective:** QA
- **Finding:** The `log.Printf("HEIC conversion failed: %v", err)` call produces output during test runs (visible in test output: `2026/03/17 18:45:06 HEIC conversion failed: imagemagick not found`). This is expected behavior being logged, not a real error during testing.
- **Reasoning:** Test output noise can mask real issues. Structured logging or injecting a logger would let tests suppress expected log lines.
- **Consequence:** Minor: cluttered test output may cause real warnings to be overlooked.
- **Suggestion:** Accept a `*log.Logger` or use `slog` with an injectable handler, or simply note this as acceptable for a personal-use app.
