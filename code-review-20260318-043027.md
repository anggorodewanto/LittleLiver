## Summary

Review of phases 40-42 covering: a new integration test for PDF report generation (`report_generation_test.go`), CSRF token handling and timezone header additions to the API client (`api.ts`), new Svelte frontend components for baby management (`BabySelector`, `CreateBabyForm`, `JoinBabyForm`, `FirstLogin`), Svelte stores for user and baby state, and a login page. The review focuses exclusively on efficiency issues.

### [MEDIUM] Repeated Intl.DateTimeFormat instantiation on every API request
- **File:** `frontend/src/lib/api.ts`
- **Line(s):** L6
- **Perspective:** Performance Engineer
- **Finding:** `getTimezone()` creates a new `Intl.DateTimeFormat` instance and resolves its options on every single API request. The user's timezone does not change during a session.
- **Reasoning:** `Intl.DateTimeFormat()` constructor does locale resolution work internally. While individually fast, it is pure waste on a hot path invoked for every API call.
- **Consequence:** Unnecessary object allocations and locale resolution on every network request. Minor but compounds across rapid-fire calls (e.g., dashboard loading multiple resources).
- **Suggestion:** Cache the timezone string at module level:
  ```typescript
  const userTimezone = Intl.DateTimeFormat().resolvedOptions().timeZone;

  function getTimezone(): string {
      return userTimezone;
  }
  ```

### [MEDIUM] Sequential fetch after create/join causes redundant round-trip
- **File:** `frontend/src/lib/stores/baby.ts`
- **Line(s):** L74-L83
- **Perspective:** Performance Engineer
- **Finding:** Both `createBaby()` and `joinBaby()` call `await fetchBabies()` after the mutation API call returns. The POST response for `createBaby` already returns the full `Baby` object, yet the code discards it for store purposes and re-fetches the entire baby list from the server.
- **Reasoning:** The create endpoint already returns the created baby. The store could append it directly instead of making a second round-trip GET request. Similarly, after joining, the baby ID is known and could be fetched individually or the join response could be enriched.
- **Consequence:** Every baby creation or join triggers two sequential API calls instead of one, doubling latency for these user-facing actions.
- **Suggestion:** After `createBaby`, update the store directly with the returned baby object:
  ```typescript
  export async function createBaby(input: CreateBabyInput): Promise<Baby> {
      const result = await apiClient.post<Baby>('/babies', input);
      babies.update((list) => [...list, result]);
      activeBaby.set(result);
      return result;
  }
  ```
  Keep `fetchBabies()` for initial load and full resync scenarios only.

### [MEDIUM] Individual row-by-row DB inserts in test seeding (no batching or transactions)
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L28-L232
- **Perspective:** Performance Engineer
- **Finding:** `seedDaysOfData` executes individual `db.Exec` calls for every row across 30 days: 120 feedings, 45 stools, 90 urine, 60 temperatures, 10 weights, 9 lab results, 4 notes, 60 UDCA med logs, 30 vitamin K med logs, and 6 photo/object-store puts -- roughly 430+ individual SQL statements with no transaction wrapping or batching.
- **Reasoning:** Each `db.Exec` without an explicit transaction triggers an implicit autocommit transaction, meaning ~430 separate transaction commits. Wrapping in a single transaction would be significantly faster.
- **Consequence:** Slow test execution. For an integration test that runs in CI, this adds unnecessary wall-clock time. With parallel test suites, the overhead multiplies.
- **Suggestion:** Wrap the inserts in a single transaction:
  ```go
  tx, err := db.Begin()
  if err != nil {
      t.Fatalf("begin tx: %v", err)
  }
  defer tx.Rollback()
  // ... use tx.Exec instead of db.Exec ...
  if err := tx.Commit(); err != nil {
      t.Fatalf("commit seed data: %v", err)
  }
  ```

### [LOW] Repeated JPEG generation for each test photo
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L211-L213
- **Perspective:** Performance Engineer
- **Finding:** `createReportTestJPEG(t)` is called inside the loop (3 iterations at L211-L231), generating an identical 20x20 JPEG image each time. The pixel-filling loop and JPEG encoding are repeated for every photo.
- **Reasoning:** The function creates the exact same image bytes every time. It could be called once before the loop and reused.
- **Consequence:** Trivial waste (3 iterations), but the pattern is worth noting for correctness of intent -- if different images were intended, the function should vary its output.
- **Suggestion:** Hoist the call outside the loop:
  ```go
  thumbPNG := createReportTestJPEG(t)
  for i := 0; i < 3; i++ {
      // reuse thumbPNG
  }
  ```

### [LOW] Unconditional activeBaby store update even when value has not changed
- **File:** `frontend/src/lib/stores/baby.ts`
- **Line(s):** L46-L51
- **Perspective:** Performance Engineer
- **Finding:** In `fetchBabies()`, when the current active baby still exists in the refreshed list, `activeBaby.set(stillExists)` is called unconditionally at L49. This fires a store update and notifies all subscribers even if the baby object is structurally identical.
- **Reasoning:** Svelte writable stores notify subscribers on every `.set()` call regardless of value equality. This can trigger unnecessary re-renders in subscribed components.
- **Consequence:** Minor unnecessary re-renders on baby list refresh. Low impact given this is not a frequently polled operation.
- **Suggestion:** Compare before setting:
  ```typescript
  if (stillExists !== currentActive) {
      activeBaby.set(stillExists);
  }
  ```
  Note: since `stillExists` is a new object from the API response, a deep equality check or returning early without updating (since the active baby data is unlikely to change) would be more precise.
