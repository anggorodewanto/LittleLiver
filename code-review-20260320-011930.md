## Summary

Phase 49 implements PWA setup: `manifest.json`, service worker with app shell caching, push event handler, notificationclick handler, install prompt handling, and push subscription registration. The implementation covers the core behaviors specified in the phase, with good test coverage across three test files. However, there is one critical gap: the service worker and install prompt are never actually registered/activated from any Svelte component, meaning the PWA functionality is dead code in production.

### [HIGH] Service worker and install prompt never registered in production code
- **File:** `frontend/src/routes/+layout.svelte`
- **Line(s):** L1-L13
- **Perspective:** Product Owner
- **Finding:** The `+layout.svelte` adds the manifest link and theme-color meta tag, but never calls `registerServiceWorker()` or `setupInstallPrompt()` from `$lib/pwa`. These functions exist but are never invoked from any Svelte component, meaning the service worker won't register, the app won't be installable, and push subscriptions cannot be created.
- **Reasoning:** The phase requires "Service worker registers successfully" and "Install prompt handling" and "Push subscription registers" as proof of progress. Without calling these functions on app startup, none of this actually works.
- **Consequence:** The PWA is non-functional in production — app shell caching doesn't happen, the app is not installable despite having a manifest, and push notifications cannot be subscribed to. The phase's proof-of-progress criteria are not met.
- **Suggestion:** Add an `onMount` block in `+layout.svelte` (or a dedicated component) that calls `registerServiceWorker()` and `setupInstallPrompt()`:
```svelte
<script lang="ts">
  import { onMount } from 'svelte';
  import { registerServiceWorker, setupInstallPrompt } from '$lib/pwa';

  onMount(() => {
    registerServiceWorker();
    setupInstallPrompt();
  });
</script>
```

### [MEDIUM] Push notification permission request not implemented
- **File:** `frontend/src/lib/pwa.ts`
- **Line(s):** N/A - missing functionality
- **Perspective:** Product Owner
- **Finding:** The phase description specifies "Push notification permission request and subscription registration (calls `POST /api/push/subscribe`)." While `subscribeToPush()` exists to call the API, there is no code that requests notification permission (`Notification.requestPermission()`), calls `pushManager.subscribe()` to create a `PushSubscription`, or orchestrates the full flow of permission -> subscribe -> register with backend.
- **Reasoning:** The spec says "The Svelte PWA registers a push subscription on install and sends it to the backend." The current `subscribeToPush` function only handles the last step (sending to backend) and expects to receive an already-created `PushSubscription` object. No code creates one.
- **Consequence:** Push notifications will never work because no subscription is ever created. Users cannot receive medication reminders, which is a core feature of the app.
- **Suggestion:** Add a function that orchestrates the full push subscription flow:
```typescript
export async function requestPushSubscription(
  registration: ServiceWorkerRegistration,
  vapidPublicKey: string
): Promise<PushSubscription | null> {
  const permission = await Notification.requestPermission();
  if (permission !== 'granted') {
    return null;
  }

  const subscription = await registration.pushManager.subscribe({
    userVisibleOnly: true,
    applicationServerKey: urlBase64ToUint8Array(vapidPublicKey)
  });

  await subscribeToPush(subscription);
  return subscription;
}
```

### [MEDIUM] Duplicate service worker logic between .js and .ts files
- **File:** `frontend/static/service-worker.js` and `frontend/src/lib/service-worker.ts`
- **Line(s):** N/A - structural issue
- **Perspective:** Tech Lead
- **Finding:** The service worker logic is implemented twice: once in `static/service-worker.js` (the file that actually runs in production as a service worker) and once in `src/lib/service-worker.ts` (a testable TypeScript module with `initServiceWorker()`). The two implementations are functionally identical but maintained separately. Tests only cover the `.ts` version via `initServiceWorker()`, but the `.js` version is what actually runs in the browser.
- **Reasoning:** If someone fixes a bug in one file but not the other, they will diverge silently. The tests give false confidence because they test the `.ts` file while the `.js` file is what runs in production.
- **Consequence:** Bug fixes or feature changes could be applied to only one copy, creating subtle production bugs that pass all tests. The test suite validates code that never runs in production.
- **Suggestion:** Either (a) build the production service worker from the TypeScript source (e.g., use a build step to compile `service-worker.ts` into `static/service-worker.js`), or (b) have the `.js` file import and call `initServiceWorker(self)` from the compiled `.ts` module. At minimum, add a comment in both files noting they must be kept in sync.

### [LOW] App shell cache only includes `/` and `/index.html`
- **File:** `frontend/static/service-worker.js`
- **Line(s):** L5
- **Perspective:** Tech Lead
- **Finding:** `APP_SHELL_URLS` only caches `/` and `/index.html`. For a SvelteKit SPA, the actual app shell includes JS bundles, CSS files, and other static assets. These are not pre-cached, so the "app shell caching" is minimal.
- **Reasoning:** The phase says "service worker for app shell caching only (HTML/JS/CSS)." Only caching the HTML without the JS/CSS means the offline fallback page will be broken (HTML without styles or scripts). The dynamic caching in the fetch handler partially mitigates this (it caches non-API responses on first load), but the pre-cache list is incomplete.
- **Consequence:** On first visit and immediate offline, the user gets an unstyled, non-functional page. After navigating around, resources get cached dynamically, but the initial offline experience is poor.
- **Suggestion:** This is acceptable for v1 since the fetch handler dynamically caches resources. Consider expanding `APP_SHELL_URLS` in a future phase to include critical JS/CSS bundles, or use SvelteKit's built-in service worker support to auto-generate the precache list.

### [LOW] Missing test for activate handler deleting old caches
- **File:** `frontend/src/tests/service-worker.test.ts`
- **Line(s):** L180-L196
- **Perspective:** QA
- **Finding:** The activate test verifies that `clients.claim()` is called but does not verify that old caches are actually deleted. The test sets up `sw.caches.keys` to return `['littleliver-v1', 'old-cache']` but never asserts that `caches.delete('old-cache')` was called.
- **Reasoning:** The activate handler's main purpose is to clean up stale caches. Without asserting deletion, the test doesn't verify the core behavior.
- **Consequence:** A regression that breaks old cache cleanup would go undetected.
- **Suggestion:** Add an assertion:
```typescript
expect(sw.caches.delete).toHaveBeenCalledWith('old-cache');
expect(sw.caches.delete).not.toHaveBeenCalledWith('littleliver-v1');
```
Note: The mock structure uses `sw.caches.delete` but the service worker code calls `caches.delete(key)` via `sw.caches` — verify the mock wiring matches.
