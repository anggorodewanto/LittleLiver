## Summary

Phase 26 implements med-log endpoints for dose logging, editing, and deletion. The implementation includes: migration DDL for `med_logs` table, store CRUD operations, handler layer with validation, route registration, and comprehensive tests at both store and handler levels. The migration DDL matches the spec. Given/skipped mutual exclusivity is enforced. Baby-medication mismatch validation is present. `logged_by` immutability is verified by test. All 14 store tests and 14 handler tests pass. The code follows early-return style and uses parameterized SQL throughout.

### [HIGH] List endpoint uses single-date filter instead of spec-required from/to date range with cursor pagination

- **File:** `backend/internal/store/med_logs.go`
- **Line(s):** L92-L143
- **Perspective:** Product Owner
- **Finding:** The spec defines `GET /api/babies/:id/med-logs?medication_id=&from=&to=&cursor=` with `from`/`to` date range filters and cursor-based pagination. The implementation uses a single `date=` query parameter with no pagination support. Every other metric endpoint in the codebase (feedings, stools, urine, weights, etc.) uses the standard `listMetricWithTZ` helper with `from`, `to`, `cursor`, and `limit` parameters returning a `MetricPage` response.
- **Reasoning:** This deviates from both the spec and the established codebase pattern. The dashboard and trends views depend on date range queries (`from`/`to`). Without cursor pagination, large result sets will be returned in a single response. The inconsistency with other metric endpoints will confuse frontend developers.
- **Consequence:** The frontend trends view and dashboard cannot query med-logs by date range. If a baby has months of med-log history, all records matching a single date are returned unbounded, and there is no way to query across a range of dates. This blocks downstream features (dashboard, reports) that depend on date-range filtering.
- **Suggestion:** Refactor `ListMedLogs` to accept `from`, `to`, `cursor *string` and `limit int` parameters, use the existing `listMetricWithTZ` pattern (or equivalent), and return `*model.MetricPage[model.MedLog]`. Update the handler to read `from`, `to`, `cursor` query params and pass a default limit. Update tests to cover range filtering and pagination. Example signature:
  ```go
  func ListMedLogs(db *sql.DB, babyID string, medicationID, from, to, cursor *string, limit int) (*model.MetricPage[model.MedLog], error) {
  ```

### [MEDIUM] med_logs table missing indexes present in spec pattern

- **File:** `backend/migrations/009_med_logs.sql`
- **Line(s):** L17-L18
- **Perspective:** Performance Engineer
- **Finding:** The migration creates indexes on `baby_id` and `medication_id`, but the spec's list endpoint supports filtering by date range (`from`/`to` on `created_at`). There is no composite index to support the common query pattern of `baby_id + date(created_at)` or `baby_id + medication_id + created_at` which will be the primary access pattern once from/to filtering is implemented.
- **Reasoning:** Once date-range filtering is added (per the HIGH issue above), queries will filter on `baby_id` and sort/filter by `created_at`. Without a composite index, SQLite will scan all rows for a baby.
- **Consequence:** For babies with many med-log entries (e.g., 3 medications x 2 doses/day x 365 days = ~2000 rows), queries will perform full scans on the baby's rows. Not critical for a personal-use app with SQLite, but worth addressing when fixing the list endpoint.
- **Suggestion:** When implementing from/to filtering, add a composite index:
  ```sql
  CREATE INDEX idx_med_logs_baby_created ON med_logs(baby_id, created_at);
  ```
