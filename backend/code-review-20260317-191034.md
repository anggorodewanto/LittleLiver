## Summary

Review of phases 16-18 production code covering six new metric CRUD handlers (stools, urine, weights, temperatures, abdomen) plus modifications to feedings and babies. The code is well-structured and consistent, but exhibits significant copy-paste duplication across both the handler and store layers that should be addressed before the pattern is replicated further.

---

### 1. [HIGH] Massively duplicated ListXxxWithTZ store functions -- identical pagination and date-filter logic copy-pasted six times

- **File(s):** `backend/internal/store/feedings.go`, `stools.go`, `urine.go`, `weights.go`, `temperatures.go`, `abdomen.go`
- **Line(s):** Every `ListXxxWithTZ` function (e.g., feedings L122-199, stools L97-171, etc.)
- **Perspective:** Tech Lead
- **Finding:** All six `ListXxxWithTZ` functions share the exact same body structure: build conditions/args from `from`, `to`, `cursor`; format the query with `Sprintf`; iterate rows; build `MetricPage` with cursor logic. The only differences are the table name, column constant, and scan function.
- **Reasoning:** Any bug fix or behavior change to pagination (e.g., fixing the off-by-one in end-of-day, adding a new filter) must be applied identically in six places. This is the most maintenance-risky pattern in the codebase.
- **Consequence:** A pagination fix applied to only 5 of 6 files creates an inconsistency that is hard to detect, leading to subtle bugs in production (e.g., one metric returns different results for the same date range).
- **Suggestion:** Extract a generic helper, for example:

```go
// buildPaginatedQuery returns (query, args) for any metric table.
func buildPaginatedQuery(table, columns string, babyID string, from, to, cursor *string, limit int, loc *time.Location) (string, []any, error) { ... }

// paginateResults builds a MetricPage from a slice, handling cursor extraction.
func paginateResults[T interface{ GetID() string }](items []T, limit int) *model.MetricPage[T] { ... }
```

Each `ListXxxWithTZ` then becomes ~10 lines: call `buildPaginatedQuery`, `db.Query`, scan loop, `paginateResults`.

### 2. [HIGH] Duplicated handler CRUD scaffolding -- five near-identical Get/Update/Delete handler functions

- **File(s):** `backend/internal/handler/abdomen.go`, `stools.go`, `urine.go`, `weights.go`, `temperatures.go`, `feedings.go`
- **Line(s):** Every `GetXxxHandler`, `UpdateXxxHandler`, `DeleteXxxHandler` across all six files
- **Perspective:** Tech Lead
- **Finding:** Each Get handler follows the identical pattern: `requireUser` -> `requireBabyAccess` -> `requireEntryID` -> `store.GetXxxByID` -> `ErrNoRows` check -> `writeJSON`. The same applies to Update (add decode+validate) and Delete. The only variance is the store function name, the response converter, and the error message noun (e.g., "stool not found" vs "weight not found").
- **Reasoning:** With six metrics and more coming (jaundice, bruising, medications, notes per the spec), this pattern will be copy-pasted 4+ more times. Each copy is a maintenance burden and a vector for inconsistency.
- **Consequence:** Future metrics will continue to accumulate boilerplate, making the handler package increasingly unwieldy and error-prone.
- **Suggestion:** Consider a lightweight generic CRUD factory that accepts a store interface and a response converter. Even a partial extraction (e.g., a `deleteMetric` helper that takes a `func(db, babyID, entryID) error` and the entity noun) would cut ~50% of the boilerplate without heavy abstraction.

### 3. [MEDIUM] Duplicated List handler boilerplate -- timezone parsing and page-to-response conversion repeated six times

- **File(s):** `backend/internal/handler/feedings.go`, `stools.go`, `urine.go`, `weights.go`, `temperatures.go`, `abdomen.go`
- **Line(s):** Every `ListXxxHandler` body (e.g., feedings L122-163, abdomen L118-158)
- **Perspective:** Tech Lead
- **Finding:** Every List handler has the identical 8-line timezone-parsing block (`loc := time.UTC; if tz := r.Header.Get("X-Timezone")...`) and the identical page-mapping loop pattern. This is copied verbatim six times.
- **Reasoning:** The timezone-parsing block is a cross-cutting concern that should be extracted once.
- **Consequence:** If the timezone header name changes or parsing behavior is updated, six files need the same edit.
- **Suggestion:** Extract `func parseLocationFromRequest(r *http.Request) *time.Location` into `helpers.go`. Similarly, consider a generic `func mapPage[S, D any](page *model.MetricPage[S], fn func(*S) D) model.MetricPage[D]` helper.

### 4. [MEDIUM] Redundant nil guard in store pagination -- `page.Data == nil` check is unreachable

- **File(s):** `backend/internal/store/feedings.go` L194-196, `stools.go` L166-168, `urine.go` L155-157, `weights.go` L155-157, `temperatures.go` L152-154, `abdomen.go` L160-162
- **Perspective:** QA
- **Finding:** Each `ListXxxWithTZ` function initializes `page.Data = make([]T, 0)` on line ~148, then in the `else` branch assigns `page.Data = observations` (which can be nil if no rows). Then afterwards checks `if page.Data == nil` and re-creates the empty slice. The initial `make([]T, 0)` on the struct literal is immediately overwritten, making it dead code. The nil guard at the end is only reachable via the else branch when the slice from `rows.Next()` iteration is nil (0 rows).
- **Reasoning:** The logic is confusing -- the initial `make` allocation is wasted. The structure could be simplified.
- **Consequence:** No runtime bug, but the dead-code pattern is confusing and is copied six times, compounding the confusion.
- **Suggestion:** Remove the initial `make([]T, 0)` from the struct literal. Keep only the nil guard at the end, or better yet, initialize the slice as `items := make([]T, 0)` before the `rows.Next()` loop so it is never nil.

### 5. [MEDIUM] Stringly-typed feed types in calories.go -- hardcoded `"breast_milk"` and `"formula"` literals

- **File(s):** `backend/internal/model/calories.go` L33, L44
- **Line(s):** L33: `feedType == "breast_milk"`, L44: `feedType == "breast_milk" || feedType == "formula"`
- **Perspective:** Tech Lead
- **Finding:** `calories.go` uses raw string literals `"breast_milk"` and `"formula"` for feed type comparisons, while `enums.go` already defines `feedTypeValues` with a validation function. However, there are no exported constants for individual feed type values.
- **Reasoning:** If a feed type string is ever renamed or a new calorie-eligible type is added, `calories.go` will silently diverge from the enum definition.
- **Consequence:** Renaming a feed type in `enums.go` without updating `calories.go` would break calorie calculation silently -- calories would come back nil for what should be calculated feeds.
- **Suggestion:** Add exported constants in `enums.go` (e.g., `FeedTypeBreastMilk = "breast_milk"`, `FeedTypeFormula = "formula"`) and reference them in `calories.go`. This also benefits handler validation messages.

### 6. [MEDIUM] Parameter sprawl in store Create/Update functions -- up to 10 positional parameters

- **File(s):** `backend/internal/store/feedings.go` L80, L204; `store/stools.go` L67, L174; `store/abdomen.go` L61, L168
- **Perspective:** Tech Lead
- **Finding:** `CreateFeeding` has 8 parameters after `db`; `UpdateFeeding` has 10 after `db`. `CreateStool` has 8 after `db`. `UpdateAbdomen` has 8 after `db`. All are positional strings, pointers, and primitives with no compile-time guard against transposition.
- **Reasoning:** Positional parameters of the same type (e.g., `babyID, loggedBy, timestamp, feedType string`) are easy to accidentally swap. Go does not catch `CreateStool(db, babyID, userID, ts, rating, label, consistency, volume, notes)` vs `CreateStool(db, babyID, userID, ts, rating, consistency, label, volume, notes)`.
- **Consequence:** A transposed call compiles and runs but inserts wrong data into wrong columns, which is extremely hard to debug.
- **Suggestion:** Introduce a typed params struct per metric (e.g., `StoolParams { Timestamp string; ColorRating int; ... }`) and have Create/Update accept `(db *sql.DB, babyID, userID string, p StoolParams)`. This makes call sites self-documenting and immune to parameter swaps.

### 7. [MEDIUM] Duplicated Delete function pattern in store -- identical logic across six files

- **File(s):** `backend/internal/store/feedings.go` L265-283, `stools.go` L202-220, `urine.go` L190-208, `weights.go` L189-207, `temperatures.go` L186-204, `abdomen.go` L195-213
- **Perspective:** Tech Lead
- **Finding:** Every `DeleteXxx` function is identical except for the table name and error message prefix: `db.Exec("DELETE FROM xxx WHERE id = ? AND baby_id = ?", ...)`, check affected, return wrapped `sql.ErrNoRows`.
- **Reasoning:** This is the simplest function to generalize since the only varying input is the table name.
- **Consequence:** Six copies of identical logic, growing with each new metric type.
- **Suggestion:** Extract:
```go
func deleteMetricByID(db *sql.DB, table, babyID, entryID string) error {
    res, err := db.Exec("DELETE FROM "+table+" WHERE id = ? AND baby_id = ?", entryID, babyID)
    // ... check affected, wrap ErrNoRows
}
```
Each `DeleteXxx` becomes a one-liner calling `deleteMetricByID`.

### 8. [LOW] Router registration boilerplate -- repetitive middleware wrapping pattern

- **File(s):** `backend/internal/handler/router.go` L83-123
- **Line(s):** L83-123 (all metric CRUD route registrations)
- **Perspective:** Tech Lead
- **Finding:** Every 5-line metric CRUD block follows the exact same pattern: POST with `rateMw(authMw(csrfMw(...)))`, GET list with `rateMw(authMw(...))`, GET single with `rateMw(authMw(...))`, PUT with `rateMw(authMw(csrfMw(...)))`, DELETE with `rateMw(authMw(csrfMw(...)))`. This is repeated for feedings, stools, urine, weights, temperatures, abdomen -- 30 lines of near-identical registrations.
- **Reasoning:** A helper function like `registerMetricCRUD(mux, path, handlers, rateMw, authMw, csrfMw)` would replace each 5-line block with a single call.
- **Consequence:** Low severity since the router is typically write-once, but adding future metrics means copying another 5-line block and risking a missed middleware (e.g., forgetting CSRF on a DELETE).
- **Suggestion:** Extract a `registerMetricCRUD` helper that accepts the base path and the five handler funcs, and applies the correct middleware pattern internally.

### 9. [LOW] Duplicate validation of `cal_density` without `volume_ml` -- checked in both handler and model

- **File(s):** `backend/internal/handler/feedings.go` L42-44, `backend/internal/model/calories.go` L24-26
- **Perspective:** QA
- **Finding:** The rule "cal_density cannot be provided without volume_ml" is validated in `feedingRequest.validate()` (handler layer) and again in `CalculateCalories()` (model layer). Both check the exact same condition.
- **Reasoning:** Dual validation is not harmful, but the model-layer check returns an `error` while the handler returns a user-facing string. If one is updated and the other is not, they could diverge in behavior.
- **Consequence:** No runtime bug, but the duplication is a minor maintenance concern. The model-layer error would surface as a 500 (internal error) rather than a 400 if the handler check were removed, so both exist for defense-in-depth. This is acceptable but worth acknowledging.
- **Suggestion:** Keep the model-layer validation as the single source of truth. In the handler, instead of duplicating the check, rely on `CalculateCalories` returning an error and map it to a 400. Alternatively, document that the handler check is intentional defense-in-depth.

### 10. [LOW] End-of-day boundary uses `24h - 1s` instead of exclusive upper bound -- loses the last second of each day

- **File(s):** `backend/internal/store/feedings.go` L145, `stools.go` L119, `urine.go` L108, `weights.go` L108, `temperatures.go` L105, `abdomen.go` L113
- **Perspective:** QA
- **Finding:** The "to" date filter computes end-of-day as `t.Add(24*time.Hour - time.Second)`, producing `23:59:59`. Any entry timestamped at exactly `23:59:59.5` (if sub-second precision is ever stored) would be excluded. More importantly, using `<=` with `23:59:59` rather than `<` with the next day's `00:00:00` is a subtle correctness gap.
- **Reasoning:** The standard pattern for date range queries is `timestamp >= startOfDay AND timestamp < startOfNextDay`, which avoids the boundary issue entirely.
- **Consequence:** Currently timestamps are stored as strings at second precision, so no data loss occurs. But if the schema ever adds sub-second precision, entries in the last second of a day could be missed.
- **Suggestion:** Change to `t.Add(24 * time.Hour)` and use `timestamp < ?` instead of `timestamp <= ?`. This is correct regardless of precision and is the standard SQL pattern.
