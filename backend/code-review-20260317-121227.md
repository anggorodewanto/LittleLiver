## Summary

Phases 22-24 add R2 object storage integration, photo upload with thumbnail generation, and photo linking/unlinking across five metric handlers (stools, abdomen, bruising, skin observations, general notes). Phase 22 adds a delete-account anonymization integration test and initializes the `AnonymizeTables` list. The review focuses strictly on efficiency issues.

### [HIGH] N+1 DB queries in `resolvePhotos` via `GetPhotoUploadsByR2Keys`
- **File:** `backend/internal/store/photo_uploads.go`
- **Line(s):** L107-L119
- **Perspective:** Performance Engineer
- **Finding:** `GetPhotoUploadsByR2Keys` issues one `SELECT` per R2 key in a loop. `resolvePhotos` calls this for every item in a list response, so listing 20 stools each with 4 photos produces 80 individual DB queries just for photo metadata, before the signed URL calls even begin.
- **Reasoning:** This is the classic N+1 query pattern. The function could use a single `WHERE r2_key IN (?, ?, ...)` query to fetch all photos in one round-trip. The max is 4 keys per entry (bounded by `MaxPhotosPerMetric`), but the outer list iteration multiplies this.
- **Consequence:** On a list of 50 entries with 4 photos each, this generates 200 DB round-trips plus 400 signed-URL calls per single API response. Latency will scale linearly with page size and photo count.
- **Suggestion:** Replace the loop with a single batched query:
  ```go
  func GetPhotoUploadsByR2Keys(db *sql.DB, keys []string) ([]model.PhotoUpload, error) {
      if len(keys) == 0 {
          return nil, nil
      }
      placeholders := strings.Repeat("?,", len(keys))
      placeholders = placeholders[:len(placeholders)-1]
      args := make([]any, len(keys))
      for i, k := range keys {
          args[i] = k
      }
      rows, err := db.Query(
          `SELECT id, baby_id, r2_key, thumbnail_key, uploaded_at, linked_at
           FROM photo_uploads WHERE r2_key IN (`+placeholders+`)`, args...)
      // ... scan rows
  }
  ```

### [HIGH] N+1 signed-URL generation in `resolvePhotos` on list endpoints
- **File:** `backend/internal/handler/helpers.go`
- **Line(s):** L129-L143
- **Perspective:** Performance Engineer
- **Finding:** `resolvePhotos` calls `objStore.SignedURL` sequentially for each photo key (once for the original, once for the thumbnail). On a list endpoint, this is invoked per list item via `mapMetricPage`, producing up to `pageSize * maxPhotos * 2` serial signed-URL calls. For a real R2/S3 client, each `SignedURL` is either a crypto operation or a network call.
- **Reasoning:** The signed-URL calls are independent of each other and could be parallelized. More importantly, for pre-signed URLs that are computed locally (HMAC-based), serialization is unnecessary overhead; for network-based signing, it becomes a serious bottleneck.
- **Consequence:** A list of 50 items with 4 photos each generates up to 400 sequential signed-URL operations on the hot path, directly blocking the HTTP response.
- **Suggestion:** If signed URLs are computed locally (typical for S3v4 pre-signing), batch all photo resolution for the entire page in a single pass. If they require network calls, use goroutines with `errgroup` to parallelize:
  ```go
  // Resolve all photos for all items in the page in one batch:
  // 1. Collect all photo keys from all items
  // 2. Single DB query for all keys
  // 3. Parallel signed URL generation
  ```

### [HIGH] N+1 DB queries in `UnlinkPhotos`
- **File:** `backend/internal/store/photo_uploads.go`
- **Line(s):** L91-L103
- **Perspective:** Performance Engineer
- **Finding:** `UnlinkPhotos` executes one `UPDATE` per key in a loop. This is called on every update and delete operation.
- **Reasoning:** A single `UPDATE ... WHERE r2_key IN (...)` would accomplish the same in one round-trip.
- **Consequence:** Deleting an entry with 4 photos issues 4 separate UPDATE statements instead of 1.
- **Suggestion:** Batch the update:
  ```go
  func UnlinkPhotos(db *sql.DB, keys []string) error {
      if len(keys) == 0 {
          return nil
      }
      placeholders := strings.Repeat("?,", len(keys))
      placeholders = placeholders[:len(placeholders)-1]
      args := make([]any, len(keys))
      for i, k := range keys {
          args[i] = k
      }
      _, err := db.Exec(
          `UPDATE photo_uploads SET linked_at = NULL WHERE r2_key IN (`+placeholders+`)`, args...)
      return err
  }
  ```

### [MEDIUM] Extra SELECT before UPDATE in all update handlers (TOCTOU / unnecessary round-trip)
- **File:** `backend/internal/handler/stools.go`
- **Line(s):** L239-L255 (same pattern in abdomen.go, bruising.go, skin_observations.go, general_notes.go)
- **Perspective:** Performance Engineer
- **Finding:** Every update handler now issues a `GetXByID` SELECT to fetch the existing record (to read old `photo_keys`), then issues a second `UpdateXWithPhotos` which internally does the UPDATE. When `photo_keys` is nil in the request (no photo change), the extra SELECT is pure waste. Even when photos change, the SELECT + UPDATE could be combined into a single query that returns the old photo_keys via `RETURNING` or by reading it in the same transaction as the update.
- **Reasoning:** This doubles the DB round-trips on every update request, even when no photo keys are provided.
- **Consequence:** Every update operation takes at least 2 DB round-trips instead of 1, plus additional queries for photo linking.
- **Suggestion:** Short-circuit when `req.PhotoKeys` is nil (no photo field in request body):
  ```go
  if req.PhotoKeys == nil {
      // No photo changes - use original update without extra SELECT
      stool, err := store.UpdateStool(db, ...)
      ...
  }
  ```
  Or combine the read-old-keys and update into one transaction.

### [MEDIUM] Extra SELECT before DELETE in all delete handlers (unnecessary existence check)
- **File:** `backend/internal/handler/stools.go`
- **Line(s):** L281-L300 (same in abdomen.go, bruising.go, skin_observations.go, general_notes.go)
- **Perspective:** Performance Engineer
- **Finding:** Delete handlers now issue a `GetXByID` SELECT before the `DELETE`, solely to check for photo keys to unlink. The DELETE itself would return "not found" if the row does not exist. When the entry has no photos (the common case), the SELECT is wasted work.
- **Reasoning:** The SELECT could be eliminated by reading photo_keys in the same operation as the delete (e.g., `DELETE ... RETURNING photo_keys` with SQLite, or a single-transaction SELECT+DELETE).
- **Consequence:** Every delete costs an additional DB query, doubling latency on the delete path.
- **Suggestion:** Use a transaction that selects photo_keys and deletes in one operation, or use SQLite's `RETURNING` clause if available.

### [MEDIUM] `resizeNearestNeighbor` uses `Set` per pixel -- slow for upload path
- **File:** `backend/internal/handler/uploads.go`
- **Line(s):** L236-L247
- **Perspective:** Performance Engineer
- **Finding:** The thumbnail generation loop calls `dst.Set(x, y, src.At(srcX, srcY))` for every pixel. `image.RGBA.Set` does bounds checking and color model conversion on every call. For a 3000x4000 source image scaled to 300px wide, this is 300*400 = 120,000 pixel operations, each with overhead.
- **Reasoning:** Using `draw.NearestNeighbor` from `golang.org/x/image/draw` or directly writing to `dst.Pix` byte slice would be significantly faster. However, since this runs on the upload path (not a hot per-request path) and is bounded to one image per request, the practical impact is moderate.
- **Consequence:** Thumbnail generation for large images will be slower than necessary, adding upload latency.
- **Suggestion:** Replace with `draw.NearestNeighbor.Scale(dst, dst.Bounds(), src, src.Bounds(), draw.Over, nil)` from `golang.org/x/image/draw`, or access `dst.Pix` directly to skip bounds checks. Note: this adds an external dep, which the project constraints discourage, so the current approach may be acceptable for a personal-use app with max 4 photos per entry.

### [CRITICAL] Delete handlers parse photo_keys with `strings.Split` instead of JSON unmarshal
- **File:** `backend/internal/handler/stools.go`
- **Line(s):** L292
- **Perspective:** QA
- **Finding:** All five delete handlers parse `existing.PhotoKeys` using `strings.Split(*existing.PhotoKeys, ",")`, but photo_keys are stored as JSON arrays (e.g., `["photos/a.jpg","photos/b.jpg"]`). This produces keys like `["photos/a.jpg"` and `"photos/b.jpg"]` (with brackets and quotes), which will never match any r2_key in the database, silently failing to unlink photos on deletion.
- **Reasoning:** The `handlePhotoLinking` function correctly uses `parsePhotoKeysJSON` for the same purpose. The delete handlers bypass that helper.
- **Consequence:** Every entry deletion silently leaks photo links -- `linked_at` is never cleared, orphaning photos and preventing their reuse. This is a correctness bug with efficiency implications (orphaned data accumulates).
- **Suggestion:** Replace `strings.Split` with `parsePhotoKeysJSON`:
  ```go
  if existing.PhotoKeys != nil && *existing.PhotoKeys != "" {
      oldKeys := parsePhotoKeysJSON(existing.PhotoKeys)
      if len(oldKeys) > 0 {
          if unlinkErr := store.UnlinkPhotos(db, oldKeys); unlinkErr != nil {
              log.Printf("unlink photos on delete: %v", unlinkErr)
          }
      }
  }
  ```
  This bug exists in: `stools.go:292`, `abdomen.go:289`, `bruising.go:285`, `skin_observations.go:286`, `general_notes.go:270`.
