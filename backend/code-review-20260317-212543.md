### [LOW] PUT without X-Timezone header clears existing timezone to NULL
- **File:** `backend/internal/handler/medications.go`
- **Line(s):** L228-L231
- **Perspective:** Product Owner
- **Finding:** When a PUT request omits the `X-Timezone` header, `tz` is nil, which causes the store layer to set `timezone = NULL` in the database. This is correct PUT semantics (full replacement), but could surprise callers who intend to update only the dose or name without changing the timezone.
- **Reasoning:** The spec says timezone is "mutable via PUT (e.g., if family moves)" which implies intentional changes. However, a client that forgets to include X-Timezone on a dose-change PUT will silently lose the timezone, breaking notification scheduling (Phase 26+).
- **Consequence:** A parent updating a medication dose without sending X-Timezone could silently disable push notification scheduling for that medication.
- **Suggestion:** Either (a) document this behavior clearly in API docs so frontend always sends X-Timezone, or (b) apply the same "nil means keep existing" pattern used for `active`:
```go
tzVal := existing.Timezone
if timezone != nil {
    tzVal = timezone
}
```

### [LOW] Schedule similarly cleared on PUT without schedule_times
- **File:** `backend/internal/handler/medications.go`
- **Line(s):** L226
- **Perspective:** Product Owner
- **Finding:** Same pattern as timezone -- if `schedule_times` is omitted or empty in the PUT body, `scheduleTimesToJSON` returns nil, and the existing schedule is cleared to NULL. This is consistent PUT semantics but could be surprising.
- **Reasoning:** A client updating only the dose would need to re-send the full schedule_times array to avoid clearing it.
- **Consequence:** Accidental loss of schedule data if the frontend omits `schedule_times` on a partial update.
- **Suggestion:** Same as above -- either document clearly or preserve existing values when nil.

### [LOW] No validation of timezone string (IANA format)
- **File:** `backend/internal/handler/medications.go`
- **Line(s):** L123-L125
- **Perspective:** QA
- **Finding:** The `X-Timezone` header value is stored as-is without validating it is a valid IANA timezone string. An invalid timezone like "NotATimezone" would be stored and could cause errors in Phase 26 (notification scheduling).
- **Reasoning:** The spec says timezone is "IANA timezone" and is "used for all notification scheduling". Storing an invalid timezone now defers the error to when it matters most (notification delivery).
- **Consequence:** Invalid timezone values could cause notification scheduling failures in Phase 26, which would be harder to debug than rejecting at input time.
- **Suggestion:** Validate using `time.LoadLocation(tzHeader)` from Go stdlib:
```go
if tzHeader := r.Header.Get("X-Timezone"); tzHeader != "" {
    if _, err := time.LoadLocation(tzHeader); err != nil {
        http.Error(w, "invalid timezone", http.StatusBadRequest)
        return
    }
    tz = &tzHeader
}
```
