## Summary

Phase 4 implements ULID generation (`internal/model/ulid.go`), domain model types for `User`, `Baby`, `BabyParent`, `Session`, `Invite` (`internal/model/types.go`), and enum validation functions (`internal/model/enums.go`). All tests pass at 98.0% coverage. The implementation is well-structured, follows Go conventions, uses early return style, and matches the spec DDL closely. The enum values are correct against the spec. The ULID implementation correctly encodes 48-bit timestamps and 80 bits of randomness using Crockford's Base32.

### [MEDIUM] Flaky sortability test due to time.Sleep

- **File:** `backend/internal/model/ulid_test.go`
- **Line(s):** L38-L47
- **Perspective:** QA
- **Finding:** `TestNewULID_Sortable` uses `time.Sleep(2 * time.Millisecond)` to ensure two ULIDs have different timestamps. On busy CI machines or systems with coarse timer resolution, 2ms sleeps can sometimes complete in under 1ms of wall-clock time, causing the test to fail intermittently.
- **Reasoning:** ULID timestamps encode milliseconds. If both ULIDs happen to land on the same millisecond (due to sleep imprecision), they could have the same timestamp prefix, and sortability depends on the random portion -- which is not guaranteed to be ordered.
- **Consequence:** Intermittent test failures in CI that are hard to diagnose, eroding trust in the test suite.
- **Suggestion:** Increase the sleep to `10 * time.Millisecond` or, better, extract the timestamp encoding into a helper that accepts a `time.Time` parameter so the test can control the timestamp deterministically:
  ```go
  func TestNewULID_Sortable(t *testing.T) {
      t.Parallel()
      id1 := model.NewULID()
      time.Sleep(10 * time.Millisecond)
      id2 := model.NewULID()
      if id1 >= id2 {
          t.Fatalf("ULIDs not sortable: %s >= %s", id1, id2)
      }
  }
  ```

### [LOW] Types tests only verify struct field assignment, not behavior

- **File:** `backend/internal/model/types_test.go`
- **Line(s):** L10-L138
- **Perspective:** QA
- **Finding:** The `types_test.go` tests instantiate structs and verify that field values can be read back. These are essentially testing Go struct assignment, which the compiler already guarantees. They do not test any meaningful behavior.
- **Reasoning:** While these tests achieve coverage, they do not provide meaningful regression protection. They would only catch issues if someone renamed a field -- but that would already cause compile errors at call sites. In strict TDD, tests should drive behavior, not just exercise syntax.
- **Consequence:** Low-severity -- the tests are not harmful, but they create a false sense of security and add maintenance burden without catching real bugs.
- **Suggestion:** These tests are acceptable as a "phase 4 proof of progress" artifact. As the project grows and these types gain methods (validation, serialization), replace these with tests that exercise actual behavior. Consider adding tests for JSON serialization correctness (e.g., verifying `Session.Token` is omitted from JSON output due to the `json:"-"` tag).

### [LOW] Mutex in ULID generation may be unnecessary

- **File:** `backend/internal/model/ulid.go`
- **Line(s):** L12-L13, L18-L19
- **Perspective:** Performance Engineer
- **Finding:** `NewULID` acquires a mutex on every call. Since `time.Now()` and `crypto/rand.Read()` are both goroutine-safe, the mutex does not protect any shared mutable state.
- **Reasoning:** In the ULID spec, monotonic ordering within the same millisecond requires coordinated incrementing of the random portion -- which this implementation does NOT do (it generates fresh random bytes each time). The mutex therefore provides no monotonic guarantee and only adds contention.
- **Consequence:** Under high concurrency, the mutex serializes all ULID generation unnecessarily. For a personal-use app this is negligible, but it is technically unnecessary overhead with no correctness benefit.
- **Suggestion:** Either remove the mutex (since no shared state is being protected) or implement proper monotonic ULID generation where the random portion is incremented within the same millisecond. For this project's scale, removing the mutex is simpler:
  ```go
  func NewULID() string {
      ms := uint64(time.Now().UnixMilli())
      // ... rest of function unchanged
  }
  ```
