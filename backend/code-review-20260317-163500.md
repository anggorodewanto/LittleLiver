## Summary

This review covers Phase 11: Baby CRUD endpoints for the LittleLiver project. The implementation adds `POST /api/babies`, `GET /api/babies`, `GET /api/babies/:id`, and `PUT /api/babies/:id` endpoints with a store layer in `internal/store/babies.go`, handler layer in `internal/handler/babies.go`, and route registration in `router.go`. Tests cover store operations (90.2% coverage) and handler behavior (79.4% coverage). The implementation correctly uses transactions for baby creation with parent linking, parameterized SQL throughout, and proper authorization checks (403 for unlinked parents).

### [MEDIUM] Handler package coverage is 79.4%, below the 90% target

- **File:** `internal/handler/babies_test.go`
- **Line(s):** N/A - structural issue
- **Perspective:** QA
- **Finding:** The handler package coverage is 79.4%, failing the project's mandatory 90%+ target. Several babies handler functions are significantly below threshold: `ListBabiesHandler` at 75%, `GetBabyHandler` at 67.9%, `UpdateBabyHandler` at 70%, and `extractBabyID` at 66.7%. The uncovered paths are internal error branches (database failures in `IsParentOfBaby`, `GetBabiesByUserID`, and JSON encoding errors), as well as the URL-parsing fallback in `extractBabyID`.
- **Reasoning:** The CLAUDE.md states "Target 90%+ code coverage across backend and frontend" as non-negotiable. While some of the uncovered lines are from pre-existing functions (CSRFTokenHandler, HealthCheck, WithAuthConfig), the new babies handler code itself has significant gaps in error path coverage.
- **Consequence:** Untested error paths may silently behave incorrectly. For instance, if `store.IsParentOfBaby` returns an error in `GetBabyHandler`, the response could expose internal details or return an unexpected status code without any test verifying the behavior.
- **Suggestion:** Add tests that inject database failures (e.g., using a closed DB) into the handler paths to cover the `log.Printf` + 500 error branches. For `extractBabyID`, test the URL-parsing fallback path explicitly. Example for GetBabyHandler with DB error on IsParentOfBaby: create a test where the DB is closed after the baby is found but before the authorization check.

### [MEDIUM] No server-side validation of `sex` field in handlers

- **File:** `internal/handler/babies.go`
- **Line(s):** L90-L93, L228-L231
- **Perspective:** QA
- **Finding:** The `CreateBabyHandler` and `UpdateBabyHandler` validate that `name`, `sex`, and `date_of_birth` are non-empty, but do not validate that `sex` is one of the allowed values (`"male"` or `"female"`). The validation relies entirely on the SQLite `CHECK (sex IN ('male', 'female'))` constraint, which surfaces as a generic 500 Internal Server Error.
- **Reasoning:** The spec defines `sex TEXT NOT NULL CHECK (sex IN ('male', 'female'))` at the database level. When an invalid sex value is submitted, the handler returns a 500 error rather than a descriptive 400 Bad Request. The existing `TestCreateBabyHandler_InvalidSex` test at line 562 explicitly asserts a 500 status, confirming this is intentional but incorrect behavior.
- **Consequence:** Clients receive an opaque "failed to create baby" 500 error for what is actually a validation issue. This makes it difficult for frontend code to display a helpful error message. It also makes the error look like a server bug in monitoring/logs.
- **Suggestion:** Add explicit validation in the handler before calling the store:
  ```go
  if req.Sex != "male" && req.Sex != "female" {
      http.Error(w, "sex must be 'male' or 'female'", http.StatusBadRequest)
      return
  }
  ```
  Update `TestCreateBabyHandler_InvalidSex` to expect 400 instead of 500.

### [MEDIUM] No validation of `date_of_birth` format in handlers

- **File:** `internal/handler/babies.go`
- **Line(s):** L90-L93, L228-L231
- **Perspective:** QA
- **Finding:** The handlers check that `date_of_birth` is non-empty but do not validate that it is a valid date string in `YYYY-MM-DD` format. An invalid date like `"not-a-date"` or `"2025-13-45"` will be inserted into SQLite without error (SQLite is permissive with DATE types) and will cause a `parseTime` failure when the baby is read back.
- **Reasoning:** SQLite stores dates as text and does not enforce date validity. The error would manifest as a 500 on the subsequent `GetBabyByID` call within `CreateBaby`, which reads back the inserted row. The same applies to `diagnosis_date` and `kasai_date`.
- **Consequence:** Malformed dates would be stored in the database and cause read failures. The error message would be confusing ("parse date_of_birth: unable to parse time") surfacing as a 500 to the client.
- **Suggestion:** Validate date fields before passing to the store:
  ```go
  if _, err := time.Parse("2006-01-02", req.DateOfBirth); err != nil {
      http.Error(w, "date_of_birth must be in YYYY-MM-DD format", http.StatusBadRequest)
      return
  }
  ```

### [LOW] `GetBabiesByUserID` is defined in `users.go` instead of `babies.go`

- **File:** `internal/store/users.go`
- **Line(s):** L62
- **Perspective:** Tech Lead
- **Finding:** The function `GetBabiesByUserID` is defined in `internal/store/users.go` rather than `internal/store/babies.go`, even though it returns `[]model.Baby` and queries the `babies` table. The phase spec explicitly states "Baby store layer in `internal/store/babies.go`."
- **Reasoning:** This function is a core part of the baby store operations (it's the implementation behind `GET /api/babies`). Placing it in `users.go` fragments the baby domain logic across multiple files, making it harder for developers to find all baby-related store operations in one place.
- **Consequence:** As more baby-related queries are added in future phases, the split location will create confusion about where to place new functions and make it harder to review baby store logic as a cohesive unit.
- **Suggestion:** Move `GetBabiesByUserID` to `internal/store/babies.go` and update its test to `babies_test.go` (where the test already lives at line 362).

### [LOW] `parseTime` does not include date-only format `"2006-01-02"`

- **File:** `internal/store/sessions.go`
- **Line(s):** L86-L98
- **Perspective:** QA
- **Finding:** The `parseTime` function handles `time.DateTime` (`"2006-01-02 15:04:05"`), `time.RFC3339`, and two timestamp variants, but does not include the date-only format `"2006-01-02"`. The babies table has `DATE` columns (`date_of_birth`, `diagnosis_date`, `kasai_date`). The tests currently pass because the `modernc.org/sqlite` driver appears to return dates with time components, but this is an implicit behavior dependency.
- **Reasoning:** If the database driver behavior changes or if raw SQL inserts use date-only strings (as `CreateTestBaby` in testutil does with `'2025-01-01'`), the parseTime function would fail. The code's correctness depends on an undocumented driver behavior rather than explicit format handling.
- **Consequence:** A driver upgrade or change in SQLite behavior could cause all baby read operations to fail with "unable to parse time" errors.
- **Suggestion:** Add `"2006-01-02"` to the parseTime format list:
  ```go
  for _, layout := range []string{
      time.DateTime,
      time.RFC3339,
      "2006-01-02T15:04:05Z",
      "2006-01-02T15:04:05",
      "2006-01-02",
  } {
  ```

### [LOW] `extractBabyID` URL parsing fallback is fragile

- **File:** `internal/handler/babies.go`
- **Line(s):** L63-L73
- **Perspective:** Tech Lead
- **Finding:** The `extractBabyID` function has a fallback that parses the URL path by splitting on `/` and taking the third segment. This assumes a fixed URL structure (`/api/babies/{id}`) and would break for any nested resource paths. Since Go 1.22+ `PathValue` is used as the primary mechanism and the routes are registered with `{id}` patterns, the fallback should never be needed.
- **Reasoning:** The router in `router.go` uses `"GET /api/babies/{id}"` and `"PUT /api/babies/{id}"` patterns, which populate `PathValue("id")`. The fallback adds dead code that creates a false sense of robustness while potentially masking routing bugs.
- **Consequence:** If `PathValue` fails unexpectedly, the fallback could extract the wrong path segment (e.g., for a future nested route like `/api/babies/{id}/feeds`), leading to authorization bypass or incorrect data access.
- **Suggestion:** Remove the URL-parsing fallback and return an empty string if `PathValue` returns empty. This makes routing issues explicit rather than silently degraded:
  ```go
  func extractBabyID(r *http.Request) string {
      return r.PathValue("id")
  }
  ```
