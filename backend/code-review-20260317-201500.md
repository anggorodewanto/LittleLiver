## Summary

Phase 24 implements photo linking, unlinking, and signed URL replacement for metric entries (stools, abdomen, skin, bruising, general_notes). The implementation adds `ValidateAndLinkPhotos` and `UnlinkPhotos` to the store layer, a shared `handlePhotoLinking` helper in the handler layer, and `resolvePhotos` for converting R2 keys to signed URLs on read. All five metric types are updated consistently. Tests cover the seven TDD scenarios specified in the phase description. All tests pass.

### [MEDIUM] Spec deviation: photo_keys stored as comma-separated instead of JSON array

- **File:** `backend/internal/handler/helpers.go`
- **Line(s):** L188
- **Perspective:** Tech Lead
- **Finding:** The spec states `photo_keys TEXT -- JSON array of R2 object keys` in the schema and refers to `photo_keys` as a "JSON array field" in the API section (SPEC.md L299, L537, L591, L607, L623, L651). However, the implementation stores photo keys as a comma-separated string (e.g., `"key1,key2"`) via `strings.Join(newPhotoKeys, ",")` and parses them with `strings.Split`. On the read side, `resolvePhotos` also uses `strings.Split`.
- **Reasoning:** While functionally equivalent for simple key strings, this deviates from the spec. If R2 keys ever contain commas (unlikely but possible), parsing would break. A JSON array format (`["key1","key2"]`) would be spec-compliant and more robust.
- **Consequence:** Minor incompatibility if other systems (cron jobs, future migration scripts) expect JSON array format in the `photo_keys` column. Could cause subtle bugs if keys contain commas.
- **Suggestion:** Use `json.Marshal(newPhotoKeys)` for storage and `json.Unmarshal` for retrieval instead of comma-separated join/split. Alternatively, if this is an intentional simplification, document the deviation.

### [MEDIUM] No guard against linking an already-linked photo to a different metric entry

- **File:** `backend/internal/store/photo_uploads.go`
- **Line(s):** L62-L83
- **Perspective:** QA
- **Finding:** `ValidateAndLinkPhotos` checks that the photo exists and belongs to the correct baby, but does not check whether `linked_at` is already set (i.e., the photo is already linked to another metric entry). A single photo could be linked to multiple metric entries simultaneously.
- **Reasoning:** While the spec does not explicitly forbid this, the `linked_at` field semantics (single timestamp, not a reference to the linking metric) suggest a one-to-one relationship. When the photo cleanup cron runs, it checks `linked_at IS NULL` to identify unlinked photos -- if a photo is linked to two metrics and removed from one, `linked_at` remains set (from the other metric), so the photo is not orphaned. However, if the second metric is also updated to remove it, `linked_at` is set to NULL even though the first metric still references it in its `photo_keys` column.
- **Consequence:** A photo referenced by metric A could be unlinked (and eventually garbage-collected) when metric B is updated to remove it, while metric A still has it in its `photo_keys`. This would cause `resolvePhotos` to fail to find the photo, resulting in a silently missing photo in the response.
- **Suggestion:** Either (1) check `linked_at IS NULL` before linking and reject already-linked photos, or (2) use a reference count instead of a timestamp for `linked_at`. Option 1 is simpler and matches the spec's implied single-use model.

### [LOW] No duplicate key validation in photo_keys input

- **File:** `backend/internal/handler/helpers.go`
- **Line(s):** L147-L190
- **Perspective:** QA
- **Finding:** `handlePhotoLinking` does not check for duplicate keys in the `newPhotoKeys` slice. A client could submit `["key1", "key1", "key1", "key1"]` and it would pass the 4-photo limit check while only linking one actual photo.
- **Reasoning:** This is a minor input validation gap. The duplicates would be stored in the comma-separated string, and `resolvePhotos` would query the same key multiple times, returning duplicate photo entries in the response.
- **Consequence:** Duplicate photo entries in API responses, wasting bandwidth and potentially confusing the frontend.
- **Suggestion:** Deduplicate `newPhotoKeys` at the start of `handlePhotoLinking`:
  ```go
  seen := make(map[string]bool, len(newPhotoKeys))
  unique := make([]string, 0, len(newPhotoKeys))
  for _, k := range newPhotoKeys {
      if !seen[k] {
          seen[k] = true
          unique = append(unique, k)
      }
  }
  newPhotoKeys = unique
  ```

### [LOW] N+1 query pattern in GetPhotoUploadsByR2Keys

- **File:** `backend/internal/store/photo_uploads.go`
- **Line(s):** L103-L116
- **Perspective:** Performance Engineer
- **Finding:** `GetPhotoUploadsByR2Keys` queries each key individually in a loop, resulting in N separate database queries for N photo keys. Similarly, `resolvePhotos` calls `objStore.SignedURL` in a loop for each photo (2N calls for N photos with thumbnails).
- **Reasoning:** With a maximum of 4 photos per metric entry, the worst case is 4 queries plus 8 signed URL calls per entry. For a single detail view, this is acceptable. However, for a list view of 20 entries with 4 photos each, this becomes 80 queries + 160 signed URL calls per list request.
- **Consequence:** List endpoints for photo-heavy metrics could become slow under moderate usage. For a personal-use app this is unlikely to be a real problem, but worth noting.
- **Suggestion:** Use a single `SELECT ... WHERE r2_key IN (?, ?, ...)` query to batch the lookups. For signed URLs, consider caching or batching if the storage backend supports it.

### [LOW] Variadic objStores parameter pattern adds unnecessary complexity

- **File:** `backend/internal/handler/stools.go` (and all other metric handlers)
- **Line(s):** L91-L95 (repeated in all handlers)
- **Perspective:** Tech Lead
- **Finding:** Every handler function uses `objStores ...storage.ObjectStore` as a variadic parameter with `if len(objStores) > 0 { objStore = objStores[0] }`. This pattern is repeated verbatim across all five metric handler files (stools, abdomen, skin, bruising, general_notes) in every CRUD function.
- **Reasoning:** This appears to be a workaround for backward compatibility so existing callers without photo support still compile. A cleaner approach would be to pass `storage.ObjectStore` directly (nil when not needed) or use a handler options struct.
- **Consequence:** Code duplication across ~25 handler functions. Each new metric type that supports photos must copy this boilerplate.
- **Suggestion:** Accept `storage.ObjectStore` as a regular parameter (callers pass `nil` when not needed), since `resolvePhotos` already handles `nil` gracefully. This eliminates the variadic boilerplate entirely.
