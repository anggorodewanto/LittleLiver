## Summary

Phase 18 implements weight, temperature, and abdomen observation endpoints for the LittleLiver post-Kasai baby health tracking application. The implementation includes: migration `004_weights_temperatures_abdomen.sql` creating three tables with `(baby_id, timestamp)` indexes; store-layer CRUD functions for all three metric types; handler-layer request/response handling with validation; route registration in `router.go`; model types in `types.go`; and enum validators in `enums.go`. All 85+ tests pass. The implementation closely follows the established metric CRUD pattern from earlier phases (feedings, stools, urine).

**Verdict: PHASE_APPROVED**

The implementation is correct, complete, and follows all project conventions. The DDL matches the spec exactly. Enum validations are correct (measurement_source: home_scale/clinic; method: rectal/axillary/ear/forehead; firmness: soft/firm/distended). Weight precision rounds to 2 decimals, temperature to 1 decimal. All SQL uses parameterized queries. Early return style is followed throughout. CRUD operations including detail endpoints are fully tested for all three metric types. The few observations below are minor and do not block approval.

### [LOW] Repeated boilerplate across three metric store files
- **File:** `backend/internal/store/weights.go`, `temperatures.go`, `abdomen.go`
- **Line(s):** N/A - structural issue
- **Perspective:** Tech Lead
- **Finding:** The List*WithTZ functions across all three files contain nearly identical pagination, date filtering, and cursor logic (approximately 40 lines each).
- **Reasoning:** This is a known pattern in the codebase -- other metric types (feedings, stools, urine) follow the same approach. The generic MetricPage type is already in use but a generic list helper could reduce duplication further.
- **Consequence:** When pagination logic needs to change (e.g., fixing the off-by-one cursor behavior), the fix must be applied in every metric store file independently.
- **Suggestion:** Consider extracting a generic list builder in a future refactor phase. This is not blocking since the pattern is already established across all metric types and consistency is maintained.

### [LOW] measurement_source column is not enforced at the database level
- **File:** `backend/migrations/004_weights_temperatures_abdomen.sql`
- **Line(s):** L9
- **Perspective:** QA
- **Finding:** The `measurement_source` column has no CHECK constraint at the database level. Validation relies solely on the handler layer's `ValidMeasurementSource` function. This matches the spec DDL exactly, which also omits a CHECK constraint.
- **Reasoning:** The same pattern is used for other enum-like columns in the codebase (e.g., feed_type, consistency). The handler-level validation is consistent and tested.
- **Consequence:** If data is inserted via direct SQL (migration scripts, manual fixes), invalid enum values could be stored. Low risk for a personal-use app.
- **Suggestion:** No change needed for spec compliance. Could add CHECK constraints in a future hardening pass if desired.

### [LOW] Handler test boilerplate could use table-driven tests
- **File:** `backend/internal/handler/weights_test.go`, `temperatures_test.go`, `abdomen_test.go`
- **Line(s):** N/A - structural issue
- **Perspective:** Tech Lead
- **Finding:** Several validation tests (missing timestamp, missing required field, invalid JSON) follow the same pattern across all three handler test files. Table-driven tests could reduce repetition.
- **Reasoning:** The current approach is readable and explicit. Each test has a clear name and intent. The verbosity is a minor cost.
- **Consequence:** Adding new validation rules requires duplicating test patterns across multiple files.
- **Suggestion:** Consider table-driven subtests for validation edge cases in future refactoring. Not blocking.
