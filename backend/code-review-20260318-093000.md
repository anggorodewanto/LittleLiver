## Summary

Phase 32 adds the `GET /api/who/percentiles?sex=&from_days=&to_days=` endpoint, which returns 5 WHO weight-for-age percentile curves (3rd, 15th, 50th, 85th, 97th). The implementation consists of a thin handler (`who_percentiles.go`) that validates query parameters and delegates to the existing `internal/who` package from Phase 31, a route registration in `router.go`, and 8 integration tests covering happy paths and validation errors.

The implementation is clean, minimal, and spec-compliant. A few minor observations follow, but none block approval.

### [LOW] Missing test for HTTP method restriction
- **File:** `backend/internal/handler/who_percentiles_test.go`
- **Line(s):** N/A - structural issue
- **Perspective:** QA
- **Finding:** No test verifies that non-GET methods (e.g., POST, PUT) return 405 Method Not Allowed on `/api/who/percentiles`.
- **Reasoning:** The Go 1.22+ `ServeMux` with the `GET` prefix handles this automatically, so this is low risk. However, having an explicit test would document the expected behavior and guard against accidental route changes.
- **Consequence:** If the route registration were accidentally changed to omit the `GET` prefix, any HTTP method would be accepted. Very unlikely but untested.
- **Suggestion:** Add a brief test:
  ```go
  func TestWHOPercentiles_PostReturns405(t *testing.T) {
      t.Parallel()
      mux, fix := setupWHOMux(t)
      req := testutil.AuthenticatedRequest(t, fix.DB, fix.User.ID, testCookieName, testSecret, http.MethodPost,
          "/api/who/percentiles?sex=male&from_days=0&to_days=10")
      rr := httptest.NewRecorder()
      mux.ServeHTTP(rr, req)
      if rr.Code != http.StatusMethodNotAllowed {
          t.Errorf("expected 405, got %d", rr.Code)
      }
  }
  ```

### [LOW] Missing test for from_days > to_days edge case
- **File:** `backend/internal/handler/who_percentiles_test.go`
- **Line(s):** N/A - structural issue
- **Perspective:** QA
- **Finding:** The `who.PercentileCurves` function validates `fromDays > toDays` and returns an error, but there is no test at the handler level for this edge case (e.g., `?sex=male&from_days=50&to_days=10`).
- **Reasoning:** The Phase 32 spec TDD approach says to test that "curves span the requested day range," which implicitly should cover invalid ranges. The `who` package itself tests this, but the handler integration path is untested.
- **Consequence:** If the handler were refactored to bypass `who.PercentileCurves` validation, this edge case would silently break.
- **Suggestion:** Add a test confirming 400 is returned when `from_days > to_days`.

### [LOW] Error message leaks internal details
- **File:** `backend/internal/handler/who_percentiles.go`
- **Line(s):** L45-L48
- **Perspective:** SecOps
- **Finding:** When `who.PercentileCurves` returns an error, the raw error message is sent to the client via `http.Error(w, err.Error(), ...)`. Error messages from the `who` package include details like `"to_days 800 is out of range (max 730)"` which reveals the internal data range.
- **Reasoning:** For this personal-use app behind auth, this is very low risk. However, in general practice, leaking internal state via error messages can help attackers probe boundaries.
- **Consequence:** An authenticated user could learn internal data structure limits. Given this is a personal app, the risk is negligible.
- **Suggestion:** No action required for a personal-use app. If this were public-facing, consider returning a generic "invalid parameters" message instead.

### [LOW] Test setup creates unnecessary DB and user
- **File:** `backend/internal/handler/who_percentiles_test.go`
- **Line(s):** L15-L31
- **Perspective:** Performance Engineer
- **Finding:** The `setupWHOMux` function creates a test database and user for every test, but the WHO percentiles handler does not use the database at all -- it only delegates to the in-memory `who` package. The DB and user are only needed because the request must pass through auth middleware.
- **Reasoning:** Each test incurs DB setup overhead unnecessarily. Tests still run fast (0.03s each) so this is not blocking, but it adds conceptual coupling.
- **Consequence:** Tests are slightly slower than they could be and create a false impression that the endpoint depends on a database.
- **Suggestion:** This is acceptable given the existing test patterns in the codebase. No change needed.
