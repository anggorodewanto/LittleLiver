## Summary

Phase 6 implements Google OAuth login, callback, session management, and user upsert for the LittleLiver project. The implementation covers: `GET /auth/google/login` (redirect to Google with CSRF state), `GET /auth/google/callback` (code exchange, user upsert, session creation, cookie setting), session store with create/get/delete/extend operations, and a logout handler. All tests pass. Coverage is 84.9% for `auth` and 82.8% for `store`.

The implementation is generally solid and well-structured. The code follows early-return style, uses parameterized SQL, mocks Google HTTP endpoints properly, and validates CSRF state. Below are findings that should be addressed.

### [MEDIUM] In-memory CSRF state map is not thread-safe
- **File:** `backend/internal/auth/auth.go`
- **Line(s):** L45-L46, L90-L98, L131-L136
- **Perspective:** QA
- **Finding:** The `states` map on `Handlers` is a plain `map[string]time.Time` with no mutex protection. It is written to in `Login` (L90) and read/deleted in `Callback` (L131-L136), and iterated with deletion in Login (L93-L98). If the server handles concurrent requests (even in single-instance deployment), concurrent map access will cause a panic.
- **Reasoning:** Go maps are not safe for concurrent read/write. The `net/http` server handles each request in its own goroutine. Two simultaneous login requests or a login and callback arriving concurrently will race on the `states` map.
- **Consequence:** Runtime panic: "concurrent map read and map write" or "concurrent map writes", crashing the server.
- **Suggestion:** Use `sync.Mutex` or `sync.RWMutex` to protect the `states` map:
```go
type Handlers struct {
    DB         *sql.DB
    Config     Config
    HTTPClient *http.Client
    mu         sync.Mutex
    states     map[string]time.Time
}

func (h *Handlers) Login(w http.ResponseWriter, r *http.Request) {
    // ...
    h.mu.Lock()
    h.states[state] = time.Now().Add(10 * time.Minute)
    now := time.Now()
    for k, v := range h.states {
        if now.After(v) {
            delete(h.states, k)
        }
    }
    h.mu.Unlock()
    // ...
}
```

### [MEDIUM] UpsertUser uses SELECT-then-INSERT/UPDATE instead of atomic upsert
- **File:** `backend/internal/store/users.go`
- **Line(s):** L12-L53
- **Perspective:** Tech Lead
- **Finding:** The `UpsertUser` function performs a SELECT, then conditionally an INSERT or UPDATE in separate statements. This is not atomic and could fail under concurrent access (two simultaneous first-logins by the same Google account).
- **Reasoning:** SQLite supports `INSERT ... ON CONFLICT(google_id) DO UPDATE SET email=excluded.email, name=excluded.name` which is atomic and simpler. The current approach has a race window between the SELECT and INSERT where a duplicate could be inserted.
- **Consequence:** In the unlikely event of two concurrent first-logins with the same Google account, the second INSERT could fail with a UNIQUE constraint violation on `google_id`, returning a 500 error.
- **Suggestion:** Use SQLite's upsert syntax:
```go
func UpsertUser(db *sql.DB, googleID, email, name string) (*model.User, error) {
    id := model.NewULID()
    _, err := db.Exec(
        `INSERT INTO users (id, google_id, email, name) VALUES (?, ?, ?, ?)
         ON CONFLICT(google_id) DO UPDATE SET email=excluded.email, name=excluded.name`,
        id, googleID, email, name,
    )
    if err != nil {
        return nil, fmt.Errorf("upsert user: %w", err)
    }
    // Fetch the actual record to get the correct ID
    var user model.User
    err = db.QueryRow("SELECT id, google_id, email, name FROM users WHERE google_id = ?", googleID).
        Scan(&user.ID, &user.GoogleID, &user.Email, &user.Name)
    if err != nil {
        return nil, fmt.Errorf("fetch upserted user: %w", err)
    }
    return &user, nil
}
```

### [MEDIUM] Cookie value stores session ID (ULID) — correct per spec but session token goes unused in cookie
- **File:** `backend/internal/auth/auth.go`
- **Line(s):** L173-L181
- **Perspective:** SecOps
- **Finding:** The cookie value is set to `session.ID` (a ULID), which is correct per the spec ("The cookie value is the `sessions.id` (ULID)"). The randomly generated `token` field is stored in the DB but not used in the cookie — it is reserved for CSRF derivation (Phase 7). This is spec-compliant.
- **Reasoning:** ULIDs are time-ordered and partially predictable (the timestamp portion). However, the spec explicitly chooses this design with the token reserved for CSRF. Since sessions are looked up by primary key and validated against expiry, this is acceptable.
- **Consequence:** None if Phase 7 implements CSRF properly using the session token.
- **Suggestion:** No change needed — this follows the spec. Ensure Phase 7 implements CSRF token derivation using `HMAC-SHA256(session.Token, ...)` as specified.

### [LOW] Duplicated `SessionDuration` constant across packages
- **File:** `backend/internal/auth/auth.go` L22, `backend/internal/store/sessions.go` L13
- **Line(s):** auth/auth.go:L22, store/sessions.go:L13
- **Perspective:** Tech Lead
- **Finding:** The 30-day session duration is defined as a constant in both `auth` (as `SessionDuration`) and `store` (as `sessionDuration`). If one is changed without the other, the cookie MaxAge and the DB expiry will diverge.
- **Reasoning:** DRY principle — a single source of truth for session duration prevents divergence.
- **Consequence:** If a developer changes the duration in one place but not the other, the cookie expiry and DB session expiry will be inconsistent, causing confusing session behavior.
- **Suggestion:** Export the constant from `store` and reference it in `auth`:
```go
// In store/sessions.go:
const SessionDuration = 30 * 24 * time.Hour

// In auth/auth.go:
MaxAge: int(store.SessionDuration.Seconds()),
```
