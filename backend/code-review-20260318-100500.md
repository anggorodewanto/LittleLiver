## Summary

Phase 33 implements Web Push subscription management for LittleLiver. The changes span four packages: a new `internal/notify/` package for VAPID key management and Web Push sending, push subscription CRUD in the store layer, HTTP handlers for `POST /api/push/subscribe` and `DELETE /api/push/subscribe`, and route registration. A `PushSubscription` model struct is added to `model/types.go`.

Overall, the implementation is solid, spec-compliant, well-tested, and follows project conventions. All tests pass. One minor dependency hygiene issue was found.

### [LOW] webpush-go Dependency Marked as Indirect
- **File:** `go.mod`
- **Line(s):** L6
- **Perspective:** DevOps
- **Finding:** The `github.com/SherClockHolmes/webpush-go` dependency was marked as `// indirect` despite being directly imported by `internal/notify/notify.go`. Running `go mod tidy` corrects this to a direct dependency.
- **Reasoning:** Misclassified dependencies can confuse dependency auditing tools and mislead developers about which packages are actually used in the codebase.
- **Consequence:** Minimal practical impact, but dependency audits and vulnerability scanners may miss this as a direct dependency, and `go mod tidy` will modify the file on next run.
- **Suggestion:** Run `go mod tidy` before committing to ensure `go.mod` accurately reflects direct vs. indirect dependencies. (Note: this was already corrected during the review.)

### [LOW] Notify Package Coverage at 80%
- **File:** `backend/internal/notify/notify.go`
- **Line(s):** L51-L76 (WebPusher.Send)
- **Perspective:** QA
- **Finding:** The `WebPusher.Send` method has 75% coverage. The uncovered path is the `webpush.SendNotification` error branch (line 71-73). Overall package coverage is 80%, below the 90% target.
- **Reasoning:** The `WebPusher` makes real HTTP calls to push services, so the error path from the underlying library is difficult to trigger in unit tests without mocking at the HTTP transport level. The `Pusher` interface with `MockPusher` provides the testability seam for consumers.
- **Consequence:** If `webpush.SendNotification` returns an error in production, the error wrapping behavior is untested. However, the logic is trivial (wrap and return), so the risk is negligible.
- **Suggestion:** This is acceptable given the architecture. If higher coverage is desired, consider injecting `http.Client` into `WebPusher` to enable transport-level mocking, but this adds complexity for minimal gain in a personal-use app.

### [MEDIUM] UpsertPushSubscription Generates New ULID on Every Call
- **File:** `backend/internal/store/push_subscriptions.go`
- **Line(s):** L13
- **Perspective:** Performance Engineer
- **Finding:** `UpsertPushSubscription` generates a new ULID (`model.NewULID()`) on every invocation, but on an upsert (conflict on endpoint), only `p256dh` and `auth` are updated -- the new ID is discarded. This means a ULID is generated and thrown away on every re-subscription.
- **Reasoning:** ULID generation is cheap (monotonic clock + random), so this is not a performance concern. However, it is slightly wasteful and could be confusing to readers who expect the generated ID to be used.
- **Consequence:** No functional impact. The test `TestUpsertPushSubscription_UpsertsOnDuplicateEndpoint` correctly verifies that the ID remains the same after upsert, confirming the `ON CONFLICT` clause preserves the original ID.
- **Suggestion:** Consider moving the `id := model.NewULID()` generation to be clearer about its purpose, or adding a brief comment noting it is only used for the INSERT path. No code change required.
