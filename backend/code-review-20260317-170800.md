## Summary

Phase 14 implements `DELETE /api/users/me` for account deletion with full cascade behavior per spec section 2.2. The implementation includes a store-level `DeleteAccount` function with transactional cascade (identify last-parent babies, delete them, delete user-created invites, anonymize `logged_by`/`updated_by` via table-driven list, anonymize `invites.used_by`, delete user record), a handler returning 204, and route registration with auth+CSRF middleware. The code is well-structured, spec-compliant, and tests cover the main behavioral paths. Two issues found.

### [MEDIUM] Missing `rows.Err()` check after iteration loop
- **File:** `internal/store/delete_account.go`
- **Line(s):** L48-L49
- **Perspective:** QA
- **Finding:** After the `rows.Next()` loop (L40-L48), `rows.Close()` is called but `rows.Err()` is never checked. If the iteration terminates due to a database error (not reaching the end of the result set), the error is silently swallowed, and the function proceeds with a potentially incomplete `lastParentBabyIDs` slice.
- **Reasoning:** Go's `database/sql` documentation specifies that `rows.Err()` must be checked after a `rows.Next()` loop to detect errors encountered during iteration. This is a standard Go best practice and a well-known gotcha.
- **Consequence:** A database error during row iteration could cause the function to skip deleting some last-parent babies, leaving orphaned baby records with no linked parents. The deletion would succeed from the caller's perspective (no error returned), but data would be left in an inconsistent state.
- **Suggestion:** Add a `rows.Err()` check after `rows.Close()`:
```go
rows.Close()
if err := rows.Err(); err != nil {
    tx.Rollback()
    return fmt.Errorf("delete account: iterate last-parent babies: %w", err)
}
```
