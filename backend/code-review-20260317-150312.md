## Summary

Phase 5 implements a multi-stage Dockerfile, docker-compose.yml for local development, a .dockerignore file, and adds static file serving to the backend router. The changes largely follow the spec's Dockerfile template, with the Go image version updated to match the project's actual `go.mod` version (1.26). The implementation is nearly correct but has a few issues worth addressing.

### [MEDIUM] Dockerfile adds ENV STATIC_DIR not present in spec
- **File:** `Dockerfile`
- **Line(s):** L19
- **Perspective:** Tech Lead
- **Finding:** The Dockerfile includes `ENV STATIC_DIR=/static` which is not in the spec's Dockerfile (section 11.2). The spec's Dockerfile relies on the Go code default or the docker-compose environment to set this variable. While functionally harmless (and arguably an improvement), it deviates from the spec.
- **Reasoning:** The spec is the source of truth. Adding env vars to the Dockerfile that aren't in the spec creates drift. If the router already defaults to `./static` and docker-compose sets `STATIC_DIR=/static`, this line is redundant when run via compose and only matters for bare `docker run`.
- **Consequence:** Minor spec drift. Could cause confusion if someone compares the Dockerfile to the spec and wonders about the discrepancy. Functionally, this is a safe addition.
- **Suggestion:** Either keep it (it is a reasonable hardening measure for bare `docker run`) and note the deviation, or remove it to match the spec exactly. If keeping it, consider updating the spec to match.

### [MEDIUM] No tests added for static file serving in router.go
- **File:** `backend/internal/handler/router.go`
- **Line(s):** L13-L20
- **Perspective:** QA
- **Finding:** The router gained non-trivial logic (env var lookup, directory existence check, file server registration) but no corresponding tests were added. The project follows strict Red-Green TDD -- production code should not be written without a failing test first.
- **Reasoning:** The CLAUDE.md and project methodology require strict TDD. The static file serving logic has multiple branches (env var set vs. not, directory exists vs. not) that should be covered by tests.
- **Consequence:** Regression risk if this logic is modified later. Violates the project's TDD discipline. Test coverage will drop below the 90% target.
- **Suggestion:** Add tests covering: (1) static dir served when STATIC_DIR env is set and directory exists, (2) graceful no-op when directory does not exist, (3) health endpoint still takes priority over file server for `/health`. Example:
```go
func TestNewMux_StaticFileServing(t *testing.T) {
    dir := t.TempDir()
    os.WriteFile(filepath.Join(dir, "index.html"), []byte("<h1>hi</h1>"), 0644)
    t.Setenv("STATIC_DIR", dir)

    mux := handler.NewMux()
    req := httptest.NewRequest(http.MethodGet, "/index.html", nil)
    rec := httptest.NewRecorder()
    mux.ServeHTTP(rec, req)

    if rec.Code != http.StatusOK {
        t.Fatalf("expected 200, got %d", rec.Code)
    }
}
```

### [LOW] docker-compose.yml uses named volume instead of bind mount
- **File:** `docker-compose.yml`
- **Line(s):** L9-L12
- **Perspective:** DevOps
- **Finding:** The phase description says "mounting a volume for SQLite" and the implementation uses a Docker named volume (`littleliver-data`). For local development, a bind mount (e.g., `./data:/data`) is often more convenient because developers can directly inspect the SQLite file, back it up, or delete it without Docker volume commands.
- **Reasoning:** Named volumes are opaque -- the database file is stored inside Docker's internal volume storage. For a local development compose file, bind mounts give better visibility and control.
- **Consequence:** Developers must use `docker volume inspect` to find the database file location. Minor inconvenience, not a correctness issue.
- **Suggestion:** Consider using a bind mount for local development:
```yaml
volumes:
  - ./data:/data
```

### [LOW] router.go uses os.Getenv at init time rather than accepting config
- **File:** `backend/internal/handler/router.go`
- **Line(s):** L13-L14
- **Perspective:** Tech Lead
- **Finding:** `NewMux()` reads `STATIC_DIR` from the environment directly. This makes the function harder to test (requires `t.Setenv`) and couples router construction to environment variables. A cleaner pattern would accept the static directory as a parameter.
- **Reasoning:** Go convention prefers explicit dependency injection over implicit environment reads. The current approach works but becomes harder to manage as more configuration is needed.
- **Consequence:** Minor testability concern. As the app grows with more env vars (DATABASE_PATH, GOOGLE_CLIENT_ID, etc.), scattered `os.Getenv` calls will become harder to track.
- **Suggestion:** Consider accepting configuration as a parameter:
```go
func NewMux(staticDir string) *http.ServeMux {
    // ...
}
```
This can be deferred to a later refactoring phase since the current approach is functional.
