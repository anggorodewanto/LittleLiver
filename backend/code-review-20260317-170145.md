## Summary

Phase 12 implements invite code generation (`POST /api/babies/:id/invite`) and join flow (`POST /api/babies/join`). The implementation adds `internal/store/invites.go` with `CreateInvite` and `RedeemInvite` functions, `internal/handler/invites.go` with corresponding HTTP handlers, and registers both routes in the router. Test coverage includes: 6-digit code generation, prior code deletion, valid redemption, expired/used/invalid code rejection, already-linked parent friendly response, multi-baby isolation, closed-DB error paths, and a full end-to-end flow test. All 20 tests pass. The code is gofmt-clean with parameterized SQL throughout.

### [MEDIUM] Already-Linked Detection Logic is Fragile and Verbose
- **File:** `backend/internal/handler/invites.go`
- **Line(s):** L124-L147
- **Perspective:** Tech Lead
- **Finding:** The `JoinBabyHandler` uses a convoluted approach to detect the already-linked case: after calling `RedeemInvite`, it calls `IsParentOfBaby`, then queries `used_at` from the database directly. The comments themselves acknowledge confusion ("Actually, we need a different approach"). The handler also contains a raw SQL query (`SELECT used_at FROM invites WHERE code = ?`) which violates the separation between handler and store layers.
- **Reasoning:** The store's `RedeemInvite` already knows whether the user was already linked (it returns `babyID, nil` without marking the code as used). The handler should not need to re-derive this information through additional queries. Raw SQL in the handler layer breaks the established pattern where all DB access goes through the store package.
- **Consequence:** Future changes to the invite schema or logic require updating both the store and handler. The raw SQL query is also silently ignoring errors (`_ = db.QueryRow(...).Scan(&usedAt)`), which could mask bugs.
- **Suggestion:** Have `RedeemInvite` return a richer result that indicates the already-linked case explicitly. For example:
  ```go
  type RedeemResult struct {
      BabyID      string
      AlreadyLinked bool
  }
  ```
  Then the handler simply checks `result.AlreadyLinked` to set the message. This eliminates the extra queries and the raw SQL in the handler.

### [MEDIUM] Declared Error `ErrAlreadyLinked` is Never Used
- **File:** `backend/internal/store/invites.go`
- **Line(s):** L24
- **Perspective:** QA
- **Finding:** `ErrAlreadyLinked` is declared as a sentinel error but is never returned by any function. The `RedeemInvite` function returns `babyID, nil` for the already-linked case instead.
- **Reasoning:** Dead code creates confusion for future developers who might expect `RedeemInvite` to return this error. The test comment on line 276 of `invites_test.go` even says "should get ErrAlreadyLinked" but the actual assertion checks for `err == nil`.
- **Consequence:** Misleading API surface. A developer integrating with `RedeemInvite` might write `errors.Is(err, store.ErrAlreadyLinked)` handling that never triggers, missing the actual already-linked signal.
- **Suggestion:** Either use `ErrAlreadyLinked` as the return value for the already-linked case (and handle it in the handler), or remove the declaration entirely. Using it would simplify the handler's already-linked detection significantly:
  ```go
  // In store:
  if linkCount > 0 {
      tx.Rollback()
      return babyID, ErrAlreadyLinked
  }
  // In handler:
  if errors.Is(err, store.ErrAlreadyLinked) {
      // friendly message
  }
  ```

### [MEDIUM] `CreateInvite` Returns Approximate `CreatedAt` Instead of DB Value
- **File:** `backend/internal/store/invites.go`
- **Line(s):** L84
- **Perspective:** QA
- **Finding:** The returned `Invite` struct sets `CreatedAt: time.Now().UTC()` after the commit, rather than reading the actual `created_at` value from the database (which uses `DEFAULT CURRENT_TIMESTAMP`). There could be a small time drift between the Go timestamp and what SQLite stored.
- **Reasoning:** While the difference is negligible in practice, the returned model does not reflect the actual persisted state. This is inconsistent with the pattern of reading back from the DB after writes.
- **Consequence:** Minor data inconsistency. If `CreatedAt` is ever used for display or comparison, it may not match the DB record.
- **Suggestion:** Either read back the inserted row with a `SELECT` after the insert, or explicitly pass the `created_at` value in the INSERT statement so the Go value and DB value are guaranteed to match.

### [LOW] Collision Retry Loop Has Subtle Control Flow Issue
- **File:** `backend/internal/store/invites.go`
- **Line(s):** L54-L73
- **Perspective:** QA
- **Finding:** When the insert succeeds (`err == nil`), the loop breaks. But if `err != nil` and it is NOT the last attempt, the loop continues. On the last attempt (`i == maxRetries-1`), it rolls back and returns. However, the `err` variable from `generateCode` or `tx.Exec` leaks out of the loop, and the code after the loop proceeds to `tx.Commit()` using whatever `err` state survived. If all 5 attempts fail on insert but the last iteration's error check is reached, the early return at line 71 handles it. But if `generateCode` fails on a non-first iteration, the early return at line 57-59 handles that. The logic is correct but the flow is hard to follow.
- **Reasoning:** The mixed use of `break` on success, early returns on `generateCode` failure, and a last-iteration check makes the loop harder to reason about than necessary.
- **Consequence:** Maintenance burden. A future developer modifying the retry logic could introduce a subtle bug.
- **Suggestion:** Consider a cleaner pattern:
  ```go
  var lastErr error
  for i := 0; i < maxRetries; i++ {
      code, err = generateCode()
      if err != nil {
          tx.Rollback()
          return nil, err
      }
      lastErr = tx.Exec(...)
      if lastErr == nil {
          break
      }
  }
  if lastErr != nil {
      tx.Rollback()
      return nil, fmt.Errorf("create invite: max retries exceeded: %w", lastErr)
  }
  ```

### [LOW] Already-Linked Path Does Not Consume the Invite Code
- **File:** `backend/internal/store/invites.go`
- **Line(s):** L139-L143
- **Perspective:** Product Owner
- **Finding:** When an already-linked parent redeems a code, the code remains unused (`used_at` stays NULL). This means the code is still valid for another user to redeem. This is actually correct behavior per the spec ("show a friendly 'You're already linked to this baby' message (no error)") -- the code should remain available since the already-linked parent did not actually need it.
- **Reasoning:** This is by design and is confirmed by the test `TestRedeemInvite_AlreadyLinked` which asserts `used_by` remains NULL. However, the `tx.Rollback()` at line 142 means the entire transaction is rolled back, which is the correct approach since no mutations need to persist.
- **Consequence:** None -- this is correct. Noting it as a verified design decision.
- **Suggestion:** No change needed. Consider adding a brief comment explaining why the code intentionally remains unused in this path.
