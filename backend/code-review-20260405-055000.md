### [LOW] ImageMagick dependency in E2E test
- **File:** `frontend/e2e/lab-extraction.spec.ts`
- **Line(s):** L25-L29
- **Perspective:** DevOps
- **Finding:** The `createTestImage()` function uses `execSync('convert -size 2x2 xc:white ...')` which requires ImageMagick to be installed on the CI/test machine.
- **Reasoning:** This creates an implicit system dependency that is not declared anywhere. If ImageMagick is not installed, the E2E tests will fail with an unhelpful error about `convert` not being found.
- **Consequence:** E2E tests silently fail in CI environments or on developer machines without ImageMagick installed. The error message would be a raw exec error rather than a clear "missing dependency" message.
- **Suggestion:** Either generate a minimal PNG programmatically using a Buffer (a valid 2x2 PNG is only ~67 bytes and can be hardcoded as a base64 constant), or add a check at the top of the test that skips with a clear message if ImageMagick is not available.

### [LOW] mockClaudeClientSequential is not thread-safe
- **File:** `backend/internal/handler/lab_extract_integration_test.go`
- **Line(s):** L328-L348
- **Perspective:** QA
- **Finding:** The `mockClaudeClientSequential` struct increments `callCount` without any synchronization (no mutex). While the current tests make sequential HTTP requests, this mock could produce data races if tests ever run concurrent requests against it.
- **Reasoning:** Go's race detector (`-race` flag) could flag this if the handler spawns goroutines internally or if the test is refactored to use concurrent requests.
- **Consequence:** Potential false test failures under `-race` flag, or incorrect behavior if concurrent requests are added later.
- **Suggestion:** Add a `sync.Mutex` to protect `callCount` access, consistent with how `ExtractRateLimiter` protects its state:
```go
type mockClaudeClientSequential struct {
    mu        sync.Mutex
    responses []mockResponse
    callCount int
}

func (m *mockClaudeClientSequential) ExtractLabResults(ctx context.Context, images []labextract.ImageData, prompt string) (string, error) {
    m.mu.Lock()
    defer m.mu.Unlock()
    // ...
}
```

### [LOW] TEST_MODE env var in production binary
- **File:** `backend/cmd/server/main.go`
- **Line(s):** L70-L74
- **Perspective:** SecOps
- **Finding:** The `TEST_MODE=1` environment variable causes the production binary to use an in-memory object store instead of R2. This test-only code path lives in the production entrypoint.
- **Reasoning:** While the risk is low (it requires explicitly setting `TEST_MODE=1` and having no R2 config), test-specific code in production binaries is generally undesirable. It could be accidentally activated in production, causing all photo uploads to be lost on restart.
- **Consequence:** If `TEST_MODE=1` is accidentally set in production (e.g., copy-paste from a test env file), all uploaded photos would be stored only in memory and lost on server restart, with no warning beyond a log line.
- **Suggestion:** This is acceptable for a personal-use app. For extra safety, consider adding a log warning like `"WARNING: TEST_MODE=1 — photo data is ephemeral (in-memory store)"` to make the risk more obvious if accidentally enabled.
