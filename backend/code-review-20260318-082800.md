## Summary

Phase 36 implements three periodic cleanup cron jobs: invite cleanup (codes older than 24h), session cleanup (expired sessions), and photo cleanup (unlinked photos older than 24h or orphaned by baby deletion). The implementation includes a `Runner` type that manages a goroutine with a ticker for hourly execution, and individual cleanup functions that are independently testable. All tests pass with 94.1% statement coverage.

**Verdict: This phase is correct and complete.** The implementation faithfully matches the spec, follows project conventions, and meets all proof-of-progress criteria. The issues below are minor observations, none blocking approval.

### [LOW] Log output during tests pollutes test output
- **File:** `backend/internal/cron/cleanup.go`
- **Line(s):** L75, L80, L86
- **Perspective:** QA
- **Finding:** The `log.Printf` calls in `CleanupPhotos` produce output during test runs (visible in test output: "simulated thumbnail delete error", "simulated R2 delete error"), which can obscure actual test failures.
- **Reasoning:** Test output should be clean and only show relevant information. Noisy logs make it harder to spot real problems when scanning test results.
- **Consequence:** Developers may miss genuine warnings buried among expected log lines during test runs.
- **Suggestion:** Accept a `*log.Logger` or use `slog.Logger` as a dependency so tests can inject a no-op logger. Alternatively, this is acceptable for a personal-use app and can be left as-is.

### [LOW] Runner test relies on time.Sleep for synchronization
- **File:** `backend/internal/cron/runner_test.go`
- **Line(s):** L22-L23
- **Perspective:** QA
- **Finding:** `TestRunner_StartStop` uses `time.Sleep(120ms)` to wait for ticker ticks, and `TestRunner_StartStopWithErrors` uses `time.Sleep(80ms)`. These are timing-dependent and could flake on slow CI machines.
- **Reasoning:** Time-based synchronization in tests is inherently fragile. Under heavy load, the goroutine scheduler may not deliver ticks within the expected window.
- **Consequence:** Intermittent test failures on resource-constrained CI runners, leading to flaky pipeline results.
- **Suggestion:** The current approach is pragmatic for a personal-use app. For robustness, the Runner could expose a callback or channel to signal each completed run, allowing tests to synchronize without sleep. Not blocking.

### [LOW] RunAll stops on first error rather than running all jobs
- **File:** `backend/internal/cron/cleanup.go`
- **Line(s):** L96-L123
- **Perspective:** Tech Lead
- **Finding:** `RunAll` returns immediately on the first error (e.g., if invite cleanup fails, session and photo cleanup are skipped entirely).
- **Reasoning:** In a cron context, it is often desirable to run all jobs even if one fails, collecting errors. A transient DB issue on one table should not prevent cleanup of others.
- **Consequence:** If the invites table has a temporary issue, expired sessions and orphaned photos accumulate until the next successful hourly run.
- **Suggestion:** Consider running all three jobs and returning a combined error (e.g., `errors.Join`). However, for an hourly cron on a personal-use app, the current fail-fast approach is acceptable since the next run will catch up.
