### [LOW] Migration adds CHECK constraint not in spec DDL
- **File:** `/home/ab/projects/LittleLiver/backend/migrations/002_feedings.sql`
- **Line(s):** L8
- **Perspective:** Tech Lead
- **Finding:** The migration includes `CHECK (feed_type IN ('breast_milk', 'formula', 'fortified_breast_milk', 'solid', 'other'))` while the spec DDL (section 6) has only `feed_type TEXT NOT NULL` without a CHECK constraint.
- **Reasoning:** This is actually a defensive improvement over the spec -- the enum values match section 3.1 exactly, and DB-level validation provides defense-in-depth alongside the handler-level `ValidFeedType()` check. However, future maintainers should be aware that adding new feed types requires updating both the migration and the Go enum set.
- **Consequence:** If a new feed type is added to the Go enum but not the CHECK constraint (or vice versa), inserts will fail at the database level. Minimal risk for a personal-use app.
- **Suggestion:** This is acceptable as-is. Document in a future migration comment if new feed types are added.

### [LOW] End-of-day boundary off by one second
- **File:** `/home/ab/projects/LittleLiver/backend/internal/store/feedings.go`
- **Line(s):** L125
- **Perspective:** QA
- **Finding:** The `to` date boundary is calculated as `t.Add(24*time.Hour - time.Second)` which produces `23:59:59`, meaning events at `23:59:59.500` would be excluded. The spec says "Both bounds are inclusive -- the range spans from 00:00:00 to 23:59:59."
- **Reasoning:** The spec explicitly states the end boundary is `23:59:59`, so this implementation matches the spec. SQLite DATETIME stores at second granularity by default (no fractional seconds in the format used here), so sub-second entries are not a practical concern.
- **Consequence:** No practical impact given the timestamp format used (`2006-01-02T15:04:05Z` has second granularity).
- **Suggestion:** No change needed. The implementation matches the spec.
