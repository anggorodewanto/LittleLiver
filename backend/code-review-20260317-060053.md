## Summary

Phase 7 implements auth middleware (session validation, sliding window extension, timezone update), CSRF token derivation (HMAC-SHA256) and validation middleware, `GET /api/csrf-token` handler, `GET /api/me` handler, and `POST /auth/logout`. The implementation adds new files in `middleware/` and `handler/` packages and extends the `store/users.go` with `GetUserByID`, `UpdateUserTimezone`, and `GetBabiesByUserID`. All tests pass with >90% coverage across all touched packages.

### [MEDIUM] CSRF middleware not wired into the router
- **File:** `backend/internal/handler/router.go`
- **Line(s):** L42-L50
- **Perspective:** Tech Lead
- **Finding:** The `middleware.CSRF` function is implemented and tested in isolation, but it is never applied in the router. No state-changing endpoint currently goes through CSRF validation. The Phase 7 description says "CSRF validation middleware for state-changing requests" should be part of this phase's deliverables.
- **Reasoning:** While the only current state-changing endpoint is `POST /auth/logout` (which arguably does not need CSRF since it only destroys the session), the phase explicitly calls for wiring the CSRF middleware. Future phases adding state-changing API routes (baby CRUD, metric logging) will need CSRF validation, and the router should be set up to apply it. At minimum, the middleware chain should be prepared so Phase 8+ can rely on it.
- **Consequence:** If subsequent phases forget to wire CSRF, state-changing endpoints will be unprotected against cross-site request forgery attacks. The foundation for the middleware chain is incomplete.
- **Suggestion:** Add the CSRF middleware to the router for all `/api/` routes that are state-changing. For example, create a middleware chain that applies both Auth and CSRF:
```go
csrfMw := middleware.CSRF(cfg.db, cookieName, sessionSecret)
// For future state-changing API routes:
// mux.Handle("POST /api/babies", authMw(csrfMw(http.HandlerFunc(CreateBabyHandler(cfg.db)))))
```
At minimum, document or prepare the wiring pattern so subsequent phases use it consistently.

### [MEDIUM] CSRF middleware re-fetches session from DB (double lookup)
- **File:** `backend/internal/middleware/middleware.go`
- **Line(s):** L87-L97
- **Perspective:** Performance Engineer
- **Finding:** The CSRF middleware independently looks up the session cookie and queries `store.GetSessionByID` (lines 87-97). When CSRF middleware is chained after Auth middleware (which already fetched the session on lines 38-43), this results in two identical database queries per state-changing request.
- **Reasoning:** Every POST/PUT/DELETE request will hit the sessions table twice - once in Auth middleware and once in CSRF middleware. While SQLite is fast for a personal-use app, this is unnecessary overhead and a design smell.
- **Consequence:** Doubled database load on every state-changing request. As metric logging features are added (Phase 15+), this adds up.
- **Suggestion:** Store the session token in context alongside the user (via Auth middleware), then have CSRF middleware retrieve it from context rather than re-querying the database:
```go
// In Auth middleware, also store session token in context:
ctx = context.WithValue(ctx, sessionTokenKey, sess.Token)

// In CSRF middleware, read from context:
sessionToken, _ := r.Context().Value(sessionTokenKey).(string)
expected := CSRFToken(sessionToken, secret)
```

### [MEDIUM] ExtendSession error silently ignored
- **File:** `backend/internal/middleware/middleware.go`
- **Line(s):** L46
- **Perspective:** QA
- **Finding:** The result of `store.ExtendSession(db, sess.ID)` is discarded with `_ = store.ExtendSession(...)`. If session extension fails (e.g., database write error), the request proceeds silently with a session that was not extended.
- **Reasoning:** The spec says "expires_at is reset on each API call" as part of the sliding window behavior. Silently failing to extend means the session could expire sooner than expected, leading to unexpected logouts.
- **Consequence:** Under database pressure or disk-full conditions, sessions may expire prematurely without any indication in logs. Users would be logged out unexpectedly.
- **Suggestion:** At minimum, log the error:
```go
if err := store.ExtendSession(db, sess.ID); err != nil {
    log.Printf("auth: failed to extend session %s: %v", sess.ID, err)
}
```

### [MEDIUM] UpdateUserTimezone error silently ignored
- **File:** `backend/internal/middleware/middleware.go`
- **Line(s):** L58
- **Perspective:** QA
- **Finding:** The error from `store.UpdateUserTimezone(db, user.ID, tz)` is silently discarded with `_ =`. If the timezone update fails, the in-memory user object still gets the new timezone value set (line 59: `user.Timezone = &tz`), creating an inconsistency between memory and database state.
- **Reasoning:** The user object passed down to handlers will show a timezone that was never persisted. If the same request reads back the user data, it will appear correct, but subsequent requests will see the old timezone.
- **Consequence:** Timezone-dependent features (medication scheduling per spec section 6.2) could use stale timezone data, causing notifications to fire at wrong times.
- **Suggestion:** Log the error and skip setting the in-memory value on failure:
```go
if err := store.UpdateUserTimezone(db, user.ID, tz); err != nil {
    log.Printf("auth: failed to update timezone for user %s: %v", user.ID, err)
} else {
    user.Timezone = &tz
}
```

### [LOW] No timezone validation in UpdateUserTimezone
- **File:** `backend/internal/middleware/middleware.go`
- **Line(s):** L56-L59
- **Perspective:** QA
- **Finding:** The `X-Timezone` header value is accepted and stored without validation. Any arbitrary string (e.g., `"not-a-timezone"`, `"<script>alert(1)</script>"`) would be persisted to the database.
- **Reasoning:** The spec says timezone is an IANA timezone string (e.g., `"America/New_York"`). Invalid timezone values could cause errors in downstream features like medication scheduling (Phase 25+) which rely on `time.LoadLocation()`.
- **Consequence:** Invalid timezone strings stored in the database could cause panics or errors when later used with `time.LoadLocation()` for medication notification scheduling.
- **Suggestion:** Validate the timezone using Go's `time.LoadLocation`:
```go
if tz != "" {
    if _, err := time.LoadLocation(tz); err == nil {
        _ = store.UpdateUserTimezone(db, user.ID, tz)
        user.Timezone = &tz
    }
}
```

### [LOW] CSRFTokenHandler not protected by auth middleware
- **File:** `backend/internal/handler/router.go`
- **Line(s):** L47
- **Perspective:** Tech Lead
- **Finding:** `GET /api/csrf-token` is registered as a plain `HandleFunc` without the auth middleware, while `GET /api/me` is wrapped with `authMw`. The CSRF handler does its own session lookup, but it does not extend the session sliding window or update the timezone.
- **Reasoning:** The spec says "expires_at is reset on each API call." If a client only calls `/api/csrf-token` (which it will do before any state-changing request), the session window is not extended, creating inconsistent behavior.
- **Consequence:** Minor inconsistency - the session might not get extended on CSRF token fetch calls. Not a functional bug since state-changing requests (which follow) would go through auth middleware, but it's inconsistent with "every API call" extending the session.
- **Suggestion:** Consider wrapping the CSRF token handler with auth middleware as well, or accept this as an intentional optimization (CSRF fetch does not need the full auth chain since it validates the session itself).
