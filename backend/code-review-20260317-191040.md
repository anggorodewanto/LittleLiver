## Summary

Efficiency-focused review of phases 16-18 production code covering six new metric CRUD modules (abdomen, stools, urine, weights, temperatures) plus modifications to feedings and babies handlers. The code follows a consistent pattern: handler validates, calls store, store executes SQL. The review targets redundant DB round-trips, wasted allocations, and unnecessary work on the hot path.

### [MEDIUM] Redundant SELECT after every INSERT and UPDATE (write-then-read anti-pattern)
- **File:** `backend/internal/store/abdomen.go`, `backend/internal/store/stools.go`, `backend/internal/store/urine.go`, `backend/internal/store/weights.go`, `backend/internal/store/temperatures.go`, `backend/internal/store/feedings.go`
- **Line(s):** abdomen.go L73, L191; stools.go L79, L198; urine.go L68, L186; weights.go L68, L185; temperatures.go L65, L182; feedings.go L102, L239
- **Perspective:** Performance Engineer
- **Finding:** Every `Create*` function does an INSERT then immediately calls `Get*ByID` to SELECT the same row back. Every `Update*` does the same after UPDATE. This doubles the DB round-trips for all write operations (12 functions total, each issuing 2 queries instead of 1).
- **Reasoning:** The data being written is already known to the caller. For INSERT, you generate the ULID, pass all field values, and SQLite sets `created_at`/`updated_at` via defaults. For UPDATE, only `updated_at` changes via `CURRENT_TIMESTAMP`. You can use `RETURNING` (SQLite 3.35+, 2021) or construct the model in-memory using `time.Now().UTC()` for the timestamps.
- **Consequence:** Every create and update request pays an unnecessary extra DB round-trip. Under concurrent use by multiple parents logging metrics simultaneously, this doubles connection pool pressure and latency for all write endpoints.
- **Suggestion:** Use SQLite's `INSERT ... RETURNING` / `UPDATE ... RETURNING` clause to get the row in a single statement:
  ```go
  row := db.QueryRow(
      `INSERT INTO temperatures (id, baby_id, logged_by, timestamp, value, method, notes)
       VALUES (?, ?, ?, ?, ?, ?, ?) RETURNING `+temperatureColumns,
      id, babyID, loggedBy, timestamp, value, method, notes,
  )
  return scanTemperature(row)
  ```

### [MEDIUM] Two separate queries in requireBabyAccess where one would suffice
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L100-L130
- **Perspective:** Performance Engineer
- **Finding:** `requireBabyAccess` runs `store.GetBabyByID` (SELECT from babies) then `store.IsParentOfBaby` (SELECT COUNT from baby_parents). This two-query pattern executes on every single metric endpoint (create, read, update, delete, list) for every metric type -- so it runs on approximately 30 route handlers.
- **Reasoning:** A single query with a JOIN can both load the baby and verify parentage: `SELECT b.* FROM babies b JOIN baby_parents bp ON b.id = bp.baby_id WHERE b.id = ? AND bp.user_id = ?`. This halves the authorization overhead on every request.
- **Consequence:** Every authenticated API call to any metric endpoint pays two DB round-trips just for authorization, before the actual business query even executes.
- **Suggestion:** Add a `store.GetBabyForParent(db, babyID, userID)` function that combines both checks:
  ```go
  func GetBabyForParent(db *sql.DB, babyID, userID string) (*model.Baby, error) {
      row := db.QueryRow(
          `SELECT b.id, b.name, b.sex, b.date_of_birth, b.diagnosis_date, b.kasai_date,
                  b.default_cal_per_feed, b.notes, b.created_at
           FROM babies b
           JOIN baby_parents bp ON b.id = bp.baby_id
           WHERE b.id = ? AND bp.user_id = ?`, babyID, userID,
      )
      return scanBaby(row)
  }
  ```
  Return `sql.ErrNoRows` when not found (baby doesn't exist or user not linked), and 403 vs 404 distinction can be handled by a follow-up `EXISTS` check only when needed.

### [MEDIUM] Extra DB query for default_cal_per_feed in CreateFeeding/UpdateFeeding
- **File:** `backend/internal/store/feedings.go`
- **Line(s):** L70-L77, L81, L205
- **Perspective:** Performance Engineer
- **Finding:** `CreateFeeding` and `UpdateFeeding` call `getDefaultCalPerFeed(db, babyID)` which runs `SELECT default_cal_per_feed FROM babies WHERE id = ?`. However, `requireBabyAccess` in the handler has already loaded the full Baby object (including `DefaultCalPerFeed`) just a few lines earlier. This value is discarded and re-fetched from the DB.
- **Reasoning:** The handler at `feedings.go` L94 already holds `baby.DefaultCalPerFeed` from the authorization step. Passing it to the store function eliminates a redundant query.
- **Consequence:** Every feeding create/update pays three authorization/setup queries (GetBabyByID + IsParentOfBaby + getDefaultCalPerFeed) before the actual write, when one or two would suffice.
- **Suggestion:** Change the store function signatures to accept `defaultCalPerFeed float64` as a parameter instead of looking it up internally:
  ```go
  func CreateFeeding(db *sql.DB, babyID, loggedBy, timestamp, feedType string,
      volumeMl, calDensity *float64, durationMin *int, notes *string,
      defaultCalPerFeed float64) (*model.Feeding, error) {
      calResult, err := model.CalculateCalories(feedType, volumeMl, calDensity, defaultCalPerFeed)
      // ...
  }
  ```
  Then in the handler: `store.CreateFeeding(db, baby.ID, user.ID, ..., baby.DefaultCalPerFeed)`.

### [LOW] Redundant nil-guard after impossible nil assignment in List* pagination
- **File:** `backend/internal/store/abdomen.go`, `backend/internal/store/feedings.go`, `backend/internal/store/stools.go`, `backend/internal/store/urine.go`, `backend/internal/store/weights.go`, `backend/internal/store/temperatures.go`
- **Line(s):** abdomen.go L148-L161; feedings.go L181-L196; stools.go L154-L170; urine.go L143-L158; weights.go L143-L158; temperatures.go L140-L156
- **Perspective:** Tech Lead
- **Finding:** Every `List*WithTZ` function initializes `page.Data` with `make([]T, 0)` on line ~148, then in the `else` branch assigns `page.Data = observations` (which could be nil from `append` on a nil slice), then checks `if page.Data == nil` and re-allocates. The initial `make([]T, 0)` on the page struct is always overwritten and wasted.
- **Reasoning:** Minor allocation waste. The fix is simple: remove the initial `make([]T, 0)` since the nil guard at the bottom already handles it and is the only path that can produce nil.
- **Consequence:** Tiny wasted allocation per list request. Not performance-critical but adds code noise across six files.
- **Suggestion:** Initialize `Data` as nil and rely solely on the final nil guard:
  ```go
  page := &model.MetricPage[model.AbdomenObservation]{}
  // ... pagination logic ...
  if page.Data == nil {
      page.Data = make([]model.AbdomenObservation, 0)
  }
  ```

### [LOW] parseTime tries 5 layouts sequentially on every scanned row
- **File:** `backend/internal/store/sessions.go` (used by all store files)
- **Line(s):** sessions.go L86-L99
- **Perspective:** Performance Engineer
- **Finding:** `parseTime` iterates through 5 format strings on every call. Each `scan*` function calls it 3 times (timestamp, created_at, updated_at). For a list of 50 rows, that is 150 calls to `parseTime`, potentially trying up to 750 `time.Parse` attempts. In practice, SQLite always returns the same format, so only the first layout matches.
- **Reasoning:** Since this codebase controls all writes and uses `CURRENT_TIMESTAMP` / stores timestamps in a consistent format, the common-case layout is predictable. The function could try the most likely format first (which it already does) but could also be replaced with a direct `time.Parse` for the known format, with fallback only on error.
- **Consequence:** Low practical impact since the first format usually matches, but the loop structure and 5-element slice allocation per call is unnecessary overhead on the list hot path.
- **Suggestion:** Use a single `time.Parse` call with the expected format and only fall back to the loop on error:
  ```go
  func parseTime(s string) (time.Time, error) {
      if t, err := time.Parse(time.DateTime, s); err == nil {
          return t, nil
      }
      // fallback for edge cases
      for _, layout := range []string{time.RFC3339, "2006-01-02T15:04:05Z", ...} {
          // ...
      }
  }
  ```

### [LOW] Double conversion in handler list endpoints (store returns []Model, handler converts to []Response)
- **File:** `backend/internal/handler/abdomen.go`, `backend/internal/handler/feedings.go`, `backend/internal/handler/stools.go`, `backend/internal/handler/urine.go`, `backend/internal/handler/weights.go`, `backend/internal/handler/temperatures.go`
- **Line(s):** abdomen.go L148-L154; feedings.go L153-L159; stools.go L152-L157; etc.
- **Perspective:** Performance Engineer
- **Finding:** Every list handler allocates a new `MetricPage[responseType]` with `make([]responseType, 0, len(page.Data))`, then loops through the store's `MetricPage[modelType].Data` converting each element. The store already allocated `MetricPage[modelType]` with its own slice. For 50 items, this means two 50-element slices live simultaneously, and each model struct is copied when dereferenced from `scanX` (which returns `*Model`) into the store's slice, then copied again into the response slice.
- **Reasoning:** For the current page size of 50 and struct sizes involved, this is a minor concern. However, the pattern of `append(observations, *a)` in the store (dereferencing a pointer into a value copy) followed by `toResponse(&page.Data[i])` (taking address of the copy) is one extra copy that could be avoided by storing pointers in the MetricPage.
- **Consequence:** Marginal memory pressure. Acceptable at current scale but worth noting if page sizes increase.
- **Suggestion:** No immediate action required at page size 50. If page sizes grow, consider having the store return `[]*Model` to avoid the dereference-copy in the scan loop.
