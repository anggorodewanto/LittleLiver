## Summary

Phase 34 implements a medication reminder scheduler in `internal/notify/scheduler.go` with comprehensive tests in `scheduler_test.go`. The scheduler queries active medications every minute, checks if any scheduled dose is due (at 0, +15, or +30 minutes), performs suppression checks against `med_logs`, and sends Web Push notifications to all subscribed parents. Tests cover all 8 required TDD scenarios plus additional edge cases. Coverage is 94.0%, all 26 tests pass.

### [MEDIUM] Notification payload lacks structured `scheduled_time` and `medication_id` fields
- **File:** `backend/internal/notify/scheduler.go`
- **Line(s):** L252-L258
- **Perspective:** Product Owner
- **Finding:** The phase spec requires the notification payload to include `scheduled_time` (UTC) and `medication_id` as identifiable fields. Currently, these values are embedded in the human-readable `Body` string (`"Time to give Ursodiol (scheduled 2026-03-18T12:00:00Z, medication_id=med1)"`). The `Payload` struct only has `Title`, `Body`, and `URL` -- there are no structured `Data` fields. The SPEC.md states: "The client passes `scheduled_time` (from the notification payload)" when logging a dose, meaning the frontend must reliably extract `scheduled_time` from the payload.
- **Reasoning:** Parsing `scheduled_time` and `medication_id` out of a human-readable body string is fragile. Web Push payloads commonly include a `data` object alongside `title`/`body` for machine-readable fields. The frontend will need to extract `scheduled_time` to pass it to `POST /api/babies/:id/med-logs`.
- **Consequence:** The frontend cannot reliably extract `scheduled_time` from the notification, potentially causing med-log entries to be created without `scheduled_time` (breaking adherence tracking) or requiring brittle regex parsing of the body text.
- **Suggestion:** Add a `Data` map to the `Payload` struct and include `scheduled_time` and `medication_id` as structured fields:
  ```go
  type Payload struct {
      Title string            `json:"title"`
      Body  string            `json:"body"`
      URL   string            `json:"url,omitempty"`
      Data  map[string]string `json:"data,omitempty"`
  }
  ```
  Then in `buildMedPayload`:
  ```go
  return Payload{
      Title: fmt.Sprintf("Medication Reminder: %s", med.Name),
      Body:  fmt.Sprintf("Time to give %s", med.Name),
      URL:   fmt.Sprintf("/log/med?medication_id=%s", med.ID),
      Data: map[string]string{
          "scheduled_time": scheduledStr,
          "medication_id":  med.ID,
      },
  }
  ```

### [LOW] Custom `contains`/`containsHelper` reimplements `strings.Contains`
- **File:** `backend/internal/notify/scheduler_test.go`
- **Line(s):** L541-L552
- **Perspective:** Tech Lead
- **Finding:** The test file defines custom `contains` and `containsHelper` functions that replicate the exact behavior of `strings.Contains` from the standard library. The `strings` package is not imported in the test file.
- **Reasoning:** Go conventions prefer stdlib usage over reimplementation. This adds unnecessary code and cognitive load for future maintainers.
- **Consequence:** Minor: future developers may be confused about why a standard function was re-implemented, and the custom implementation lacks the optimizations of `strings.Contains`.
- **Suggestion:** Replace with `strings.Contains`:
  ```go
  import "strings"
  // Then use:
  if !strings.Contains(payloadStr, expectedScheduledUTC) {
  ```

### [LOW] Test uses `time.Sleep` for goroutine synchronization
- **File:** `backend/internal/notify/scheduler_test.go`
- **Line(s):** L535
- **Perspective:** QA
- **Finding:** `TestScheduler_StartStop` uses `time.Sleep(10 * time.Millisecond)` to wait for the goroutine to start before calling `Stop()`. While this is a common pattern for simple lifecycle tests and the duration is short, it introduces a minor flakiness risk on very slow CI runners.
- **Reasoning:** Sleep-based synchronization can cause flaky tests under load. A channel-based "started" signal would be deterministic.
- **Consequence:** Rare test flakiness on slow machines. Low risk given the 10ms duration.
- **Suggestion:** Accept as-is for now given the simplicity, or add a "ready" channel to Scheduler that closes after the first ticker setup completes.
