## Summary

Phase 35 adds two integration tests for the medication flow: (1) `TestMedicationNotificationLifecycle` exercising the full notification lifecycle with mock time and mock push sender, and (2) `TestAccountDeletionAnonymizesMedicationTables` verifying that deleting a user anonymizes `logged_by`/`updated_by` in `medications` and `med_logs` tables. The only production code change is adding `"medications"` and `"med_logs"` to `AnonymizeTables` in `delete_account.go`.

Both tests pass. The full test suite passes with no regressions. The implementation is well-structured, uses parameterized SQL, follows early-return style, and covers all scenarios described in the phase spec.

### [LOW] Phase spec deviation: skipped dose suppression behavior
- **File:** `backend/internal/integration/medication_flow_test.go`
- **Line(s):** L232-L239
- **Perspective:** Product Owner
- **Finding:** The phase description states "log as skipped -> verify +30 min suppressed", implying the +30 minute follow-up should NOT fire after logging a skipped dose. However, the test asserts the opposite: the +30 min follow-up DOES fire (sendCount == 1) because `isDoseSuppressed` checks `given_at`, which is NULL for skipped logs.
- **Reasoning:** The test behavior is arguably more clinically correct (a skipped dose means the medication was not administered, so continued reminders make sense). The test includes a clear comment explaining this decision. However, this contradicts the literal phase spec text.
- **Consequence:** If the spec was authoritative and the intent was truly to suppress after skipped, this would be a behavioral mismatch. In practice, the implemented behavior is the safer clinical choice.
- **Suggestion:** Update the phase description in `docs/PHASES.md` to match the actual behavior: "log as skipped -> verify +30 min NOT suppressed (skipped has NULL given_at)". This eliminates future confusion when auditing phases against implementation.

### [LOW] Adherence ratio tested via direct DB query rather than dashboard API
- **File:** `backend/internal/integration/medication_flow_test.go`
- **Line(s):** L243-L269
- **Perspective:** QA
- **Finding:** The phase spec says "verify adherence ratio in dashboard", but the test computes adherence by querying the database directly (`SELECT COUNT(*) FROM med_logs ...`) rather than verifying it through the dashboard API response. The dashboard check at step 11 only verifies `upcoming_meds`, not an adherence ratio field.
- **Reasoning:** If the dashboard endpoint is supposed to return an adherence ratio, this test does not verify that the API computes and returns it correctly. Testing via direct SQL only proves the data exists, not that the API presents it correctly.
- **Consequence:** A bug in the dashboard's adherence calculation would go undetected by this test. However, the dashboard endpoint may not yet expose an adherence field (it may be a future phase), in which case the direct SQL approach is a reasonable interim verification.
- **Suggestion:** If the dashboard API already returns an adherence metric, add an assertion against that field in the dashboard response. If not yet implemented, this is acceptable as-is and can be revisited when the dashboard is enhanced.

### [LOW] New mockPusher per tick step creates isolated send histories
- **File:** `backend/internal/integration/medication_flow_test.go`
- **Line(s):** L161-L239
- **Perspective:** Tech Lead
- **Finding:** Each tick step creates a fresh `mockPusher` and `Scheduler`. While this isolates each step's send count (making assertions straightforward), it also means the test does not verify that a single long-running scheduler correctly handles state across multiple ticks.
- **Reasoning:** Since `Scheduler` is documented as stateless (each tick re-derives what to send), creating fresh instances per step is actually correct and matches the design. The isolation is a feature, not a bug.
- **Consequence:** No real consequence given the stateless design. This is purely a style observation.
- **Suggestion:** No change needed. The pattern is appropriate for the stateless scheduler design.
