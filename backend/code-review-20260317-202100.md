Phase 27 implements `GET /api/babies/{id}/dashboard` with three sections: `summary_cards`, `stool_color_trend`, and `upcoming_meds`. The route is registered in the router with auth and rate-limiting middleware. Store-level functions perform the SQL aggregation. ### [MEDIUM] JSON field names deviate from spec schema
- **File:** `backend/internal/store/dashboard.go`
- **Line(s):** L11-L17
- **Perspective:** Product Owner
- **Finding:** The spec defines `summary_cards` fields as `total_wet_diapers`, `total_stools`, `worst_stool_color`, and `last_temperature`. The implementation uses `wet_diapers`, `stools`, `color_indicator`, and `last_temp` instead.
- **Reasoning:** Frontend code (and any future API consumers) will be built against the spec. Diverging field names create confusion and will require either spec updates or code changes when the frontend integrates.
- **Consequence:** Frontend integration will fail or require ad-hoc mapping, and the API contract becomes unclear. If multiple consumers exist, this becomes a breaking inconsistency.
- **Suggestion:** Either align the JSON tags to match the spec exactly (`total_wet_diapers`, `total_stools`, `worst_stool_color`, `last_temperature`) or update the spec to reflect the chosen names. Additionally, `worst_stool_color` semantically differs from `color_indicator` (most recent vs worst) -- clarify which is intended.

### [MEDIUM] `from` query parameter not validated before use
- **File:** `backend/internal/store/dashboard.go`
- **Line(s):** L40
- **Perspective:** QA
- **Finding:** The `from` parameter is concatenated directly as `from + "T00:00:00Z"` without being parsed or validated. The `to` parameter is properly parsed with `time.Parse`. If a user passes an invalid `from` value (e.g., `from=garbage`), it becomes `garbageT00:00:00Z` in the SQL query, which will silently match zero rows rather than returning an error.
- **Reasoning:** Asymmetric validation is a bug smell. The `to` date is validated but `from` is not, meaning invalid `from` values produce silent empty results instead of a 400 error.
- **Consequence:** Users could unknowingly query with an invalid date range and receive empty dashboard data, making debugging difficult.
- **Suggestion:** Parse `from` the same way `to` is parsed:
```go
fromDate, err := time.Parse("2006-01-02", from)
if err != nil {
    return nil, fmt.Errorf("parse from date: %w", err)
}
fromTime := fromDate.Format("2006-01-02T15:04:05Z")
```

### [MEDIUM] Handler does not validate `from`/`to` format or return 400 on bad input
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L46-L53
- **Perspective:** QA
- **Finding:** The handler passes user-supplied `from` and `to` query parameters directly to the store without format validation. If the store's `time.Parse` fails for `to`, it returns a 500 error rather than a 400. The `from` parameter is not validated at all (see previous finding).
- **Reasoning:** User input validation should happen at the handler layer, returning 400 for malformed dates rather than propagating errors as 500s.
- **Consequence:** Clients receive misleading 500 errors for simple input mistakes, complicating debugging and monitoring (500s trigger alerts in most setups).
- **Suggestion:** Add date format validation in the handler before calling store functions:
```go
if from != "" {
    if _, err := time.Parse("2006-01-02", from); err != nil {
        http.Error(w, "invalid from date format, expected YYYY-MM-DD", http.StatusBadRequest)
        return
    }
}
```

### [LOW] `computeNextDoseAt` uses `time.Now()` directly, making it untestable
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L127
- **Perspective:** Tech Lead
- **Finding:** `computeNextDoseAt` calls `time.Now()` internally. The handler test (`TestDashboardHandler_WithData`) checks that `next_dose_at` exists but cannot assert its exact value because the function depends on wall-clock time.
- **Reasoning:** Time-dependent functions are difficult to test deterministically. Injecting a `now` parameter or using a clock interface would allow precise assertions.
- **Consequence:** Subtle bugs in the countdown logic (e.g., DST transitions, edge of day) cannot be caught by tests.
- **Suggestion:** Accept a `now time.Time` parameter: `func computeNextDoseAt(scheduleTimes []string, tz *string, now time.Time) *string`. This also applies to `GetStoolColorTrend` at `store/dashboard.go:129` which has the same `time.Now()` testability issue.

### [LOW] Multiple sequential SQL queries for summary data
- **File:** `backend/internal/store/dashboard.go`
- **Line(s):** L50-L121
- **Perspective:** Performance Engineer
- **Finding:** `GetDashboardSummary` executes 6 separate SQL queries sequentially (feeds, urine, stools count, stool color, temperature, weight). These could be combined into fewer queries or a single query with subselects.
- **Reasoning:** For a personal-use app with SQLite, this is acceptable. However, each query involves a round trip and lock acquisition. As data grows, this adds latency.
- **Consequence:** Dashboard load time increases linearly with query count. With SQLite's single-writer model, this holds the connection longer than necessary.
- **Suggestion:** Consider combining the date-range queries into a single query using subselects, and the non-date queries into another. Not urgent for a personal-use app but worth noting for future optimization.

### [LOW] No test for `computeNextDoseAt` edge cases
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L117-L166
- **Perspective:** QA
- **Finding:** The `computeNextDoseAt` function handles today's remaining times and tomorrow's first time, but there are no unit tests for: (1) all schedule times already passed today, (2) invalid timezone string, (3) invalid schedule time format, (4) DST transition day. The handler integration test only verifies the field is present.
- **Reasoning:** This is countdown logic that users rely on for medication adherence. Edge cases in time-zone-aware code are notoriously tricky.
- **Consequence:** A bug in next-dose calculation could cause a parent to miss or double-administer medication.
- **Suggestion:** Add focused unit tests for `computeNextDoseAt` with a `now` parameter (per the testability suggestion above) covering these edge cases.
