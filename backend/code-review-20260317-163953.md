## Summary

Phase 10 implements per-session rate limiting middleware (100 req/min) with 429 responses when exceeded. The implementation adds a `RateLimiter` struct with a fixed-window algorithm keyed by session cookie, wires it into the router middleware chain ahead of auth, and includes tests covering all four required behaviors (under-limit success, over-limit 429, window reset, session independence). All tests pass with 92.5% coverage on the middleware package. `go vet` is clean.

The core implementation is correct, clean, and follows project conventions (early return, stdlib-only, no over-engineering). Two issues worth addressing before approval:

### [MEDIUM] Unbounded session map growth (memory leak)

- **File:** `internal/middleware/ratelimit.go`
- **Line(s):** L16-L21
- **Perspective:** Performance Engineer
- **Finding:** The `sessions` map in `RateLimiter` grows indefinitely. Entries are created for every unique session but never removed, even after their window expires. Over time, stale entries accumulate.
- **Reasoning:** While this is a personal-use app with limited users, the map holds pointers to `sessionBucket` structs for every session that has ever made a request. Sessions that logged out or expired months ago still occupy memory. This also means the mutex contention grows as the map gets larger (though negligibly for personal use).
- **Consequence:** In long-running deployments without restarts, memory grows monotonically. For a personal app this is unlikely to matter in practice, but it represents a latent resource leak.
- **Suggestion:** Add a periodic cleanup or lazy eviction. The simplest approach is to evict expired entries during `allow()`:

```go
// In allow(), after acquiring the lock, periodically sweep stale entries.
// Or more simply, just delete the entry when resetting:
if now.Sub(b.windowStart) >= rl.window {
    b.count = 1
    b.windowStart = now
    return true
}
```

The current reset-in-place approach is fine for active sessions. For truly stale sessions (ones that never call `allow()` again), consider a background goroutine or a sweep triggered every N calls. Given this is personal-use, this is non-blocking but worth a TODO comment.

### [LOW] Missing `Retry-After` header on 429 responses

- **File:** `internal/middleware/ratelimit.go`
- **Line(s):** L71-L73
- **Perspective:** QA
- **Finding:** RFC 6585 recommends (though does not require) including a `Retry-After` header in 429 responses to tell clients when they can retry.
- **Reasoning:** Without this header, clients have no programmatic way to know when the rate limit window resets. For a browser-based personal app this is low-impact, but it is good practice and trivial to add.
- **Consequence:** Clients must guess when to retry, which could lead to unnecessary retry storms or poor UX.
- **Suggestion:** Calculate remaining window time and set the header:

```go
if b.count >= rl.limit {
    remaining := rl.window - now.Sub(b.windowStart)
    w.Header().Set("Retry-After", fmt.Sprintf("%d", int(remaining.Seconds())+1))
    http.Error(w, "rate limit exceeded", http.StatusTooManyRequests)
    return
}
```

This requires passing `w` and window info into the middleware layer (which already has access to both). Non-blocking.

### [LOW] Test uses `time.Sleep` for window reset verification

- **File:** `internal/middleware/ratelimit_test.go`
- **Line(s):** L81
- **Perspective:** Tech Lead
- **Finding:** `TestRateLimit_ResetsAfterWindow` uses `time.Sleep(60 * time.Millisecond)` to wait for the rate limit window to expire. While the sleep is short (60ms) and the window is 50ms, this introduces a real-time dependency.
- **Reasoning:** The sleep is kept minimal and the test runs in ~60ms total, which is acceptable. However, injecting a clock interface (e.g., `func() time.Time`) into `RateLimiter` would make this fully deterministic and avoid any flakiness on slow CI runners.
- **Consequence:** Rare test flakiness on heavily loaded CI systems. Currently acceptable given the generous margin (60ms sleep vs 50ms window).
- **Suggestion:** Consider accepting a `nowFunc func() time.Time` in `NewRateLimiter` (defaulting to `time.Now`). Tests can then advance time without sleeping. Non-blocking for this phase.
