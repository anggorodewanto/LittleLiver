## Summary

Review of phases 16-18 production code across 16 files (6 handler files, 6 store files, 2 model files, 1 router file, 1 helper file). The changes add CRUD endpoints for feedings, stools, urine, weights, temperatures, and abdomen observations. The code is well-structured and correctly uses existing helpers (`requireUser`, `requireBabyAccess`, `requireEntryID`, `optionalQuery`, `writeJSON`, `parseTime`, `scanner`). However, there are significant code reuse opportunities -- the same patterns are copy-pasted across all 6 metric types with only table names, column lists, and scan targets varying.

---

### [HIGH] Duplicated List Handler Boilerplate (Timezone Parsing + Pagination Query Params)
- **File:** `handler/feedings.go`, `handler/stools.go`, `handler/urine.go`, `handler/weights.go`, `handler/temperatures.go`, `handler/abdomen.go`
- **Line(s):** Every `List*Handler` function (e.g., feedings.go L130-L144, stools.go L134-L143, urine.go L111-L120, weights.go L121-L130, temperatures.go L124-L133, abdomen.go L130-L139)
- **Perspective:** Tech Lead
- **Finding:** The following 10-line block is copy-pasted identically in all 6 `List*Handler` functions:
  ```go
  from := optionalQuery(r, "from")
  to := optionalQuery(r, "to")
  cursor := optionalQuery(r, "cursor")
  loc := time.UTC
  if tz := r.Header.Get("X-Timezone"); tz != "" {
      if parsed, err := time.LoadLocation(tz); err == nil {
          loc = parsed
      }
  }
  ```
- **Reasoning:** This is identical boilerplate in 6 places. Any change to pagination parameter names or timezone header logic requires updating all 6 files.
- **Consequence:** When adding new metric types (jaundice, bruising, medications, notes -- all defined in enums.go), each will copy this block again, compounding maintenance risk. A bug fix (e.g., changing the timezone header name) could easily miss one file.
- **Suggestion:** Extract to `helpers.go`:
  ```go
  type listParams struct {
      From   *string
      To     *string
      Cursor *string
      Loc    *time.Location
  }

  func parseListParams(r *http.Request) listParams {
      loc := time.UTC
      if tz := r.Header.Get("X-Timezone"); tz != "" {
          if parsed, err := time.LoadLocation(tz); err == nil {
              loc = parsed
          }
      }
      return listParams{
          From:   optionalQuery(r, "from"),
          To:     optionalQuery(r, "to"),
          Cursor: optionalQuery(r, "cursor"),
          Loc:    loc,
      }
  }
  ```

### [HIGH] Duplicated Store List/Pagination Logic (6 near-identical functions)
- **File:** `store/feedings.go`, `store/stools.go`, `store/urine.go`, `store/weights.go`, `store/temperatures.go`, `store/abdomen.go`
- **Line(s):** Each `List*WithTZ` function (~40 lines each): feedings.go L122-L199, stools.go L97-L171, urine.go L86-L159, weights.go L86-L159, temperatures.go L83-L157, abdomen.go L91-L165
- **Perspective:** Tech Lead
- **Finding:** All 6 `List*WithTZ` functions share identical logic: build conditions/args from `babyID`/`from`/`to`/`cursor`, construct a `SELECT ... WHERE ... ORDER BY id DESC LIMIT ?` query, iterate rows with a scan function, and paginate with `limit+1` trick. The only differences are: (a) the table name, (b) the column list, (c) the scan function, and (d) the concrete type parameter.
- **Reasoning:** ~240 lines of nearly identical pagination code across 6 files. This is the single largest code reuse opportunity in this codebase.
- **Consequence:** Every new metric type adds another ~40 lines of copy-paste. Pagination bugs (e.g., the cursor logic or timezone conversion) must be fixed in 6+ places.
- **Suggestion:** Create a generic helper in `store/pagination.go`:
  ```go
  // buildPaginatedQuery constructs a paginated SELECT query with timezone-aware date filtering.
  func buildPaginatedQuery(table, columns, babyID string, from, to, cursor *string, limit int, loc *time.Location) (string, []any, error) {
      var conditions []string
      var args []any
      conditions = append(conditions, "baby_id = ?")
      args = append(args, babyID)
      // ... shared from/to/cursor logic ...
      query := fmt.Sprintf("SELECT %s FROM %s WHERE %s ORDER BY id DESC LIMIT ?", columns, table, strings.Join(conditions, " AND "))
      args = append(args, limit+1)
      return query, args, nil
  }

  // paginateResults applies the limit+1 pagination pattern to a slice of items.
  func paginateResults[T interface{ GetID() string }](items []T, limit int) *model.MetricPage[T] {
      page := &model.MetricPage[T]{Data: make([]T, 0)}
      if len(items) > limit {
          page.Data = items[:limit]
          nc := items[limit-1].GetID()
          page.NextCursor = &nc
      } else if len(items) > 0 {
          page.Data = items
      }
      return page
  }
  ```
  Each `List*WithTZ` then becomes ~10 lines: call `buildPaginatedQuery`, scan rows, call `paginateResults`.

### [HIGH] Duplicated Store Delete Pattern (6 identical functions)
- **File:** `store/feedings.go`, `store/stools.go`, `store/urine.go`, `store/weights.go`, `store/temperatures.go`, `store/abdomen.go`
- **Line(s):** Each `Delete*` function: feedings.go L265-L283, stools.go L202-L220, urine.go L190-L208, weights.go L189-L207, temperatures.go L186-L204, abdomen.go L195-L213
- **Perspective:** Tech Lead
- **Finding:** All 6 `Delete*` functions are structurally identical -- they execute `DELETE FROM <table> WHERE id = ? AND baby_id = ?`, check `RowsAffected`, and return `sql.ErrNoRows` if 0. The only difference is the table name and the error message prefix.
- **Reasoning:** ~18 lines x 6 = ~108 lines of pure duplication.
- **Consequence:** A change to delete semantics (e.g., soft-delete migration) requires modifying 6 functions identically.
- **Suggestion:** Extract to a shared helper:
  ```go
  func deleteByID(db *sql.DB, table, babyID, entryID string) error {
      res, err := db.Exec(
          fmt.Sprintf("DELETE FROM %s WHERE id = ? AND baby_id = ?", table),
          entryID, babyID,
      )
      if err != nil {
          return fmt.Errorf("delete %s: %w", table, err)
      }
      affected, err := res.RowsAffected()
      if err != nil {
          return fmt.Errorf("delete %s: rows affected: %w", table, err)
      }
      if affected == 0 {
          return fmt.Errorf("delete %s: %w", table, sql.ErrNoRows)
      }
      return nil
  }
  ```

### [MEDIUM] Duplicated Handler Get/Delete/Update Error Handling Pattern
- **File:** `handler/feedings.go`, `handler/stools.go`, `handler/urine.go`, `handler/weights.go`, `handler/temperatures.go`, `handler/abdomen.go`
- **Line(s):** Every `Get*Handler`, `Update*Handler`, `Delete*Handler` (e.g., feedings.go L183-L192, stools.go L183-L192, etc.)
- **Perspective:** Tech Lead
- **Finding:** The `sql.ErrNoRows` check + 500 fallback pattern is repeated in every Get, Update, and Delete handler (18 occurrences):
  ```go
  if errors.Is(err, sql.ErrNoRows) {
      http.Error(w, "<thing> not found", http.StatusNotFound)
      return
  }
  log.Printf("<op> <thing>: %v", err)
  http.Error(w, "internal error", http.StatusInternalServerError)
  ```
- **Reasoning:** 18 copies of the same 6-line error handling block with only the entity name varying.
- **Consequence:** Inconsistency risk -- some say "internal error" while others say "failed to update X". A logging format change must touch all 18 spots.
- **Suggestion:** Extract to `helpers.go`:
  ```go
  func handleStoreError(w http.ResponseWriter, err error, notFoundMsg, logPrefix string) {
      if errors.Is(err, sql.ErrNoRows) {
          http.Error(w, notFoundMsg, http.StatusNotFound)
          return
      }
      log.Printf("%s: %v", logPrefix, err)
      http.Error(w, "internal error", http.StatusInternalServerError)
  }
  ```

### [MEDIUM] Duplicated Timestamp Validation in Every Request Validator
- **File:** `handler/feedings.go`, `handler/stools.go`, `handler/urine.go`, `handler/weights.go`, `handler/temperatures.go`, `handler/abdomen.go`

  if req.Timestamp == "" {

  }
  if _, err := time.Parse(model.DateTimeFormat, req.Timestamp); err != nil {

- **Line(s):** Every `scan*` function (e.g., feedings.go L29-L40, stools.go L31-L42, etc.)
- **Perspective:** Tech Lead
- **Finding:** Every scan function has the same triplet of `parseTime` calls for `timestamp`, `created_at`, and `updated_at`, each with identical error wrapping:

  x.Timestamp, err = parseTime(tsStr)
  if err != nil { return nil, fmt.Errorf("parse timestamp: %w", err) }
  x.CreatedAt, err = parseTime(createdStr)

  ```
- **Reasoning:** 6 copies x 9 lines = ~54 lines of identical time-parsing boilerplate.
- **Consequence:** If the time-parsing logic changes (e.g., adding timezone awareness), 6 files need modification.
- **Suggestion:** Extract a helper struct or function:

      Timestamp, CreatedAt, UpdatedAt time.Time
  }

  func parseMetricTimes(tsStr, createdStr, updatedStr string) (metricTimes, error) {
      ts, err := parseTime(tsStr)
      if err != nil { return metricTimes{}, fmt.Errorf("parse timestamp: %w", err) }

  ```

### [MEDIUM] Duplicated NullString-to-Pointer Conversion in Scan Functions
- **File:** `store/feedings.go`, `store/stools.go`, `store/urine.go`, `store/weights.go`, `store/temperatures.go`, `store/abdomen.go`
- **Line(s):** Every `scan*` function's nullable field handling blocks
- **Perspective:** Tech Lead
- **Finding:** Every scan function repeats the pattern `if x.Valid { field = &x.String }` for `sql.NullString` and `if x.Valid { field = &x.Float64 }` for `sql.NullFloat64`. The `updatedBy` and `notes` nullable fields appear in all 6 scan functions.
- **Reasoning:** While each scan function has different nullable fields, the conversion pattern is always the same. At minimum `updatedBy` and `notes` are universal.
- **Consequence:** Minor -- this is idiomatic Go. However, a small helper would reduce visual noise.
- **Suggestion:** Add nullable conversion helpers:
  ```go
  func nullStr(ns sql.NullString) *string {
      if ns.Valid { return &ns.String }
      return nil
  }
  func nullFloat(nf sql.NullFloat64) *float64 {
      if nf.Valid { return &nf.Float64 }
      return nil
  }
  ```

### [MEDIUM] Duplicated List-Handler Response Mapping Loop
- **File:** `handler/feedings.go`, `handler/stools.go`, `handler/urine.go`, `handler/weights.go`, `handler/temperatures.go`, `handler/abdomen.go`
- **Line(s):** Every `List*Handler` post-query block (e.g., feedings.go L153-L159, stools.go L152-L158, etc.)
- **Perspective:** Tech Lead
- **Finding:** Every List handler has the same pattern:
  ```go
  resp := model.MetricPage[XResponse]{
      Data:       make([]XResponse, 0, len(page.Data)),
      NextCursor: page.NextCursor,
  }
  for i := range page.Data {
      resp.Data = append(resp.Data, toXResponse(&page.Data[i]))
  }
  ```
- **Reasoning:** 6 copies of the same map-and-rebuild pattern. Since Go 1.21 has `slices.Map` patterns and the project already uses generics (`MetricPage[T]`), this could be consolidated.
- **Consequence:** Boilerplate noise. Each new metric type copies this block.
- **Suggestion:** Add a generic mapping helper in `helpers.go`:
  ```go
  func mapMetricPage[M any, R any](page *model.MetricPage[M], convert func(*M) R) model.MetricPage[R] {
      resp := model.MetricPage[R]{
          Data:       make([]R, 0, len(page.Data)),
          NextCursor: page.NextCursor,
      }
      for i := range page.Data {
          resp.Data = append(resp.Data, convert(&page.Data[i]))
      }
      return resp
  }
  ```

### [LOW] Redundant Nil Check on page.Data After Initialization
- **File:** `store/feedings.go`, `store/stools.go`, `store/urine.go`, `store/weights.go`, `store/temperatures.go`, `store/abdomen.go`
- **Line(s):** feedings.go L194-L196, stools.go L166-L168, urine.go L155-L157, weights.go L155-L157, temperatures.go L152-L154, abdomen.go L160-L162
- **Perspective:** QA
- **Finding:** Each `List*WithTZ` function initializes `page.Data = make([]T, 0)` at creation, then checks `if page.Data == nil` at the end. The `page.Data` is only set to `nil` if `observations` is `nil` and the else branch runs (`page.Data = observations` where observations is nil since `append` to a nil slice was used). However, the initial `make([]T, 0)` is immediately overwritten in both branches of the `if len > limit` check.
- **Reasoning:** The initial `make([]T, 0)` on the MetricPage is wasted -- it is immediately overwritten. The nil check at the bottom is the real safety net.
- **Consequence:** No bug, but the dead initial allocation is confusing and hides the actual nil-safety logic. When refactored to a shared helper, this should be cleaned up.
- **Suggestion:** Remove the initial `make([]T, 0)` from the struct literal and keep only the final nil guard, or better yet, consolidate this into the shared `paginateResults` helper from finding #2.

### [LOW] Router Registration Boilerplate
- **File:** `handler/router.go`
- **Line(s):** L83-L123
- **Perspective:** Tech Lead
- **Finding:** Each metric CRUD block in the router follows an identical 5-line pattern (POST with csrf, GET list, GET by ID, PUT with csrf, DELETE with csrf). The middleware chain `rateMw(authMw(csrfMw(...)))` vs `rateMw(authMw(...))` is the only variation (state-changing vs read-only).
- **Reasoning:** 6 metric types x 5 routes = 30 lines of highly repetitive registration. Each new metric type adds another 5-line block.
- **Consequence:** Copy-paste risk when adding new metrics -- forgetting CSRF middleware on a state-changing route would be a security issue.
- **Suggestion:** Extract a `registerMetricCRUD` helper:
  ```go
  func registerMetricCRUD(mux *http.ServeMux, path string, db *sql.DB, rateMw, authMw, csrfMw func(http.Handler) http.Handler,
      create, list, get, update, delete http.HandlerFunc) {
      mux.Handle("POST "+path, rateMw(authMw(csrfMw(create))))
      mux.Handle("GET "+path, rateMw(authMw(list)))
      mux.Handle("GET "+path+"/{entryId}", rateMw(authMw(get)))
      mux.Handle("PUT "+path+"/{entryId}", rateMw(authMw(csrfMw(update))))
      mux.Handle("DELETE "+path+"/{entryId}", rateMw(authMw(csrfMw(delete))))
  }
  ```
