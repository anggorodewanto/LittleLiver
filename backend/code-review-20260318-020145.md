## Summary

Phase 39 adds chart rendering (stool color distribution, weight with WHO percentiles, lab trends) as PNG images using gonum/plot, a photo appendix section that fetches thumbnails from object storage, and integrates both into the PDF report. The `ObjectStore` interface gains a `Get` method, the `MemoryStore` implements it (with the old `Get` renamed to `GetWithMeta`), and the report handler is updated to accept `ObjectStore`. Tests cover all chart types (valid PNG output, empty input, edge cases), full PDF generation with charts and photos, and handler-level tests. Coverage is 90.7% on the report package and 97.1% on storage. All tests pass across the entire test suite (13 packages).

### [HIGH] Phantom data points at origin when weight timestamp parsing fails
- **File:** `backend/internal/report/charts.go`
- **Line(s):** L145-L154
- **Perspective:** QA
- **Finding:** The `pts` slice is pre-allocated with `make(plotter.XYs, len(weights))`, initializing all points to `{X: 0, Y: 0}`. When `parseTimestampMulti` fails and `continue` is executed, the point at index `i` retains its zero value, creating a phantom scatter point at the chart origin (age 0 days, weight 0 kg).
- **Reasoning:** This is the same pattern correctly handled in `renderLabTrendsChart` (line 197), where `pts` is initialized with `make(plotter.XYs, 0, len(entries))` and points are appended only on success. The weight chart does not follow this consistent pattern.
- **Consequence:** A single unparseable timestamp in weight data produces a visually misleading data point at (0, 0) on the weight chart, which could confuse a clinician reviewing the report. A zero-weight data point is clinically alarming and could trigger unnecessary concern.
- **Suggestion:** Use the append pattern consistent with `renderLabTrendsChart`:
  ```go
  pts := make(plotter.XYs, 0, len(weights))
  for _, w := range weights {
      ts, parseErr := parseTimestampMulti(w.Timestamp)
      if parseErr != nil {
          continue
      }
      ageDays := ts.Sub(dobTime).Hours() / 24
      pts = append(pts, plotter.XY{X: ageDays, Y: w.WeightKg})
  }
  if len(pts) == 0 {
      // skip scatter if no points parsed
  }
  ```

### [MEDIUM] Unused `stoolColors` map -- dead code
- **File:** `backend/internal/report/charts.go`
- **Line(s):** L29-L37
- **Perspective:** Tech Lead
- **Finding:** The `stoolColors` map, which maps color scores 1-7 to display colors, is defined but never referenced. The bar chart at line 96 uses a single uniform color (`color.RGBA{R: 100, G: 150, B: 200, A: 255}`) for all bars.
- **Reasoning:** The phase specifies a "stool color distribution chart." Coloring each bar by its stool score would make the chart more clinically useful and is likely the intended behavior. The map exists but the wiring is missing.
- **Consequence:** Dead code in the codebase. The stool chart loses its primary visual distinction -- bars for "pale/clay" (score 1, clinically concerning) look identical to "brown" (score 7, normal), reducing the chart's diagnostic value for hepatologists.
- **Suggestion:** Either use per-bar coloring (gonum/plot supports custom bar coloring) or remove the unused map. If per-bar coloring is deferred, remove the dead code and track it as a follow-up.

### [MEDIUM] Photo appendix fetches thumbnails via `Get` rather than signed URLs as spec states
- **File:** `backend/internal/report/report.go`
- **Line(s):** L156-L208
- **Perspective:** Tech Lead
- **Finding:** The phase description states "fetch thumbnails from R2 (using signed URLs)" but the implementation uses `objStore.Get(ctx, key)` to fetch raw bytes directly. The `Get` method was added to the `ObjectStore` interface specifically for this use case.
- **Reasoning:** For server-side PDF embedding, fetching bytes directly via `Get` is actually the correct approach -- signed URLs are designed for browser-side access. The implementation is arguably better than what the phase described.
- **Consequence:** No functional impact. This is a spec/implementation mismatch in description only. The approach is sound.
- **Suggestion:** No code change needed. Consider noting this deviation in phase documentation for accuracy.

### [LOW] No dedicated test for `detectImageExtension`
- **File:** `backend/internal/report/report.go`
- **Line(s):** L397-L405
- **Perspective:** QA
- **Finding:** The `detectImageExtension` function handles PNG, JPEG, and a default fallback, but has no dedicated unit tests. It is only exercised indirectly through the full PDF generation test with PNG photos. The JPEG detection path and the edge case of very short input (`len(data) < 2`) are not directly tested.
- **Reasoning:** While overall coverage meets the 90% threshold, this helper function's edge cases could regress silently. The TDD methodology calls for tests covering every behavior.
- **Consequence:** Regressions in image format detection (especially JPEG) could go unnoticed.
- **Suggestion:** Add a small table-driven test covering PNG bytes, JPEG bytes, unknown format, and short/empty input.
