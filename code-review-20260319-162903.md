## Summary

Phase 47 adds four new Chart.js chart components to the Trends view: AbdomenGirthChart (line), FeedingChart (daily kcal bar), DiaperChart (stacked bar for wet + stool counts), and LabTrendsChart (multi-line with normal range shading). The TrendsView parent component is updated to render all seven charts. All 274 frontend tests pass with 98.17% statement coverage. The implementation is clean, follows project conventions, and covers all four required chart types with appropriate tests.

### [HIGH] Lab value type mismatch between backend and frontend
- **File:** `frontend/src/lib/components/LabTrendsChart.svelte`
- **Line(s):** L5-L9
- **Perspective:** QA
- **Finding:** The `LabDataPoint` interface declares `value: number`, but the backend `LabTrendEntry` struct defines `Value` as `string` (Go type `string`, JSON serialized as a string). The database schema (`lab_results.value TEXT NOT NULL`) stores lab values as text because lab results can include non-numeric values. The frontend will receive `"3.2"` (string) rather than `3.2` (number).
- **Reasoning:** While JavaScript's type coercion may silently convert numeric strings to numbers in Chart.js data points (e.g., `"3.2"` becomes `3.2` in a math context), this is fragile. If a lab value is truly non-numeric (e.g., "positive", "reactive"), the chart will render `NaN` data points silently. The TypeScript interface is also misleading to future developers.
- **Consequence:** Charts may render incorrectly or show NaN data points for non-numeric lab results. TypeScript strict mode provides false confidence since the runtime type differs from the declared interface.
- **Suggestion:** Change the interface to `value: string` and explicitly parse numeric values:
  ```typescript
  interface LabDataPoint {
      timestamp: string;
      test_name: string;
      value: string;
      unit: string;
  }
  // In the mapping:
  data: points
      .map((p) => ({ x: new Date(p.timestamp).getTime(), y: parseFloat(p.value) }))
      .filter((p) => !isNaN(p.y))
  ```

### [MEDIUM] Hardcoded normal ranges may drift from clinical standards
- **File:** `frontend/src/lib/components/LabTrendsChart.svelte`
- **Line(s):** L23-L31
- **Perspective:** Product Owner
- **Finding:** Normal ranges for lab values (total_bilirubin, ALT, AST, GGT, albumin, INR, platelets) are hardcoded in the frontend component. The spec's `lab_results` table has a `normal_range` column per entry, suggesting the backend may store per-result normal ranges.
- **Reasoning:** Pediatric normal ranges vary by age and institution. Hardcoding them in the frontend means they cannot be updated without a code change, and they may conflict with ranges provided by the baby's specific lab.
- **Consequence:** Parents may see misleading normal range shading that differs from their hepatologist's reference ranges, potentially causing unnecessary alarm or false reassurance.
- **Suggestion:** Consider using the `normal_range` field from the backend data when available, falling back to the hardcoded defaults. This is acceptable for v1 as a personal-use app but should be noted as technical debt.

### [MEDIUM] AbdomenGirthChart renders empty chart with no data guard
- **File:** `frontend/src/lib/components/AbdomenGirthChart.svelte`
- **Line(s):** L18-L55
- **Perspective:** QA
- **Finding:** When `data` is an empty array, the component still creates a full Chart.js instance with an empty dataset. Unlike `LabTrendsChart` which has an explicit empty-data guard (line 37-44), `AbdomenGirthChart`, `FeedingChart`, and `DiaperChart` all create charts with configured axes and labels even when there is no data to show.
- **Reasoning:** While this does not crash (as verified by tests), it presents empty chart axes with labels like "Date" and "Girth (cm)" to the user with no data, which is a suboptimal UX.
- **Consequence:** Users see empty chart frames with axis labels and no indication that there is simply no data for this metric, which may be confusing.
- **Suggestion:** Consider adding a "No data available" message when the data array is empty, or render the canvas only when data exists. This is a low-priority UX improvement.

### [LOW] LabTrendsChart uses generic `Record<string, unknown>[]` for datasets
- **File:** `frontend/src/lib/components/LabTrendsChart.svelte`
- **Line(s):** L46
- **Perspective:** Tech Lead
- **Finding:** The `datasets` array is typed as `Record<string, unknown>[]` instead of using Chart.js's `ChartDataset` type or a more specific interface.
- **Reasoning:** This bypasses TypeScript's type checking for the dataset configuration objects, making it easier to introduce subtle configuration errors (e.g., misspelled property names) without compile-time detection.
- **Consequence:** Future modifications to the dataset configuration could introduce runtime errors that TypeScript would otherwise catch.
- **Suggestion:** Use a more specific type, such as Chart.js's `ChartDatasetCustomTypesPerDataset` or define a local interface matching the expected shape.

### [LOW] TrendsView does not handle `unit` field for lab trend display
- **File:** `frontend/src/lib/components/LabTrendsChart.svelte`
- **Line(s):** L96-L110
- **Perspective:** Product Owner
- **Finding:** The `LabDataPoint` interface includes a `unit` field, but it is never used in the chart configuration. The y-axis label is simply "Value" with no unit indication. When multiple lab tests with different units (mg/dL, U/L, etc.) are plotted on the same y-axis, the chart becomes difficult to interpret.
- **Reasoning:** Mixing units on a single y-axis (e.g., bilirubin in mg/dL alongside GGT in U/L) makes the chart misleading since the scales are fundamentally different.
- **Consequence:** Parents or clinicians viewing the chart may misinterpret relative values of different lab tests due to incompatible units sharing one axis.
- **Suggestion:** This is a known limitation of multi-line lab charts. For v1, consider at minimum including units in the dataset labels (e.g., "total_bilirubin (mg/dL)") so users understand the scale differences. A future improvement could use separate y-axes or separate charts per unit.

### [LOW] Test mock data uses `by_type` as Record with number values
- **File:** `frontend/src/tests/feeding-chart.test.ts`
- **Line(s):** L13-L15
- **Perspective:** QA
- **Finding:** The `by_type` field in mock feeding data uses `number` values (e.g., `breast_milk: 300`), but the backend `FeedingDailyEntry.ByType` is a `map[string]float64` representing volume in mL. The frontend interface declares it as `Record<string, number>` which matches, but the `FeedingChart` component does not actually use the `by_type` field at all.
- **Reasoning:** The `by_type` data is available but unused. The chart only shows total calories, not a breakdown by feed type.
- **Consequence:** No functional issue, but the available feed type breakdown data is unused, which could be valuable for parents tracking breast milk vs formula ratios.
- **Suggestion:** This is fine for the current phase scope (which only requires "daily kcal" bar chart). Consider using `by_type` in a future enhancement for stacked bars showing caloric contribution by feed type.
