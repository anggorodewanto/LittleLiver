## Summary

Phases 25-27 add medication CRUD, medication administration logging (med-logs), and a dashboard aggregation endpoint. The changes introduce 6 new production files (3 handlers, 3 store files), 2 migration files, and model type additions. The code generally follows existing project conventions but has several copy-paste deviations, a duplicated JSON parsing pattern, inconsistent use of helper functions, and an off-by-one bug in the `to` date filter for med-logs.

### [HIGH] Med-log `to` filter is exclusive of the target day (off-by-one)
- **File:** `backend/internal/store/med_logs.go`
- **Line(s):** L125-L133
- **Perspective:** QA
- **Finding:** The `to` date filter appends `T00:00:00Z` directly, making the filter `created_at < toT00:00:00Z` -- this means the entire `to` day is excluded. In contrast, `GetDashboardSummary` in `backend/internal/store/dashboard.go` (L46-L49) correctly adds 24 hours to make `to` inclusive of the full day.
- **Reasoning:** Every other date-range filter in this codebase treats `to` as inclusive of the whole day. The dashboard store does it correctly. The med-log store does not, leading to silently missing data when filtering by date.
- **Consequence:** Users querying med-logs for a specific day (e.g., `from=2026-03-17&to=2026-03-17`) will get zero results because the filter becomes `created_at < 2026-03-17T00:00:00Z`, excluding the entire day.
- **Suggestion:** Parse the `to` date and add 24 hours, matching the dashboard pattern:
  ```go
  toDate, err := time.Parse(model.DateFormat, *to)
  if err != nil {
      return nil, fmt.Errorf("parse to date: %w", err)
  }
  toTime := toDate.Add(24 * time.Hour).Format("2006-01-02T15:04:05Z")
  conditions = append(conditions, "created_at < ?")
  args = append(args, toTime)
  ```

### [MEDIUM] Duplicated schedule JSON parsing across dashboard and medications handlers
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L87-L103
- **Perspective:** Tech Lead
- **Finding:** The pattern of parsing `m.Schedule` from a JSON string into `[]string`, with a nil/empty guard and fallback to `[]string{}`, is duplicated verbatim between `toMedicationResponse` in `medications.go` (L56-L63) and the dashboard handler loop (L87-L103). Both also silently log and continue on parse error.
- **Reasoning:** This is a textbook copy-paste-with-slight-variation pattern. The logic is identical but lives in two separate places. If the schedule format changes, both must be updated.
- **Consequence:** Divergent behavior if one location is updated but not the other. Maintenance burden increases as more consumers of medication schedule data are added.
- **Suggestion:** Extract a shared helper function, e.g., `parseScheduleTimes(schedule *string) []string` in the handler package or on the model, and call it from both locations.

### [MEDIUM] Med-log handlers inline `r.PathValue("entryId")` instead of using `requireEntryID`
- **File:** `backend/internal/handler/med_logs.go`
- **Line(s):** L202-L206, L231-L235, L271-L275
- **Perspective:** Tech Lead
- **Finding:** All three med-log handlers (Get, Update, Delete) manually extract the entry ID via `r.PathValue("entryId")` with an inline empty check and `http.Error`. Every other handler in the codebase (feedings, stools, temperatures, weights, urine, lab_results, general_notes, bruising, abdomen, skin_observations) uses the shared `requireEntryID(w, r)` helper from `helpers.go`.
- **Reasoning:** The project has an established pattern for extracting path parameters via `requireEntryID`. Deviating from it creates inconsistency and duplicates the same 4-line block three times within a single file.
- **Consequence:** If the entry ID extraction logic needs to change (e.g., adding ULID format validation), the med-log handlers will be missed. Inconsistent error messages across endpoints.
- **Suggestion:** Replace all three inline blocks with `entryID, ok := requireEntryID(w, r)`.

### [MEDIUM] Medication handlers inline path value extraction instead of extracting a helper
- **File:** `backend/internal/handler/medications.go`
- **Line(s):** L180-L184, L209-L213
- **Perspective:** Tech Lead
- **Finding:** `GetMedicationHandler` and `UpdateMedicationHandler` both inline `r.PathValue("medId")` with an empty check. While the path parameter name differs from `entryId`, the pattern is identical to `requireEntryID`. The same 4-line block is repeated twice.
- **Reasoning:** Either a `requirePathValue(w, r, "medId")` generic helper should be created, or the existing `requireEntryID` should be generalized to accept a parameter name. Other handlers in the codebase handle a single path parameter shape (`{entryId}`), but medications introduce `{medId}` as a new variant.
- **Consequence:** Continued proliferation of inline path-value-checking boilerplate as more resource types are added.
- **Suggestion:** Create a `requirePathParam(w http.ResponseWriter, r *http.Request, name string) (string, bool)` helper that generalizes `requireEntryID`, then use it for both `medId` and `entryId` extractions.

### [MEDIUM] `DashboardSummary` and `UpcomingMed` types defined in store package leak into handler response
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L31-L35
- **Perspective:** Tech Lead
- **Finding:** `dashboardResponseJSON` directly embeds `*store.DashboardSummary` and `[]store.StoolColorEntry` as JSON response fields. This means the store layer's struct field names and JSON tags directly dictate the API contract. All other handlers in this codebase define their own response types and use explicit `toXxxResponse` mapping functions (e.g., `toFeedingResponse`, `toMedicationResponse`).
- **Reasoning:** This couples the API response shape to the store implementation. If the store struct changes for internal reasons (e.g., renaming a field for clarity), the API contract silently changes too. Other endpoints in this same diff (medications, med-logs) correctly use separate response types.
- **Consequence:** A refactor of the dashboard store types would unintentionally break the API contract with no compile-time detection. Inconsistent architecture pattern within the same set of changes.
- **Suggestion:** Define dedicated `dashboardSummaryResponse` and `stoolColorTrendResponse` types in the handler layer and map from the store types, consistent with every other handler in the codebase.

### [LOW] Hard-coded date format strings instead of using `model.DateFormat`
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L45
- **Perspective:** Tech Lead
- **Finding:** The dashboard handler uses the literal string `"2006-01-02"` for date formatting. The model package already defines `DateFormat` (or similar) for this purpose, and `store/med_logs.go` (L117, L127) uses `model.DateFormat` for the same pattern.
- **Reasoning:** The codebase has an established constant for date format strings. Inconsistent usage means some code will be missed if the format ever needs to change.
- **Consequence:** Minor maintenance burden; if date format constants are updated, this file would be overlooked.
- **Suggestion:** Replace `"2006-01-02"` with `model.DateFormat` in `dashboard.go`, and do the same in `store/dashboard.go` (L46, L132).
