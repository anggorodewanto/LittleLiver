## Summary

Phases 28-30 add chart data series (7 query functions), active alerts (4 alert sub-functions + IsDoseCovered), and wire them into the DashboardHandler. The integration test file provides good end-to-end coverage. The code is functional and readable, but contains several code quality issues: duplicated response types, a redundant mapping layer, repetitive handler boilerplate, stringly-typed alert identifiers, and repetitive query function structure.

### [MEDIUM] Redundant `alertResponse` type duplicates `store.Alert`
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L54-L60, L174-L183
- **Perspective:** Tech Lead
- **Finding:** `alertResponse` (handler) has identical fields and JSON tags to `store.Alert`. The handler then performs a field-by-field copy in a loop at L174-L183 that adds no value.
- **Reasoning:** This is a pure redundant mapping layer. Neither type adds, removes, renames, or transforms any field. If the response shape ever diverges from the store shape, the mapping would be justified, but right now it is dead weight.
- **Consequence:** Every future field added to Alert must be added in three places (store.Alert, alertResponse, and the mapping loop), increasing the chance of a missed-field bug.
- **Suggestion:** Remove `alertResponse` and use `[]store.Alert` directly in `dashboardResponseJSON`. The JSON tags on `store.Alert` already produce the correct output. Delete the mapping loop and assign `storeAlerts` directly (with the nil-to-empty-slice guard):
```go
if storeAlerts == nil {
    storeAlerts = make([]store.Alert, 0)
}
// ...
ActiveAlerts: storeAlerts,
```

### [MEDIUM] DashboardHandler has 8 sequential fetch-error-return blocks that are nearly identical
- **File:** `backend/internal/handler/dashboard.go`
- **Line(s):** L117-L164
- **Perspective:** Tech Lead
- **Finding:** Lines 117-164 contain 7 chart-data fetches (plus 1 alerts fetch) that each follow the identical pattern: `result, err := store.GetX(db, baby.ID, from, to); if err != nil { log.Printf("label: %v", err); http.Error(w, "internal error", 500); return }`. This is 7 copies of the same 5-line block.
- **Reasoning:** While Go is explicit about error handling, this level of repetition makes the handler harder to scan and increases the risk of a copy-paste error (wrong log label, wrong variable name).
- **Consequence:** Maintenance burden grows linearly with each new chart series. A subtle copy-paste bug (e.g., logging "temperature series" but returning `diaperDaily`) would be hard to spot in review.
- **Suggestion:** Extract a small helper closure inside the handler that captures `w`:
```go
var fetchErr error
fetch := func(label string, fn func() error) {
    if fetchErr != nil {
        return
    }
    if err := fn(); err != nil {
        log.Printf("%s: %v", label, err)
        http.Error(w, "internal error", http.StatusInternalServerError)
        fetchErr = err
    }
}

var feedingDaily []store.FeedingDailyEntry
fetch("feeding daily", func() error {
    var err error
    feedingDaily, err = store.GetFeedingDaily(db, baby.ID, from, to)
    return err
})
// ... repeat for others ...
if fetchErr != nil {
    return
}
```
This reduces each fetch to 4 lines and makes the error-handling pattern impossible to get wrong.

### [MEDIUM] Stringly-typed alert types should be constants
- **File:** `backend/internal/store/alerts.go`
- **Line(s):** L128, L160, L202, L294
- **Perspective:** Tech Lead
- **Finding:** Alert type strings `"acholic_stool"`, `"fever"`, `"jaundice_worsening"`, and `"missed_medication"` are used as raw string literals in alerts.go and then checked by string comparison in at least 3 test files (alerts_test.go, dashboard_test.go, dashboard_aggregation_test.go).
- **Reasoning:** Stringly-typed identifiers are fragile. A typo in any of the ~30+ comparison sites would silently produce a false-green test or broken alert logic. Constants provide compile-time safety and IDE autocomplete.
- **Consequence:** A future developer adding a new alert type or checking for an existing one could misspell the string, resulting in alerts being silently dropped or tests passing incorrectly.
- **Suggestion:** Define constants in `store/alerts.go`:
```go
const (
    AlertTypeAcholicStool      = "acholic_stool"
    AlertTypeFever             = "fever"
    AlertTypeJaundiceWorsening = "jaundice_worsening"
    AlertTypeMissedMedication  = "missed_medication"
)
```
Then use `AlertTypeAcholicStool` etc. everywhere.

### [MEDIUM] Jaundice severity value strings are also stringly-typed
- **File:** `backend/internal/store/alerts.go`
- **Line(s):** L186, L192-L198
- **Perspective:** Tech Lead
- **Finding:** The jaundice alert value uses raw strings like `"severe_limbs_and_trunk"`, `"scleral_icterus"`, and `"severe_limbs_and_trunk+scleral_icterus"`. The `"severe_limbs_and_trunk"` string also appears in the query comparison at L186 and in test code.
- **Reasoning:** Same fragility as alert type strings. The concatenated value `"severe_limbs_and_trunk+scleral_icterus"` introduces a custom encoding that is easy to get wrong.
- **Consequence:** A consumer parsing the `value` field would need to know the `+` delimiter convention, which is undocumented and error-prone.
- **Suggestion:** Define constants for jaundice levels. Consider returning the value as a structured type (e.g., `map[string]bool{"severe": true, "scleral_icterus": true}`) rather than a concatenated string, or at minimum document the format.

### [LOW] Chart data functions in chart_data.go follow a highly repetitive pattern
- **File:** `backend/internal/store/chart_data.go`
- **Line(s):** L85-L359
- **Perspective:** Tech Lead
- **Finding:** 6 of the 7 functions (all except GetLabTrends) follow the exact same structure: parseDateRange, db.Query, defer rows.Close, scan loop, rows.Err check, nil-to-empty-slice. The only differences are the SQL query string and the scan target.
- **Reasoning:** This is a judgment call. Go's explicit style and the project's "no over-engineering" constraint favor keeping the functions separate. Each function has different scan targets and column counts, making a generic helper require `interface{}` or generics, which adds complexity. The repetition is mechanical and low-risk.
- **Consequence:** Minor maintenance overhead when adding a new chart series (copy a function, change 3 things). Low bug risk since the pattern is simple.
- **Suggestion:** Acceptable as-is given the project's constraints. If Go 1.18+ generics are available, a generic `querySlice[T any](db, query, args, scanFn) ([]T, error)` helper could reduce boilerplate, but this is a nice-to-have, not a requirement.

### [LOW] `parseDateRange` returns formatted strings, not `time.Time` values
- **File:** `backend/internal/store/chart_data.go`
- **Line(s):** L70-L82
- **Perspective:** Tech Lead
- **Finding:** `parseDateRange` parses date strings into `time.Time`, then immediately formats them back to strings. It returns `(string, string, error)` rather than `(time.Time, time.Time, error)`.
- **Reasoning:** Since every caller immediately passes the results as SQL query parameters (which accept strings), this is not incorrect. However, returning `time.Time` would be more flexible and would allow callers to do time arithmetic if needed.
- **Consequence:** Minimal. If a future caller needs `time.Time` values, the function signature would need to change.
- **Suggestion:** Acceptable as-is. The function is internal and all callers use strings.

### [LOW] `getMissedMedicationAlerts` uses `time.Now()` directly, making it untestable
- **File:** `backend/internal/store/alerts.go`
- **Line(s):** L242-L243
- **Perspective:** QA
- **Finding:** `getMissedMedicationAlerts` calls `time.Now().UTC()` directly at L242, making it impossible to test time-sensitive edge cases (e.g., doses exactly at the 30-minute boundary) without real-time waits.
- **Reasoning:** This function's behavior depends on the current time relative to scheduled doses. Without a clock abstraction, tests cannot reliably verify boundary conditions.
- **Consequence:** Flaky or incomplete tests for missed medication logic. Edge cases around midnight, timezone transitions, and the 30-minute window boundary are difficult to cover.
- **Suggestion:** Accept a `now time.Time` parameter (or use a `Clock` interface) so tests can inject a fixed time. The same applies to `computeNextDoseAt` in dashboard.go (L251).
