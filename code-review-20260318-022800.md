## Summary

This review covers phases 37-39 of LittleLiver: a photo flow integration test (Phase 37), PDF report generation with text/table sections (Phase 38), and PDF report charts plus photo appendix (Phase 39). The review focuses specifically on **code reuse** opportunities -- identifying duplicated logic, reinvented utilities, and patterns that already exist elsewhere in the codebase.

---

### [MEDIUM] Duplicated `photoTestClient` is a near-clone of `testClient`
- **File:** `backend/internal/integration/photo_flow_test.go`
- **Line(s):** L57-L167
- **Perspective:** Tech Lead
- **Finding:** `photoTestClient` (fields: t, srv, userID, sessionID, csrfToken) and its `doJSON` method are structurally identical to the existing `testClient` in `multi_parent_test.go` (L21-L118) and `medication_flow_test.go`. The `newPhotoTestClient` constructor is also a line-for-line duplicate of `newTestClient`. The only addition is the `uploadPhoto` method.
- **Reasoning:** Three copies of the same struct and HTTP helper methods exist across integration test files. This violates DRY and means bug fixes to request handling (e.g., adding a header) must be replicated in multiple places.
- **Consequence:** When a shared behavior changes (e.g., adding a new auth header, changing cookie names), each duplicate must be updated independently. Missed updates cause subtle test failures.
- **Suggestion:** Extract the shared `testClient` struct, `newTestClient`, and `doJSON`/`doJSONList`/`doRaw` into a shared file (e.g., `backend/internal/integration/helpers_test.go`). Then embed `testClient` inside `photoTestClient` to add the `uploadPhoto` method:
  ```go
  // helpers_test.go
  type testClient struct { ... } // single definition
  func newTestClient(t *testing.T, srv *httptest.Server, db *sql.DB) *testClient { ... }
  func (tc *testClient) doJSON(...) (int, map[string]any) { ... }

  // photo_flow_test.go
  type photoTestClient struct {
      *testClient
  }
  func (pc *photoTestClient) uploadPhoto(...) (int, map[string]any) { ... }
  ```

### [MEDIUM] Duplicated `setupPhotoServer` mostly clones `setupMultiParentServer`
- **File:** `backend/internal/integration/photo_flow_test.go`
- **Line(s):** L29-L54
- **Perspective:** Tech Lead
- **Finding:** `setupPhotoServer` is nearly identical to `setupMultiParentServer` from `multi_parent_test.go` (L31-L55). The only difference is adding `handler.WithObjectStore(objStore)` and returning the `*storage.MemoryStore`. The auth config is copied verbatim.

### [MEDIUM] `parseTimestampMulti` duplicates `store.parseTime`
- **File:** `backend/internal/report/charts.go`
- **Line(s):** L267-L279
- **Perspective:** Tech Lead
- **Finding:** `parseTimestampMulti` in `charts.go` tries three timestamp formats: `model.DateTimeFormat`, `"2006-01-02 15:04:05"`, and `"2006-01-02T15:04:05"`. Meanwhile, `store.parseTime` in `sessions.go` (L86-L98) does the same thing with a superset of formats (also includes `time.DateTime`, `time.RFC3339`, and `"2006-01-02"`). The `report` package has reinvented this utility with fewer format options.
- **Reasoning:** Having two multi-format time parsers that handle overlapping but different format sets is error-prone. If a new timestamp format appears in the data, only one might be updated.
- **Consequence:** Timestamp parsing inconsistencies between the report package and the store package. A timestamp that parses correctly in one context may fail in another.
- **Suggestion:** Export `parseTime` from the store package (or move it to a shared `internal/timeutil` package) and use it in `charts.go` instead of `parseTimestampMulti`. Alternatively, since `parseTime` is unexported in `store`, promote it:
  ```go
  // store/metric_helpers.go (or a new timeutil package)
  func ParseTime(s string) (time.Time, error) { ... }
  ```

- **Suggestion:** Replace with `model.DateFormat`:
  ```go

  ```

### [LOW] Manual nil-to-empty-slice pattern instead of `store.emptySliceIfNil`
- **File:** `backend/internal/report/report.go`
- **Line(s):** L246-L248, L283-L285
- **Perspective:** Tech Lead
- **Finding:** Both `queryMedAdherence` and `queryNotes` use a manual `if result == nil { result = make([]T, 0) }` pattern. The store package already has a generic `emptySliceIfNil[T]` helper (`metric_helpers.go` L161-L166) that does exactly this.
- **Reasoning:** The `report` package already imports `store` extensively. Using the existing helper reduces visual noise and keeps the nil-to-empty-slice pattern consistent.
- **Consequence:** Trivial -- just a missed reuse opportunity and minor code bloat.
- **Suggestion:** Note that `emptySliceIfNil` is currently unexported. Either export it (rename to `EmptySliceIfNil`) or add a similar helper to the report package. If exported:
  ```go
  return store.EmptySliceIfNil(result), nil
  ```

### [LOW] Duplicated age-in-days calculation
- **File:** `backend/internal/report/report.go`
- **Line(s):** L129, L453
- **Perspective:** Tech Lead
- **Finding:** The expression `int(now.Sub(baby.DateOfBirth).Hours() / 24)` appears twice in the same file (once in `queryReportData` L129, once in `addHeader` L453). The same pattern also appears at L476 for Kasai date. This is a small internal duplication.
- **Reasoning:** While the calculation is simple, extracting it to a tiny helper (e.g., `daysBetween(a, b time.Time) int`) would make intent clearer and prevent copy-paste drift if the calculation needs to account for leap seconds or other edge cases in the future.
- **Consequence:** Minimal. But the pattern is used in three places in one file, suggesting a local helper would pay for itself.
- **Suggestion:**
  ```go
  func daysSince(from time.Time, now time.Time) int {
      return int(now.Sub(from).Hours() / 24)
  }
  ```

### [LOW] Three chart embed functions share identical boilerplate
- **File:** `backend/internal/report/report.go`
- **Line(s):** L321-L366
- **Perspective:** Tech Lead
- **Finding:** `addChartSection`, `addWeightChartSection`, and `addLabTrendsSection` all follow the same structure: call render function, check for nil/error, add section title row, add image row with identical `props.Rect`, add spacer. The only differences are the title string and which render function is called.
- **Reasoning:** This is a minor structural duplication. The pattern could be extracted to a single `embedChart(m, title, pngBytes)` helper, with each caller just responsible for rendering.
- **Consequence:** If the chart embedding layout changes (e.g., adjusting dimensions, adding borders), three functions must be updated in lockstep.
- **Suggestion:**
  ```go
  func embedChartPNG(m core.Maroto, title string, pngData []byte) {
      if pngData == nil { return }
      m.AddRows(text.NewRow(8, title, sectionStyle))
      m.AddRows(row.New(80).Add(
          image.NewFromBytesCol(12, pngData, extension.Png, props.Rect{Percent: 100, Center: true}),
      ))
      m.AddRows(spacerRow(4))
  }
  ```
  Then each caller becomes:
  ```go
  if png, err := renderStoolChart(data.stools); err == nil {
      embedChartPNG(m, "Stool Color Distribution", png)
  }
  ```
