## Summary

Phase 42 implements the baby selector, create baby form, join flow UI, and first-login screen for the LittleLiver frontend. The implementation includes a Svelte store (`baby.ts`) for managing baby state, four Svelte 5 components (`CreateBabyForm`, `JoinBabyForm`, `BabySelector`, `FirstLogin`), and comprehensive test suites for all modules. All 68 tests pass. Line coverage is 100% for all new files. The code is clean, follows early-return style, uses TypeScript strict mode, and aligns with the project's API endpoints and data model from the spec. One bug was found related to form submission behavior.

### [HIGH] Missing preventDefault on Form Submission
- **File:** `frontend/src/lib/components/CreateBabyForm.svelte`, `frontend/src/lib/components/JoinBabyForm.svelte`
- **Line(s):** CreateBabyForm L48, JoinBabyForm L25
- **Perspective:** QA
- **Finding:** Both forms use `onsubmit={handleSubmit}` but the `handleSubmit` functions do not accept an event parameter or call `event.preventDefault()`. In Svelte 5 runes mode, `onsubmit` is a native DOM event binding and does not automatically prevent the browser's default form submission behavior (unlike Svelte 4's `on:submit|preventDefault`).
- **Reasoning:** Without `preventDefault()`, the browser will perform a default form submission (GET request to the current URL), causing a full page reload. The `handleSubmit` function will execute but its effects will be immediately lost when the page reloads. This bug may not surface in tests because jsdom's `requestSubmit()` is not fully implemented (as shown in test output warnings), meaning the form submission path in tests bypasses actual DOM submission behavior.
- **Consequence:** Every form submission will trigger a page reload, making both the create baby and join baby flows completely non-functional in production. Users will be unable to create babies or join via invite codes.
- **Suggestion:** Either accept the event and call `preventDefault()`, or use Svelte 5's `onsubmit` with `preventDefault`:
  ```svelte
  <!-- Option A: Accept event in handler -->
  <script>
  function handleSubmit(event: SubmitEvent) {
      event.preventDefault();
      // ... rest of validation logic
  }
  </script>

  <!-- Option B: Use Svelte 5 event modifier -->
  <form onsubmit|preventDefault={handleSubmit}>
  ```

### [LOW] Synchronous Store Read Pattern Could Be Simplified
- **File:** `frontend/src/lib/stores/baby.ts`
- **Line(s):** L39-L40, L59-L60
- **Perspective:** Tech Lead
- **Finding:** The pattern of subscribing and immediately unsubscribing to read a store value synchronously (`activeBaby.subscribe((v) => (currentActive = v))()`) works but is non-idiomatic. Svelte provides `get()` from `svelte/store` for this purpose, which is already imported in the test file.
- **Reasoning:** The subscribe-and-immediately-unsubscribe pattern is harder to read and understand. The `get()` utility from `svelte/store` is the standard approach for synchronous store reads outside of components and is already used extensively in the test file.
- **Consequence:** Minor readability concern. Future contributors may not immediately understand the subscribe/unsubscribe pattern, increasing cognitive load during maintenance.
- **Suggestion:**
  ```typescript
  import { writable, derived, get } from 'svelte/store';

  // In fetchBabies:
  const currentActive = get(activeBaby);

  // In setActiveBaby:
  const currentBabies = get(babies);
  ```

### [LOW] CreateBabyInput sex Field Uses string Instead of Union Type
- **File:** `frontend/src/lib/stores/baby.ts`
- **Line(s):** L16
- **Perspective:** Tech Lead
- **Finding:** The `CreateBabyInput` interface defines `sex` as `string`, while the `Baby` interface correctly types it as `'male' | 'female'`. The input type should be at least as restrictive to prevent invalid values from being sent to the API.
- **Reasoning:** The spec schema enforces `CHECK (sex IN ('male', 'female'))` on the `babies` table. Using a plain `string` type allows invalid values like `'other'` or empty strings to pass TypeScript compilation and only fail at the API level.
- **Consequence:** TypeScript will not catch invalid sex values at compile time. The server will reject them, but the error will be a generic API error rather than a clear type error during development.
- **Suggestion:**
  ```typescript
  export interface CreateBabyInput {
      name: string;
      date_of_birth: string;
      sex: 'male' | 'female';
      diagnosis_date?: string;
      kasai_date?: string;
  }
  ```
