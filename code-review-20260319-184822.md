## Summary

Phase 51 implements the Settings page for the LittleLiver app, including baby profile editing (name, DOB, sex, diagnosis/kasai dates, default_cal_per_feed with recalculate option), invite code generation with copy-to-clipboard, self-unlink from baby with confirmation dialog, and account deletion with confirmation dialog. The implementation spans 5 Svelte components, 1 store file with new functions, and 6 test files totaling 47 passing tests.

Overall the implementation is solid: components are well-structured, tests cover the key behaviors, confirmation dialogs guard destructive actions, and API endpoints match the spec. However, there is one spec-compliance bug in the response handling for recalculate mode, and a few completeness gaps.

---

### [HIGH] Envelope Response Not Handled for Recalculate Mode
- **File:** `frontend/src/lib/stores/baby.ts`
- **Line(s):** L104-L116
- **Perspective:** QA
- **Finding:** When `recalculate=true`, the spec states the API returns an envelope `{ "baby": {...}, "recalculated_count": N }`, not a plain `Baby` object. The `updateBaby` function always treats the response as `Baby`, meaning when recalculate is enabled, it will store the envelope object (with `baby` and `recalculated_count` fields) directly into the `babies` and `activeBaby` stores instead of the actual baby data.
- **Reasoning:** The spec explicitly says: "Normal PUT returns just the baby object. When `recalculate_calories=true`, response uses an envelope: `{ "baby": {...}, "recalculated_count": N }`." The current code does not differentiate between the two response shapes.
- **Consequence:** When a user changes `default_cal_per_feed` and checks the recalculate box, the baby store will contain `{ baby: {...}, recalculated_count: N }` instead of the actual baby object. This will break the entire UI -- name, DOB, sex, and all other baby fields will be `undefined` throughout the app until the page is refreshed.
- **Suggestion:** Handle the envelope response when `recalculate` is true:
  ```typescript
  interface RecalculateResponse {
    baby: Baby;
    recalculated_count: number;
  }

  export async function updateBaby(
    id: string,
    input: UpdateBabyInput,
    recalculate = false
  ): Promise<Baby> {
    const path = recalculate
      ? `/babies/${id}?recalculate_calories=true`
      : `/babies/${id}`;

    let baby: Baby;
    if (recalculate) {
      const envelope = await apiClient.put<RecalculateResponse>(path, input);
      baby = envelope.baby;
    } else {
      baby = await apiClient.put<Baby>(path, input);
    }

    babies.update((current) =>
      current.map((b) => (b.id === id ? baby : b))
    );
    const currentActive = get(activeBaby);
    if (currentActive?.id === id) {
      activeBaby.set(baby);
    }
    return baby;
  }
  ```
  Add a corresponding test in `settings-store.test.ts` that verifies the envelope is unwrapped.

### [MEDIUM] Missing Expiry Countdown (Spec Requirement)
- **File:** `frontend/src/lib/components/InviteSection.svelte`
- **Line(s):** L31-L37
- **Perspective:** Product Owner
- **Finding:** The Phase 51 spec explicitly calls for an "expiry countdown" for invite codes. The current implementation only shows a static formatted date/time via `toLocaleString()`. There is no live countdown timer showing remaining time until expiration.
- **Reasoning:** The phase description says "invite code generation + share UI (display code, copy button, expiry countdown)". A countdown gives users a clear sense of urgency and helps them understand how much time remains for the invite to be used.
- **Consequence:** Users see a static timestamp (e.g., "Expires 3/21/2026, 12:00:00 AM") instead of a dynamic countdown (e.g., "Expires in 23h 45m"). This is a missing feature from the phase requirements.
- **Suggestion:** Add a reactive countdown that updates periodically:
  ```typescript
  let remainingTime = $state('');
  let intervalId: ReturnType<typeof setInterval> | undefined;

  function updateCountdown(iso: string) {
    const diff = new Date(iso).getTime() - Date.now();
    if (diff <= 0) {
      remainingTime = 'Expired';
      clearInterval(intervalId);
      return;
    }
    const hours = Math.floor(diff / 3600000);
    const minutes = Math.floor((diff % 3600000) / 60000);
    remainingTime = `${hours}h ${minutes}m`;
  }
  ```

### [MEDIUM] Missing Baby Selector Management
- **File:** `frontend/src/lib/components/SettingsPage.svelte`
- **Line(s):** N/A - structural issue
- **Perspective:** Product Owner
- **Finding:** The Phase 51 spec includes "Baby selector management" as a feature to build, but the SettingsPage only shows settings for the currently active baby. There is no UI for switching between babies or managing which baby is selected within the settings context.
- **Reasoning:** The spec (section 2.3) says "A parent can switch between baby profiles via a selector in the app header." While this may exist in a header component, the phase description explicitly lists "Baby selector management" as part of this phase's scope.
- **Consequence:** If the baby selector is not implemented anywhere else, parents with multiple babies cannot switch between profiles to manage settings for each.
- **Suggestion:** Either add a baby selector dropdown within the SettingsPage, or verify that a global baby selector already exists in the app header from a prior phase. If it exists elsewhere, this can be considered covered.

### [MEDIUM] No Test for Recalculate Envelope Unwrapping
- **File:** `frontend/src/tests/settings-store.test.ts`
- **Line(s):** L62-L73
- **Perspective:** QA
- **Finding:** The test for the recalculate path only verifies that `PUT` is called with the correct URL and body. It does not verify that the envelope response `{ baby: {...}, recalculated_count: N }` is correctly unwrapped. The mock returns a plain baby object, masking the bug described above.
- **Reasoning:** The test mock at L63 (`mockResolvedValue(updatedBaby)`) returns a `Baby` directly, but the real API will return `{ baby: updatedBaby, recalculated_count: 5 }`. This means the test passes but the production code will fail.
- **Consequence:** The bug in response handling goes undetected by the test suite, making it a false green.
- **Suggestion:** Update the mock to return the envelope format and add assertions that the store contains the unwrapped baby:
  ```typescript
  it('unwraps envelope response when recalculate is true', async () => {
    const updatedBaby = { ...mockBabies[0], default_cal_per_feed: 80 };
    vi.spyOn(apiClient, 'put').mockResolvedValue({
      baby: updatedBaby,
      recalculated_count: 5
    });
    vi.spyOn(apiClient, 'get').mockResolvedValue({ babies: mockBabies });
    await fetchBabies();

    await updateBaby('baby-1', { default_cal_per_feed: 80 }, true);

    expect(get(activeBaby)?.default_cal_per_feed).toBe(80);
    expect(get(activeBaby)).not.toHaveProperty('recalculated_count');
  });
  ```

### [LOW] Svelte Compiler Warnings for State Referenced Locally
- **File:** `frontend/src/lib/components/BabySettingsForm.svelte`
- **Line(s):** L13-L22
- **Perspective:** Tech Lead
- **Finding:** The Svelte compiler emits 7 `state_referenced_locally` warnings for `BabySettingsForm.svelte`. The `baby` prop is destructured into local `$state` variables at component initialization (lines 13-18, 22), capturing only the initial value. If the `baby` prop were to change after mount, the form fields would not update.
- **Reasoning:** While this is functionally acceptable for the current usage (the baby prop is not expected to change during the form's lifecycle), the warnings indicate a code pattern that Svelte considers problematic. Clean compiler output is valuable for detecting real issues.
- **Consequence:** If the active baby is switched while the settings page is open, the form would continue showing the previous baby's data. Minor UX confusion.
- **Suggestion:** Use `$derived` for the initial values or acknowledge the pattern is intentional by restructuring to use `$effect` for synchronization:
  ```typescript
  let name = $state('');
  $effect(() => { name = baby.name; });
  ```

### [LOW] Account Deletion Does Not Clear Application State
- **File:** `frontend/src/lib/components/SettingsPage.svelte`
- **Line(s):** L72-L79
- **Perspective:** SecOps
- **Finding:** After successful account deletion, the code redirects to `/login` via `window.location.href` but does not clear Svelte stores (babies, activeBaby, or any other cached user data). While the hard navigation will reset the SPA state, if there is any persistence layer (localStorage, sessionStorage), stale data could remain.
- **Reasoning:** A full page navigation (`window.location.href = '/login'`) will indeed destroy the in-memory Svelte store state. However, best practice for account deletion is to explicitly clear all application state before redirecting.
- **Consequence:** Low risk since the page navigation resets in-memory state. Only an issue if stores are persisted to localStorage in a future phase.
- **Suggestion:** Add explicit cleanup before redirect for defense in depth:
  ```typescript
  async function handleDeleteAccount() {
    deleteError = '';
    try {
      await deleteAccount();
      _resetBabyStores();
      window.location.href = '/login';
    } catch (err) {
      deleteError = err instanceof Error ? err.message : 'Failed to delete account';
    }
  }
  ```

  it('shows error when invite generation fails', async () => {
