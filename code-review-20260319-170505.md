## Summary

Efficiency review of Phases 46-48: TrendsView with 7 Chart.js chart components, DateRangeSelector, MedicationList, MedicationForm, DoseLogForm, and MedLogList. The review focuses on redundant API calls, Chart.js lifecycle management, fetch parallelization, and unnecessary re-renders.

### [HIGH] Charts do not update when props change -- they are mount-only

- **File:** `frontend/src/lib/components/AbdomenGirthChart.svelte` (and all 6 other chart components)
- **Line(s):** L20-L61 (AbdomenGirthChart as representative example)
- **Perspective:** Tech Lead
- **Finding:** Every chart component creates the Chart.js instance in `onMount` and never updates it. When TrendsView receives new data (e.g., user changes date range from 7d to 30d), TrendsView re-fetches data and updates its `dashboard` state, but because Svelte 5 will re-render child components with new props, the old Chart.js instance on the existing canvas is stale. The charts only initialize once in `onMount` and have no `$effect` watching the `data` prop for updates.
- **Reasoning:** In Svelte 5 runes mode, when the parent conditionally renders `{#if dashboard}` and the dashboard object reference changes, the block is re-evaluated. However, if Svelte reuses the component instance (which it does when the `{#if}` branch stays the same), `onMount` does NOT re-fire. The chart shows stale data from the first fetch.
- **Consequence:** When users switch date ranges (7d to 30d to 90d), charts may display the original 7d data instead of the newly fetched data. The entire Trends feature appears broken for any range switch after the first load.
- **Suggestion:** Either (a) add a reactive `$effect` that calls `chart.update()` with new data when props change, or (b) key the chart components so Svelte destroys and recreates them on data change:
  ```svelte
  <!-- Option B: in TrendsView.svelte, force remount with a key -->
  {#key dashboard}
    <StoolColorChart data={dashboard.chart_data_series.stool_color} />
  {/key}
  ```
  Option A is more efficient (avoids full Chart.js teardown/rebuild) but requires more code in each chart component.

### [MEDIUM] MedicationList re-fetches entire list after toggle instead of local state update

- **File:** `frontend/src/lib/components/MedicationList.svelte`
- **Line(s):** L63-L69 (toggleActive function)
- **Perspective:** Performance Engineer
- **Finding:** `toggleActive` calls `apiClient.put(...)` and then immediately calls `fetchMedications()` to re-fetch the entire list from the API, just to toggle a single boolean field locally.
- **Reasoning:** For a small personal-use app this is tolerable, but it is an unnecessary round-trip. The PUT response likely contains the updated medication, which could be used to update the local state directly.
- **Consequence:** Extra network request and full re-render of the list on every toggle. With slow networks, the UI flashes through a loading state unnecessarily.
- **Suggestion:** Update the local array optimistically or from the PUT response:
  ```typescript
  async function toggleActive(med: Medication): Promise<void> {
    try {
      await apiClient.put(`/babies/${babyId}/medications/${med.id}`, { active: !med.active });
      medications = medications.map(m => m.id === med.id ? { ...m, active: !m.active } : m);
    } catch {
      error = 'Failed to update medication';
    }
  }
  ```

### [MEDIUM] DoseLogForm and MedicationList both fetch the same medications endpoint independently

- **File:** `frontend/src/lib/components/DoseLogForm.svelte` and `frontend/src/lib/components/MedicationList.svelte`
- **Line(s):** DoseLogForm L67-L75, MedicationList L50-L61
- **Perspective:** Performance Engineer
- **Finding:** Both components independently call `GET /babies/${babyId}/medications`. If both are rendered on the same page (or in quick succession during navigation), the same data is fetched twice.
- **Reasoning:** There is no shared data layer, cache, or store. Each component owns its own fetch. For a personal-use app this is acceptable, but it is a clear duplication.
- **Consequence:** Two identical API calls when both components coexist, doubling server load for that endpoint.
- **Suggestion:** Either (a) lift medication fetching to the parent and pass medications as props, or (b) introduce a simple Svelte store/cache for medications. For now this is low-priority given the app's scale.

### [MEDIUM] Chart.js `registerables` imported globally as side effect on every page

- **File:** `frontend/src/lib/chart-setup.ts`
- **Line(s):** L1-L3
- **Perspective:** Performance Engineer
- **Finding:** `chart-setup.ts` calls `Chart.register(...registerables)` at module level. This is imported by TrendsView.svelte. The `registerables` array includes ALL Chart.js controllers, elements, scales, and plugins. This is approximately 60KB of registration overhead.
- **Reasoning:** Chart.js recommends tree-shakeable imports for production: import only the specific controllers/elements/scales you actually use (LineController, BarController, ScatterController, CategoryScale, LinearScale, PointElement, LineElement, BarElement, Filler, Legend, Tooltip).
- **Consequence:** Bundle includes the full Chart.js library even if only line/bar/scatter charts are used. Adds ~30-40KB to the final gzipped bundle that could be avoided.
- **Suggestion:** Replace the blanket registration with specific imports:
  ```typescript
  import { Chart, LineController, BarController, ScatterController,
           CategoryScale, LinearScale, PointElement, LineElement,
           BarElement, Filler, Legend, Tooltip } from 'chart.js';
  Chart.register(LineController, BarController, ScatterController,
                 CategoryScale, LinearScale, PointElement, LineElement,
                 BarElement, Filler, Legend, Tooltip);
  ```

### [LOW] LabTrendsChart duplicates timestamp parsing for normal range x-bounds

- **File:** `frontend/src/lib/components/LabTrendsChart.svelte`
- **Line(s):** L42-L45 (inside the loop, after dataset creation)
- **Perspective:** Performance Engineer
- **Finding:** For each test name that has a normal range, the code re-maps all points to extract timestamps via `points.map(p => new Date(p.timestamp).getTime())` a second time, even though the same timestamps were already parsed 5 lines earlier when building the `data` array for the dataset.
- **Reasoning:** The parsed x-values from the dataset could be reused.
- **Consequence:** Minor: parses Date objects twice per test name. Negligible for the typical data volumes (tens to low hundreds of lab results).
- **Suggestion:** Reuse the already-mapped points array:
  ```typescript
  const mappedPoints = points
    .map(p => ({ x: new Date(p.timestamp).getTime(), y: parseFloat(p.value) }))
    .filter(p => !isNaN(p.y));
  // ...use mappedPoints for dataset data...
  // For normal range:
  const xValues = mappedPoints.map(p => p.x);
  ```

### [LOW] StoolColorChart and TemperatureChart create Chart.js even with empty data (inconsistent with other charts)

- **File:** `frontend/src/lib/components/StoolColorChart.svelte`, `frontend/src/lib/components/TemperatureChart.svelte`
- **Line(s):** StoolColorChart L24-L73, TemperatureChart L18-L60
- **Perspective:** QA
- **Finding:** AbdomenGirthChart, DiaperChart, FeedingChart, and LabTrendsChart all guard against empty data with `if (isEmpty) return` in `onMount` and show a "No data" message. StoolColorChart and TemperatureChart do not -- they create a Chart.js instance with an empty dataset array.
- **Reasoning:** Inconsistent behavior. Creating an empty chart wastes a small amount of memory and renders a blank canvas with axes instead of a user-friendly "No data" message.
- **Consequence:** Users see empty chart canvases with axis labels for stool color and temperature when no data exists, while other charts show "No data" text. Minor UX inconsistency.
- **Suggestion:** Add the same empty-data guard to StoolColorChart and TemperatureChart for consistency.
