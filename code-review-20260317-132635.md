## Summary

Review of efficiency concerns across 3 phases of backend changes: Auth middleware (session validation, CSRF, timezone update), API handlers (`/api/csrf-token`, `/api/me`), store functions (`GetUserByID`, `GetBabiesByUserID`, `UpdateUserTimezone`), test utilities, and integration tests. The diff introduces session-based authentication, CSRF protection, user/baby retrieval endpoints, and a shared `testutil` package.

### [MEDIUM] Auth middleware issues 3 sequential DB queries on every authenticated request
- **File:** `backend/internal/middleware/middleware.go`
- **Line(s):** L49-L76
- **Perspective:** Performance Engineer
- **Finding:** Every authenticated request executes up to 4 sequential DB round-trips: (1) `GetSessionByID`, (2) `ExtendSession`, (3) `GetUserByID`, and (4) `UpdateUserTimezone` (when `X-Timezone` header is present). These are all serialized on the hot path.
- **Reasoning:** The session lookup already joins on `user_id`; the user data could be fetched in the same query via a JOIN, reducing the critical path from 2 mandatory queries to 1. Additionally, `ExtendSession` and `UpdateUserTimezone` are fire-and-forget writes that do not need to block the response.
- **Consequence:** Each authenticated API request pays for at least 3 DB round-trips before the handler even begins. With SQLite (single-writer), the writes also serialize against each other, adding latency under concurrency.
- **Suggestion:** Combine `GetSessionByID` and `GetUserByID` into a single `SELECT ... JOIN` query. Run `ExtendSession` and `UpdateUserTimezone` asynchronously (goroutine) since their failures are already non-fatal and only logged. Example:
  ```go
  // Combined query
  sess, user, err := store.GetSessionWithUser(db, cookie.Value)

  // Fire-and-forget writes
  go func() {
      if err := store.ExtendSession(db, sess.ID); err != nil {
          log.Printf("auth: extend session: %v", err)
      }
  }()
  ```

### [MEDIUM] Timezone UPDATE fires on every request even when the value has not changed
- **File:** `backend/internal/middleware/middleware.go`
- **Line(s):** L68-L77
- **Perspective:** Performance Engineer
- **Finding:** When `X-Timezone` is present and valid, `UpdateUserTimezone` writes to the DB unconditionally, even if the user's timezone already matches the header value.
- **Reasoning:** Browsers send the same timezone on every request. This means every authenticated request with the header triggers a redundant `UPDATE users SET timezone = ?` that writes the same value, causing unnecessary SQLite WAL churn.
- **Consequence:** Wasted write I/O on every request. In SQLite's single-writer model, this contention is amplified.
- **Suggestion:** Compare `user.Timezone` with the incoming header value before issuing the UPDATE:
  ```go
  if tz != "" && (user.Timezone == nil || *user.Timezone != tz) {
      if _, err := time.LoadLocation(tz); err == nil {
          // only write if actually changed
          if err := store.UpdateUserTimezone(db, user.ID, tz); err != nil {
              log.Printf("auth: failed to update timezone for user %s: %v", user.ID, err)
          } else {
              user.Timezone = &tz
          }
      }
  }
  ```

### [LOW] UpsertUser generates a ULID that is discarded on conflict, then issues a second SELECT
- **File:** `backend/internal/store/users.go`
- **Line(s):** L12-L33
- **Perspective:** Performance Engineer
- **Finding:** `UpsertUser` always generates a new ULID (`model.NewULID()`) and then executes an `INSERT ... ON CONFLICT DO UPDATE`. On the conflict (update) path, the generated ULID is thrown away. After the upsert, a second `SELECT` query fetches the row back by `google_id`.
- **Reasoning:** SQLite supports `RETURNING`, which can return the row in a single statement, eliminating the second round-trip. The wasted ULID generation is minor but unnecessary on the update path.
- **Consequence:** Two DB round-trips instead of one on every OAuth callback. Low severity because OAuth callbacks are infrequent.
- **Suggestion:** Use `INSERT ... ON CONFLICT ... RETURNING id, google_id, email, name` with `QueryRow` to combine into a single statement. If the SQLite driver does not support `RETURNING`, at minimum use the generated `id` for the follow-up SELECT (indexed primary key) instead of `google_id`.

### [LOW] CreateTestBaby fetches ALL babies for a user to find the one just inserted
- **File:** `backend/internal/testutil/testutil.go`
- **Line(s):** L70-L103
- **Perspective:** Performance Engineer
- **Finding:** `CreateTestBaby` calls `store.GetBabiesByUserID(db, userID)` after inserting a single baby, then linearly scans the returned slice to find the newly created record by ID.
- **Reasoning:** This is test-only code so it does not affect production performance, but it grows O(n) with the number of babies per user in test scenarios and is needlessly wasteful. A direct `SELECT ... WHERE id = ?` would be O(1).
- **Consequence:** Negligible in practice (test data is small), but sets a poor pattern. If a store function like `GetBabyByID` is added later, this helper should use it.
- **Suggestion:** Query directly by the known `babyID`:
  ```go
  babies, err := store.GetBabiesByUserID(db, userID) // replace with GetBabyByID(db, babyID)
  ```
  Or accept the current approach given it is test-only code.

### [LOW] CSRFTokenHandler performs a redundant DB session lookup that Auth middleware could provide
- **File:** `backend/internal/handler/api.go`
- **Line(s):** L16-L37
- **Perspective:** Performance Engineer
- **Finding:** `CSRFTokenHandler` is registered at `GET /api/csrf-token` *without* the Auth middleware (per `router.go` L61). It independently reads the session cookie and calls `store.GetSessionByID`. If it were placed behind the Auth middleware, it could read the session token from context at zero DB cost (the Auth middleware already does the lookup).
- **Reasoning:** The router comment says "needs session but not auth middleware context," but the Auth middleware does exactly what this handler needs and would eliminate a duplicate DB query.
- **Consequence:** One extra DB query per CSRF token fetch. Minor since this endpoint is called infrequently (once per page load).
- **Suggestion:** Place `CSRFTokenHandler` behind `authMw` and read the session token from context via `middleware.SessionTokenFromContext(r.Context())`, removing the manual cookie/session lookup inside the handler.
