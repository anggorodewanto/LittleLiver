## Summary

This review covers phases 31-33 of the LittleLiver project, which introduce three features: WHO Child Growth Standards percentile calculations (`who` package), a WHO percentiles HTTP endpoint, Web Push notification infrastructure (`notify` package), and push subscription CRUD operations (handler + store layers). The code is generally well-structured with good test coverage and proper error handling. The issues found are primarily around code quality patterns rather than bugs.

### [LOW] Dead Code: Empty Test Body
- **File:** `backend/internal/who/who_test.go`
- **Line(s):** L359-L369
- **Perspective:** QA
- **Finding:** `TestLoadLMS_ErrorCases` contains a subtest `"invalid CSV data"` with an empty body -- just comments explaining why it is difficult to test, but no actual assertions.
- **Reasoning:** Empty test bodies create a false sense of coverage. The test name implies error paths are verified when they are not. This is effectively dead test code.
- **Consequence:** Future developers may assume CSV parsing error paths are tested and skip writing proper tests, leaving those code paths permanently unverified.
- **Suggestion:** Either remove the empty test entirely (it provides no value) or use `fs/fstest.MapFS` to construct an in-memory FS with malformed CSV data and pass it to `loadLMS` to actually exercise the error branches.

### [MEDIUM] Stringly-Typed Sex Parameter
- **File:** `backend/internal/who/who.go`
- **Line(s):** L104-L113
- **Perspective:** Tech Lead
- **Finding:** The `sex` parameter is accepted as a raw string throughout the `who` package (`ZScore`, `Percentile`, `PercentileCurves`), with validation done via string comparison (`"male"`, `"female"`). The `model` package already has `ValidSex()` and a `sexValues` set in `enums.go`, but the `who` package duplicates these magic strings independently.
- **Reasoning:** Two separate sources of truth for valid sex values creates a maintenance risk. If the model layer ever changes its vocabulary (or adds validation), the `who` package would remain out of sync.
- **Consequence:** A casing or naming mismatch between what the handler accepts (via model validation) and what the `who` package expects could silently produce incorrect results or confusing errors.
- **Suggestion:** Either import and use `model.ValidSex()` in the `who` package, or define exported constants (e.g., `SexMale = "male"`, `SexFemale = "female"`) in `model` and reference them in both places.

### [MEDIUM] Duplicate Subscription Type
- **File:** `backend/internal/notify/notify.go`
- **Line(s):** L19-L24
- **Perspective:** Tech Lead
- **Finding:** `notify.Subscription` duplicates the same three fields (`Endpoint`, `P256dh`, `Auth`) that already exist in `model.PushSubscription`. The handler must manually map between these two representations.
- **Reasoning:** Having two structs representing the same concept (push subscription keys) in different packages creates a leaky abstraction boundary. Changes to the subscription schema must be synchronized in both places.
- **Consequence:** If a field is added or renamed in `model.PushSubscription`, the `notify.Subscription` struct could fall out of sync, causing silent data loss when converting between them.
- **Suggestion:** Either have `notify.Send` accept `model.PushSubscription` directly, or define a minimal interface/struct in one place and reuse it. Alternatively, accept the three fields as parameters to `Send` rather than wrapping them in a near-duplicate struct.

### [MEDIUM] Copy-Paste Middleware Setup in Push Tests
- **File:** `backend/internal/handler/push_test.go`
- **Line(s):** L35-L38, L73-L76, L140-L143, L173-L175, L206-L208, L253-L255, L276-L278, L301-L303
- **Perspective:** Tech Lead
- **Finding:** The three-line middleware wrapping pattern (`authMw := middleware.Auth(...)` / `csrfMw := middleware.CSRF(...)` / `h := authMw(csrfMw(...))`) is repeated 8 times across the test file with only the inner handler varying.
- **Reasoning:** This repetition inflates the test file and makes it fragile -- if the middleware chain changes, all 8 locations need updating. Other test files (e.g., `who_percentiles_test.go`) use a `setupMux` helper that avoids this.
- **Suggestion:** Extract a test helper similar to `setupWHOMux` used in `who_percentiles_test.go`, or create a generic `wrapWithAuth(handler http.HandlerFunc) http.Handler` helper in a shared test utility.

### [MEDIUM] Parameter Sprawl in UpsertPushSubscription
- **File:** `backend/internal/store/push_subscriptions.go`
- **Line(s):** L12
- **Perspective:** Tech Lead
- **Finding:** `UpsertPushSubscription` takes 4 string parameters after `db`: `userID, endpoint, p256dh, auth`. All are bare strings with no type distinction, making call-site mistakes easy (e.g., swapping `p256dh` and `auth`).
- **Reasoning:** Four positional string parameters are error-prone. The function already returns a `*model.PushSubscription`, so accepting a partial struct or a dedicated input type would be more self-documenting.
- **Consequence:** A caller could accidentally swap `p256dh` and `auth` with no compile-time or runtime error, causing push notifications to silently fail.
- **Suggestion:** Accept a struct parameter:
  ```go
  type PushSubscriptionInput struct {
      UserID   string
      Endpoint string
      P256dh   string
      Auth     string
  }
  func UpsertPushSubscription(db *sql.DB, input PushSubscriptionInput) (*model.PushSubscription, error)
  ```

### [LOW] Potentially Dead Code: DeleteAllPushSubscriptionsByUserID
- **File:** `backend/internal/store/push_subscriptions.go`
- **Line(s):** L72-L78
- **Perspective:** Tech Lead
- **Finding:** `DeleteAllPushSubscriptionsByUserID` is exported but has no callers outside its own test file. Given the cascade delete tested in `TestPushSubscription_CascadeDeleteOnUser`, the database already handles bulk deletion when a user is removed.
- **Reasoning:** Unused exported functions increase API surface area and maintenance burden. If user deletion cascades already clean up subscriptions, this function may never be needed.
- **Consequence:** Minor: dead code that must be maintained, tested, and understood by future developers.
- **Suggestion:** If this is intended for future use (e.g., a "log out of all devices" feature), add a comment noting the intended purpose. Otherwise, consider removing it until it is actually needed (YAGNI).

### [LOW] Hardcoded Percentile List
- **File:** `backend/internal/who/who.go`
- **Line(s):** L291
- **Perspective:** Tech Lead
- **Finding:** The percentile values `[]float64{3, 15, 50, 85, 97}` are defined inline within `PercentileCurves`. These same values are asserted against in multiple test files.
- **Reasoning:** If the percentile set ever needs to change (e.g., adding 5th/95th), the values must be updated in the function and in every test that hard-codes the expected list.
- **Consequence:** Minor maintenance burden if percentile curves need to be extended.
- **Suggestion:** Define the standard percentiles as a package-level variable:
  ```go
  var StandardPercentiles = []float64{3, 15, 50, 85, 97}
  ```
  Tests can then reference `who.StandardPercentiles` directly.
