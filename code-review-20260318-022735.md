## Summary

Phases 37-39 add three capabilities: a photo flow integration test (Phase 37), PDF report generation with text/table sections (Phase 38), and PDF charts plus photo appendix (Phase 39). The changes introduce ~2,700 new lines across a `report` package (report.go, charts.go), a handler, storage changes, and comprehensive tests. Overall the code is well-structured with good test coverage. The issues below are mostly about copy-paste duplication and a couple of subtle bugs.

### [MEDIUM] Copy-paste: Three chart-embedding functions are near-identical
- **File:** `backend/internal/report/report.go`
- **Line(s):** L321-L366
- **Perspective:** Tech Lead
- **Finding:** `addChartSection`, `addWeightChartSection`, and `addLabTrendsSection` each follow the exact same pattern: render chart PNG, early-return on error/nil, add section title row, add image row with identical props, add spacer. The only differences are the title string and which render function is called.
- **Reasoning:** Three copies of the same 8-line layout template creates a maintenance burden. If the chart embedding layout changes (e.g., row height, percent, spacing), all three must be updated in lockstep.
- **Consequence:** A future change to chart layout (e.g., adding captions or adjusting sizing) will require editing three places, inviting inconsistency.
- **Suggestion:** Extract a helper like `addChartImage(m, title string, pngData []byte)` that handles the common title-row + image-row + spacer pattern, and call it from each section after rendering:
  ```go
  func addChartImage(m core.Maroto, title string, chartPNG []byte) {
      if chartPNG == nil { return }
      m.AddRows(text.NewRow(8, title, sectionStyle))
      m.AddRows(row.New(80).Add(
          image.NewFromBytesCol(12, chartPNG, extension.Png, props.Rect{Percent: 100, Center: true}),
      ))
      m.AddRows(spacerRow(4))
  }
  ```

### [MEDIUM] Duplicate test helper: `seedBabyWithKasai` vs `seedBabyWithKasaiForHandler`
- **File:** `backend/internal/report/report_test.go` and `backend/internal/handler/report_test.go`
- **Line(s):** report_test.go L20-L43, report_test.go L46-L72 vs handler/report_test.go L17-L37
- **Perspective:** Tech Lead
- **Finding:** `seedBabyWithKasai` in report_test.go and `seedBabyWithKasaiForHandler` in handler/report_test.go are near-identical. Both insert a baby named "Report Baby" with the same DOB and kasai date. The only difference is that the report_test version also fetches the full baby via `getBabyByID`, while the handler version returns a minimal `model.Baby` struct.
- **Reasoning:** Duplicated test fixtures across packages diverge over time and increase maintenance cost.
- **Consequence:** Schema changes to `babies` will require updating both helpers independently, risking missed updates.
- **Suggestion:** Move `seedBabyWithKasai` into the `testutil` package so both test files can share it. The handler test can ignore fields it does not need.

### [MEDIUM] `time.Now()` called twice, producing potentially inconsistent age values
- **File:** `backend/internal/report/report.go`
- **Line(s):** L128 and L452
- **Perspective:** QA
- **Finding:** `time.Now().UTC()` is called once in `queryReportData` (L128) to compute the WHO percentile age range, and again in `addHeader` (L452) to compute the displayed age. These two calls can return different times, especially under load or if data queries are slow.
- **Reasoning:** In extreme cases the baby's computed age could differ between the header and the WHO chart data range, producing a subtle inconsistency in the report.
- **Consequence:** The header could show "3 months, 1 day" while the WHO chart uses data for "3 months, 0 days" -- a minor but confusing discrepancy.
- **Suggestion:** Compute `now` once in `Generate()` or `buildPDF()` and pass it through, rather than calling `time.Now()` in two separate locations. This also improves testability.

### [LOW] No validation that `from <= to` in the report handler
- **File:** `backend/internal/handler/report.go`
- **Line(s):** L28-L43
- **Perspective:** QA
- **Finding:** The handler validates that `from` and `to` are present and well-formatted, but does not check that `from <= to`. A request with `from=2025-09-01&to=2025-08-01` would proceed to generate a report with an empty (or nonsensical) date range.
- **Reasoning:** Other handlers or store functions may silently return empty results for inverted ranges, but the user gets a valid-looking empty PDF with no indication of the error.
- **Consequence:** Users could receive a blank report without understanding why, leading to confusion or support requests.
- **Suggestion:** Add a check after parsing:
  ```go
  fromDate, _ := time.Parse("2006-01-02", from)
  toDate, _ := time.Parse("2006-01-02", to)
  if fromDate.After(toDate) {
      http.Error(w, "from date must not be after to date", http.StatusBadRequest)
      return
  }
  ```

### [LOW] `detectImageExtension` uses fragile magic-byte check
- **File:** `backend/internal/report/report.go`
- **Line(s):** L397-L405
- **Perspective:** QA
- **Finding:** PNG detection checks `string(data[:8]) == "\x89PNG\r\n\x1a\n"` which compares binary bytes via string conversion. While this works, `net/http.DetectContentType` or `bytes.HasPrefix` would be more idiomatic and handle edge cases better. The function also defaults to PNG for unrecognized formats, which could produce corrupt images in the PDF.
- **Reasoning:** If a JPEG thumbnail has a corrupted or truncated header that does not match the 2-byte check, it would be incorrectly labeled as PNG, causing the PDF library to fail or produce a broken image.
- **Consequence:** Corrupt or WebP images silently default to PNG extension, potentially producing broken image rendering in the PDF.
- **Suggestion:** Use `http.DetectContentType(data)` and map MIME types to extensions, or at minimum log a warning when falling back to the default.

### [LOW] `fetchPhotoThumbnails` silently swallows errors
- **File:** `backend/internal/report/report.go`
- **Line(s):** L156-L208
- **Perspective:** QA
- **Finding:** The function returns `nil` on date parsing errors (L163) and continues silently on row scan errors (L186) and thumbnail fetch errors (L201). While graceful degradation is reasonable for a report, the date parsing failure at L163 is a bug symptom that should surface.
- **Reasoning:** If `ParseDateRange` fails, it means the caller passed invalid dates. This is a programming error in the report generation pipeline, not a runtime condition to silently handle.
- **Consequence:** If date format assumptions change, this function silently produces empty photo sections with no log output, making debugging difficult.
- **Suggestion:** At minimum, log the date parsing error like the other error paths: `log.Printf("parse date range for photos: %v", err)`.

### [LOW] Stringly-typed date format used inconsistently
- **File:** `backend/internal/report/charts.go`
- **Line(s):** L100
- **Perspective:** Tech Lead
- **Finding:** `renderWeightChart` accepts `dob string` and parses it with the hardcoded format `"2006-01-02"` (L100), while the rest of the codebase uses `model.DateFormat` for the same pattern. The caller at report.go L311 formats with `model.DateFormat`, so this works, but the parsing side should use the constant too.
- **Reasoning:** Using a raw string literal instead of the existing `model.DateFormat` constant creates a risk of divergence if the date format ever changes.
- **Consequence:** If `model.DateFormat` is updated, this parsing will break silently.
- **Suggestion:** Replace `time.Parse("2006-01-02", dob)` with `time.Parse(model.DateFormat, dob)`.
