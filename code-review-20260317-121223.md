## Summary

This review covers the last 3 commits (refactor of phases 19-21, plus new phases 22-24) which add photo upload, photo linking/unlinking across 5 metric types (stools, abdomen, bruising, skin_observations, general_notes), and account deletion anonymization. The review is scoped strictly to **code reuse** issues: duplicated patterns, missed use of existing utilities, and copy-paste that should be extracted.

---

### [HIGH] Delete handlers use `strings.Split` instead of existing `parsePhotoKeysJSON`
- **File:** `backend/internal/handler/abdomen.go`, `bruising.go`, `general_notes.go`, `skin_observations.go`, `stools.go`
- **Line(s):** abdomen.go L289, bruising.go L285, stools.go L292, skin_observations.go L286, general_notes.go L270
- **Perspective:** Tech Lead
- **Finding:** All 5 delete handlers parse `existing.PhotoKeys` with `strings.Split(*existing.PhotoKeys, ",")`, but `photo_keys` is stored as a **JSON array** (e.g., `["photos/a.jpg","photos/b.jpg"]`). The existing helper `parsePhotoKeysJSON` in `helpers.go` (L222) correctly uses `json.Unmarshal` for this format. The delete handlers are using the wrong parsing method.
- **Reasoning:** `handlePhotoLinking` (used in create/update paths) stores photo keys as a JSON array via `json.Marshal`. `parsePhotoKeysJSON` correctly unmarshals this format. But the delete handlers use `strings.Split(value, ",")` on a JSON string like `["photos/a.jpg","photos/b.jpg"]`, which will produce malformed keys containing `[`, `]`, and `"` characters.
- **Consequence:** Photo unlinking on delete will silently fail for every entry that has photos, because the split keys will not match any `r2_key` in the `photo_uploads` table. Photos will become orphaned and never have their `linked_at` cleared. This is also a **correctness bug**, not just a style issue.
- **Suggestion:** Replace all 5 instances with the existing `parsePhotoKeysJSON` helper:
  ```go
  // BEFORE (in all 5 delete handlers):
  if existing.PhotoKeys != nil && *existing.PhotoKeys != "" {
      oldKeys := strings.Split(*existing.PhotoKeys, ",")
      if unlinkErr := store.UnlinkPhotos(db, oldKeys); unlinkErr != nil {
          log.Printf("unlink photos on delete: %v", unlinkErr)
      }
  }

  // AFTER:
  if oldKeys := parsePhotoKeysJSON(existing.PhotoKeys); len(oldKeys) > 0 {
      if unlinkErr := store.UnlinkPhotos(db, oldKeys); unlinkErr != nil {
          log.Printf("unlink photos on delete: %v", unlinkErr)
      }
  }
  ```

### [HIGH] Delete handler photo-unlink block is copy-pasted 5 times instead of being extracted
- **File:** `backend/internal/handler/abdomen.go`, `bruising.go`, `general_notes.go`, `skin_observations.go`, `stools.go`
- **Line(s):** abdomen.go L285-296, bruising.go L281-292, stools.go L284-296, skin_observations.go L282-293, general_notes.go L266-277
- **Perspective:** Tech Lead
- **Finding:** The entire "fetch existing entry, parse photo keys, unlink photos, then delete" block is copied verbatim across all 5 delete handlers. This is the same pattern that was already centralized for create/update via `handlePhotoLinking` in `helpers.go`.
- **Reasoning:** The project already demonstrates a pattern of extracting shared logic into helpers (e.g., `handlePhotoLinking`, `resolvePhotos`, `mapMetricPage`). The delete-time unlinking follows an identical structure across all 5 files and should follow the same approach.
- **Consequence:** Any fix to the parsing bug above (or future changes to unlink behavior) must be applied in 5 places. Each copy is a divergence risk.
- **Suggestion:** Add a helper to `helpers.go`:
  ```go
  // unlinkAllPhotos parses and unlinks all photos from a photo_keys JSON string.
  func unlinkAllPhotos(db *sql.DB, photoKeys *string) {
      keys := parsePhotoKeysJSON(photoKeys)
      if len(keys) == 0 {
          return
      }
      if err := store.UnlinkPhotos(db, keys); err != nil {
          log.Printf("unlink photos on delete: %v", err)
      }
  }
  ```
  Then each delete handler reduces to: `unlinkAllPhotos(db, existing.PhotoKeys)`.

### [MEDIUM] Variadic `objStores ...storage.ObjectStore` extraction pattern repeated 20 times
- **File:** `backend/internal/handler/abdomen.go`, `bruising.go`, `general_notes.go`, `skin_observations.go`, `stools.go`
- **Line(s):** 20 occurrences (4 handlers x 5 files), e.g., abdomen.go L82-85, L137-140, L171-174, L204-207
- **Perspective:** Tech Lead
- **Finding:** Every handler function across all 5 metric types repeats the exact same 4-line variadic extraction:
  ```go
  func XxxHandler(db *sql.DB, objStores ...storage.ObjectStore) http.HandlerFunc {
      var objStore storage.ObjectStore
      if len(objStores) > 0 {
          objStore = objStores[0]
      }
  ```
  This pattern appears 20 times (Create, List, Get, Update for each of the 5 types).
- **Reasoning:** This is a textbook case for a one-line helper function. The variadic pattern was chosen for backward compatibility (avoiding breaking existing callers that don't pass an ObjectStore), but the extraction logic is identical every time.
- **Consequence:** 20 copies of boilerplate that obscure the actual handler logic and increase line count. Any change to how the optional ObjectStore is resolved requires modifying 20 call sites.
- **Suggestion:** Add a helper to `helpers.go`:
  ```go
  // firstObjStore returns the first ObjectStore from a variadic list, or nil.
  func firstObjStore(stores []storage.ObjectStore) storage.ObjectStore {
      if len(stores) > 0 {
          return stores[0]
      }
      return nil
  }
  ```
  Then each handler becomes: `objStore := firstObjStore(objStores)`.

### [MEDIUM] Five identical `toXxxResponseWithPhotos` functions follow the same pattern
- **File:** `backend/internal/handler/abdomen.go`, `bruising.go`, `general_notes.go`, `skin_observations.go`, `stools.go`
- **Line(s):** abdomen.go L75-78, bruising.go L81-84, stools.go L84-87, skin_observations.go L72-75, general_notes.go L66-69
- **Perspective:** Tech Lead
- **Finding:** Each of the 5 `toXxxResponseWithPhotos` functions follows the exact same 3-line pattern:
  ```go
  func toXxxResponseWithPhotos(m *model.Xxx, db *sql.DB, objStore storage.ObjectStore, r *http.Request) xxxResponse {
      resp := toXxxResponse(m)
      resp.Photos = resolvePhotos(r.Context(), db, objStore, m.PhotoKeys)
      return resp
  }
  ```
  The only difference is the concrete model/response types.
- **Reasoning:** Go's type system makes full generalization difficult without interfaces, and each response struct is different. However, these functions are so trivially mechanical that they could be eliminated by having each `toXxxResponse` function accept optional photo-resolution parameters, or by inlining the `resolvePhotos` call at the point of use (since it is always called immediately after `toXxxResponse`).
- **Consequence:** Moderate. Five 4-line functions is tolerable, but the pattern will grow as more photo-enabled metrics are added. Currently acceptable given Go's type constraints.
- **Suggestion:** This is borderline acceptable. If more metric types gain photo support, consider an interface-based approach or code generation. For now, document the pattern so future developers copy it consistently. Alternatively, add a `Photos` field setter:
  ```go
  // Each toXxxResponse could inline the resolvePhotos call directly,
  // eliminating the WithPhotos variant entirely.
  ```

### [MEDIUM] Create handler photo-linking boilerplate duplicated across 5 files
- **File:** `backend/internal/handler/abdomen.go`, `bruising.go`, `general_notes.go`, `skin_observations.go`, `stools.go`
- **Line(s):** e.g., abdomen.go L115-122, bruising.go L116-123, stools.go L120-128, skin_observations.go L112-119, general_notes.go L101-108
- **Perspective:** Tech Lead
- **Finding:** Every Create handler repeats the same 8-line block:
  ```go
  var photoKeysStr *string
  if len(req.PhotoKeys) > 0 {
      var errMsg string
      photoKeysStr, errMsg, ok = handlePhotoLinking(db, baby.ID, nil, req.PhotoKeys)
      if !ok {
          http.Error(w, "invalid photo_keys: "+errMsg, http.StatusBadRequest)
          return
      }
  }
  ```
  This is identical across all 5 create handlers. The update handlers have a similar (though slightly different) pattern.
- **Reasoning:** The `handlePhotoLinking` helper already centralizes the complex linking logic, but the boilerplate around calling it and writing the HTTP error is duplicated.
- **Consequence:** Changes to error formatting or the guard condition before calling `handlePhotoLinking` would need to be replicated in 5 places (10 counting update handlers).
- **Suggestion:** Extract into a helper that returns the photoKeysStr and writes the error response if needed:
  ```go
  // linkPhotosForCreate validates and links photos for a create operation.
  // Returns the photo keys JSON string and true, or writes a 400 response and returns false.
  func linkPhotosForCreate(w http.ResponseWriter, db *sql.DB, babyID string, photoKeys []string) (*string, bool) {
      if len(photoKeys) == 0 {
          return nil, true
      }
      result, errMsg, ok := handlePhotoLinking(db, babyID, nil, photoKeys)
      if !ok {
          http.Error(w, "invalid photo_keys: "+errMsg, http.StatusBadRequest)
          return nil, false
      }
      return result, true
  }
  ```

### [MEDIUM] Router does not pass `objStore` to metric handlers
- **File:** `backend/internal/handler/router.go`
- **Line(s):** L85-114
- **Perspective:** QA
- **Finding:** The `registerMetricCRUD` calls for stools, abdomen, skin, bruising, and general_notes do not pass `cfg.objStore` to the handler constructors. For example:
  ```go
  registerMetricCRUD(mux, "/api/babies/{id}/stools", rateMw, authMw, csrfMw,
      CreateStoolHandler(cfg.db), ListStoolsHandler(cfg.db),
      GetStoolHandler(cfg.db), UpdateStoolHandler(cfg.db), DeleteStoolHandler(cfg.db))
  ```
  Since the variadic `objStores` parameter receives nothing, `objStore` will be `nil`, and `resolvePhotos` will always return an empty photos array in production.
- **Reasoning:** The handlers were updated to accept `...storage.ObjectStore` but the router was not updated to pass `cfg.objStore`. The photo upload endpoint correctly receives it, but the CRUD endpoints do not.
- **Consequence:** In production, all metric responses will have `"photos": []` even when photos are linked, because `resolvePhotos` returns early when `objStore == nil`. Photo linking works (create/update store the keys), but photo resolution in responses is broken.
- **Suggestion:** Pass `cfg.objStore` to each photo-enabled metric handler:
  ```go
  registerMetricCRUD(mux, "/api/babies/{id}/stools", rateMw, authMw, csrfMw,
      CreateStoolHandler(cfg.db, cfg.objStore), ListStoolsHandler(cfg.db, cfg.objStore),
      GetStoolHandler(cfg.db, cfg.objStore), UpdateStoolHandler(cfg.db, cfg.objStore),
      DeleteStoolHandler(cfg.db, cfg.objStore))
  ```
