## Summary

Review of phases 34-36 covering the medication notification scheduler (`notify/scheduler.go`), cron cleanup jobs (`cron/cleanup.go`, `cron/runner.go`), and the medication flow integration test. The code is generally clean and well-tested. The issues below focus on duplicated mock types, parameter sprawl, stringly-typed patterns, and a subtle correctness concern.

### [MEDIUM] Duplicated MockPusher across packages
- **File:** `backend/internal/notify/notify_test.go` and `backend/internal/integration/medication_flow_test.go`
- **Line(s):** notify_test.go L13-32, medication_flow_test.go L20-47
- **Perspective:** Tech Lead
- **Finding:** `MockPusher` in the `notify` package and `mockPusher` in the `integration` package are near-identical implementations of the same test double (record sends, return a canned response). The integration version adds `sendCount()` and `lastPayload()` helpers but is structurally the same.
- **Reasoning:** The `notify.MockPusher` is already exported (capitalized). The integration package could import and use it directly, adding convenience wrappers if needed, rather than maintaining a second copy.
- **Consequence:** If the `Pusher` interface changes (e.g., adding a method or changing the signature), both mocks must be updated independently. Easy to forget one.
- **Suggestion:** Use `notify.MockPusher` in the integration tests. If `sendCount()`/`lastPayload()` are needed, add them to the exported mock or wrap it:
  ```go
  type pusherHelper struct {
      *notify.MockPusher
  }
  func (h *pusherHelper) sendCount() int { ... }
  ```

### [MEDIUM] Parameter sprawl in checkScheduleTime
- **File:** `backend/internal/notify/scheduler.go`
- **Line(s):** L126
- **Perspective:** Tech Lead
- **Finding:** `checkScheduleTime` takes 5 parameters: `(med activeMed, schedTime string, nowLocal, nowUTC time.Time, loc *time.Location)`. The `nowLocal` and `loc` parameters are redundant since `nowLocal.Location()` already provides `loc`, and `nowUTC` can be derived via `nowLocal.UTC()`. Additionally, `nowLocal` itself is only used for `.Year()`, `.Month()`, `.Day()` to construct `scheduledLocal`, which could be done from `nowUTC` converted using `loc`.
- **Reasoning:** Passing the same information in multiple forms (UTC time, local time, and location separately) creates implicit coupling -- callers must ensure these are consistent. This is a classic symptom of parameter sprawl.
- **Consequence:** A future caller could pass mismatched `nowLocal`/`nowUTC`/`loc` values, leading to silent scheduling bugs that only manifest in certain timezones.
- **Suggestion:** Reduce to `(med activeMed, schedTime string, now time.Time, loc *time.Location)` where `now` is always UTC, and derive `nowLocal := now.In(loc)` inside the function:
  ```go
  func (s *Scheduler) checkScheduleTime(med activeMed, schedTime string, nowUTC time.Time, loc *time.Location) {
      nowLocal := nowUTC.In(loc)
      // ...
  }
  ```

### [MEDIUM] Stringly-typed time format repeated as bare literal
- **File:** `backend/internal/notify/scheduler.go`
- **Line(s):** L167-168, L253
- **Perspective:** Tech Lead
- **Finding:** The time format string `"2006-01-02T15:04:05Z"` appears 3 times in `scheduler.go` (lines 167, 168, 253) and also in `scheduler_test.go` (lines 109-110). This is effectively `time.RFC3339` without fractional seconds, but it is written as a bare string literal each time.
- **Reasoning:** `time.RFC3339` is `"2006-01-02T15:04:05Z07:00"` which includes timezone offset formatting, so it is not a direct substitute. However, the repeated literal should be a package-level constant to avoid typo risk and signal intent.
- **Consequence:** A typo in one occurrence (e.g., `"2006-01-02T15:04:05"` missing the trailing `Z`) would cause silent formatting mismatches between the suppression query and the payload data, potentially breaking suppression logic.
- **Suggestion:** Define a constant:
  ```go
  const utcTimeFormat = "2006-01-02T15:04:05Z"
  ```

### [LOW] Duplicate test setup boilerplate in scheduler_test.go
- **File:** `backend/internal/notify/scheduler_test.go`
- **Line(s):** L135-161, L193-216, L220-243, L247-281 (and many more)
- **Perspective:** Tech Lead
- **Finding:** Nearly every test in `scheduler_test.go` repeats the same 6-line preamble: `testDB`, `seedUser`, `seedBaby`, `seedPushSubscription`, `seedMedication`, then construct `now` from a location. At least 15 tests share this identical setup with only minor variations (timezone, schedule times, active flag).
- **Reasoning:** While some repetition in tests is acceptable for clarity, this level of duplication obscures what each test uniquely verifies. A table-driven approach or a shared `setupSchedulerTest` helper would reduce noise.
- **Consequence:** Adding a new seed step (e.g., if the schema adds a required column) requires editing 15+ test functions.
- **Suggestion:** Extract a helper that returns the common fixtures:
  ```go
  func setupSchedulerFixtures(t *testing.T) (*sql.DB, *MockPusher, *Scheduler, time.Time) {
      db := testDB(t)
      seedUser(t, db, "u1", "g1", "u1@test.com")
      seedBaby(t, db, "b1", "u1")
      seedPushSubscription(t, db, "u1", "https://push.example.com/u1")
      // ...
  }
  ```

### [LOW] Separate testDB helper duplicates testutil.SetupTestDB
- **File:** `backend/internal/notify/scheduler_test.go`
- **Line(s):** L19-58
- **Perspective:** Tech Lead
- **Finding:** The `notify` package has its own `testDB` function that opens an in-memory SQLite database and manually runs migrations. The rest of the codebase uses `testutil.SetupTestDB(t)` for the same purpose (as seen in the `cron` tests).
- **Reasoning:** The comment says "to avoid circular imports we use database/sql directly," but `testutil` is a test-only package that should not cause circular imports with `notify`.
- **Consequence:** If a migration changes, the `testDB` function might behave differently from `testutil.SetupTestDB` if they diverge in how they apply migrations. Also, the `projectRoot` function using `runtime.Caller` is fragile if files are moved.
- **Suggestion:** Verify whether `testutil.SetupTestDB` can be used here. If there is a genuine circular import issue, document it clearly. Otherwise, switch to the shared helper.

### [LOW] verifyEntryCountByTable uses fmt.Sprintf for SQL table name
- **File:** `backend/internal/integration/medication_flow_test.go`
- **Line(s):** L499-509
- **Perspective:** SecOps
- **Finding:** `verifyEntryCountByTable` constructs SQL with `fmt.Sprintf("SELECT COUNT(*) FROM %s WHERE baby_id = ?", table)`. While this is a test helper with hardcoded table names and no user input, it establishes a pattern of string-interpolated SQL that contradicts the project convention of "parameterized queries only."
- **Reasoning:** Table names cannot be parameterized in SQL, so this is technically necessary. However, the function could validate the table name against an allow-list to be defensive.
- **Consequence:** Minimal in practice since this is test-only code with hardcoded callers, but it sets a precedent.
- **Suggestion:** Add a brief comment acknowledging the pattern, or validate the table name:
  ```go
  allowedTables := map[string]bool{"medications": true, "med_logs": true, ...}
  if !allowedTables[table] {
      t.Fatalf("unexpected table: %s", table)
  }
  ```
