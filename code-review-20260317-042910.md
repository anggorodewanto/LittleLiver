## Summary

Review of three new files in `backend/internal/model/`: `ulid.go` (ULID generator), `enums.go` (enum validation helpers), and `types.go` (domain struct definitions). These files establish the core model layer for the LittleLiver application. The code is generally clean and idiomatic Go, with good test coverage across all three files. Five issues were identified, ranging from an unnecessary mutex to stringly-typed domain fields.

### [MEDIUM] Unnecessary Mutex in ULID Generator
- **File:** `backend/internal/model/ulid.go`
- **Line(s):** L12, L17-L18
- **Perspective:** Performance Engineer
- **Finding:** The `ulidMu sync.Mutex` guards `NewULID()`, but the function has no shared mutable state. Each call reads `time.Now()` (safe for concurrent use) and generates fresh random bytes from `crypto/rand` (also safe for concurrent use). The mutex serializes all ULID generation unnecessarily.
- **Reasoning:** A mutex on a function with no shared state is pure overhead. Under concurrent request handling (e.g., multiple parents logging entries simultaneously), this creates an artificial bottleneck where every ULID generation must wait for the previous one to complete.
- **Consequence:** Under moderate concurrency (10+ goroutines generating ULIDs), this serialization adds latency to every request that creates an entity. While unlikely to be severe for a personal-use app, it is unnecessary complexity that misleads future readers into thinking shared state exists.
- **Suggestion:** Remove the mutex entirely:
  ```go
  func NewULID() string {
      ms := uint64(time.Now().UnixMilli())
      // ... rest of function unchanged
  }
  ```

### [MEDIUM] Stringly-Typed Domain Fields Where Type Aliases Would Add Safety
- **File:** `backend/internal/model/types.go`
- **Line(s):** L19 (Baby.Sex), L32 (BabyParent.Role)
- **Perspective:** Tech Lead
- **Finding:** Fields like `Sex string` and `Role string` accept any string value at the type level. The enum validators in `enums.go` exist but are disconnected from the struct definitions -- nothing in the type system ties them together. A caller can construct `Baby{Sex: "banana"}` without any compile-time feedback.
- **Reasoning:** Go lacks sum types, but a named type alias (e.g., `type Sex string` with exported constants) would make the code self-documenting, enable IDE autocomplete for valid values, and make it clear which validator function corresponds to which field. The current approach requires the developer to know the convention that `Baby.Sex` uses `ValidSex()`.
- **Consequence:** As more contributors or handlers are added, the disconnect between struct fields and their validators increases the risk of passing unvalidated strings. Bugs would only surface at runtime (or worse, as corrupt data in SQLite).
- **Suggestion:** Introduce named string types for the most critical enums:
  ```go
  type Sex string

  const (
      SexMale   Sex = "male"
      SexFemale Sex = "female"
  )

  func (s Sex) Valid() bool { return s == SexMale || s == SexFemale }
  ```
  This can be adopted incrementally for the most important enums without rewriting everything at once.

### [LOW] Panic on crypto/rand Failure Instead of Returning Error
- **File:** `backend/internal/model/ulid.go`
- **Line(s):** L38-L40
- **Perspective:** QA
- **Finding:** The function panics if `crypto/rand.Read` fails. While `crypto/rand.Read` failing is extremely rare (it would indicate a broken OS entropy source), panicking in a library function is generally discouraged in Go. The conventional approach is to return `(string, error)`.
- **Reasoning:** A panic in a request handler crashes the entire goroutine and, depending on the HTTP framework's recovery middleware, could produce a 500 error with no meaningful context. Returning an error allows the caller to handle it gracefully (e.g., log and return a proper error response). That said, on modern Linux systems, `crypto/rand.Read` effectively never fails, and many ULID libraries do panic here, so this is a pragmatic trade-off.
- **Consequence:** If the OS entropy pool is somehow exhausted or unavailable (e.g., in a constrained container environment during early boot), the application would crash rather than degrade gracefully. For a personal-use app, this is low risk.
- **Suggestion:** Consider changing the signature to `func NewULID() (string, error)` and letting callers decide how to handle the (near-impossible) failure. Alternatively, document the panic contract in the function's doc comment so callers are aware. The current approach is acceptable if the team agrees on the convention.

### [LOW] Enum Validators Accept Empty String Silently as False
- **File:** `backend/internal/model/enums.go`
- **Line(s):** L29-L65 (all validator functions)
- **Perspective:** QA
- **Finding:** All validator functions return `false` for empty strings, which is correct behavior. However, there is no distinction between "value is missing" (empty string) and "value is invalid" (e.g., "banana"). Callers cannot differentiate between a missing required field and an invalid enum value to provide user-friendly error messages like "sex is required" vs "sex must be one of: male, female".
- **Reasoning:** For a personal-use app, generic "invalid value" errors are probably sufficient. However, as the API matures and form validation becomes important for the frontend, distinguishing "missing" from "invalid" improves the user experience for both parents.
- **Consequence:** Users receive the same error message whether they forgot to fill in a field or entered an unsupported value, making troubleshooting slightly harder.
- **Suggestion:** This is a low-priority enhancement. If desired later, the validators could return a richer result, or the handler layer could check for empty strings before calling the validator. No immediate action needed.

### [LOW] Test Coverage for Types File Tests Only Field Assignment
- **File:** `backend/internal/model/types_test.go`
- **Line(s):** L10-L138
- **Perspective:** QA
- **Finding:** The `types_test.go` tests verify that struct fields can be assigned and read back, which essentially tests Go's struct initialization -- something guaranteed by the language. These tests do not verify any behavior (there is none on these structs yet), JSON serialization, or the `omitempty` / `json:"-"` tag behavior (e.g., that `Session.Token` is excluded from JSON output).
- **Reasoning:** While having tests is better than not, these tests provide a false sense of coverage without testing anything meaningful. The most valuable thing to test on these structs would be the JSON tags -- specifically that `Token` is excluded from marshaled output (the `json:"-"` tag) and that `omitempty` fields behave correctly with nil pointers.
- **Consequence:** If someone accidentally removes the `json:"-"` tag from `Session.Token`, the session secret would leak in API responses with no test catching it.
- **Suggestion:** Replace or supplement the field-assignment tests with JSON serialization tests:
  ```go
  func TestSessionTokenExcludedFromJSON(t *testing.T) {
      s := model.Session{Token: "secret-token"}
      data, _ := json.Marshal(s)
      if strings.Contains(string(data), "secret-token") {
          t.Fatal("Session.Token should not appear in JSON output")
      }
  }
  ```
