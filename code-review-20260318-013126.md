## Summary

This review examines code reuse issues across phases 34-36, which introduced the medication reminder `Scheduler`, the cron `Runner` for periodic cleanup, an integration test for the medication notification lifecycle, and the `delete_account` handler update. The focus is on duplicated helpers, reimplemented utilities, and structurally duplicated patterns.

### [MEDIUM] Duplicated `testDB()` helper reimplements `testutil.SetupTestDB`
- **File:** `backend/internal/notify/scheduler_test.go`
- **Line(s):** L19-L51
- **Perspective:** Tech Lead
- **Finding:** The `testDB()` function manually opens a SQLite database, reads migration files from disk, and applies them one by one. This is a reimplementation of `testutil.SetupTestDB()` (in `backend/internal/testutil/testutil.go` L32-L50), which does the same thing via `store.OpenDB` and `store.RunMigrations`.
- **Reasoning:** There is no circular import preventing the use of `testutil.SetupTestDB` here. The `scheduler_test.go` file is in `package notify`, and `testutil` imports `store`, `model`, and `middleware` -- none of which import `notify`. The test file can safely import `testutil`. The comment on L22 ("To avoid circular imports we use database/sql directly") is incorrect.
- **Consequence:** When migration logic changes (e.g., new pragma settings, migration ordering rules), `testDB()` must be updated separately from `testutil.SetupTestDB`, creating a maintenance burden and a risk of inconsistent test database setup across packages.
- **Suggestion:** Replace the local `testDB()` with `testutil.SetupTestDB(t)` and remove the `projectRoot()` helper on L53-L58, which is also duplicated from `testutil`.
```go
import "github.com/ablankz/LittleLiver/backend/internal/testutil"

// Replace all calls to testDB(t) with:
db := testutil.SetupTestDB(t)
```

### [MEDIUM] Duplicated `projectRoot()` helper
- **File:** `backend/internal/notify/scheduler_test.go`
- **Line(s):** L53-L58
- **Perspective:** Tech Lead
- **Finding:** The `projectRoot()` function uses `runtime.Caller(0)` and filepath manipulation to locate the backend root directory. An identical implementation exists in `testutil.projectRoot()` (`backend/internal/testutil/testutil.go` L25-L29). Both use the same `runtime.Caller` + `filepath.Join(filepath.Dir(filename), "..", "..")` pattern.
- **Reasoning:** This function only exists to support the local `testDB()` helper. Removing `testDB()` in favor of `testutil.SetupTestDB` eliminates the need for this function entirely.
- **Consequence:** Two parallel implementations of the same path-resolution logic that must be kept in sync when project structure changes.
- **Suggestion:** Remove this function when switching to `testutil.SetupTestDB`.

### [MEDIUM] Duplicated `seedUser` and `seedBaby` helpers vs `testutil.CreateTestUser` / `testutil.CreateTestBaby`
- **File:** `backend/internal/notify/scheduler_test.go`
- **Line(s):** L61-L81
- **Perspective:** Tech Lead
- **Finding:** `seedUser()` and `seedBaby()` manually insert rows into `users`, `babies`, and `baby_parents` tables using raw SQL. The `testutil` package already provides `CreateTestUser()` and `CreateTestBaby()` which do the same thing but additionally use the `store` layer (ensuring consistency with production code paths) and generate unique IDs via an atomic counter to avoid collisions.
- **Reasoning:** The local seed helpers hardcode IDs like `"u1"`, `"b1"` which works for these tests but is a less robust pattern. The `testutil` versions use `model.NewULID()` and auto-incrementing counters. Using the existing utilities would reduce code and ensure test setup stays consistent with schema evolution.
- **Consequence:** Schema changes to `users` or `babies` tables (e.g., new NOT NULL columns) require updates in both `testutil` and `scheduler_test.go` seed helpers separately.
- **Suggestion:** Replace `seedUser` and `seedBaby` with `testutil.CreateTestUser` and `testutil.CreateTestBaby`. The tests would need minor adjustments to use the returned user/baby IDs instead of hardcoded strings, but this is a net improvement in maintainability.

### [MEDIUM] Duplicated `mockPusher` in integration test vs `MockPusher` in notify package
- **File:** `backend/internal/integration/medication_flow_test.go`
- **Line(s):** L20-L47
- **Perspective:** Tech Lead
- **Finding:** The `mockPusher` struct in `medication_flow_test.go` duplicates the `MockPusher` from `backend/internal/notify/notify_test.go` (L13-L32). Both record sends with subscription and payload, both use a mutex for thread safety, and both return `http.StatusCreated`. The integration version adds `sendCount()` and `lastPayload()` convenience methods, but the core structure is identical.
- **Reasoning:** The `MockPusher` in `notify_test.go` is exported (capital M) and could be imported by integration tests. The integration test is in `package integration_test`, so it can import `notify` without issues.
- **Consequence:** Bug fixes or interface changes to `Pusher` require updating two independent mock implementations. For example, if `Send` gains a context parameter, both mocks must be updated.
- **Suggestion:** Use the exported `notify.MockPusher` in the integration test. Add the `sendCount()` and `lastPayload()` convenience methods to the canonical `MockPusher`, or wrap it in the integration test.
```go
// In medication_flow_test.go:
mock := &notify.MockPusher{}
s := notify.NewScheduler(db, mock)
s.Tick(t0800)
if len(mock.Sends) != 1 { ... }
```

### [MEDIUM] Duplicated `seedPushSub` / `seedPushSubscription` across test files
- **File:** `backend/internal/integration/medication_flow_test.go`
- **Line(s):** L486-L496
- **Perspective:** Tech Lead
- **Finding:** `seedPushSub()` in the integration test (L486-L496) is functionally identical to `seedPushSubscription()` in `scheduler_test.go` (L122-L131). Both insert a row into `push_subscriptions` with the same columns and dummy values for `p256dh` and `auth`.
- **Reasoning:** This helper is used in two packages and should be centralized. Since push subscriptions are now a core test fixture (used by both scheduler and integration tests), it belongs in `testutil`.
- **Consequence:** Adding a new column to `push_subscriptions` (e.g., `user_agent`, `created_at`) requires updating both seed functions separately.
- **Suggestion:** Add a `CreateTestPushSubscription(t, db, userID, endpoint)` function to `backend/internal/testutil/testutil.go` and use it from both test files.

### [LOW] Duplicated ticker+stop channel pattern between `Runner` and `Scheduler`
- **File:** `backend/internal/cron/runner.go` L36-L61 and `backend/internal/notify/scheduler.go` L32-L49
- **Line(s):** N/A - structural issue
- **Perspective:** Tech Lead
- **Finding:** Both `Runner.run()` and `Scheduler.Start()` implement the same pattern: create a `time.Ticker`, select on `ticker.C` and a `stop` channel, and clean up with `defer ticker.Stop()`. The `Runner` additionally has a `done` channel for synchronous shutdown and runs the task immediately on start.
- **Reasoning:** While the pattern is a common Go idiom and each is only ~15 lines, the two structs serve the same structural purpose (periodic task execution with graceful shutdown). However, they have different enough requirements (Runner runs in a goroutine with `done` signaling, Scheduler blocks the caller; Runner takes an interval parameter, Scheduler hardcodes 1 minute; Runner runs immediately on start) that extracting a shared abstraction would add complexity without proportional benefit for a personal-use app.
- **Consequence:** Minimal -- both implementations are simple and correct. A shared abstraction would be over-engineering for this codebase size.
- **Suggestion:** No action needed. This is noted for awareness but does not warrant a refactor given the project's "minimal dependencies, no over-engineering" constraint.
