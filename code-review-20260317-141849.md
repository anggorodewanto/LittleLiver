## Summary

This review covers phases 10-12 of the LittleLiver project: rate limiting middleware, baby CRUD endpoints, and invite code generation/join flow. The focus is on **code reuse** opportunities -- identifying duplicated logic, missed existing utilities, and inline code that could leverage shared helpers.

### [HIGH] Duplicated baby row scanning logic between `GetBabyByID` and `GetBabiesByUserID`
- **File:** `backend/internal/store/babies.go`
- **Line(s):** L108-L148 (GetBabyByID) and L193-L243 (GetBabiesByUserID)
- **Perspective:** Tech Lead
- **Finding:** The baby row scanning logic -- scanning columns into local variables (`dobStr`, `diagStr`, `kasaiStr`, `notesStr`, `createdStr`), then calling `parseTime` on each, handling `sql.NullString` for optional date fields and notes -- is duplicated verbatim between `GetBabyByID` and the inner loop of `GetBabiesByUserID`. This is approximately 30 lines of identical logic.
- **Reasoning:** Any schema change to the babies table (adding a column, renaming a field) requires updating both scan sites in lockstep. This is a classic maintenance trap and violates DRY within the same file.
- **Consequence:** A future column addition to the babies table will require modifying two scan blocks. If only one is updated, queries will silently fail or return incorrect data.
- **Suggestion:** Extract a `scanBaby` helper function:
  ```go
  func scanBaby(scanner interface{ Scan(dest ...any) error }) (*model.Baby, error) {
      var b model.Baby
      var dobStr string
      var diagStr, kasaiStr, notesStr sql.NullString
      var createdStr string
      err := scanner.Scan(&b.ID, &b.Name, &b.Sex, &dobStr, &diagStr, &kasaiStr,
          &b.DefaultCalPerFeed, &notesStr, &createdStr)
      if err != nil {
          return nil, err
      }
      // ... parse fields ...
      return &b, nil
  }
  ```
  Both `GetBabyByID` and `GetBabiesByUserID` can then call `scanBaby(row)` or `scanBaby(rows)`.

### [HIGH] Duplicated validation logic between `CreateBabyHandler` and `UpdateBabyHandler`
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L87-L112 (CreateBabyHandler) and L247-L272 (UpdateBabyHandler)
- **Perspective:** Tech Lead
- **Finding:** The validation block in `CreateBabyHandler` (required field checks, sex enum validation, date format parsing for `date_of_birth`, `diagnosis_date`, and `kasai_date`) is copy-pasted identically into `UpdateBabyHandler`. This is approximately 25 lines of duplicated validation.
- **Reasoning:** Both handlers validate the same `babyRequest` struct with the same rules. When validation rules change (e.g., adding a new allowed sex value, changing date format), both handlers must be updated in lockstep.
- **Consequence:** A validation rule change applied to only one handler creates inconsistent behavior between create and update paths, leading to data integrity issues.
- **Suggestion:** Extract a `validateBabyRequest` method on `babyRequest`:
  ```go
  func (req *babyRequest) validate() (string, bool) {
      if req.Name == "" || req.Sex == "" || req.DateOfBirth == "" {
          return "name, sex, and date_of_birth are required", false
      }
      if req.Sex != "male" && req.Sex != "female" {
          return "sex must be 'male' or 'female'", false
      }
      if _, err := time.Parse("2006-01-02", req.DateOfBirth); err != nil {
          return "date_of_birth must be in YYYY-MM-DD format", false
      }
      // ... optional date checks ...
      return "", true
  }
  ```

### [MEDIUM] Duplicated authorization check pattern (get baby + check parent) across 3 handlers
- **File:** `backend/internal/handler/babies.go` and `backend/internal/handler/invites.go`
- **Line(s):** `babies.go` L167-L193 (GetBabyHandler), L218-L239 (UpdateBabyHandler), `invites.go` L48-L69 (CreateInviteHandler)
- **Perspective:** Tech Lead
- **Finding:** The pattern of (1) call `extractBabyID`, (2) check empty, (3) call `store.GetBabyByID`, (4) check `sql.ErrNoRows` for 404, (5) call `store.IsParentOfBaby`, (6) check false for 403 is repeated three times across `GetBabyHandler`, `UpdateBabyHandler`, and `CreateInviteHandler`. Each instance is 20+ lines of nearly identical boilerplate.
- **Reasoning:** This is the most repeated pattern in the diff. Any change to authorization logic (e.g., adding role-based access) must be applied to all three locations.
- **Consequence:** A security fix to the authorization check applied to only some handlers would leave others vulnerable. As more baby-related endpoints are added (delete, metrics), this pattern will continue to proliferate.
- **Suggestion:** Extract a helper that returns the baby or writes the error response:
  ```go
  func requireBabyAccess(w http.ResponseWriter, r *http.Request, db *sql.DB, userID string) (*model.Baby, bool) {
      babyID := extractBabyID(r)
      if babyID == "" {
          http.Error(w, "missing baby ID", http.StatusBadRequest)
          return nil, false
      }
      baby, err := store.GetBabyByID(db, babyID)
      if err != nil {
          if errors.Is(err, sql.ErrNoRows) {
              http.Error(w, "baby not found", http.StatusNotFound)
              return nil, false
          }
          log.Printf("get baby: %v", err)
          http.Error(w, "internal error", http.StatusInternalServerError)
          return nil, false
      }
      linked, err := store.IsParentOfBaby(db, userID, baby.ID)
      if err != nil {
          log.Printf("check parent: %v", err)
          http.Error(w, "internal error", http.StatusInternalServerError)
          return nil, false
      }
      if !linked {
          http.Error(w, "forbidden", http.StatusForbidden)
          return nil, false
      }
      return baby, true
  }
  ```

### [MEDIUM] `meBaby` struct in `api.go` overlaps with `babyResponse` in `babies.go`
- **File:** `backend/internal/handler/api.go` (L45-L51) and `backend/internal/handler/babies.go` (L34-L44)
- **Line(s):** `api.go` L45-L51, `babies.go` L34-L44
- **Perspective:** Tech Lead
- **Finding:** The `meBaby` struct in `api.go` has fields `ID`, `Name`, `Sex`, `DateOfBirth`, `Notes` -- which are a strict subset of `babyResponse` in `babies.go`. The `MeHandler` manually converts `model.Baby` to `meBaby` with inline format logic (L79-L87), while `babies.go` has the reusable `toBabyResponse` converter that does the same thing (plus additional fields).
- **Reasoning:** The `MeHandler` was written before the baby CRUD handlers. Now that `babyResponse` and `toBabyResponse` exist, the `/api/me` endpoint should use them instead of maintaining its own parallel struct and conversion logic.
- **Consequence:** Adding a new baby field (e.g., `kasai_date`) would require updating `meBaby` separately from `babyResponse`, and the `/api/me` response would silently omit new fields until someone notices the inconsistency.
- **Suggestion:** Replace `meBaby` usage in `MeHandler` with `babyResponse` and `toBabyResponse`:
  ```go
  type meResponse struct {
      User   meUser         `json:"user"`
      Babies []babyResponse `json:"babies"`
  }
  // In MeHandler:
  for i := range babies {
      resp.Babies = append(resp.Babies, toBabyResponse(&babies[i]))
  }
  ```
  Then delete the `meBaby` struct entirely.

### [MEDIUM] Duplicated user-extraction + nil-check boilerplate across all handlers
- **File:** `backend/internal/handler/babies.go` and `backend/internal/handler/invites.go`
- **Line(s):** `babies.go` L75-L79, L133-L137, L161-L165, L205-L209; `invites.go` L36-L40, L94-L98
- **Perspective:** Tech Lead
- **Finding:** Every handler starts with the identical 5-line block: `user := middleware.UserFromContext(r.Context())` followed by a nil check returning 401. This appears 6 times in the new code (plus once in existing `api.go`), totaling 7 occurrences across the handler package.
- **Reasoning:** This is technically guarded by the auth middleware, so the nil check is defensive. But given the middleware chain ensures the user is always present, this could be simplified. If the pattern is kept for defense-in-depth, it should at least be a one-liner.
- **Consequence:** Moderate -- the boilerplate is small per-instance but adds up. More significantly, if the unauthorized response format changes (e.g., to JSON error bodies), all 7 locations need updating.
- **Suggestion:** Extract a helper:
  ```go
  func requireUser(w http.ResponseWriter, r *http.Request) (*model.User, bool) {
      user := middleware.UserFromContext(r.Context())
      if user == nil {
          http.Error(w, "unauthorized", http.StatusUnauthorized)
          return nil, false
      }
      return user, true
  }
  ```

### [MEDIUM] Test boilerplate: repeated middleware chain setup across all handler tests
- **File:** `backend/internal/handler/babies_test.go` and `backend/internal/handler/invites_test.go`
- **Line(s):** Throughout both files (e.g., `babies_test.go` L324-L327, L384-L386, L431-L433; `invites_test.go` L275-L277, L316-L317)
- **Perspective:** QA
- **Finding:** Nearly every test function in both `babies_test.go` and `invites_test.go` repeats the same 3-line middleware chain setup: `authMw := middleware.Auth(db, testCookieName)`, `csrfMw := middleware.CSRF(db, testCookieName, testSecret)`, `h := authMw(csrfMw(http.HandlerFunc(...)))`. This appears approximately 20 times across the two test files.
- **Reasoning:** The `testutil` package already provides `AuthenticatedRequest` to handle request setup. A similar helper for wrapping a handler with the standard test middleware chain would eliminate this repetition.
- **Consequence:** When a new middleware is added to the chain (like rate limiting was in this phase), all test middleware chains need manual updating. This is error-prone and verbose.
- **Suggestion:** Add a helper to `testutil` or a test-local helper:
  ```go
  func wrapWithAuth(db *sql.DB, h http.HandlerFunc) http.Handler {
      authMw := middleware.Auth(db, testCookieName)
      csrfMw := middleware.CSRF(db, testCookieName, testSecret)
      return authMw(csrfMw(h))
  }
  ```

### [MEDIUM] `store` package tests use local `setupTestDB` instead of `testutil.SetupTestDB`
- **File:** `backend/internal/store/babies_test.go` and `backend/internal/store/invites_test.go`
- **Line(s):** Every test function (e.g., `babies_test.go` L266, `invites_test.go` L895)
- **Perspective:** Tech Lead
- **Finding:** The store-level tests (`babies_test.go`, `invites_test.go`) use a local `setupTestDB` function defined in `db_test.go` rather than `testutil.SetupTestDB`. The `testutil.SetupTestDB` was specifically created to be the shared DB setup helper across test packages.
- **Reasoning:** This is understandable since the store tests are in the `store` package (not `store_test`), and `testutil` imports `store`, which would create a circular dependency. However, the duplication means the two `setupTestDB` implementations could drift.
- **Consequence:** If the DB setup logic changes (e.g., different migration strategy), it must be updated in both `testutil.SetupTestDB` and `store.setupTestDB`.
- **Suggestion:** This is a known limitation of Go's circular import rules. Document the relationship with a comment in `store/db_test.go`:
  ```go
  // setupTestDB mirrors testutil.SetupTestDB but is defined locally to avoid
  // a circular import (testutil imports store). Keep these in sync.
  ```

### [LOW] Hardcoded date format string `"2006-01-02"` repeated throughout
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L51, L57, L60, L97, L102, L108, L257, L262, L268
- **Perspective:** Tech Lead
- **Finding:** The Go date format string `"2006-01-02"` appears 9 times in `babies.go` alone, plus additional uses in `api.go`. There is no package-level constant for this format.
- **Reasoning:** While this is a well-known Go idiom, a single constant would improve grep-ability and reduce the chance of typos (e.g., `"2006-01-2"` would be hard to spot).
- **Consequence:** Low risk -- Go developers recognize this format. But a constant would be a minor readability improvement.
- **Suggestion:** Define once in the model or handler package:
  ```go
  const DateFormat = "2006-01-02"
  const DateTimeFormat = "2006-01-02T15:04:05Z"
  ```
