## Summary

Phase 40 implements an integration test for PDF report generation. The test file `backend/internal/integration/report_generation_test.go` seeds 30 days of comprehensive health data (feedings, stools, urine, temperatures, weights, lab results, notes, medications with med logs, and photos) and then generates a PDF report via the API endpoint. Three test functions cover: full integration with all data types, an empty date range, and a scenario without photos. All tests pass and run in parallel in ~0.2 seconds.

The implementation is thorough and covers the phase requirements well. The tests validate PDF structure (magic bytes, EOF marker), section headers, baby header info, Kasai date, data content fragments, medication names/doses, and conditional photo appendix presence. A few minor issues are noted below.

### [MEDIUM] Hardcoded cookie name instead of using `auth.CookieName` constant
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L258
- **Perspective:** Tech Lead
- **Finding:** The `doRawBytes` method uses a hardcoded string `"session_id"` for the cookie name, while every other integration test file uses `auth.CookieName`.
- **Reasoning:** The rest of the codebase consistently references the `auth.CookieName` constant. If the cookie name ever changes, this method would silently break while all other helpers would continue working, making the failure confusing to diagnose.
- **Consequence:** If `auth.CookieName` is changed in the future, this test would fail with an authentication error rather than a clear compile-time or linkage issue.
- **Suggestion:** Import `auth` and use the constant:
  ```go
  import "github.com/ablankz/LittleLiver/backend/internal/auth"
  // ...
  req.AddCookie(&http.Cookie{Name: auth.CookieName, Value: tc.sessionID})
  ```

### [LOW] `doRawBytes` helper defined in test file instead of shared helpers
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L252-L271
- **Perspective:** Tech Lead
- **Finding:** The `doRawBytes` method is defined on `testClient` in the report test file, but it is a general-purpose helper (GET request returning raw bytes) that could be reused by other integration tests. All other `testClient` methods are defined in `helpers_test.go`.
- **Reasoning:** The project already centralizes test helpers in `helpers_test.go`. Placing this method in a specific test file breaks the established pattern and makes it harder to discover and reuse.
- **Consequence:** Future test files needing raw byte responses would either duplicate this helper or have difficulty finding it.
- **Suggestion:** Move `doRawBytes` to `backend/internal/integration/helpers_test.go` alongside the other `testClient` methods.

### [LOW] `createReportTestJPEG` duplicates JPEG creation pattern from photo_flow_test.go
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L236-L249
- **Perspective:** Tech Lead
- **Finding:** The `createReportTestJPEG` function creates a small test JPEG image. The `photo_flow_test.go` file likely has a similar helper for creating test images. This creates minor duplication.
- **Reasoning:** If the test image creation approach needs to change (e.g., different encoding options), it would need to be updated in multiple places.
- **Consequence:** Minor maintenance burden from duplicated test utilities.
- **Suggestion:** Consider extracting a shared `createTestJPEG` helper into `helpers_test.go` that both test files can use.

### [LOW] PDF text assertion approach is fragile for binary content
- **File:** `backend/internal/integration/report_generation_test.go`
- **Line(s):** L318-L367
- **Perspective:** QA
- **Finding:** The test treats the entire PDF byte stream as a string and searches for text fragments using `strings.Contains`. PDF is a binary format where text may be encoded, compressed, or split across internal stream objects depending on the PDF library's output.
- **Reasoning:** Currently this works because maroto v2 apparently embeds text in a way that is searchable in the raw bytes. However, if the maroto library changes its compression or encoding strategy in an update, these assertions could break even though the PDF visually contains all expected content.
- **Consequence:** A maroto library update could cause false test failures without any actual regression in report content.
- **Suggestion:** This is an acceptable tradeoff for an integration test in a personal-use app. No immediate action needed, but a comment acknowledging this fragility would be helpful for future reference. If issues arise, consider using a PDF text extraction library.
