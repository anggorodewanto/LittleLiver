## Summary

Phase 41 implements the frontend auth flow and API client for LittleLiver. The changes include: a login page with a "Sign in with Google" link, an API client (`lib/api.ts`) that attaches `X-Timezone` and `X-CSRF-Token` headers with session cookie credentials, a 401 response interceptor that redirects to `/login`, and a Svelte store for current user state. All 23 tests pass with 100% coverage on `api.ts` and `user.ts`. The implementation is minimal and well-tested.

### [MEDIUM] `fetchCurrentUser` in user store bypasses the API client
- **File:** `frontend/src/lib/stores/user.ts`
- **Line(s):** L16
- **Perspective:** Tech Lead
- **Finding:** `fetchCurrentUser()` calls `fetch('/api/me', ...)` directly instead of using `apiClient.get('/me')`. This bypasses the API client's header attachment (X-Timezone), CSRF handling, and 401 redirect logic.
- **Reasoning:** The API client was specifically built to centralize request handling. Using raw `fetch` for `/api/me` means: (1) the `X-Timezone` header is not sent, so the user's timezone will not be updated on `GET /api/me` calls (the spec says timezone is "updated on every API call via X-Timezone header"); (2) if a future change adds interceptor logic, `/api/me` would be missed.
- **Consequence:** The user's `timezone` field in the database will not be updated when `fetchCurrentUser` is called, violating the spec requirement that timezone updates on every API call. Additionally, the 401 handling in user store (set user to null) differs from the API client's 401 handling (redirect to /login), creating inconsistent behavior.
- **Suggestion:** Use `apiClient.get` instead of raw `fetch`:
  ```typescript
  import { apiClient } from '$lib/api';
  
  export async function fetchCurrentUser(): Promise<void> {
    try {
      const data = await apiClient.get<{ user: User }>('/me');
      currentUser.set(data.user);
    } catch {
      currentUser.set(null);
    }
  }
  ```
  Note: This would mean a 401 on `/api/me` triggers both the API client's redirect AND sets the store to null. That's actually fine -- the redirect takes the user to login, and the null state is consistent. The test would need updating to mock via `apiClient` or via the underlying fetch with the CSRF + timezone headers present.

### [LOW] Content-Type header always set even for GET requests
- **File:** `frontend/src/lib/api.ts`
- **Line(s):** L36
- **Perspective:** Tech Lead
- **Finding:** The `Content-Type: application/json` header is set on all requests including GET and DELETE, which typically have no body.
- **Reasoning:** While this does not cause functional issues (servers generally ignore Content-Type on bodyless requests), it is technically incorrect per HTTP semantics.
- **Consequence:** No functional impact. Minor HTTP hygiene issue.
- **Suggestion:** Conditionally set Content-Type only when there is a body:
  ```typescript
  const headers: Record<string, string> = {
    'X-Timezone': getTimezone(),
    ...(options?.headers as Record<string, string>)
  };
  if (options?.body) {
    headers['Content-Type'] = 'application/json';
  }
  ```

### [LOW] No PATCH method exposed on apiClient
- **File:** `frontend/src/lib/api.ts`
- **Line(s):** L67-L95
- **Perspective:** QA
- **Finding:** The `STATE_CHANGING_METHODS` array includes `PATCH` (line 31), but the `apiClient` object does not expose a `patch()` method. This is not currently needed but creates a minor inconsistency.
- **Reasoning:** The CSRF guard correctly covers PATCH, but there is no convenient method to make PATCH requests through the client. If PATCH is needed later, developers would need to use `request()` directly (which is not exported).
- **Consequence:** No immediate impact since no API endpoints use PATCH. Minor gap.
- **Suggestion:** Either add a `patch` method or remove PATCH from `STATE_CHANGING_METHODS`. Given the spec does not use PATCH endpoints, removing it would be the minimal approach.

### [MEDIUM] API client `response.json()` called on all successful responses without checking content
- **File:** `frontend/src/lib/api.ts`
- **Line(s):** L60
- **Perspective:** QA
- **Finding:** The `request` function always calls `response.json()` on success (line 60). Some endpoints return 204 No Content (e.g., `DELETE /api/users/me`, `DELETE /api/babies/:id/parents/me`), which has no body. Calling `.json()` on a 204 response will throw.
- **Reasoning:** The spec defines multiple endpoints that return 204 No Content. When the frontend eventually calls these via `apiClient.del()`, the call will fail with a JSON parse error despite the operation succeeding.
- **Consequence:** Future DELETE calls to 204-returning endpoints will throw unexpected errors, requiring a workaround or fix at that time.
- **Suggestion:** Handle empty responses:
  ```typescript
  if (response.status === 204) {
    return undefined as T;
  }
  return response.json() as Promise<T>;
  ```
