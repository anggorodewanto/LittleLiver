## Summary

Phases 43-45 add 13 new Svelte form components (FeedingForm, UrineForm, TemperatureForm, WeightForm, StoolForm, AbdomenForm, SkinForm, BruisingForm, LabForm, NotesForm, PhotoUpload, QuickLogButtons) and a TodayDashboard component, along with comprehensive test suites. The code is functional and well-tested, but contains significant copy-paste duplication across forms and uses raw strings where typed constants should exist.

### [HIGH] Copy-paste duplication: `defaultTimestamp()` duplicated verbatim across 10 form components
- **File:** `frontend/src/lib/components/FeedingForm.svelte`, `UrineForm.svelte`, `TemperatureForm.svelte`, `WeightForm.svelte`, `StoolForm.svelte`, `AbdomenForm.svelte`, `SkinForm.svelte`, `BruisingForm.svelte`, `LabForm.svelte`, `NotesForm.svelte`
- **Line(s):** Varies per file (e.g., FeedingForm L18-L22, UrineForm L16-L20, etc.)
- **Perspective:** Tech Lead
- **Finding:** The exact same `defaultTimestamp()` function is copied into every single form component. It computes a local-time ISO string for `datetime-local` inputs.
- **Reasoning:** This is the textbook case of copy-paste with slight variation (actually no variation at all). Any fix (e.g., timezone bug, format change) must be applied to 10 files independently.
- **Consequence:** A future bug fix or format change will be applied inconsistently across forms, leading to subtle timestamp mismatches between different metric types.
- **Suggestion:** Extract to a shared utility, e.g., `frontend/src/lib/utils.ts`:
  ```typescript
  export function defaultTimestamp(): string {
    const now = new Date();
    const offset = now.getTimezoneOffset();
    const local = new Date(now.getTime() - offset * 60000);
    return local.toISOString().slice(0, 16);
  }
  ```
  Import in each form: `import { defaultTimestamp } from '$lib/utils';`

### [HIGH] Copy-paste duplication: identical form structure (error display, submit button, validation pattern)
- **File:** All form components in `frontend/src/lib/components/`
- **Line(s):** N/A - structural issue
- **Perspective:** Tech Lead
- **Finding:** Every form repeats the identical pattern: `{#if validationError}<p role="alert">...{/if}`, `{#if error}<p role="alert">...{/if}`, and `<button type="submit" disabled={submitting}>{submitting ? 'Logging...' : 'Log X'}</button>`. The Props interface also repeats `submitting?: boolean; error?: string;` in every single form.
- **Reasoning:** This is roughly 10-15 lines of boilerplate duplicated across 10 forms. The error display and submit button rendering are identical in every form with only the label changing.
- **Consequence:** If you later want to change error styling (e.g., add an icon, change ARIA attributes, add a dismiss button), you must update 10 files. A missed update causes inconsistent UX.
- **Suggestion:** Create a shared `FormFooter.svelte` component:
  ```svelte
  <script lang="ts">
    interface Props { validationError?: string; error?: string; submitting?: boolean; label: string; }
    let { validationError = '', error = '', submitting = false, label }: Props = $props();
  </script>
  {#if validationError}<p role="alert">{validationError}</p>{/if}
  {#if error}<p role="alert">{error}</p>{/if}
  <button type="submit" disabled={submitting}>
    {submitting ? 'Logging...' : `Log ${label}`}
  </button>
  ```

### [MEDIUM] Stringly-typed code: `onselect` callback in QuickLogButtons uses raw strings for metric types
- **File:** `frontend/src/lib/components/QuickLogButtons.svelte`
- **Line(s):** L3, L9-L13
- **Perspective:** Tech Lead
- **Finding:** The `onselect` prop is typed as `(type: string) => void`, and the button handlers pass raw strings like `'feeding'`, `'urine'`, `'stool'`, `'temperature'`, `'medication'`. The consumer (TodayDashboard) also uses a stringly-typed `handleQuickLog(type: string)`.
- **Reasoning:** There is no compile-time guarantee that a consumer handles all metric types or that the strings match expected values. A typo like `'temperture'` would silently route to the wrong page.
- **Consequence:** A misspelling or a newly added metric type that is not handled by the consumer would fail silently at runtime with no TypeScript error.
- **Suggestion:** Define a string union type and export it:
  ```typescript
  export type MetricType = 'feeding' | 'urine' | 'stool' | 'temperature' | 'medication';
  interface Props {
    onselect: (type: MetricType) => void;
  }
  ```

### [MEDIUM] Stringly-typed code: select option values are raw strings with no shared constants
- **File:** `frontend/src/lib/components/FeedingForm.svelte`, `TemperatureForm.svelte`, `AbdomenForm.svelte`, `StoolForm.svelte`, `UrineForm.svelte`, `BruisingForm.svelte`, `SkinForm.svelte`, `NotesForm.svelte`, `WeightForm.svelte`
- **Line(s):** Varies (e.g., FeedingForm L52-L58, TemperatureForm L72-L77, AbdomenForm L44-L48)
- **Perspective:** Tech Lead
- **Finding:** Every `<select>` uses hardcoded string values like `'breast_milk'`, `'formula'`, `'rectal'`, `'axillary'`, `'soft'`, `'firm'`, `'distended'`, etc. These same strings appear in the payload types, but there is no shared constant or union type connecting the option values to the payload type definitions.
- **Reasoning:** The payload interface says `feed_type: string` but the valid values are only the 5 options in the select. If the backend changes or adds an option, there is no type safety to catch mismatches.
- **Consequence:** Payload types accept any string, so the TypeScript compiler cannot detect invalid values. A backend API change (e.g., renaming `'breast_milk'` to `'breastmilk'`) would require finding every occurrence manually.
- **Suggestion:** Use string literal union types in payloads:
  ```typescript
  export type FeedType = 'breast_milk' | 'formula' | 'fortified_breast_milk' | 'solid' | 'other';
  export interface FeedingPayload {
    timestamp: string;
    feed_type: FeedType;
    // ...
  }
  ```

### [MEDIUM] Redundant state: `selectedQuickPick` in LabForm duplicates `testName`
- **File:** `frontend/src/lib/components/LabForm.svelte`
- **Line(s):** L44, L48-L50
- **Perspective:** Tech Lead
- **Finding:** `selectedQuickPick` is a separate `$state` that tracks which quick-pick button is selected. It is always set to `pick.testName` alongside `testName = pick.testName`. The `aria-pressed` attribute checks `selectedQuickPick === pick.testName`, which is functionally identical to checking `testName === pick.testName`.
- **Reasoning:** Two pieces of state that always hold the same value create a synchronization risk. If a user manually edits the test name input, `selectedQuickPick` still shows the old quick-pick as selected, creating a misleading UI state.
- **Consequence:** If a user taps "ALT" quick-pick then manually changes the test name to "GGT", the ALT button still appears pressed (aria-pressed="true") while the form contains "GGT". This is a UX bug.
- **Suggestion:** Remove `selectedQuickPick` and derive the pressed state from `testName`:
  ```typescript
  // Remove: let selectedQuickPick = $state('');
  // In selectQuickPick, remove: selectedQuickPick = pick.testName;
  // In template:
  aria-pressed={testName === pick.testName ? 'true' : 'false'}
  ```

### [MEDIUM] Redundant state: `colorLabel` in StoolForm is derivable from `colorRating`
- **File:** `frontend/src/lib/components/StoolForm.svelte`
- **Line(s):** L41-L42, L49-L51
- **Perspective:** Tech Lead
- **Finding:** `colorRating` and `colorLabel` are kept as separate state variables, but `colorLabel` is always the `ref` value from the `COLOR_SWATCHES` array entry matching `colorRating`. Given `COLOR_SWATCHES` is a const array, `colorLabel` can be derived.
- **Reasoning:** Two state variables that must always be updated together in `selectColor()` create a coupling that can break if one is updated without the other.
- **Consequence:** If future code updates `colorRating` without calling `selectColor()`, `colorLabel` becomes stale, causing the payload to contain a mismatched rating/label pair sent to the backend.
- **Suggestion:** Use `$derived` instead:
  ```typescript
  let colorRating = $state(0);
  let colorLabel = $derived(COLOR_SWATCHES.find(s => s.rating === colorRating)?.ref ?? '');
  ```

### [MEDIUM] Leaky abstraction: PhotoUpload uses a hardcoded `id="photo-upload"` causing DOM ID collisions
- **File:** `frontend/src/lib/components/PhotoUpload.svelte`
- **Line(s):** L23, L24
- **Perspective:** QA
- **Finding:** The `<label for="photo-upload">` and `<input id="photo-upload">` use a static ID. When multiple forms with PhotoUpload are rendered on the same page (or the same form is rendered twice), the IDs collide, breaking label-input association and accessibility.
- **Reasoning:** The NotesForm, StoolForm, AbdomenForm, BruisingForm, and SkinForm all include PhotoUpload. If a dashboard or modal renders more than one at a time, the DOM will have duplicate IDs.
- **Consequence:** Clicking the "Photo" label could activate the wrong file input on pages with multiple photo-enabled forms, causing uploads to go to the wrong metric entry.
- **Suggestion:** Accept an `id` prop or generate a unique ID:
  ```typescript
  interface Props {
    id?: string;
    // ...
  }
  let { id = `photo-upload-${Math.random().toString(36).slice(2)}`, ... } = $props();
  ```

### [MEDIUM] Leaky abstraction: TodayDashboard uses `window.location.href` for navigation
- **File:** `frontend/src/lib/components/TodayDashboard.svelte`
- **Line(s):** L147-L149
- **Perspective:** Tech Lead
- **Finding:** `handleQuickLog` sets `window.location.href = '/log/${type}'` which causes a full page reload in a SvelteKit SPA. This bypasses SvelteKit's client-side router.
- **Reasoning:** SvelteKit apps should use `goto()` from `$app/navigation` or `<a>` tags for navigation. Direct `window.location` assignment forces a full reload, losing all client-side state (stores, component state, any in-memory data).
- **Consequence:** Every quick-log button click will cause a full page reload, destroying all SPA state. This will feel sluggish and will re-trigger authentication checks, API calls, and component re-initialization.
- **Suggestion:** Use SvelteKit's `goto`:
  ```typescript
  import { goto } from '$app/navigation';
  function handleQuickLog(type: string): void {
    goto(`/log/${type}`);
  }
  ```

### [LOW] TodayDashboard `$effect` will re-fetch on every render cycle due to missing dependency tracking
- **File:** `frontend/src/lib/components/TodayDashboard.svelte`
- **Line(s):** L171-L174
- **Perspective:** Performance Engineer
- **Finding:** The `$effect` block calls `void fetchDashboard()` and the comment says "Re-fetch when babyId changes", but `fetchDashboard` reads `babyId` inside it (via the API URL). In Svelte 5, `$effect` tracks reactive reads, but since `babyId` is read inside the async function (not directly in the effect body), tracking behavior depends on when the read occurs relative to the microtask boundary.
- **Reasoning:** If `babyId` is not tracked correctly, the effect may not re-run when the baby changes, or it may run more often than expected.
- **Consequence:** Switching babies could show stale dashboard data for the previous baby.
- **Suggestion:** Explicitly read `babyId` in the effect body to ensure tracking:
  ```typescript
  $effect(() => {
    const id = babyId; // explicit tracking
    void fetchDashboard();
  });
  ```
  Or better, pass `babyId` as a parameter: `void fetchDashboard(babyId);`

### [LOW] Alert type strings in TodayDashboard are not type-safe
- **File:** `frontend/src/lib/components/TodayDashboard.svelte`
- **Line(s):** L125-L138
- **Perspective:** Tech Lead
- **Finding:** The `alertLabel` function uses a `switch` on raw strings (`'acholic_stool'`, `'fever'`, etc.) with a `default` that returns the raw `alertType`. The `Alert` interface types `alert_type` as `string` rather than a union.
- **Reasoning:** If the backend adds a new alert type, it silently falls through to the default case, displaying the raw snake_case identifier to the user (e.g., `weight_loss_rapid`).
- **Consequence:** Users see raw technical identifiers instead of human-readable labels for any new alert type added on the backend.
- **Suggestion:** Define a union type and use a const map:
  ```typescript
  type AlertType = 'acholic_stool' | 'fever' | 'jaundice_worsening' | 'missed_medication';
  const ALERT_LABELS: Record<AlertType, string> = {
    acholic_stool: 'Acholic Stool',
    fever: 'Fever',
    jaundice_worsening: 'Jaundice Worsening',
    missed_medication: 'Missed Medication'
  };
  ```
