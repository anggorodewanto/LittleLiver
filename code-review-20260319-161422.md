### [MEDIUM] TemperatureChart crashes on empty data array
- **File:** `/home/ab/projects/LittleLiver/frontend/src/lib/components/TemperatureChart.svelte`
- **Line(s):** L29-L31
- **Perspective:** QA
- **Finding:** When `data` is an empty array, `tempPoints` will be empty, causing `Math.min(...[])` to return `Infinity` and `Math.max(...[])` to return `-Infinity`. The fever threshold line would then have points at `{x: Infinity, y: 38}` and `{x: -Infinity, y: 38}`, which may produce unexpected Chart.js behavior or rendering artifacts.
- **Reasoning:** The dashboard API could legitimately return an empty temperature array for a given date range (e.g., no temperature readings logged in the last 7 days). The TrendsView always passes `dashboard.chart_data_series.temperature` to this component without guarding for empty arrays.
- **Consequence:** Chart.js may render incorrectly, throw a runtime error, or display a confusing chart when no temperature data exists for the selected range.
- **Suggestion:** Guard against empty data before constructing the threshold points:
  ```typescript
  const thresholdPoints = xValues.length > 0
    ? [{ x: Math.min(...xValues), y: FEVER_THRESHOLD }, { x: Math.max(...xValues), y: FEVER_THRESHOLD }]
    : [];
  ```
  Alternatively, skip rendering the chart entirely when `data` is empty and show a "No temperature data" message. Also add a test for the empty data case.

### [MEDIUM] StoolColorChart y-axis max is 8 but valid scores are 1-7
- **File:** `/home/ab/projects/LittleLiver/frontend/src/lib/components/StoolColorChart.svelte`
- **Line(s):** L59
- **Perspective:** Product Owner
- **Finding:** The y-axis is configured with `max: 8`, but the spec defines stool color ratings as 1-7 only. Setting max to 8 leaves a blank gap at the top of the chart with no corresponding label (the y-axis callback returns `''` for value 8 since no swatch exists for that rating).
- **Reasoning:** The extra unused space at the top is visually misleading and wastes chart real estate. The y-axis min is 0, which also creates a gap below rating 1 ("White").
- **Consequence:** Users see unlabeled empty space at the top and bottom of the chart, reducing readability. Minor visual issue.
- **Suggestion:** Set `min: 1` and `max: 7` to exactly match the valid rating range, or use `min: 0.5` and `max: 7.5` for padding without showing unlabeled ticks.

### [LOW] No validation on custom date range inputs
- **File:** `/home/ab/projects/LittleLiver/frontend/src/lib/components/DateRangeSelector.svelte`
- **Line(s):** L23-L25
- **Perspective:** QA
- **Finding:** The `applyCustom` function calls `onchange('custom', customFrom, customTo)` without validating that both dates are filled or that `customFrom` is before `customTo`. While the `TrendsView.computeDateRange` gracefully falls back to 7d when custom dates are empty, a reversed date range (from > to) would be sent to the API as-is.
- **Reasoning:** Users could click "Apply" with incomplete or reversed date inputs, producing a confusing API request with from > to.
- **Consequence:** API call with invalid date range returns empty or unexpected data. The user experience degrades but no crash occurs since `TrendsView` handles the error state.
- **Suggestion:** Disable the "Apply" button until both inputs are filled and `customFrom <= customTo`:
  ```svelte
  <button type="button" onclick={applyCustom} disabled={!customFrom || !customTo || customFrom > customTo}>Apply</button>
  ```
  Add a test verifying the button is disabled when dates are incomplete.

### [LOW] Chart.register called on every component mount
- **File:** `/home/ab/projects/LittleLiver/frontend/src/lib/components/StoolColorChart.svelte`
- **Line(s):** L25
- **Perspective:** Performance Engineer
- **Finding:** Each chart component calls `Chart.register(...registerables)` in its `onMount`. When TrendsView renders all three charts, this registration is called three times. While Chart.js handles duplicate registration idempotently, it is unnecessary repeated work.
- **Reasoning:** Registration is a one-time global operation. Calling it per-component is a minor inefficiency and a code duplication concern across the three chart components.
- **Consequence:** Negligible performance impact since Chart.js deduplicates internally. Primarily a code cleanliness issue.
- **Suggestion:** Move `Chart.register(...registerables)` to a shared initialization module (e.g., `$lib/chart-setup.ts`) that is imported once at the app level, or call it once in `TrendsView` before rendering child charts.
