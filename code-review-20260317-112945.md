## Summary

Review of three new files in `backend/internal/model/`: `ulid.go` (ULID generation), `enums.go` (enum validation via map lookups), and `types.go` (domain structs). The review scope is limited to **efficiency concerns only** as requested.

### [MEDIUM] Unnecessary Mutex in NewULID — Serializes All ID Generation With No Benefit
- **File:** `backend/internal/model/ulid.go`
- **Line(s):** L12, L17-L18
- **Perspective:** Performance Engineer
- **Finding:** `ulidMu sync.Mutex` is locked on every call to `NewULID()`, but the function has no shared mutable state. `time.Now()` is goroutine-safe. `crypto/rand.Read()` is goroutine-safe (it uses its own internal locking). `buf` and `rnd` are stack-local variables. The `crockfordBase32` constant string is immutable. There is literally nothing for the mutex to protect.
- **Reasoning:** Under concurrent request load, every goroutine generating a ULID must wait in line for this mutex. This turns a naturally parallel, zero-contention operation into a serial bottleneck. For a single-user baby-tracking app the practical impact is negligible, but the mutex is still wrong in principle -- it adds ~25ns of overhead per call and prevents concurrent ID generation for no correctness benefit.
- **Consequence:** Under concurrent writes (e.g., bulk import, or multiple parents logging events simultaneously), ULID generation becomes a serialization point. The performance ceiling is artificially lowered for zero safety gain.
- **Suggestion:** Remove the mutex entirely. The function is already safe for concurrent use without it.
  ```go
  // Remove these two lines:
  // var ulidMu sync.Mutex

  func NewULID() string {
      // Remove Lock/Unlock — no shared mutable state exists.
      ms := uint64(time.Now().UnixMilli())
      // ... rest unchanged ...
  }
  ```

### [LOW] Map Lookup Overhead for Tiny Enum Sets (2-3 Values)
- **File:** `backend/internal/model/enums.go`
- **Line(s):** L13, L19, L21, L23
- **Perspective:** Performance Engineer
- **Finding:** Several enum sets contain only 2-3 values (`sexValues` has 2, `stoolVolumeValues` has 3, `firmnessValues` has 3, `bruisingSizeValues` has 3). For sets this small, a `map[string]bool` lookup involves hashing the input string, finding the bucket, and comparing -- which is slower than a simple chain of string equality comparisons or a `switch` statement. For 2 values, a direct `v == "male" || v == "female"` is measurably faster (the Go compiler can optimize short string comparisons to length check + word comparison).
- **Reasoning:** Map-based lookup has a fixed overhead (hashing cost) that only pays off when the set is large enough for O(1) amortization to beat O(n) linear scan. For n=2-3, the linear scan always wins. That said, the current approach is consistent and readable, and these validators are called on HTTP request boundaries, not in tight loops.
- **Consequence:** Minimal real-world impact. The overhead per call is roughly 20-40ns extra versus a switch statement. This is negligible at request-handling scale. The consistency of the current pattern has maintainability value.
- **Suggestion:** No change required -- this is informational. If micro-optimization were ever needed, the smallest sets (2-3 values) could use switch statements. For sets of 4-7 values, the map approach is already reasonable. The uniform pattern is a fair trade-off for this codebase.
