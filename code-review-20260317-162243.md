## Summary

This review examines Phases 13-15 production code (UnlinkSelf, DeleteAccount, Feedings CRUD) for **code reuse opportunities**. The codebase is compact and follows consistent patterns. Several repeated patterns across handler and store files could benefit from extraction into shared helpers, reducing boilerplate and ensuring consistency as more metric types are added in future phases.

### [MEDIUM] Repeated entryID extraction and validation pattern
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L198-L202, L236-L240, L285-L289
- **Perspective:** Tech Lead
- **Finding:** The pattern of extracting `entryID` via `r.PathValue("entryId")` followed by an empty-string check and 400 response is repeated identically in `GetFeedingHandler`, `UpdateFeedingHandler`, and `DeleteFeedingHandler`. This mirrors the existing `extractBabyID` helper in `babies.go` but lacks an equivalent extraction+validation helper.
- **Reasoning:** `extractBabyID` already exists as a named helper for the `{id}` path parameter. The `{entryId}` parameter uses the same pattern but is inlined three times. As more metric types are added (medications, med_logs per the spec), this pattern will multiply further.
- **Consequence:** Each new metric CRUD resource will duplicate this 4-line block, increasing the surface area for inconsistent error messages or missing validation.
- **Suggestion:** Extract a generic path-value helper analogous to `extractBabyID`:
  ```go
  // extractPathValue extracts a named path parameter and writes a 400 if missing.
  func extractPathValue(w http.ResponseWriter, r *http.Request, name string) (string, bool) {
      v := r.PathValue(name)
      if v == "" {
          http.Error(w, "missing "+name, http.StatusBadRequest)
          return "", false
      }
      return v, true
  }
  ```
  This also allows `extractBabyID` to be refactored to use it, or be replaced entirely.

### [MEDIUM] Repeated ErrNoRows-to-404 handling pattern
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L205-L213, L254-L262, L291-L299
- **Perspective:** Tech Lead
- **Finding:** The pattern `if errors.Is(err, sql.ErrNoRows) { http.Error(w, "X not found", 404); return } ... log; http.Error(w, "internal error", 500)` appears 3 times in `feedings.go` and once in `babies.go` (L114-L120 in `requireBabyAccess`). Each instance differs only in the log prefix and "not found" entity name.
- **Reasoning:** This is the most common error-handling pattern in the handler layer. As more resources are added, it will proliferate. A small helper would reduce boilerplate and enforce consistent log formatting.
- **Consequence:** Inconsistent error messages or missing log statements when future developers copy-paste this block. For example, some instances say "internal error" while others say "failed to X" for 500 responses.
- **Suggestion:** Extract a helper:
  ```go
  func handleDBError(w http.ResponseWriter, err error, logPrefix, notFoundMsg string) {
      if errors.Is(err, sql.ErrNoRows) {
          http.Error(w, notFoundMsg, http.StatusNotFound)
          return
      }
      log.Printf("%s: %v", logPrefix, err)
      http.Error(w, "internal error", http.StatusInternalServerError)
  }
  ```

### [MEDIUM] feedingListResponse duplicates generic MetricPage structure
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L82-L85
- **Perspective:** Tech Lead
- **Finding:** `feedingListResponse` is defined as `struct { Data []feedingResponse; NextCursor *string }` which mirrors the generic `model.MetricPage[T]` struct. The handler manually constructs this response type, mapping from `model.MetricPage[model.Feeding]` to `feedingListResponse`. The generic `MetricPage` type exists in `model/types.go` but is not leveraged in the response layer.
- **Reasoning:** When medications, med_logs, and other metric types are added, each will need its own `XxxListResponse` struct with identical `Data`+`NextCursor` shape. The generic `MetricPage` was clearly designed to prevent this duplication.
- **Consequence:** Every new metric handler will duplicate this list-response struct and the manual conversion loop (L169-L175), increasing code size linearly with metric types.
- **Suggestion:** Consider using `MetricPage[feedingResponse]` directly as the response type, or create a generic helper:
  ```go
  func toMetricPageResponse[T any, R any](page *model.MetricPage[T], convert func(*T) R) MetricPage[R] {
      resp := MetricPage[R]{
          Data:       make([]R, 0, len(page.Data)),
          NextCursor: page.NextCursor,
      }
      for i := range page.Data {
          resp.Data = append(resp.Data, convert(&page.Data[i]))
      }
      return resp
  }
  ```

### [MEDIUM] Repeated JSON response writing boilerplate
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L120-L123, L177-L179, L215-L217, L264-L266
- **Perspective:** Tech Lead
- **Finding:** The pattern `w.Header().Set("Content-Type", "application/json"); [optional w.WriteHeader(status)]; if err := json.NewEncoder(w).Encode(v); err != nil { log.Printf(...) }` is repeated 14 times across all handler files (babies.go, feedings.go, invites.go, api.go, health.go). Each instance is 3-4 lines of identical structure varying only in the value encoded and the log prefix.
- **Reasoning:** This is the single most repeated pattern in the handler layer. A `writeJSON` helper would reduce each call site to one line and ensure the Content-Type header is never forgotten.
- **Consequence:** Missing Content-Type headers or inconsistent error logging if any call site deviates. Adds noise that makes the actual handler logic harder to read.
- **Suggestion:** Extract a helper:
  ```go
  func writeJSON(w http.ResponseWriter, status int, v any) {
      w.Header().Set("Content-Type", "application/json")
      w.WriteHeader(status)
      if err := json.NewEncoder(w).Encode(v); err != nil {
          log.Printf("encode response: %v", err)
      }
  }
  ```

### [LOW] Repeated tx.Rollback() without defer pattern in store layer
- **File:** `backend/internal/store/delete_account.go`
- **Line(s):** L36, L44, L51, L59, L67, L78, L86, L94, L101
- **Perspective:** Tech Lead
- **Finding:** `delete_account.go` calls `tx.Rollback()` 9 times across error paths. The same pattern appears in `babies.go` (UnlinkParent, CreateBaby) and `invites.go`. A `defer tx.Rollback()` after `tx.Begin()` is safe (rollback on a committed tx is a no-op in Go) and eliminates all manual rollback calls.
- **Reasoning:** The Go `database/sql` documentation confirms that calling `Rollback()` on an already-committed transaction returns `sql.ErrTxDone` and has no effect. Using `defer` is idiomatic and eliminates the risk of missing a rollback on a new error path.
- **Consequence:** If a new error path is added to any of these functions and `tx.Rollback()` is forgotten, the transaction will leak and hold database locks until the connection is reclaimed.
- **Suggestion:** Use `defer tx.Rollback()` immediately after `tx.Begin()`:
  ```go
  tx, err := db.Begin()
  if err != nil {
      return fmt.Errorf("delete account: begin tx: %w", err)
  }
  defer tx.Rollback() // no-op after successful commit
  ```
  Then remove all individual `tx.Rollback()` calls. This applies to `delete_account.go`, `babies.go` (CreateBaby, UnlinkParent), and `invites.go`.

### [LOW] Redundant babyID extraction after requireBabyAccess
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L100, L141, L197, L235, L284
- **Perspective:** Tech Lead
- **Finding:** Every feeding handler calls `requireBabyAccess(w, r, db, user.ID)` (which internally calls `extractBabyID`) and then immediately calls `extractBabyID(r)` again on the next line. The same pattern appears in `UnlinkSelfHandler` in `babies.go` (L235-L240). The baby ID is extracted twice per request.
- **Reasoning:** `requireBabyAccess` already loads the baby by its ID and returns the `*model.Baby` which contains `baby.ID`. The second call to `extractBabyID` is redundant.
- **Consequence:** Minor inefficiency. More importantly, it obscures the data flow -- a reader might wonder if the two extractions could return different values.
- **Suggestion:** Use `baby.ID` from the returned `*model.Baby` instead of re-extracting from the path. For handlers that discard the baby (using `_`), either capture it or have `requireBabyAccess` return the baby ID as well:
  ```go
  baby, ok := requireBabyAccess(w, r, db, user.ID)
  if !ok {
      return
  }
  babyID := baby.ID
  ```

### [LOW] Optional query param extraction could use a small helper
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L143-L152
- **Perspective:** Tech Lead
- **Finding:** The pattern `if v := r.URL.Query().Get("X"); v != "" { ptr = &v }` is repeated 3 times for `from`, `to`, and `cursor`. As more list endpoints are added (medications, med_logs), each will need the same optional query parameter extraction.
- **Reasoning:** This is a minor repetition now but will scale with the number of list endpoints.
- **Consequence:** Low impact currently, but worth noting for future phases.
- **Suggestion:** A small `optionalQuery` helper:
  ```go
  func optionalQuery(r *http.Request, key string) *string {
      if v := r.URL.Query().Get(key); v != "" {
          return &v
      }
      return nil
  }
  ```
