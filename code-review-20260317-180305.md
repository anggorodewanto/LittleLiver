## Summary

This review analyzes phases 19-21 of the LittleLiver project, which add four new metric types: skin observations, bruising, lab results, and general notes. Each metric follows the same five-handler CRUD pattern (Create, List, Get, Update, Delete) at both the handler and store layers, plus an integration test file for multi-parent lifecycle scenarios. The new code correctly uses existing helpers (`registerMetricCRUD`, `mapMetricPage`, `validateTimestamp`, `requireUser`, `requireBabyAccess`, `requireEntryID`, `handleStoreError`, `writeJSON`, `parseMetricTimes`, `nullStr`, `nullFloat`, `deleteByID`, `checkRowsAffected`), which is good. However, massive structural duplication exists in the store layer's `ListXxxWithTZ` functions -- identical pagination/filtering logic is copy-pasted across all 10 metric store files (4 new + 6 existing).

---

### [HIGH] Duplicated ListWithTZ pagination/filtering logic across all store files
- **File:** `backend/internal/store/bruising.go`, `backend/internal/store/skin_observations.go`, `backend/internal/store/lab_results.go`, `backend/internal/store/general_notes.go`
- **Line(s):** bruising.go L76-L148, skin_observations.go L76-L148, lab_results.go L74-L145, general_notes.go L73-L144
- **Perspective:** Tech Lead
- **Finding:** Every `ListXxxWithTZ` function contains an identical ~70-line block that: (1) builds conditions/args for `baby_id`, `from` date, `to` date, and `cursor`; (2) constructs the query with `ORDER BY id DESC LIMIT ?`; (3) iterates rows via a scan function; (4) builds the `MetricPage` with cursor pagination logic (including the `limit+1` trick and nil-data guard). The only differences across all 10 store files are the table name, column constant, and scan function. This pattern was already present in the 6 existing files (feedings, stools, urine, weights, temperatures, abdomen) and the 4 new files perpetuate it.
- **Reasoning:** This is 700+ lines of nearly identical code across 10 files. Any change to pagination logic (e.g., switching cursor strategy, adding a new filter, fixing an edge case) must be applied to all 10 files in lockstep.
- **Consequence:** A bug fix or pagination enhancement applied to one metric but missed in another leads to inconsistent behavior across API endpoints. Maintenance cost scales linearly with each new metric type added.
- **Suggestion:** Extract a generic `listMetricWithTZ` helper using Go generics. The scan function and table/column names vary per metric, so pass them as parameters:
  ```go
  // In metric_helpers.go
  type ScanFunc[T any] func(scanner) (*T, error)
  type IDFunc[T any] func(*T) string

  func listMetricWithTZ[T any](
      db *sql.DB, table, columns, babyID string,
      from, to, cursor *string, limit int, loc *time.Location,
      scan ScanFunc[T], getID IDFunc[T],
  ) (*model.MetricPage[T], error) {
      // single implementation of condition building, query, iteration, pagination
  }
  ```
  Each store file's `ListXxxWithTZ` becomes a one-line call to this helper.

### [MEDIUM] Duplicated handler CRUD boilerplate across 4 new handler files
- **File:** `backend/internal/handler/bruising.go`, `backend/internal/handler/skin_observations.go`, `backend/internal/handler/lab_results.go`, `backend/internal/handler/general_notes.go`
- **Line(s):** N/A - structural issue
- **Perspective:** Tech Lead
- **Finding:** Each of the 4 new handler files (and the 6 existing ones) follows the exact same 5-handler structure: `Create` (requireUser, requireBabyAccess, decode JSON, validate, call store, writeJSON 201), `List` (requireUser, requireBabyAccess, parseListParams, call store, mapMetricPage), `Get` (requireUser, requireBabyAccess, requireEntryID, call store, writeJSON), `Update` (same as Create + requireEntryID), `Delete` (requireUser, requireBabyAccess, requireEntryID, call store, 204). The `List`, `Get`, and `Delete` handlers are structurally identical across all metrics -- only the store function name and response mapper differ. The route registration already uses `registerMetricCRUD`, but the handler bodies themselves are fully duplicated.
- **Reasoning:** While Go's lack of inheritance makes some duplication natural, the `List`, `Get`, and `Delete` handlers could be generated or abstracted via higher-order functions since their logic is completely formulaic. `Create` and `Update` have slight variations (validation, store call signatures) that make full abstraction harder.
- **Consequence:** Adding a new metric requires ~220 lines of near-identical handler code. Any cross-cutting change (e.g., adding audit logging to all List handlers) must be manually applied to 10+ files.
- **Suggestion:** Consider higher-order handler constructors for the three fully generic operations:
  ```go
  func makeListHandler[M any, R any](
      db *sql.DB,
      listFn func(*sql.DB, string, *string, *string, *string, int, *time.Location) (*model.MetricPage[M], error),
      toResp func(*M) R,
      logLabel string,
  ) http.HandlerFunc { ... }

  func makeGetHandler[M any, R any](
      db *sql.DB,
      getFn func(*sql.DB, string, string) (*M, error),
      toResp func(*M) R,
      notFoundMsg string,
  ) http.HandlerFunc { ... }

  func makeDeleteHandler(
      db *sql.DB,
      deleteFn func(*sql.DB, string, string) error,
      notFoundMsg string,
  ) http.HandlerFunc { ... }
  ```
  This would cut ~60 lines per metric for List+Get+Delete. Create and Update can remain bespoke since they have per-metric validation.

### [MEDIUM] Response mapping functions duplicate common field formatting
- **File:** `backend/internal/handler/bruising.go`, `backend/internal/handler/skin_observations.go`, `backend/internal/handler/lab_results.go`, `backend/internal/handler/general_notes.go`
- **Line(s):** bruising.go L66-L82, skin_observations.go L62-L78, lab_results.go L53-L68, general_notes.go L47-L60
- **Perspective:** Tech Lead
- **Finding:** Every `toXxxResponse` function manually formats the same 5 common fields: `ID`, `BabyID`, `LoggedBy`, `UpdatedBy`, `Timestamp` (via `model.DateTimeFormat`), `CreatedAt` (via `model.DateTimeFormat`), `UpdatedAt` (via `model.DateTimeFormat`). These 7 field assignments are repeated verbatim in all 10 metric response mappers.
- **Reasoning:** If the timestamp format changes or a new common field is added (e.g., `deleted_at`), every mapper must be updated individually.
- **Consequence:** Drift between mappers if a format change is missed in one. Currently 10 places that format timestamps identically.
- **Suggestion:** Consider embedding a common `baseMetricResponse` struct that handles the shared fields, and have each response struct embed it:
  ```go
  type baseMetricResponse struct {
      ID        string  `json:"id"`
      BabyID    string  `json:"baby_id"`
      LoggedBy  string  `json:"logged_by"`
      UpdatedBy *string `json:"updated_by,omitempty"`
      Timestamp string  `json:"timestamp"`
      CreatedAt string  `json:"created_at"`
      UpdatedAt string  `json:"updated_at"`
  }
  ```
  Plus a helper to populate it from any model that has these common fields (via an interface or a shared embedded struct in the model layer).

### [LOW] Test boilerplate is heavily duplicated across handler test files
- **File:** `backend/internal/handler/bruising_test.go`, `backend/internal/handler/skin_observations_test.go`, `backend/internal/handler/lab_results_test.go`, `backend/internal/handler/general_notes_test.go`
- **Line(s):** N/A - structural issue
- **Perspective:** QA
- **Finding:** Each handler test file repeats the same ~15-line setup pattern for every test case: `SetupTestDB`, `CreateTestUser`, `CreateTestBaby`, `AuthenticatedRequest`, create middleware chain (`authMw`, `csrfMw`), create `mux`, handle route, `ServeHTTP`, check status code. The middleware chain construction (`authMw(csrfMw(http.HandlerFunc(...)))`) is repeated 40+ times across the 4 new test files. Tests for InvalidJSON, MissingTimestamp, NotFound, and Delete+NotFound are structurally identical across all metric types.
- **Reasoning:** This is test code so the impact is lower, but it increases the friction of writing tests for new metrics and makes it harder to spot meaningful test differences.
- **Consequence:** Adding a new metric requires copying ~400 lines of test boilerplate. A test infrastructure change (e.g., new middleware) requires editing dozens of test functions.
- **Suggestion:** Create a test helper that encapsulates the common setup and request execution:
  ```go
  func setupMetricTest(t *testing.T) (*sql.DB, *model.User, *model.Baby, func(http.HandlerFunc, string, string, string) *httptest.ResponseRecorder)
  ```
  Or consider table-driven tests for the generic error cases (InvalidJSON, MissingTimestamp, NotFound) that are identical across metrics.

### [LOW] Store tests in bruising and lab_results use internal package access while general_notes uses external
- **File:** `backend/internal/store/bruising_test.go`, `backend/internal/store/lab_results_test.go`, `backend/internal/store/general_notes_test.go`
- **Line(s):** bruising_test.go L1 (`package store`), lab_results_test.go L1 (`package store`), general_notes_test.go L1 (`package store_test`)
- **Perspective:** Tech Lead
- **Finding:** `bruising_test.go` and `lab_results_test.go` use `package store` (white-box testing with `setupTestDB`), while `general_notes_test.go` uses `package store_test` (black-box testing with `testutil.SetupTestDB`). This is inconsistent within the same batch of new code.
- **Reasoning:** Mixing test package conventions within the same layer makes the codebase harder to navigate. White-box tests can access unexported functions and may mask API surface issues.
- **Consequence:** Developers adding new metrics will be unsure which convention to follow. The inconsistency could mask import cycle issues or unexported API problems.
- **Suggestion:** Standardize on one convention. Since the store layer exposes a public API, `package store_test` (black-box) is generally preferred and aligns with `general_notes_test.go`. Update `bruising_test.go` and `lab_results_test.go` to use `package store_test` with `testutil.SetupTestDB`.
