## Summary

Phase 38 implements PDF report generation using maroto v2 for the `GET /api/babies/:id/report?from=&to=` endpoint. The implementation includes a `report` package with text/table sections (header, summary, stool color log, temperature log with fever flags, feeding summary, medication adherence ratio, and notable observations), plus a handler and route registration. Tests cover PDF generation with data, empty date ranges, expected text content, edge cases (no Kasai date, old baby age formatting, notes without category, invalid dates). Coverage is 91.0% on the report package. All tests pass.

### [MEDIUM] Headers set before PDF generation can produce corrupt response on error

- **File:** `backend/internal/handler/report.go`
- **Line(s):** L32-L38
- **Perspective:** QA
- **Finding:** `Content-Type: application/pdf` and `Content-Disposition` headers are set at L32-33 before `report.Generate` is called at L35. If `Generate` fails after partially writing to `w`, the response will have PDF headers but contain a mix of partial PDF bytes followed by an HTTP error text body. Even if no bytes were written, the `Content-Type` header is already set to `application/pdf` before the 500 error response.
- **Reasoning:** `http.Error` will attempt to set `Content-Type` to `text/plain` but if `WriteHeader` was already called implicitly by a prior `Write`, the status code and headers are already sent. In the partial-write case, the client receives a corrupt response.
- **Consequence:** Clients receiving a corrupt PDF with mixed binary/text content. The error response may silently fail to communicate the actual error to the caller.
- **Suggestion:** Buffer the PDF into a `bytes.Buffer` first, then set headers and write the buffer only on success:
  ```go
  var buf bytes.Buffer
  if err := report.Generate(db, baby, from, to, &buf); err != nil {
      log.Printf("generate report: %v", err)
      http.Error(w, "internal error", http.StatusInternalServerError)
      return
  }
  w.Header().Set("Content-Type", "application/pdf")
  w.Header().Set("Content-Disposition", "inline; filename=\"report.pdf\"")
  w.Write(buf.Bytes())
  ```

### [MEDIUM] No date format validation in handler before passing to report

- **File:** `backend/internal/handler/report.go`
- **Line(s):** L25-L29
- **Perspective:** QA
- **Finding:** The handler only checks for empty `from`/`to` parameters but does not validate that they are valid `YYYY-MM-DD` dates. Invalid date strings are passed directly to `report.Generate`, which will fail inside `store.ParseDateRange` and return a 500 error instead of a 400.
- **Reasoning:** Other handlers in this codebase (e.g., dashboard) likely have the same pattern, but for correctness, malformed input should return 400 Bad Request, not 500 Internal Server Error.
- **Consequence:** Users providing malformed dates (e.g., `from=abc`) get a 500 error instead of a descriptive 400 error, making debugging harder and potentially triggering false alerts in monitoring.
- **Suggestion:** Validate date format before calling Generate:
  ```go
  if _, err := time.Parse("2006-01-02", from); err != nil {
      http.Error(w, "invalid from date format, expected YYYY-MM-DD", http.StatusBadRequest)
      return
  }
  if _, err := time.Parse("2006-01-02", to); err != nil {
      http.Error(w, "invalid to date format, expected YYYY-MM-DD", http.StatusBadRequest)
      return
  }
  ```

### [LOW] Grammar issue in FormatAge output

- **File:** `backend/internal/report/report.go`
- **Line(s):** L290-L302
- **Perspective:** Product Owner
- **Finding:** `FormatAge` produces grammatically incorrect output for singular values: "1 months, 0 days", "1 years, 0 months". The tests explicitly assert these grammatically incorrect strings.
- **Reasoning:** This text appears in a clinical PDF report meant for hepatologist appointments. Grammatical errors in a medical document reduce perceived professionalism.
- **Consequence:** Minor but noticeable grammar errors on clinical reports shared with healthcare professionals.
- **Suggestion:** Handle singular/plural:
  ```go
  func pluralize(n int, singular string) string {
      if n == 1 {
          return fmt.Sprintf("%d %s", n, singular)
      }
      return fmt.Sprintf("%d %ss", n, singular)
  }
  ```

### [LOW] Stool color table only shows numeric score, not color label

- **File:** `backend/internal/report/report.go`
- **Line(s):** L338-L362
- **Perspective:** Product Owner
- **Finding:** The stool color log table only displays the numeric `ColorScore` (e.g., 3, 4) without the `color_label` (e.g., "yellow", "pale"). The spec says "Stool color log -- table of entries" which implies the color label should be visible for clinical readability.
- **Reasoning:** The `StoolColorSeriesEntry` struct from the store only provides `Timestamp` and `ColorScore`. Adding `color_label` would require updating the store query (out of scope for this phase potentially), but the report would be more useful with it.
- **Consequence:** Clinicians reading the report must mentally map numeric scores to colors, reducing the report's clinical utility.
- **Suggestion:** Consider adding `color_label` to `StoolColorSeriesEntry` in the store layer, or map the score to a label in the report package. This could be deferred to a follow-up if modifying the store is out of scope.
