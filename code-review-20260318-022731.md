## Summary

Review of Phases 37-39 which add PDF report generation (text/table sections, charts with WHO percentile overlays, and a photo appendix). The new code spans ~1000 lines across `report.go`, `charts.go`, and `handler/report.go`. The focus is on efficiency: unnecessary work, missed concurrency, hot-path bloat, memory usage, and overly broad operations.

### [HIGH] Sequential DB queries that could run concurrently
- **File:** `backend/internal/report/report.go`
- **Line(s):** L83-L151
- **Perspective:** Performance Engineer
- **Finding:** `queryReportData` issues 8 sequential database queries plus an object-storage fetch loop. Every query is independent of the others -- none uses the result of a prior query.
- **Reasoning:** PDF generation is a user-facing request. Serializing 8+ round-trips to the database (plus N object-storage GETs) adds unnecessary latency. With an `errgroup`, all queries could execute concurrently, reducing wall-clock time to the latency of the slowest single query rather than the sum.
- **Consequence:** For a date range with substantial data, the endpoint could take several seconds to respond when it could take a fraction of that. This will feel sluggish to the user.
- **Suggestion:** Use `golang.org/x/sync/errgroup` to fan out all independent queries:
  ```go
  g, ctx := errgroup.WithContext(ctx)
  g.Go(func() error { summary, err = store.GetDashboardSummary(db, babyID, from, to); return err })
  g.Go(func() error { stools, err = store.GetStoolColorSeries(db, babyID, from, to); return err })
  // ... etc
  if err := g.Wait(); err != nil { return nil, err }
  ```

### [HIGH] Sequential object-storage fetches for photo thumbnails
- **File:** `backend/internal/report/report.go`
- **Line(s):** L198-L205
- **Perspective:** Performance Engineer
- **Finding:** Up to 20 thumbnail images are fetched from R2 one at a time in a serial loop. Each `objStore.Get` is a network round-trip.
- **Reasoning:** Object storage GETs are typically 50-200ms each. Fetching 20 thumbnails serially could add 1-4 seconds of pure I/O wait. These fetches are entirely independent.
- **Consequence:** Report generation time scales linearly with photo count, up to multiple seconds of avoidable latency.
- **Suggestion:** Fan out the fetches with a bounded `errgroup` or a simple goroutine pool:
  ```go
  g, ctx := errgroup.WithContext(ctx)
  g.SetLimit(5) // limit concurrency to avoid overwhelming R2
  thumbs := make([][]byte, len(keys))
  for i, key := range keys {
      g.Go(func() error {
          data, err := objStore.Get(ctx, key)
          if err != nil { return nil }
          thumbs[i] = data
          return nil
      })
  }
  g.Wait()
  ```

### [MEDIUM] Redundant date-range parsing across every query
- **File:** `backend/internal/report/report.go`
- **Line(s):** L86-L138 (callers), L161 and L213 and L255 (callees)
- **Perspective:** Tech Lead
- **Finding:** `ParseDateRange(from, to)` is called independently inside `GetDashboardSummary`, `GetStoolColorSeries`, `GetTemperatureSeries`, `GetFeedingDaily`, `GetWeightSeries`, `GetLabTrends`, `queryMedAdherence`, `queryNotes`, and `fetchPhotoThumbnails` -- at least 9 times total for the same `from`/`to` pair. Each call re-parses and re-formats the same two date strings.
- **Reasoning:** While individually cheap, this is a pattern of redundant work that also duplicates error handling. Parsing once at the top and passing the parsed values through would be cleaner and marginally faster.
- **Consequence:** Minor CPU waste, but more importantly it is a maintainability concern -- if the date format changes, every call site must handle the error the same way.
- **Suggestion:** Parse `from` and `to` once in `queryReportData` and pass the parsed `fromTime`/`toTime` strings to all sub-queries. This would require refactoring the store functions to accept pre-parsed time boundaries, or adding parallel `*WithRange` variants.

### [MEDIUM] Entire PDF buffered in memory before writing to response
- **File:** `backend/internal/handler/report.go`
- **Line(s):** L44-L52
- **Perspective:** Performance Engineer
- **Finding:** The handler allocates a `bytes.Buffer`, generates the entire PDF into it, then writes the whole buffer to the HTTP response. Additionally, `report.Generate` itself calls `doc.GetBytes()` (another full copy) and then writes to the buffer. So the PDF bytes exist in at least two copies in memory simultaneously.
- **Reasoning:** For a report with 20 embedded thumbnail images and multiple chart PNGs, the PDF could be several megabytes. Two full copies doubles the memory footprint per concurrent request.
- **Consequence:** Under concurrent report generation, memory usage spikes. With enough concurrent requests, this could cause OOM pressure.
- **Suggestion:** This is partly a limitation of the maroto library (it materializes the full PDF). At minimum, write directly from `doc.GetBytes()` to `w` without the intermediate `bytes.Buffer`:
  ```go
  // In handler: pass w (http.ResponseWriter) directly
  // In Generate: write doc.GetBytes() directly to w, avoiding double-copy
  ```

### [MEDIUM] `GetDashboardSummary` runs 6 separate queries that overlap with other report queries
- **File:** `backend/internal/store/dashboard.go`
- **Line(s):** L41-L123
- **Perspective:** Performance Engineer
- **Finding:** `GetDashboardSummary` queries stools count + latest color, feedings count + calories, and latest weight. The report also separately queries `GetStoolColorSeries` (full stool data), `GetFeedingDaily` (full feeding data), and `GetWeightSeries` (full weight data). The summary aggregates could be computed from the detail data already being fetched.
- **Reasoning:** This is redundant work: the stools count is just `len(stools)`, the latest color is the last element of the sorted stool series, feeding totals can be summed from the daily entries, and latest weight is the last element of the weight series.
- **Consequence:** 6 extra DB round-trips per report that could be eliminated entirely by deriving summary values from the detail data.
- **Suggestion:** Compute summary values in Go from the detail query results instead of issuing separate DB queries. For example:
  ```go
  summary.Stools = len(stools)
  if len(stools) > 0 { summary.ColorIndicator = &stools[len(stools)-1].ColorLabel }
  ```

### [LOW] `fetchPhotoThumbnails` swallows all errors silently
- **File:** `backend/internal/report/report.go`
- **Line(s):** L156-L208
- **Perspective:** QA
- **Finding:** Both the DB query error (L178) and individual thumbnail fetch errors (L200-L201) are logged but otherwise silently ignored, returning partial or empty results. While graceful degradation is reasonable for thumbnails, the caller has no way to distinguish "no photos exist" from "storage is down."
- **Reasoning:** If object storage is misconfigured or experiencing an outage, the report will silently omit all photos with no indication to the user.
- **Consequence:** Users may not realize their report is incomplete when there is an infrastructure issue, leading to incorrect medical records being shared with doctors.
- **Suggestion:** Consider returning a boolean or error alongside the thumbnails so the report can include a "Photos unavailable" notice rather than silently omitting the section.

### [LOW] No context propagation or cancellation support
- **File:** `backend/internal/report/report.go`
- **Line(s):** L65 (Generate signature), L196 (context.Background)
- **Perspective:** Tech Lead
- **Finding:** `Generate` does not accept a `context.Context`. Internally, `fetchPhotoThumbnails` creates a `context.Background()` on L196. None of the DB queries use context-aware variants (`db.QueryContext`).
- **Reasoning:** If the HTTP client disconnects mid-generation, all the DB queries and R2 fetches will continue to completion, wasting resources. This also prevents implementing request timeouts.
- **Consequence:** Abandoned requests continue consuming DB connections and network I/O. Under load, this could exhaust connection pools.
- **Suggestion:** Thread `context.Context` from the HTTP request through `Generate` and all sub-calls. Use `db.QueryContext` and `db.QueryRowContext` variants. This also enables the concurrency improvements suggested above via `errgroup`.
