## Summary

Review of phases 31-33 covering WHO growth percentile calculations (`who` package), push notification subscription handlers and store layer, and the `notify` package for Web Push delivery. The review is scoped exclusively to **efficiency** concerns.

### [MEDIUM] Extra SELECT after every UpsertPushSubscription
- **File:** `backend/internal/store/push_subscriptions.go`
- **Line(s):** L12-L28
- **Perspective:** Performance Engineer
- **Finding:** `UpsertPushSubscription` executes an INSERT/ON CONFLICT, then immediately performs a second SELECT query via `getPushSubscriptionByEndpoint` to return the resulting row. This means every subscribe call issues two round-trips to the database.
- **Reasoning:** The upsert already knows the values being written (`endpoint`, `p256dh`, `auth`). The only unknown is `created_at` (a server-default column) and potentially the `id` on conflict. SQLite supports `RETURNING` (since 3.35.0, 2021), which would allow retrieving the final row in a single statement. Alternatively, the function could use the known input values and query only `created_at` (or set it explicitly in Go).
- **Consequence:** Under normal single-user personal-app load this is negligible. However, it sets a pattern that could be replicated across the codebase, doubling DB round-trips for every upsert-style operation.
- **Suggestion:** Use SQLite's `RETURNING` clause to retrieve the row in the same statement:
  ```go
  row := db.QueryRow(
      `INSERT INTO push_subscriptions (id, user_id, endpoint, p256dh, auth)
       VALUES (?, ?, ?, ?, ?)
       ON CONFLICT (endpoint) DO UPDATE SET
         p256dh = excluded.p256dh,
         auth = excluded.auth
       RETURNING id, user_id, endpoint, p256dh, auth, created_at`,
      id, userID, endpoint, p256dh, auth,
  )
  return scanPushSubscription(row)
  ```

### [LOW] WHO percentile curves recomputed on every request with no caching
- **File:** `backend/internal/handler/who_percentiles.go`
- **Line(s):** L44
- **Perspective:** Performance Engineer
- **Finding:** `WHOPercentilesHandler` calls `who.PercentileCurves` on every request. The WHO LMS data is static and the output for a given `(sex, fromDays, toDays)` tuple is deterministic. Each call performs `5 * numDays` floating-point computations (including `math.Pow` and `math.Round`) and allocates `5 * numDays` `CurvePoint` structs.
- **Reasoning:** For a full 0-730 range request, that is 3,655 `CurvePoint` allocations and math operations per request. The data never changes. For a personal-use app this is fast enough (sub-millisecond), but since there are only a handful of valid query combinations (male/female, and the app likely always requests the same range), a simple cache or pre-computation would eliminate all per-request work.
- **Consequence:** Minimal real-world impact for a personal app. If the endpoint were ever called repeatedly (e.g., by a charting component on every render), it creates unnecessary GC pressure and CPU work.
- **Suggestion:** Consider setting `Cache-Control` headers (e.g., `Cache-Control: public, max-age=86400`) on the HTTP response so browsers and intermediaries cache the response. Alternatively, pre-compute the two most common curve sets (male 0-730, female 0-730) at init time alongside the LMS data.

### [LOW] Redundant byte-to-string conversion in CSV loading
- **File:** `backend/internal/who/who.go`
- **Line(s):** L67
- **Perspective:** Performance Engineer
- **Finding:** `loadLMS` reads embedded file bytes via `fs.ReadFile(path)` returning `[]byte`, then converts to `string(data)` to create a `strings.NewReader`, which the CSV reader then reads back as bytes internally. This creates an unnecessary copy of the entire CSV content.
- **Reasoning:** `csv.NewReader` accepts any `io.Reader`. Using `bytes.NewReader(data)` instead of `strings.NewReader(string(data))` would avoid allocating a duplicate copy of the CSV content in memory.
- **Consequence:** Since this runs once at init time with small CSV files (~731 rows), the impact is trivial. It is a style/hygiene point rather than a meaningful performance concern.
- **Suggestion:** Replace `strings.NewReader(string(data))` with `bytes.NewReader(data)` and import `bytes` instead of `strings`.

### [LOW] ULID generated but discarded on upsert conflict path
- **File:** `backend/internal/store/push_subscriptions.go`
- **Line(s):** L13
- **Perspective:** Performance Engineer
- **Finding:** `model.NewULID()` is called unconditionally in `UpsertPushSubscription`, but on the conflict (update) path the generated ID is discarded because the existing row's ID is preserved. ULID generation involves reading from `crypto/rand` and performing timestamp encoding.
- **Reasoning:** For the upsert-update path, the ULID generation work is wasted. This is a minor concern since ULID generation is fast, but it is conceptually unnecessary work.
- **Consequence:** Negligible performance impact. Noted for completeness.
- **Suggestion:** No change needed unless this pattern is replicated at scale. If using `RETURNING` as suggested above, the concern becomes moot since you get back whichever ID was actually used.
