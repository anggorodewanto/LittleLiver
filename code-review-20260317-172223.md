## Summary

Efficiency review of Phases 13-15 production code covering baby unlinking, account deletion, and feeding CRUD. The codebase is generally well-structured with clean pagination and proper transaction usage. Findings focus on redundant database round-trips, a TOCTOU pattern in `requireBabyAccess`, unnecessary per-request work in the delete-account path, and a minor cursor-setting bug.

### [MEDIUM] Redundant database queries in UnlinkSelfHandler (TOCTOU + wasted work)
- **File:** `backend/internal/handler/babies.go`
- **Line(s):** L228-L248
- **Perspective:** Performance Engineer
- **Finding:** `UnlinkSelfHandler` calls `requireBabyAccess` which executes two queries (GetBabyByID + IsParentOfBaby), then discards the baby result (assigned to `_`), then calls `extractBabyID` again to get the same baby ID already available from the request. More importantly, `requireBabyAccess` performs a full baby SELECT (9 columns, time parsing) just to confirm existence, when only the parent-link check is needed here.
- **Reasoning:** Two queries are executed where one would suffice. The baby row is fetched, fully parsed (including three `time.Parse` calls), and thrown away. The authorization check only needs to confirm a row exists in `baby_parents`.
- **Consequence:** Every unlink request performs one unnecessary SELECT with associated memory allocation and time parsing overhead. Minor for a personal-use app, but it is still wasted work on every request.
- **Suggestion:** For handlers that only need authorization (not the baby data), consider a single-query authorization check that validates baby existence and parent linkage in one step, or at minimum use `IsParentOfBaby` directly (which already confirms the baby exists implicitly via the join table). Example:
  ```go
  func UnlinkSelfHandler(db *sql.DB) http.HandlerFunc {
      return func(w http.ResponseWriter, r *http.Request) {
          user, ok := requireUser(w, r)
          if !ok { return }
          babyID := extractBabyID(r)
          if babyID == "" {
              http.Error(w, "missing baby ID", http.StatusBadRequest)
              return
          }
          linked, err := store.IsParentOfBaby(db, user.ID, babyID)
          if err != nil { /* handle */ }
          if !linked {
              http.Error(w, "forbidden", http.StatusForbidden)
              return
          }
          if err := store.UnlinkParent(db, babyID, user.ID); err != nil { /* handle */ }
          w.WriteHeader(http.StatusNoContent)
      }
  }
  ```

### [MEDIUM] Same redundant two-query pattern in all feeding handlers
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L89-L100, L129-L141, L185-L197, L223-L235, L272-L284
- **Perspective:** Performance Engineer
- **Finding:** Every feeding handler (Create, List, Get, Update, Delete) calls `requireBabyAccess` which runs two queries (GetBabyByID + IsParentOfBaby), discards the baby result (`_, ok = requireBabyAccess(...)`), then extracts babyID again from the path. None of these handlers use the baby data.
- **Reasoning:** This is 5 handlers each performing an unnecessary full baby row SELECT + parse on every request, purely for authorization. The baby data is never used in any feeding handler.
- **Consequence:** 10 unnecessary database round-trips per 5 feeding endpoints (2 wasted queries each). For a personal-use app this is tolerable but represents a systematic inefficiency pattern across the entire API surface.
- **Suggestion:** Create a lightweight `requireBabyAccessID` helper that returns only the babyID after a single existence+authorization check, or combine the check into a single query:
  ```sql
  SELECT 1 FROM baby_parents WHERE baby_id = ? AND user_id = ?
  ```
  This single query validates baby existence (via FK) and parent linkage simultaneously.

### [MEDIUM] Insert-then-SELECT pattern for Create and Update operations
- **File:** `backend/internal/store/feedings.go`
- **Line(s):** L70-L83, L183-L207
- **Perspective:** Performance Engineer
- **Finding:** Both `CreateFeeding` and `UpdateFeeding` perform a write operation followed by a separate `GetFeedingByID` SELECT to return the result. The same pattern exists in `store/babies.go` for `CreateBaby` (L91) and `UpdateBaby` (L142). Since SQLite triggers compute `calories`, `used_default_cal`, `created_at`, and `updated_at`, a re-read is needed -- but for `CreateFeeding`, the INSERT could use `RETURNING` (SQLite 3.35+) to avoid the extra round-trip.
- **Reasoning:** Each create/update operation costs two database round-trips when one could suffice with `RETURNING`. The second query re-scans 14 columns after just writing them.
- **Consequence:** Doubles the database I/O for every write operation. For a personal-use app the latency impact is minimal, but it adds up if multiple feedings are logged in quick succession.
- **Suggestion:** If the project targets SQLite 3.35+, use `INSERT ... RETURNING *` or `UPDATE ... RETURNING *` to get the computed columns in a single round-trip. If triggers make this complex, the current approach is acceptable but should be documented as a known trade-off.

### [LOW] Correlated subquery in DeleteAccount last-parent detection
- **File:** `backend/internal/store/delete_account.go`
- **Line(s):** L29-L33
- **Perspective:** Performance Engineer
- **Finding:** The query to find babies where the user is the last parent uses a correlated subquery: `(SELECT COUNT(*) FROM baby_parents bp2 WHERE bp2.baby_id = bp.baby_id) = 1`. This executes the inner COUNT for every row in baby_parents matching the user.
- **Reasoning:** A more efficient approach would use a GROUP BY with HAVING, or a NOT EXISTS anti-join to find babies with no other parents. However, for a personal-use app with a small number of babies per user (likely 1-3), this has negligible real-world impact.
- **Consequence:** Minimal -- the correlated subquery is fine for small datasets. Would become problematic only at scale far beyond this app's intended use.
- **Suggestion:** If ever needed, rewrite as:
  ```sql
  SELECT bp.baby_id FROM baby_parents bp
  WHERE bp.user_id = ?
  AND NOT EXISTS (
      SELECT 1 FROM baby_parents bp2
      WHERE bp2.baby_id = bp.baby_id AND bp2.user_id != ?
  )
  ```

### [LOW] Pagination cursor set from wrong index
- **File:** `backend/internal/store/feedings.go`
- **Line(s):** L165-L168
- **Perspective:** QA
- **Finding:** When there are more results than the limit, the next cursor is set from `feedings[limit-1].ID` (the last item on the current page) rather than `feedings[limit].ID` (the first item of the next page). Since the cursor condition is `id < ?`, this means the cursor points to the last item *included* in the current response, and the next page query `id < lastIncludedID` would correctly skip it. However, using `feedings[limit-1]` as the cursor is semantically confusing -- it is the last included item, not a pointer to "where to resume." The data slice is `feedings[:limit]` (indices 0 through limit-1), so it does work correctly, but would be clearer as the ID of the last returned item.
- **Reasoning:** This is functionally correct but could confuse future maintainers. The `limit+1` fetch pattern typically sets the cursor from the last *returned* item (index `limit-1`), which is what this does. No actual bug.
- **Consequence:** None -- the pagination works correctly. This is purely a readability note.
- **Suggestion:** No change needed. Optionally, add a comment clarifying that the cursor is the ID of the last returned item, so the next page fetches `id < cursor`.

### [LOW] `time.LoadLocation` called on every paginated request
- **File:** `backend/internal/handler/feedings.go`
- **Line(s):** L155-L159
- **Perspective:** Performance Engineer
- **Finding:** `ListFeedingsHandler` calls `time.LoadLocation(tz)` on every request with an `X-Timezone` header. While Go caches timezone data internally after the first load, the lookup still involves map access and string processing on each call.
- **Reasoning:** For a personal-use app, this is entirely negligible. The Go runtime's timezone cache makes repeated calls cheap.
- **Consequence:** No measurable impact for this use case.
- **Suggestion:** No change needed. If this ever became a hot path, a `sync.Map` cache could be added, but that would be over-engineering for this project.
